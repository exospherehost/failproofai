failproofai JSON कॉन्फ़िगरेशन फ़ाइलों का उपयोग करता है जो यह नियंत्रित करती हैं कि कौन-सी policies सक्रिय हैं, वे कैसे काम करती हैं, और कस्टम policies कहाँ से लोड की जाती हैं। कॉन्फ़िगरेशन को आपकी टीम के साथ साझा करना आसान बनाया गया है — इसे अपने repo में commit करें और हर developer को एक समान agent safety net मिल जाती है।

---

## कॉन्फ़िगरेशन स्कोप

तीन कॉन्फ़िगरेशन स्कोप हैं, जिन्हें प्राथमिकता क्रम में मूल्यांकित किया जाता है:

| स्कोप | फ़ाइल पथ | उद्देश्य |
|-------|-----------|---------|
| **project** | `.failproofai/policies-config.json` | प्रति-repo सेटिंग्स, version control में committed |
| **local** | `.failproofai/policies-config.local.json` | व्यक्तिगत प्रति-repo ओवरराइड, gitignored |
| **global** | `~/.failproofai/policies-config.json` | सभी projects में user-level डिफ़ॉल्ट |

जब failproofai किसी hook इवेंट को प्राप्त करता है, तो वह वर्तमान working directory के लिए मौजूद तीनों फ़ाइलों को लोड और मर्ज करता है।

### मर्ज नियम

**`enabledPolicies`** — तीनों स्कोप का union। किसी भी स्तर पर enabled की गई policy सक्रिय रहती है।

```text
project:  ["block-sudo"]
local:    ["block-rm-rf"]
global:   ["block-sudo", "sanitize-api-keys"]

resolved: ["block-sudo", "block-rm-rf", "sanitize-api-keys"]  ← deduplicated union
```

**`policyParams`** — जो पहला स्कोप किसी दी गई policy के लिए params परिभाषित करता है, वही पूरी तरह जीतता है। किसी policy के params के भीतर values की deep merging नहीं होती।

```text
project:  block-sudo → { allowPatterns: ["sudo apt-get update"] }
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo apt-get update"] }   ← project जीता, global अनदेखा
```

```text
project:  (block-sudo एंट्री नहीं)
local:    (block-sudo एंट्री नहीं)
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo systemctl status"] }  ← global तक पहुँचता है
```

**`customPoliciesPath`** — जो पहला स्कोप इसे परिभाषित करता है वह जीतता है।

**`llm`** — जो पहला स्कोप इसे परिभाषित करता है वह जीतता है।

## कॉन्फ़िग फ़ाइल फ़ॉर्मेट

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "sanitize-jwt",
    "block-env-files",
    "block-read-outside-cwd"
  ],
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    },
    "block-push-master": {
      "protectedBranches": ["main", "release", "prod"]
    },
    "block-rm-rf": {
      "allowPaths": ["/tmp"]
    },
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company"]
    },
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo API key" }
      ]
    },
    "warn-large-file-write": {
      "thresholdKb": 512
    }
  },
  "customPoliciesPath": "/home/alice/myproject/my-policies.js"
}
```

## फ़ील्ड संदर्भ

### `enabledPolicies`

प्रकार: `string[]`

enable करने के लिए policy नामों की सूची। नाम `failproofai policies` द्वारा दिखाए गए policy identifiers से बिल्कुल मेल खाने चाहिए। पूरी सूची के लिए [Built-in Policies](/hi/built-in-policies) देखें।

`enabledPolicies` में नहीं आई policies निष्क्रिय रहती हैं, भले ही `policyParams` में उनकी एंट्री हो।

### `policyParams`

प्रकार: `Record<string, Record<string, unknown>>`

प्रति-policy पैरामीटर ओवरराइड। बाहरी key policy का नाम है; आंतरिक keys policy-specific हैं। प्रत्येक policy के उपलब्ध पैरामीटर [Built-in Policies](/hi/built-in-policies) में दस्तावेज़ीकृत हैं।

यदि किसी policy के पैरामीटर हैं लेकिन आपने उन्हें specify नहीं किया, तो policy के built-in defaults उपयोग होते हैं। जो उपयोगकर्ता `policyParams` बिल्कुल कॉन्फ़िगर नहीं करते, उन्हें पिछले versions जैसा ही व्यवहार मिलता है।

किसी policy के params block में अज्ञात keys को hook-fire के समय चुपचाप नज़रअंदाज़ किया जाता है, लेकिन जब आप `failproofai policies` चलाते हैं तो उन्हें warnings के रूप में flagged किया जाता है।

### `customPoliciesPath`

प्रकार: `string` (absolute path)

कस्टम hook policies वाली JavaScript फ़ाइल का पथ। इसे `failproofai policies --install --custom <path>` द्वारा स्वचालित रूप से सेट किया जाता है (पथ को store करने से पहले absolute में resolve किया जाता है)।

फ़ाइल हर hook इवेंट पर नए सिरे से लोड होती है — कोई caching नहीं होती। authoring विवरण के लिए [Custom Policies](/hi/custom-policies) देखें।

### `llm`

प्रकार: `object` (वैकल्पिक)

AI calls करने वाली policies के लिए LLM client कॉन्फ़िगरेशन। अधिकांश setups के लिए आवश्यक नहीं।

```json
{
  "llm": {
    "model": "claude-sonnet-4-6",
    "apiKey": "sk-ant-..."
  }
}
```

## CLI से कॉन्फ़िगरेशन प्रबंधन

`policies --install` और `policies --uninstall` कमांड Claude Code के `settings.json` (hook entry points) में लिखते हैं, जबकि `policies-config.json` वह फ़ाइल है जिसे आप सीधे प्रबंधित करते हैं। ये दोनों अलग हैं:

- **`settings.json`** — Claude Code को बताता है कि प्रत्येक tool use पर `failproofai --hook <event>` को call करे
- **`policies-config.json`** — failproofai को बताता है कि कौन-सी policies evaluate करनी हैं और किन params के साथ

आप `policies-config.json` को किसी भी समय सीधे edit कर सकते हैं; बदलाव अगले hook इवेंट पर तुरंत प्रभावी हो जाते हैं, बिना किसी restart के।

## उदाहरण: टीम के डिफ़ॉल्ट के साथ project-level कॉन्फ़िग

`.failproofai/policies-config.json` को अपने repo में commit करें:

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "block-env-files"
  ],
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "release", "hotfix"]
    }
  }
}
```

इसके बाद प्रत्येक developer टीम के साथियों को प्रभावित किए बिना व्यक्तिगत ओवरराइड के लिए `.failproofai/policies-config.local.json` (gitignored) बना सकता है।
