failproofai में दो टेस्ट सूट हैं: **यूनिट टेस्ट** (तेज़, मॉक्ड) और **एंड-टू-एंड टेस्ट** (वास्तविक subprocess इनवोकेशन)।

---

## टेस्ट चलाना

```bash
# सभी यूनिट टेस्ट एक बार चलाएं
bun run test:run

# यूनिट टेस्ट watch mode में चलाएं
bun run test

# E2E टेस्ट चलाएं (सेटअप ज़रूरी है - नीचे देखें)
bun run test:e2e

# बिना बिल्ड किए टाइप-चेक करें
bunx tsc --noEmit

# Lint
bun run lint
```

### एक policy यूनिट टेस्ट लिखना

```typescript
import { describe, it, expect, beforeEach } from "vitest";
import { getBuiltinPolicies } from "../../src/hooks/builtin-policies";
import { allow, deny } from "../../src/hooks/policy-types";

describe("block-sudo", () => {
  const policy = getBuiltinPolicies().find((p) => p.name === "block-sudo")!;

  it("denies sudo commands", () => {
    const ctx = {
      eventType: "PreToolUse" as const,
      payload: {},
      toolName: "Bash",
      toolInput: { command: "sudo apt install nodejs" },
      params: { allowPatterns: [] },
    };
    expect(policy.fn(ctx)).toEqual(deny("sudo command blocked by failproofai"));
  });

  it("allows non-sudo commands", () => {
    const ctx = {
      eventType: "PreToolUse" as const,
      payload: {},
      toolName: "Bash",
      toolInput: { command: "ls -la" },
      params: { allowPatterns: [] },
    };
    expect(policy.fn(ctx)).toEqual(allow());
  });

  it("allows patterns in allowPatterns", () => {
    const ctx = {
      eventType: "PreToolUse" as const,
      payload: {},
      toolName: "Bash",
      toolInput: { command: "sudo systemctl status nginx" },
      params: { allowPatterns: ["sudo systemctl status"] },
    };
    expect(policy.fn(ctx)).toEqual(allow());
  });
});
```

## एंड-टू-एंड टेस्ट

E2E टेस्ट असली `failproofai` बाइनरी को subprocess के रूप में इनवोक करते हैं, stdin को JSON payload पाइप करते हैं, और stdout आउटपुट तथा exit code पर assert करते हैं। यह उस पूर्ण integration path को टेस्ट करता है जिसे Claude Code उपयोग करता है।

### सेटअप

E2E टेस्ट रिपो सोर्स से सीधे बाइनरी चलाते हैं। पहली बार चलाने से पहले, वह CJS बंडल बिल्ड करें जिसे custom hook फ़ाइलें `'failproofai'` से इम्पोर्ट करते समय उपयोग करती हैं:

```bash
bun build src/index.ts --outdir dist --target node --format cjs
```

फिर टेस्ट चलाएं:

```bash
bun run test:e2e
```

जब भी आप public hook API (`src/hooks/custom-hooks-registry.ts`, `src/hooks/policy-helpers.ts`, या `src/hooks/policy-types.ts`) बदलें, `dist/` को रीबिल्ड करें।

### E2E टेस्ट संरचना

```text
__tests__/e2e/
  helpers/
    hook-runner.ts      # बाइनरी spawn करें, payload JSON पाइप करें, exit code + stdout + stderr कैप्चर करें
    fixture-env.ts      # config फ़ाइलों के साथ प्रति-टेस्ट isolated temp directories
    payloads.ts         # प्रत्येक event type के लिए Claude-accurate payload factories
  hooks/
    builtin-policies.e2e.test.ts   # वास्तविक subprocess के साथ प्रत्येक builtin policy
    custom-hooks.e2e.test.ts       # Custom hook लोडिंग और evaluation
    config-scopes.e2e.test.ts      # project/local/global में config मर्जिंग
    policy-params.e2e.test.ts      # प्रत्येक parameterized policy के लिए parameter इंजेक्शन
```

### E2E हेल्पर्स का उपयोग

**`FixtureEnv`** - प्रति-टेस्ट isolated environment:

```typescript
import { createFixtureEnv } from "../helpers/fixture-env";

const env = createFixtureEnv();
// env.cwd    - temp dir; .failproofai/policies-config.json पिक अप करने के लिए payload.cwd के रूप में पास करें
// env.home   - isolated home dir; असली ~/.failproofai लीक नहीं होता

env.writeConfig({
  enabledPolicies: ["block-sudo"],
  policyParams: {
    "block-sudo": { allowPatterns: ["sudo systemctl status"] },
  },
});
```

`createFixtureEnv()` स्वचालित रूप से `afterEach` क्लीनअप रजिस्टर करता है।

**`runHook`** - बाइनरी इनवोक करें:

```typescript
import { runHook } from "../helpers/hook-runner";
import { Payloads } from "../helpers/payloads";

const result = await runHook(
  "PreToolUse",
  Payloads.preToolUse.bash("sudo apt install nodejs", env.cwd),
  { homeDir: env.home }
);

expect(result.exitCode).toBe(0);
expect(result.parsed?.hookSpecificOutput?.permissionDecision).toBe("deny");
```

**`Payloads`** - तैयार payload factories:

```typescript
Payloads.preToolUse.bash(command, cwd)
Payloads.preToolUse.write(filePath, content, cwd)
Payloads.preToolUse.read(filePath, cwd)
Payloads.postToolUse.bash(command, output, cwd)
Payloads.postToolUse.read(filePath, content, cwd)
Payloads.notification(message, cwd)
Payloads.stop(cwd)
```

### एक E2E टेस्ट लिखना

```typescript
import { describe, it, expect } from "vitest";
import { createFixtureEnv } from "../helpers/fixture-env";
import { runHook } from "../helpers/hook-runner";
import { Payloads } from "../helpers/payloads";

describe("block-rm-rf (E2E)", () => {
  it("denies rm -rf", async () => {
    const env = createFixtureEnv();
    env.writeConfig({ enabledPolicies: ["block-rm-rf"] });

    const result = await runHook(
      "PreToolUse",
      Payloads.preToolUse.bash("rm -rf /", env.cwd),
      { homeDir: env.home }
    );

    expect(result.exitCode).toBe(0);
    expect(result.parsed?.hookSpecificOutput?.permissionDecision).toBe("deny");
  });

  it("allows non-recursive rm", async () => {
    const env = createFixtureEnv();
    env.writeConfig({ enabledPolicies: ["block-rm-rf"] });

    const result = await runHook(
      "PreToolUse",
      Payloads.preToolUse.bash("rm /tmp/file.txt", env.cwd),
      { homeDir: env.home }
    );

    expect(result.exitCode).toBe(0);
    expect(result.stdout).toBe("");  // allow → empty stdout
  });
});
```

### E2E रिस्पॉन्स शेप्स

| निर्णय | Exit code | stdout |
|--------|-----------|--------|
| `PreToolUse` deny | `0` | `{"hookSpecificOutput":{"permissionDecision":"deny","permissionDecisionReason":"..."}}` |
| `PostToolUse` deny | `0` | `{"hookSpecificOutput":{"additionalContext":"Blocked ... because: ..."}}` |
| Instruct (non-Stop) | `0` | `{"hookSpecificOutput":{"additionalContext":"Instruction from failproofai: ..."}}` |
| Stop instruct | `2` | खाली stdout; कारण stderr में |
| Allow | `0` | खाली string |

### Vitest कॉन्फ़िग

E2E टेस्ट `vitest.config.e2e.mts` का उपयोग करते हैं जिसमें:

- `environment: "node"` - ब्राउज़र globals की ज़रूरत नहीं
- `pool: "forks"` - सच्ची process isolation (टेस्ट subprocess spawn करते हैं)
- `testTimeout: 20_000` - प्रति टेस्ट 20 सेकंड (बाइनरी स्टार्टअप + hook eval)

`forks` pool महत्वपूर्ण है: thread-based workers `globalThis` शेयर करते हैं, जो subprocess-spawning टेस्ट में बाधा डाल सकता है। Process-based forks इससे बचते हैं।
