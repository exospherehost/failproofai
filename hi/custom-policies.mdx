Custom policies आपको किसी भी agent behavior के लिए rules लिखने देती हैं: project conventions enforce करें, drift रोकें, destructive operations को gate करें, stuck agents detect करें, या Slack, approval workflows वगैरह के साथ integrate करें। ये built-in policies की तरह ही hook event system और `allow`, `deny`, `instruct` decisions का उपयोग करती हैं।

---

## Quick example

```js
// my-policies.js
import { customPolicies, allow, deny, instruct } from "failproofai";

customPolicies.add({
  name: "no-production-writes",
  description: "Block writes to paths containing 'production'",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Write" && ctx.toolName !== "Edit") return allow();
    const path = ctx.toolInput?.file_path ?? "";
    if (path.includes("production")) {
      return deny("Writes to production paths are blocked");
    }
    return allow();
  },
});
```

Install करें:

```bash
failproofai policies --install --custom ./my-policies.js
```

## Install और update करना

```bash
# Custom policies file के साथ install करें
failproofai policies --install --custom ./my-policies.js

# Policies file path बदलें
failproofai policies --install --custom ./new-policies.js

# Config से custom policies path हटाएं
failproofai policies --uninstall --custom
```

Resolved absolute path को `policies-config.json` में `customPoliciesPath` के रूप में store किया जाता है। File हर hook event पर fresh load होती है - events के बीच कोई caching नहीं होती।

## API

### Import

```js
import { customPolicies, allow, deny, instruct } from "failproofai";
```

### `customPolicies.add(hook)`

एक policy register करता है। एक ही file में multiple policies के लिए इसे जितनी बार जरूरी हो call करें।

```ts
customPolicies.add({
  name: string;                         // required - unique identifier
  description?: string;                 // `failproofai policies` output में दिखता है
  match?: { events?: HookEventType[] }; // event type से filter करें; सभी match करने के लिए छोड़ें
  fn: (ctx: PolicyContext) => PolicyResult | Promise<PolicyResult>;
});
```

### Decision helpers

| Function | Effect | कब उपयोग करें |
|----------|--------|----------|
| `allow()` | Operation को चुपचाप permit करें | Action safe है, कोई message जरूरी नहीं |
| `deny(message)` | Operation block करें | Agent को यह action नहीं लेनी चाहिए |
| `instruct(message)` | बिना block किए context दें | Agent को track पर रखने के लिए extra context दें |

`deny(message)` - message Claude को `"Blocked by failproofai:"` prefix के साथ दिखता है। एक भी `deny` सभी आगे की evaluation को short-circuit कर देता है।

`instruct(message)` - message current tool call के लिए Claude के context में append होता है। पहला `instruct` जीतता है — दूसरी policies से आए subsequent `instruct` returns को ignore किया जाता है।

### Informational allow messages (beta)

<Note>
`allow(message)` एक beta feature है जो v0.0.2-beta.3 से available है। API भविष्य के releases में बदल सकता है। पुराने versions में केवल बिना arguments के `allow()` support होता है।
</Note>

`allow(message)` operation को permit करता है **और** Claude को एक informational message भेजता है। Message hook handler के stdout response में `additionalContext` के रूप में deliver होता है — यही mechanism `instruct` भी उपयोग करता है, लेकिन semantically अलग है: यह warning नहीं, बल्कि status update है।

| Function | Effect | कब उपयोग करें |
|----------|--------|----------|
| `allow(message)` | Permit करें और Claude को context भेजें | Confirm करें कि check pass हुई, या बताएं कि check क्यों skip हुई |

Use cases:
- **Status confirmations:** `allow("All CI checks passed.")` — Claude को बताता है कि सब ठीक है
- **Fail-open explanations:** `allow("GitHub CLI not installed, skipping CI check.")` — Claude को बताता है कि check क्यों skip हुई ताकि उसके पास पूरा context हो
- **Multiple messages accumulate:** अगर कई policies `allow(message)` return करती हैं, तो सभी messages newlines से जोड़कर एक साथ deliver होते हैं

```js
customPolicies.add({
  name: "confirm-branch-status",
  match: { events: ["Stop"] },
  fn: async (ctx) => {
    const cwd = ctx.session?.cwd;
    if (!cwd) return allow("No working directory, skipping branch check.");

    // ... check branch status ...
    if (allPushed) {
      return allow("Branch is up to date with remote.");
    }
    return deny("Unpushed changes detected.");
  },
});
```

### `PolicyContext` fields

| Field | Type | Description |
|-------|------|-------------|
| `eventType` | `string` | `"PreToolUse"`, `"PostToolUse"`, `"Notification"`, `"Stop"` |
| `toolName` | `string \| undefined` | जो tool call हो रहा है (जैसे `"Bash"`, `"Write"`, `"Read"`) |
| `toolInput` | `Record<string, unknown> \| undefined` | Tool के input parameters |
| `payload` | `Record<string, unknown>` | Claude Code से पूरा raw event payload |
| `session` | `SessionMetadata \| undefined` | Session context (नीचे देखें) |

### `SessionMetadata` fields

| Field | Type | Description |
|-------|------|-------------|
| `sessionId` | `string` | Claude Code session identifier |
| `cwd` | `string` | Claude Code session की working directory |
| `transcriptPath` | `string` | Session की JSONL transcript file का path |

### Event types

| Event | कब fire होता है | `toolInput` contents |
|-------|--------------|----------------------|
| `PreToolUse` | Claude के tool चलाने से पहले | Tool का input (जैसे Bash के लिए `{ command: "..." }`) |
| `PostToolUse` | Tool complete होने के बाद | Tool का input + `tool_result` (output) |
| `Notification` | जब Claude notification भेजता है | `{ message: "...", notification_type: "idle" \| "permission_prompt" \| ... }` - hooks को हमेशा `allow()` return करना चाहिए, ये notifications block नहीं कर सकते |
| `Stop` | जब Claude session समाप्त होता है | Empty |

## Evaluation order

Policies इस क्रम में evaluate होती हैं:

1. Built-in policies (definition order में)
2. Custom policies (`.add()` order में)

<Note>
पहला `deny` सभी बाद की policies को short-circuit कर देता है। पहला `instruct` जीतता है — subsequent `instruct` returns को ignore किया जाता है।
</Note>

## Transitive imports

Custom policy files relative paths का उपयोग करके local modules import कर सकती हैं:

```js
// my-policies.js
import { isBlockedPath } from "./utils.js";
import { checkApproval } from "./approval-client.js";

customPolicies.add({
  name: "approval-gate",
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const approved = await checkApproval(ctx.toolInput?.command, ctx.session?.sessionId);
    return approved ? allow() : deny("Approval required for this command");
  },
});
```

Entry file से reachable सभी relative imports resolve होते हैं। यह `from "failproofai"` imports को actual dist path पर rewrite करके और ESM compatibility सुनिश्चित करने के लिए temporary `.mjs` files बनाकर implement किया जाता है।

## Event type filtering

Policy कब fire हो, यह limit करने के लिए `match.events` का उपयोग करें:

```js
customPolicies.add({
  name: "require-summary-on-stop",
  match: { events: ["Stop"] },
  fn: async (ctx) => {
    // केवल तब fire होता है जब session समाप्त हो
    // ctx.session.transcriptPath में पूरा session log होता है
    return allow();
  },
});
```

हर event type पर fire करने के लिए `match` पूरी तरह छोड़ दें।

## Error handling और failure modes

Custom policies **fail-open** हैं: errors कभी built-in policies को block नहीं करतीं या hook handler को crash नहीं करतीं।

| Failure | Behavior |
|---------|----------|
| `customPoliciesPath` set नहीं है | कोई custom policies नहीं चलती; built-ins सामान्य रूप से जारी रहते हैं |
| File नहीं मिली | Warning `~/.failproofai/hook.log` में log होती है; built-ins जारी रहते हैं |
| Syntax/import error | Error `~/.failproofai/hook.log` में log होती है; सभी custom policies skip होती हैं |
| `fn` runtime में throw करता है | Error log होती है; वह hook `allow` माना जाता है; दूसरे hooks जारी रहते हैं |
| `fn` 10 सेकंड से अधिक लेता है | Timeout log होता है; `allow` माना जाता है |

<Tip>
Custom policy errors debug करने के लिए, log file देखते रहें:

```bash
tail -f ~/.failproofai/hook.log
```
</Tip>

## Full example: multiple policies

```js
// my-policies.js
import { customPolicies, allow, deny, instruct } from "failproofai";

// Prevent agent from writing to secrets/ directory
customPolicies.add({
  name: "block-secrets-dir",
  description: "Prevent agent from writing to secrets/ directory",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (!["Write", "Edit"].includes(ctx.toolName ?? "")) return allow();
    const path = ctx.toolInput?.file_path ?? "";
    if (path.includes("secrets/")) return deny("Writing to secrets/ is not permitted");
    return allow();
  },
});

// Keep the agent on track: verify tests before committing
customPolicies.add({
  name: "remind-test-before-commit",
  description: "Keep the agent on track: verify tests pass before committing",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const cmd = ctx.toolInput?.command ?? "";
    if (/git\s+commit/.test(cmd)) {
      return instruct("Verify all tests pass before committing. Run `bun test` if you haven't already.");
    }
    return allow();
  },
});

// Prevent unplanned dependency changes during freeze
customPolicies.add({
  name: "dependency-freeze",
  description: "Prevent unplanned dependency changes during freeze period",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const cmd = ctx.toolInput?.command ?? "";
    const isInstall = /^(npm install|yarn add|bun add|pnpm add)\s+\S/.test(cmd);
    if (isInstall && process.env.DEPENDENCY_FREEZE === "1") {
      return deny("Package installs are frozen. Unset DEPENDENCY_FREEZE to allow.");
    }
    return allow();
  },
});

export { customPolicies };
```
