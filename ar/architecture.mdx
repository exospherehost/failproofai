يشرح هذا المستند كيفية عمل failproofai داخلياً: كيف يعترض نظام الخطافات استدعاءات أدوات الوكيل، وكيف يتم تحميل الإعدادات ودمجها، وكيف يتم تقييم السياسات، وكيف تراقب لوحة التحكم نشاط الوكيل.

---

## نظرة عامة

يتكون failproofai من نظامين فرعيين مستقلين:

1. **معالج الخطافات** - عملية فرعية سريعة من CLI يستدعيها Claude Code عند كل استدعاء لأداة الوكيل. يقيّم السياسات ويعيد قراراً.
2. **مراقب الوكيل (لوحة التحكم)** - تطبيق ويب مبني بـ Next.js لمراقبة جلسات الوكيل وإدارة السياسات.

يتشارك النظامان الفرعيان ملفات الإعداد في `~/.failproofai/` ودليل `.failproofai/` الخاص بالمشروع، لكنهما يعملان كعمليتين منفصلتين ويتواصلان فقط عبر نظام الملفات.

## معالج الخطافات

### التكامل مع Claude Code

عند تشغيل `failproofai policies --install`، يكتب إدخالات كالتالي في `~/.claude/settings.json`:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "failproofai --hook PreToolUse"
          }
        ]
      }
    ],
    "PostToolUse": [ ... ]
  }
}
```

يستدعي Claude Code بعدها `failproofai --hook PreToolUse` كعملية فرعية قبل كل استدعاء أداة، مرسلاً حمولة JSON عبر stdin.

### صيغة الحمولة

```json
{
  "session_id": "abc123",
  "transcript_path": "/home/user/.claude/projects/myproject/sessions/abc123.jsonl",
  "cwd": "/home/user/myproject",
  "permission_mode": "default",
  "hook_event_name": "PreToolUse",
  "tool_name": "Bash",
  "tool_input": { "command": "sudo apt install nodejs" }
}
```

في أحداث `PostToolUse`، تحتوي الحمولة أيضاً على `tool_result` مع مخرجات الأداة.

يفرض المعالج حداً أقصى قدره 1 ميغابايت لـ stdin. تُهمل الحمولات التي تتجاوز هذا الحجم وتسمح جميع السياسات ضمنياً.

### صيغة الاستجابة

**الرفض (PreToolUse):**
```json
{
  "hookSpecificOutput": {
    "permissionDecision": "deny",
    "permissionDecisionReason": "Blocked by failproofai: sudo command blocked"
  }
}
```

**الرفض (PostToolUse):**
```json
{
  "hookSpecificOutput": {
    "additionalContext": "Blocked by failproofai because: API key detected in output"
  }
}
```

**التعليمات (أي حدث باستثناء Stop):**
```json
{
  "hookSpecificOutput": {
    "additionalContext": "Instruction from failproofai: Verify tests pass before committing."
  }
}
```

**تعليمات حدث Stop:**
- كود الخروج: `2`
- يُكتب السبب إلى stderr (وليس stdout)

**السماح:**
- كود الخروج: `0`
- stdout فارغ

**السماح مع رسالة (تجريبي):**

منذ الإصدار v0.0.2-beta.3، يتيح `allow(message)` للسياسة إرسال سياق معلوماتي إلى Claude حتى عند السماح بالعملية. يكتب معالج الخطافات JSON التالي إلى **stdout** (ليس ملف إعداد — هذه هي استجابة المعالج إلى Claude Code، تماماً مثل استجابات الرفض والتعليمات أعلاه):

```json
// Written to stdout by the hook handler process
{
  "hookSpecificOutput": {
    "additionalContext": "All CI checks passed on branch 'feat/my-feature'."
  }
}
```
- كود الخروج: `0` (العملية مسموح بها)
- عندما تُعيد سياسات متعددة `allow` مع رسالة، تُدمج رسائلها بسطور جديدة في سلسلة `additionalContext` واحدة
- إذا لم تُقدّم أي سياسة رسالة، يكون stdout فارغاً (كما كان سابقاً)

### خط معالجة البيانات

يُنفذ `src/hooks/handler.ts` خط المعالجة الكامل:

```text
stdin JSON
  → parse payload (max 1 MB)
  → extract session metadata (session_id, cwd, tool_name, tool_input, etc.)
  → readMergedHooksConfig(cwd)    ← merges project + local + global config
  → register enabled builtin policies with resolved params
  → load custom policies from customPoliciesPath (if set)
  → register custom policies into policy registry
  → evaluate all policies (builtins first, then custom)
      → first deny short-circuits
      → instruct decisions accumulate
      → allow messages accumulate
  → write JSON decision to stdout
  → persist event to ~/.failproofai/hook-activity.jsonl
  → exit
```

تعمل العملية بأكملها في أقل من 100 ميلي ثانية للحمولات النموذجية دون أي استدعاءات لنماذج اللغة الكبيرة.

## تحميل الإعدادات

يُنفذ `src/hooks/hooks-config.ts` تحميل الإعدادات على ثلاثة نطاقات.

```text
[1] {cwd}/.failproofai/policies-config.json        ← المشروع  (أعلى أولوية)
[2] {cwd}/.failproofai/policies-config.local.json  ← محلي
[3] ~/.failproofai/policies-config.json             ← عام   (أدنى أولوية)
```

منطق الدمج:
- `enabledPolicies` - اتحاد مكرر من الملفات الثلاثة
- `policyParams` - لكل سياسة، أول ملف يُعرّفها يفوز بالكامل
- `customPoliciesPath` - أول ملف يُعرّفه يفوز
- `llm` - أول ملف يُعرّفه يفوز

تستخدم لوحة التحكم `readHooksConfig()` (العام فقط) للقراءة والكتابة، لأنها لا تُستدعى مع مسار cwd للمشروع.

## تقييم السياسات

يُشغّل `src/hooks/policy-evaluator.ts` السياسات بالترتيب.

لكل سياسة:

1. البحث عن مخطط `params` للسياسة (إن وجد).
2. قراءة `policyParams[policy.name]` من الإعداد المدمج.
3. دمج القيم التي حددها المستخدم فوق القيم الافتراضية للمخطط لإنتاج `ctx.params`.
4. استدعاء `policy.fn(ctx)` مع السياق المُحلَّل.
5. إذا كانت النتيجة `deny`، يتوقف فوراً ويُعيد ذلك القرار.
6. إذا كانت النتيجة `instruct`، يجمع الرسالة ويكمل.
7. إذا كانت النتيجة `allow`، ينتقل إلى السياسة التالية.

بعد تشغيل جميع السياسات:
- إذا أعيد أي `deny`، يُصدر استجابة الرفض.
- إذا جُمعت أي استجابات `instruct`، يُصدر استجابة تعليمات واحدة بجميع الرسائل مدمجة.
- وإلا، يُصدر استجابة سماح (stdout فارغ، كود الخروج 0).

## السياسات المدمجة

يُعرّف `src/hooks/builtin-policies.ts` جميع السياسات المدمجة الـ 26 كعناصر `BuiltinPolicyDefinition`:

```typescript
interface BuiltinPolicyDefinition {
  name: string;
  description: string;
  fn: (ctx: PolicyContext) => PolicyResult;
  match: {
    events: HookEventType[];
    tools?: string[];
  };
  defaultEnabled: boolean;
  category: string;
  beta?: boolean;
  params?: PolicyParamsSchema;
}
```

السياسات التي تقبل `params` تُعلن عن `PolicyParamsSchema` بأنواع وقيم افتراضية لكل معامل. يُدخل مُقيّم السياسات القيم المُحلَّلة في `ctx.params` قبل استدعاء `fn`. تقرأ دوال السياسات `ctx.params` دون الحاجة للتحقق من القيم الفارغة لأن القيم الافتراضية تُطبَّق دائماً أولاً.

يستخدم مطابقة الأنماط داخل السياسات رموز الأمر المُحللة (argv)، وليس مطابقة السلاسل الخام. هذا يمنع التحايل عبر حقن عوامل الصدفة (مثلاً، نمط لـ `sudo systemctl status *` لا يمكن تجاوزه بإضافة `; rm -rf /` إلى الأمر).

## السياسات المخصصة

يُنفذ `src/hooks/custom-hooks-registry.ts` سجلاً مدعوماً بـ `globalThis`:

```typescript
const REGISTRY_KEY = "__failproofai_custom_hooks__";

export const customPolicies = {
  add(hook: CustomHook): void { ... }
};

export function getCustomHooks(): CustomHook[] { ... }
export function clearCustomHooks(): void { ... }  // used in tests
```

يحمّل `src/hooks/custom-hooks-loader.ts` ملف السياسة الخاص بالمستخدم:

1. قراءة `customPoliciesPath` من الإعداد؛ التخطي إذا لم يكن موجوداً.
2. التحليل إلى مسار مطلق؛ التحقق من وجود الملف.
3. إعادة كتابة جميع استيرادات `from "failproofai"` إلى مسار dist الفعلي كي يحلّ `customPolicies` إلى نفس سجل `globalThis`.
4. إعادة كتابة الاستيرادات المحلية المتعدية بشكل تكراري لضمان توافق ESM.
5. كتابة ملفات `.mjs` مؤقتة و`import()` ملف الإدخال.
6. استدعاء `getCustomHooks()` لاسترداد الخطافات المُسجَّلة.
7. تنظيف جميع الملفات المؤقتة في كتلة `finally`.

عند أي خطأ (ملف غير موجود، خطأ في الصياغة، فشل الاستيراد)، يُسجَّل الخطأ في `~/.failproofai/hook.log` ويُعيد المحمّل مصفوفة فارغة. لا تتأثر السياسات المدمجة.

تُقيَّم السياسات المخصصة بعد جميع السياسات المدمجة. قرار `deny` من سياسة مخصصة لا يزال يوقف السياسات المخصصة الأخرى (لكن جميع السياسات المدمجة تكون قد عملت بالفعل عند تلك النقطة).

## تسجيل النشاط

بعد كل حدث خطاف، يُلحق المعالج سطر JSONL بـ `~/.failproofai/hook-activity.jsonl`:

```json
{
  "timestamp": "2026-04-06T12:34:56.789Z",
  "sessionId": "abc123",
  "eventType": "PreToolUse",
  "toolName": "Bash",
  "policyName": "block-sudo",
  "decision": "deny",
  "reason": "sudo command blocked by failproofai",
  "durationMs": 12
}
```

سطر واحد لكل سياسة أصدرت قراراً غير سماح. لا تُسجَّل قرارات السماح (للحفاظ على صغر حجم الملف).

## بنية لوحة التحكم

لوحة التحكم هي تطبيق **Next.js 16** يستخدم App Router مع React Server Components وServer Actions.

```text
app/
  layout.tsx                  ← التخطيط الجذري (السمة، القياس، التنقل)
  projects/page.tsx           ← مكوّن خادم: عرض جميع مشاريع Claude
  project/[name]/page.tsx     ← مكوّن خادم: عرض الجلسات في مشروع
  project/[name]/session/
    [sessionId]/page.tsx      ← مكوّن خادم: عرض مشاهد الجلسة
  policies/page.tsx           ← مكوّن عميل: إدارة السياسات + سجل النشاط
  actions/
    get-hooks-config.ts       ← قراءة الإعداد + قائمة السياسات
    update-hooks-config.ts    ← تفعيل/تعطيل السياسة
    update-policy-params.ts   ← تحديث معاملات السياسة
    get-hook-activity.ts      ← تصفح/بحث سجل النشاط
    install-hooks-web.ts      ← تثبيت/إزالة الخطافات من المتصفح
  api/
    download/[project]/[session]/route.ts   ← تصدير الجلسة كـ ZIP/JSONL
```

**تدفق البيانات:**

- تستدعي مكونات الصفحة `lib/projects.ts` و`lib/log-entries.ts` لقراءة بيانات المشروع/الجلسة مباشرة من نظام الملفات (لا توجد طبقة API للقراءات).
- تستخدم صفحة السياسات Server Actions لجميع عمليات التعديل (التبديل، تحديث المعاملات، التثبيت/الإزالة).
- يحلل مشاهد الجلسة صيغة JSONL الخاصة بـ Claude ويعرض جدولاً زمنياً للرسائل واستدعاءات الأدوات.

**قرارات التصميم الرئيسية:**

- لا قاعدة بيانات - جميع الحالات الدائمة في ملفات عادية (`~/.failproofai/`، `~/.claude/projects/`).
- Server Actions للتعديلات - لا حاجة لـ REST API لعمليات CRUD.
- React Server Components لصفحات القراءة - تحميل أولي أسرع، لا حزمة عميل لجلب البيانات.
- مكونات العميل فقط عند الحاجة للتفاعل (تبديل السياسات، بحث النشاط، مشاهد السجل).
