يأتي failproofai مع 39 سياسة مدمجة تتعامل مع أنماط الفشل الشائعة للعوامل. تعمل كل سياسة على حدث خطاف معين واسم أداة محدد. تقبل تسع عشرة سياسة معاملات تتيح لك ضبط سلوكها دون كتابة أكواد. خمس سياسات سير عمل تفرض خط أنابيب commit → push → PR → CI قبل توقف Claude.

---

## نظرة عامة

يتم تجميع السياسات في فئات:

| الفئة | السياسات | نوع الخطاف |
|----------|----------|-----------|
| [الأوامر الخطرة](#dangerous-commands) | block-sudo, block-rm-rf, block-curl-pipe-sh, block-failproofai-commands | PreToolUse |
| [أوامر البنية التحتية](#infra-commands) | block-kubectl, block-terraform, block-aws-cli, block-gcloud, block-az-cli, block-helm, block-gh-pipeline | PreToolUse |
| [الأسرار (معقمات)](#secrets-sanitizers) | sanitize-jwt, sanitize-api-keys, sanitize-connection-strings, sanitize-private-key-content, sanitize-bearer-tokens | PostToolUse |
| [البيئة](#environment) | block-env-files, protect-env-vars | PreToolUse |
| [الوصول إلى الملفات](#file-access) | block-read-outside-cwd, block-secrets-write | PreToolUse |
| [Git](#git) | block-push-master, block-work-on-main, block-force-push, warn-git-amend, warn-git-stash-drop, warn-all-files-staged | PreToolUse |
| [قاعدة البيانات](#database) | warn-destructive-sql, warn-schema-alteration | PreToolUse |
| [التحذيرات](#warnings) | warn-large-file-write, warn-package-publish, warn-background-process, warn-global-package-install | PreToolUse |
| [مديري الحزم](#package-managers) | prefer-package-manager | PreToolUse |
| [سير العمل](#workflow) | require-commit-before-stop, require-push-before-stop, require-pr-before-stop, require-no-conflicts-before-stop, require-ci-green-before-stop | Stop |

- **`block-`** — وقف العامل عن المتابعة.
- **`warn-`** — إعطاء العامل سياق إضافي حتى يتمكن من التصحيح الذاتي.
- **`sanitize-`** — مسح البيانات الحساسة من مخرجات الأداة قبل أن يراها العامل.

### المساحات

توجد كل سياسة في فتحة `<namespace>/<name>`. تنتمي السياسات المدمجة إلى مساحة
**`exospherehost/`** — على سبيل المثال، `exospherehost/sanitize-jwt`. تمنع المساحة
التصادمات عند تحميل سياسات مخصصة أو تابعة لطرف ثالث بأسماء قصيرة مشابهة.

في إعدادك يمكنك الإشارة إلى سياسة مدمجة إما باسمها القصير أو اسمها المؤهل؛ تحل كلا الصيغتين إلى نفس السياسة:

```json
{
  "enabledPolicies": [
    "sanitize-jwt",
    "exospherehost/block-rm-rf"
  ]
}
```

إذا كان الاسم بدون `/`، يتعامل failproofai معه على أنه ينتمي إلى المساحة الافتراضية
`exospherehost`. الأسماء التي تحتوي بالفعل على `/` (على سبيل المثال `myorg/foo`،
`custom/my-hook`) يتم الاحتفاظ بها كما هي.
- **`require-`** — منع حدث Stop حتى استيفاء الشروط.

---

<Tip>
تدعم كل سياسة حقل اختياري `hint` في `policyParams`. يتم إضافة التلميح إلى رسالة الرفض أو التعليمات التي يراها Claude، مما يوفر إرشادات قابلة للتنفيذ دون تعديل كود السياسة. يعمل مع السياسات المدمجة والمخصصة والاتفاقية. راجع [التكوين → hint](/ar/configuration#hint-cross-cutting) للتفاصيل.
</Tip>

## الأوامر الخطرة

منع العوامل من تشغيل العمليات التي يصعب التراجع عنها أو التي قد تضر بنظام الاستضافة.

### `block-sudo`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يرفض أي أمر `sudo`.

يمنع الاستدعاءات التي تتضمن كلمة مفتاحية `sudo`. يتم مطابقة النمط على رموز الأوامر المحللة، وليس السلسلة الخام، لمنع الالتفاف عبر حقن عامل shell.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | بادئات الأوامر الدقيقة المسموحة. يتم مطابقة كل إدخال مقابل رموز argv المحللة. |

**مثال:**

```json
{
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    }
  }
}
```

مع هذا الإعداد، يُسمح بـ `sudo systemctl status nginx`، لكن `sudo rm /etc/hosts` مرفوض.

<Note>
يتم مطابقة الأنماط مقابل الرموز المحللة، وليس السلسلة الخام للأمر. يمنع هذا الالتفاف عبر عوامل shell الملحقة (على سبيل المثال، `sudo systemctl status x; rm -rf /` لا يطابق `sudo systemctl status *`).
</Note>

## أوامر البنية التحتية

منع عوامل الترميز من تشغيل أوامر CLI للبنية التحتية أو تشغيل خطوط أنابيب CI/CD. جميع السياسات في هذه الفئة اختيارية **(`defaultEnabled: false`)** — العوامل التي تحتاج حقاً إلى استدعاء `kubectl`, `terraform`, إلخ. لن تتأثر إلا إذا قمت بتفعيل السياسة. عند التفعيل، يتم رفض كل استدعاء للـ CLI المطابقة إلا إذا كان الأمر يطابق إدخالاً في `allowPatterns`.

قواعد نحو النمط هي نفسها كما في [`block-sudo`](#block-sudo): يتم مطابقة الرموز مقابل argv المحللة، `*` هو بدل لرمز واحد، وأي أمر يحتوي على عامل shell مستقل (`&&`, `||`, `|`, `;`) أو رمز به أحرف metacharacter مضمنة يتم رفضه قبل مطابقة القائمة البيضاء لمنع التجاوزات بالحقن.

### `block-kubectl`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يرفض أي استدعاء `kubectl`.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | بادئات أوامر kubectl المسموحة. |

**مثال:**

```json
{
  "policyParams": {
    "block-kubectl": {
      "allowPatterns": ["kubectl get *", "kubectl describe *", "kubectl logs *"]
    }
  }
}
```

مع هذا الإعداد، يُسمح بـ `kubectl get pods` لكن `kubectl apply -f deploy.yaml` مرفوض.

### `block-gh-pipeline`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يرفض أوامر فرعية `gh` CLI التالية التي تغير الحالة أو تشغل خطوط الأنابيب:

- `gh workflow run`, `gh workflow enable`, `gh workflow disable`
- `gh run rerun`, `gh run cancel`
- `gh pr merge`
- `gh release create`, `gh release delete`
- `gh cache delete`
- `gh secret set`, `gh secret delete`

أوامر `gh` الفرعية للقراءة فقط مثل `gh pr view`, `gh pr list`, `gh run list`, `gh release view`, و `gh api repos/.../...` **لا** يتم مطابقتها بهذه السياسة — فهي مطلوبة بشكل منتظم لفحوصات سير العمل (بما في ذلك `require-ci-green-before-stop` الخاص بـ failproofai).

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | استدعاءات برمجية محددة مراد السماح بها على الرغم من أنها ستكون مرفوضة بخلاف ذلك. |

**مثال:**

```json
{
  "policyParams": {
    "block-gh-pipeline": {
      "allowPatterns": ["gh run rerun *"]
    }
  }
}
```

## الأسرار (معقمات)

منع العوامل من تسريب بيانات الاعتماد إلى سياقها أو إخراجها. تعمل سياسات المعقم على أحداث **PostToolUse**. عندما يشغل Claude أمر Bash، أو يقرأ ملفاً، أو يستدعي أي أداة، تفحص هذه السياسات الإخراج قبل إرجاعه إلى Claude. إذا تم اكتشاف نمط سري، تعود السياسة بقرار رفض يمنع الإخراج من العودة.

### `sanitize-jwt`

**الحدث:** PostToolUse (جميع الأدوات)  
**الافتراضي:** يعيد تشفير رموز JWT (ثلاث قطاعات base64url مفصولة بـ `.`).

بدون معاملات.

---

### `sanitize-api-keys`

**الحدث:** PostToolUse (جميع الأدوات)  
**الافتراضي:** يعيد تشفير صيغ مفاتيح API الشائعة: Anthropic (`sk-ant-`), OpenAI (`sk-`), GitHub PATs (`ghp_`), مفاتيح الوصول AWS (`AKIA`), مفاتيح Stripe (`sk_live_`, `sk_test_`), ومفاتيح Google API (`AIza`).

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `additionalPatterns` | `{ regex: string; label: string }[]` | `[]` | أنماط regex إضافية يجب التعامل معها كأسرار. |

**مثال:**

```json
{
  "policyParams": {
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo internal API key" },
        { "regex": "pat_[0-9a-f]{40}", "label": "Internal PAT" }
      ]
    }
  }
}
```

---

### `sanitize-connection-strings`

**الحدث:** PostToolUse (جميع الأدوات)  
**الافتراضي:** يعيد تشفير سلاسل الاتصال بقاعدة البيانات التي تحتوي على بيانات اعتماد مضمنة (على سبيل المثال، `postgresql://user:password@host/db`).

بدون معاملات.

---

### `sanitize-private-key-content`

**الحدث:** PostToolUse (جميع الأدوات)  
**الافتراضي:** يعيد تشفير كتل PEM (`-----BEGIN PRIVATE KEY-----`, `-----BEGIN RSA PRIVATE KEY-----`, إلخ).

بدون معاملات.

---

### `sanitize-bearer-tokens`

**الحدث:** PostToolUse (جميع الأدوات)  
**الافتراضي:** يعيد تشفير رؤوس `Authorization: Bearer <token>` حيث يكون الرمز 20 حرفاً أو أكثر.

بدون معاملات.

## البيئة

حماية إعدادات البيئة الحساسة من القراءة أو التعريض من قبل العوامل.

### `block-env-files`

**الحدث:** PreToolUse (Bash, Read)  
**الافتراضي:** يرفض قراءة ملفات `.env` عبر `cat .env`، استدعاءات أداة Read بـ `.env` كمسار الملف، إلخ.

لا يمنع `.envrc` أو ملفات أخرى مجاورة للبيئة - فقط الملفات المسماة بدقة `.env`.

بدون معاملات.

---

### `protect-env-vars`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يرفض الأوامر التي تطبع متغيرات البيئة: `printenv`, `env`, `echo $VAR`.

بدون معاملات.

## الوصول إلى الملفات

إبقاء العوامل تعمل داخل حدود المشروع والبعيدة عن الملفات الحساسة.

### `block-read-outside-cwd`

**الحدث:** PreToolUse (Read, Bash)  
**الافتراضي:** يرفض قراءة الملفات خارج جذر المشروع. الحد هو `CLAUDE_PROJECT_DIR` (تعيينه مرة واحدة لكل جلسة بواسطة Claude Code)، مع عودة إلى دليل العمل الحالي للجلسة عندما لا يتم تعيين هذا المتغير. استخدام جذر المشروع بدلاً من `cwd` الحي يعني أن الحد يبقى مستقراً حتى بعد قيام Claude بـ `cd` إلى مجلد فرعي.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `allowPaths` | `string[]` | `[]` | بادئات المسار المطلق المسموحة حتى لو كانت خارج جذر المشروع. |

**مثال:**

```json
{
  "policyParams": {
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company/config"]
    }
  }
}
```

---

### `block-secrets-write`

**الحدث:** PreToolUse (Write, Edit)  
**الافتراضي:** يرفض الكتابة إلى الملفات الشائع استخدامها للمفاتيح الخاصة والشهادات: `id_rsa`, `id_ed25519`, `*.key`, `*.pem`, `*.p12`, `*.pfx`.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `additionalPatterns` | `string[]` | `[]` | أنماط أسماء ملفات إضافية (بنمط glob) يجب منعها. |

**مثال:**

```json
{
  "policyParams": {
    "block-secrets-write": {
      "additionalPatterns": [".token", ".secret"]
    }
  }
}
```

## Git

منع عمليات الدفع العرضية، والدفع القسري، والأخطاء في الفروع التي يصعب التراجع عنها.

### `block-push-master`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يرفض `git push origin main` و `git push origin master`.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `protectedBranches` | `string[]` | `["main", "master"]` | أسماء الفروع التي لا يمكن الدفع إليها مباشرة. |

**مثال:**

```json
{
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "master", "release", "prod"]
    }
  }
}
```

<Tip>
للسماح بالدفع إلى جميع الفروع (بفعالية تعطيل هذه السياسة دون إزالتها من `enabledPolicies`)، عيّن `protectedBranches: []`.
</Tip>

---

### `block-work-on-main`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يرفض `git commit`, `git merge`, `git rebase`, و `git cherry-pick` بينما شجرة العمل على `main` أو `master`. إنشاء الفروع والتبديل بينها (`git checkout`, `git checkout -b`, `git switch`, `git switch -c`) لا تتأثر.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `protectedBranches` | `string[]` | `["main", "master"]` | أسماء الفروع التي يتم فيها رفض commit/merge/rebase/cherry-pick. |

---

### `block-force-push`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يرفض `git push --force` و `git push -f`.

بدون معاملات خاصة بالسياسة. استخدم التقاطع [`hint`](/ar/configuration#hint-cross-cutting) لاقتراح بدائل:

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "Create a new branch from your current HEAD (e.g. `git checkout -b <new-branch>`) and push that instead."
    }
  }
}
```

---

## مديري الحزم

فرض أي مديري حزم مسموح للعامل باستخدامهم.

### `prefer-package-manager`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** معطل. عند التفعيل، يمنع أي أمر مدير حزم غير في قائمة `allowed` ويخبر Claude بإعادة كتابة الأمر باستخدام مدير مسموح به.

يكتشف: pip, pip3, python -m pip, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo.

| المعامل | النوع | الافتراضي | الوصف |
|-----------|------|---------|-------------|
| `allowed` | string[] | `[]` | أسماء مديري الحزم المسموحة. أي مدير مكتشف ليس في هذه القائمة يتم منعه. عند الترك فارغاً، السياسة بلا تأثير. |
| `blocked` | string[] | `[]` | أسماء مديري إضافية يجب منعها بعيداً عن القائمة المدمجة (على سبيل المثال `['pdm', 'pipx']`). |

قائمة الحجب المدمجة تغطي: pip, pip3, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo. استخدم `blocked` لإضافة مديرين غير موجودين في هذه القائمة.

**إعداد مثال:**

```json
{
  "enabledPolicies": ["prefer-package-manager"],
  "policyParams": {
    "prefer-package-manager": {
      "allowed": ["uv", "bun"],
      "blocked": ["pdm", "pipx"]
    }
  }
}
```

مع هذا الإعداد، يتم رفض `pip install flask` و `pdm install flask` برسالة تخبر Claude بأن يستخدم `uv` أو `bun` بدلاً من ذلك. أوامر مثل `uv pip install flask` مسموحة لأن `uv` موجود في قائمة السماح ويتم فحصه أولاً.

## سير العمل

فرض سير عمل منضبط للنهاية. تعمل هذه السياسات على حدث **Stop** وترفض العامل من التوقف حتى استيفاء كل شرط. تتبع سلسلة تبعية طبيعية: commit → push → PR → CI. إذا رفضت سياسة، يتم تخطي السياسات اللاحقة في السلسلة (الرفض يقطع المسار).

جميع سياسات سير العمل **fail-open**: إذا لم تكن الأداة المطلوبة متاحة (على سبيل المثال `gh` غير مثبت، لا جهاز تحكم عن بعد)، السياسة تسمح برسالة إعلامية تشرح سبب تخطي الفحص.

### دلالات Stop لكل CLI

يبدو فرض التوقف مختلفاً قليلاً عبر CLI السبع المدعومة لأن كل واحد يفضح عقد خطاف end-of-agent مختلف. **النتيجة** هي نفسها — العامل لا يهرب بالتوقف بينما بوابة سير العمل فاشلة — لكن **الميكانيكا** تختلف. الجدول أدناه يلخصها؛ فقط Pi له دقة مرئية تستحق الفهم قبل تفعيل سياسة `require-*-before-stop`.

| CLI | عند إطلاق البوابة | ما تراه |
|---|---|---|
| Claude Code | نفس حلقة العامل، فوراً | يستمر Claude في العمل — يصلح المشكلة، ثم يحاول الانتهاء مرة أخرى. لا انقطاع مرئي لك. |
| Codex | نفس حلقة العامل، فوراً | نفس Claude. |
| GitHub Copilot CLI | نفس حلقة العامل، فوراً | نفس Claude (يستخدم قناة إعادة محاولة `{decision:"block", reason}` من Copilot — تم التحقق تجريبياً مقابل Copilot CLI 1.0.41). |
| Cursor Agent | نفس حلقة العامل، فوراً | نفس Claude (يستخدم قناة `{followup_message}` من Cursor — مغطاة بـ `loop_limit`، الافتراضي 5 إعادة محاولات). |
| Gemini CLI | نفس حلقة العامل، فوراً | نفس Claude (يستخدم قناة `{decision:"block", reason}` من Gemini على `AfterAgent`). |
| OpenCode | نفس حلقة العامل، فوراً | نفس Claude (يستخدم استدعاء SDK `client.session.prompt(...)` من OpenCode الموجه عبر `hookSpecificOutput.additionalContext`). |
| **Pi (pi-coding-agent)** | **دور المستخدم التالي** | **يتوقف Pi بوضوح** عند إطلاق البوابة — حلقة وكيله تخرج وتعود إلى الفحص. البوابة تطلق عند المرة التالية التي ترسل فيها فحص: يضيف failproofai موجهة `MANDATORY ACTION REQUIRED` إلى موجه النظام لذلك الدور، مما يعطي التعليمات للـ LLM لإكمال خطوة سير العمل (commit, push, إلخ) قبل فعل ما طلبته. |

<Note>
**قيد Pi.** `AgentEndEvent` من Pi (المكافئ أعلى مستوى لـ`Stop` hook من Claude) لا له نوع Result — في الوقت الذي يطلق فيه، حلقة عامل Pi قد خرج بالفعل. لا يمكن فرض Pi على إعادة محاولة نفس الحلقة بالطريقة التي يمكن بها Claude / Copilot / Cursor / Gemini / OpenCode. يحول failproofai البوابة إلى حدث `before_agent_start` من Pi (الذي يطلق بعد الفحص التالي للمستخدم) حتى فحص سير العمل يفرض، فقط على الدور التالي بدلاً من الحالي.

**ما هذا يعني عملياً:**

- بعد توقف Pi، يتم التقاط سبب الرفض في الذاكرة مفهرساً بمعرّف جلسة Pi. الفحص التالي بالضبط الذي تقدمه في نفس عملية Pi يصفيه: يرى الـ LLM موجهة `MANDATORY ACTION REQUIRED` في أعلى موجه نظامه، يرتكب (أو يدفع / يفتح الـ PR / ينتظر CI)، وفقط بعد ذلك يستمر مع طلبك. سبب الرفض الملتقط هو لمرة واحدة — بمجرد تصفيته، البوابة واضحة.
- البوابة محدودة بعمر عملية Pi. إذا قمت بـ `Ctrl+C` Pi أو تركت بين الأدوار، الإدخال في الذاكرة يتم إسقاطه جنباً إلى جنب مع العملية والبوابة تُفتقد. Claude, Copilot, Cursor, Gemini, و OpenCode لها نفس الحد (اقتل العامل والبوابة تُفتقد) — Pi يجعله فقط أكثر وضوحاً لأن العامل يخرج بوضوح قبل إطلاق البوابة.
- رفض معلق يتم مسحه أيضاً على `session_shutdown` لأي سبب (`new` / `resume` / `fork` / `quit`)، لذا بوابة قديمة من جلسة سابقة لا يمكنها أن تسرب إلى جلسة جديدة بدأت في نفس عملية Pi.

إذا احتجت إلى إعادة محاولة نفس الحلقة على طريقة Claude، شغّل سياسات `Stop` الخاصة بك تحت أي من CLIs الستة المدعومة الأخرى. نحن نتابع Pi أعلى مستوى بحثاً عن نوع Result مستقبلي على `AgentEndEvent` الذي سيسمح لنا بإغلاق هذه الفجوة.
</Note>

### `require-no-conflicts-before-stop`

**الحدث:** Stop  
**الافتراضي:** يرفض التوقف عندما لا يمكن دمج الفرع الحالي بنظافة في الفرع الأساسي. تتحقق السياسة أولاً من وجود PR `OPEN` على GitHub للفرع — بدونه، لا هدف دمج لفرضه، لذا السياسة بأكملها تقصر للسماح. بمجرد تأكيد PR `OPEN`، يتم تشغيل مسحان مستقلان:

1. **محلي** — `git merge-tree --write-tree --name-only origin/<baseBranch> HEAD`. في حالة النزاع، رسالة الرفض تسمي الملفات المتنازع عليها حتى يعرف Claude بالضبط ما يجب حله.
2. **GitHub** — يعيد استخدام نتيجة `gh pr view --json mergeable,state` المجلوبة بالفعل في الفحص المسبق. يلتقط التنازعات التي قد يفتقدها `origin/<baseBranch>` القديم محلياً (على سبيل المثال شخص ما هبط PR متنازع على `main` منذ آخر جلب). نتيجة `CONFLICTING` ترفض. نتيجة `UNKNOWN` ترفض أيضاً وتعطي Claude تعليمات للانتظار ~10 ثوان والتحقق مرة أخرى قبل محاولة التوقف — يمنع هذا السلبيات الكاذبة بينما GitHub يعيد الحساب.

تقفز تماماً (تسمح) عندما: `gh` غير مثبت، لا PR موجود للفرع، حالة الـ PR ليست `OPEN` (على سبيل المثال `MERGED`, `CLOSED`)، أو `gh pr view` يعود بمخرجات غير قابلة للتحليل. تفشل أيضاً مفتوحة عندما `origin/<baseBranch>` مفقود محلياً أو عندما لا التزام قدماً من القاعدة — سقوط طبقة 1 هذه تستشير mergeability المخزن مؤقتاً للـ PR قبل السماح.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `baseBranch` | `string` | `"main"` | الفرع الأساسي للفحص للتنازعات ضده. |

<Note>
GitHub CLI (`gh`) مطلوب لهذه السياسة. تستخدم السياسة `gh pr view` لتأكيد
وجود PR `OPEN` قبل تشغيل أي مسح تنازع — بدون `gh`، السياسة
تقصر للسماح. شغّل `gh auth login` برمز وصول شخصي له
نطاق `repo` للوصول للقراءة إلى طلبات الدمج.
</Note>

أزل سياسة محددة من `enabledPolicies` في إعدادك، أو أيقفها في tab الـ Policies للـ dashboard.

```json
{
  "enabledPolicies": [
```

السياسات غير المدرجة في `enabledPolicies` لا تعمل، حتى إذا وجدت إدخالات `policyParams` لها.
