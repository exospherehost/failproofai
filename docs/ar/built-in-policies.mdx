يأتي failproofai مع 39 سياسة مدمجة تكتشف أوضاع فشل الوكيل الشائعة. تعمل كل سياسة على نوع حدث hook معين واسم أداة محددة. تقبل تسع عشرة سياسة معاملات تسمح لك بضبط سلوكها دون كتابة أي أكواد. تفرض خمس سياسات سير عمل خط أنابيب commit → push → PR → CI قبل توقف Claude.

---

## نظرة عامة

يتم تجميع السياسات في فئات:

| الفئة | السياسات | نوع Hook |
|----------|----------|-----------|
| [الأوامر الخطرة](#dangerous-commands) | block-sudo, block-rm-rf, block-curl-pipe-sh, block-failproofai-commands | PreToolUse |
| [أوامر البنية التحتية](#infra-commands) | block-kubectl, block-terraform, block-aws-cli, block-gcloud, block-az-cli, block-helm, block-gh-pipeline | PreToolUse |
| [الأسرار (منظفات)](#secrets-sanitizers) | sanitize-jwt, sanitize-api-keys, sanitize-connection-strings, sanitize-private-key-content, sanitize-bearer-tokens | PostToolUse |
| [البيئة](#environment) | block-env-files, protect-env-vars | PreToolUse |
| [الوصول إلى الملفات](#file-access) | block-read-outside-cwd, block-secrets-write | PreToolUse |
| [Git](#git) | block-push-master, block-work-on-main, block-force-push, warn-git-amend, warn-git-stash-drop, warn-all-files-staged | PreToolUse |
| [قاعدة البيانات](#database) | warn-destructive-sql, warn-schema-alteration | PreToolUse |
| [تحذيرات](#warnings) | warn-large-file-write, warn-package-publish, warn-background-process, warn-global-package-install | PreToolUse |
| [مديرو الحزم](#package-managers) | prefer-package-manager | PreToolUse |
| [سير العمل](#workflow) | require-commit-before-stop, require-push-before-stop, require-pr-before-stop, require-no-conflicts-before-stop, require-ci-green-before-stop | Stop |

- **`block-`** — يوقف الوكيل عن المتابعة.
- **`warn-`** — يعطي الوكيل سياقاً إضافياً حتى يتمكن من تصحيح نفسه.
- **`sanitize-`** — ينظف البيانات الحساسة من مخرجات الأداة قبل أن يراها الوكيل.

### Namespaces

تعيش كل سياسة في فتحة `<namespace>/<name>`. تنتمي السياسات المدمجة إلى namespace
**`failproofai/`** — على سبيل المثال، `failproofai/sanitize-jwt`. يمنع
namespace التصادمات عند تحميل سياسات مخصصة أو من طرف ثالث
بأسماء قصيرة مشابهة.

في ملف الإعدادات الخاص بك، يمكنك الإشارة إلى سياسة مدمجة باستخدام اسمها القصير أو
اسمها المؤهل؛ كلا النموذجين ينقلان إلى نفس السياسة:

```json
{
  "enabledPolicies": [
    "sanitize-jwt",
    "failproofai/block-rm-rf"
  ]
}
```

إذا لم يحتوي الاسم على `/`، يتعامل failproofai معه كأنه ينتمي إلى namespace الافتراضي
`failproofai`. الأسماء التي تحتوي بالفعل على `/` (مثل `myorg/foo`,
`custom/my-hook`) يتم الاحتفاظ بها كما هي.
- **`require-`** — يوقف حدث Stop حتى تتحقق الشروط.

---

<Tip>
تدعم كل سياسة حقل `hint` اختياري في `policyParams`. يتم إلحاق hint برسالة deny أو instruct التي يراها Claude، مما يوفر إرشادات قابلة للتطبيق دون تعديل أكواد السياسة. يعمل مع السياسات المدمجة والمخصصة والاتفاقية. راجع [التكوين → hint](/ar/configuration#hint-cross-cutting) للمزيد من التفاصيل.
</Tip>

## الأوامر الخطرة

منع الوكلاء من تشغيل عمليات يصعب التراجع عنها أو قد تلحق الضرر بنظام المضيف.

### `block-sudo`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** ينكر أي أمر `sudo`.

يحجب الاستدعاءات التي تتضمن كلمة `sudo`. يتم إجراء مطابقة الأنماط على tokens الأوامر التي تم تحليلها، وليس على السلسلة الخام، لمنع الالتفاف عبر حقن عوامل shell.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | بادئات الأوامر الدقيقة المسموحة. يتم مطابقة كل إدخال مقابل tokens argv المحللة. |

**مثال:**

```json
{
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    }
  }
}
```

مع هذا الإعداد، يُسمح بـ `sudo systemctl status nginx`، لكن `sudo rm /etc/hosts` يُنكر.

<Note>
يتم مطابقة الأنماط مقابل tokens المحللة، وليس سلسلة الأوامر الخام. يمنع الالتفاف عبر عوامل shell الملحقة (على سبيل المثال، `sudo systemctl status x; rm -rf /` لا تطابق `sudo systemctl status *`).
</Note>

**الحدث:** PreToolUse (Bash)

**المعاملات:**

|-------|------|---------|-------------|
| `allowPaths` | `string[]` | `[]` | المسارات الآمنة للحذف المتكرر (على سبيل المثال، `/tmp`). |

**مثال:**

```json
{

    "block-rm-rf": {
      "allowPaths": ["/tmp", "/var/cache"]
    }
  }
}
```

## أوامر البنية التحتية

وقف وكلاء الترميز من تشغيل CLIs البنية التحتية أو تشغيل خطوط أنابيب CI/CD. جميع السياسات في هذه الفئة **opt-in** (`defaultEnabled: false`) — الوكلاء الذين يحتاجون بشكل شرعي إلى استدعاء `kubectl`, `terraform`, وغيرها لن يتم إزعاجهم ما لم تفعل السياسة. عند التفعيل، يتم نكر كل استدعاء للـ CLI المطابق ما لم تطابق الأوامر إدخالاً في `allowPatterns`.

قواعد نحو النمط هي نفسها [`block-sudo`](#block-sudo): يتم مطابقة tokens مقابل argv المحللة، `*` wildcard لـ token واحد، وأي أمر يحتوي على عامل shell مستقل (`&&`, `||`, `|`, `;`) أو token بها metacharacters shell مدمجة يتم رفضه قبل مطابقة قائمة السماح لمنع bypasses الحقن.

### `block-kubectl`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** ينكر أي استدعاء `kubectl`.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | بادئات أوامر kubectl المسموحة. |

**مثال:**

```json
{
  "policyParams": {
    "block-kubectl": {
      "allowPatterns": ["kubectl get *", "kubectl describe *", "kubectl logs *"]
    }
  }
}
```

مع هذا الإعداد، يُسمح بـ `kubectl get pods` لكن `kubectl apply -f deploy.yaml` يُنكر.

### `block-gh-pipeline`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** ينكر subcommands `gh` CLI التالية التي تغير الحالة أو تشغل خطوط الأنابيب:

- `gh workflow run`, `gh workflow enable`, `gh workflow disable`
- `gh run rerun`, `gh run cancel`
- `gh pr merge`
- `gh release create`, `gh release delete`
- `gh cache delete`
- `gh secret set`, `gh secret delete`

subcommands `gh` ذات الوصول القراءة فقط مثل `gh pr view`, `gh pr list`, `gh run list`, `gh release view`, و `gh api repos/.../...` **لم** يتم مطابقتها بواسطة هذه السياسة — فهي مطلوبة بشكل روتيني لفحوصات سير العمل (بما في ذلك `require-ci-green-before-stop` الخاص بـ failproofai).

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | استدعاءات نصية محددة للسماح بها حتى لو تم نكرها بخلاف ذلك. |

**مثال:**

```json
{
  "policyParams": {
    "block-gh-pipeline": {
      "allowPatterns": ["gh run rerun *"]
    }
  }
}
```

## الأسرار (منظفات)

منع الوكلاء من تسريب بيانات الاعتماد في سياقهم أو مخرجاتهم. تعمل سياسات المنظفات على أحداث **PostToolUse**. عندما يقوم Claude بتشغيل أمر Bash أو قراءة ملف أو استدعاء أي أداة، تفحص هذه السياسات المخرجات قبل إرجاعها إلى Claude. إذا تم اكتشاف نمط سري، تعيد السياسة قرار deny الذي يمنع المخرجات من الإرجاع.

### `sanitize-jwt`

**الحدث:** PostToolUse (جميع الأدوات)  
**الافتراضي:** يحذف JWT tokens (ثلاث segments base64url مفصولة بـ `.`).

لا توجد معاملات.

---

### `sanitize-api-keys`

**الحدث:** PostToolUse (جميع الأدوات)  
**الافتراضي:** يحذف صيغ API key الشائعة: Anthropic (`sk-ant-`), OpenAI (`sk-`), GitHub PATs (`ghp_`), AWS access keys (`AKIA`), Stripe keys (`sk_live_`, `sk_test_`), و Google API keys (`AIza`).

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `additionalPatterns` | `{ regex: string; label: string }[]` | `[]` | أنماط regex إضافية لمعاملتها كأسرار. |

**مثال:**

```json
{
  "policyParams": {
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo internal API key" },
        { "regex": "pat_[0-9a-f]{40}", "label": "Internal PAT" }
      ]
    }
  }
}
```

---

### `sanitize-connection-strings`

**الحدث:** PostToolUse (جميع الأدوات)  
**الافتراضي:** يحذف سلاسل اتصال قاعدة البيانات التي تحتوي على بيانات اعتماد مدمجة (على سبيل المثال، `postgresql://user:password@host/db`).

لا توجد معاملات.

---

### `sanitize-private-key-content`

**الحدث:** PostToolUse (جميع الأدوات)  
**الافتراضي:** يحذف PEM blocks (`-----BEGIN PRIVATE KEY-----`, `-----BEGIN RSA PRIVATE KEY-----`، إلخ).

لا توجد معاملات.

---

### `sanitize-bearer-tokens`

**الحدث:** PostToolUse (جميع الأدوات)  
**الافتراضي:** يحذف رؤوس `Authorization: Bearer <token>` حيث يكون الـ token 20 حرفاً أو أكثر.

لا توجد معاملات.

## البيئة

حماية إعدادات البيئة الحساسة من القراءة أو الفضح من قبل الوكلاء.

### `block-env-files`

**الحدث:** PreToolUse (Bash, Read)  
**الافتراضي:** ينكر قراءة ملفات `.env` عبر `cat .env`, استدعاءات أداة Read مع `.env` كمسار الملف، إلخ.

لا يحجب `.envrc` أو ملفات أخرى ذات صلة بالبيئة - فقط الملفات المسماة بالضبط `.env`.

لا توجد معاملات.

---

### `protect-env-vars`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** ينكر الأوامر التي تطبع متغيرات البيئة: `printenv`, `env`, `echo $VAR`.

لا توجد معاملات.

## الوصول إلى الملفات

جعل الوكلاء يعملون داخل حدود المشروع بعيداً عن الملفات الحساسة.

### `block-read-outside-cwd`

**الحدث:** PreToolUse (Read, Bash)  
**الافتراضي:** ينكر قراءة الملفات خارج جذر المشروع. الحد هو `CLAUDE_PROJECT_DIR` (يتم تعيينه مرة واحدة لكل جلسة بواسطة Claude Code)، مع رجوع إلى دليل العمل الحالي للجلسة عندما لا يتم تعيين هذا المتغير. استخدام جذر المشروع بدلاً من `cwd` الفعلي يعني بقاء الحد مستقراً حتى بعد أن يقوم Claude بـ `cd` إلى مجلد فرعي.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `allowPaths` | `string[]` | `[]` | بادئات المسار المطلق المسموحة حتى لو كانت خارج جذر المشروع. |

**مثال:**

```json
{
  "policyParams": {
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company/config"]
    }
  }
}
```

---

### `block-secrets-write`

**الحدث:** PreToolUse (Write, Edit)  
**الافتراضي:** ينكر الكتابة إلى الملفات المستخدمة بشكل شائع للمفاتيح الخاصة والشهادات: `id_rsa`, `id_ed25519`, `*.key`, `*.pem`, `*.p12`, `*.pfx`.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `additionalPatterns` | `string[]` | `[]` | أنماط أسماء ملفات إضافية (نمط glob) لحجبها. |

**مثال:**

```json
{
  "policyParams": {
    "block-secrets-write": {
      "additionalPatterns": [".token", ".secret"]
    }
  }
}
```

## Git

منع الدفع العرضي والدفع القسري وأخطاء الفروع التي يصعب التراجع عنها.

### `block-push-master`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** ينكر `git push origin main` و `git push origin master`.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `protectedBranches` | `string[]` | `["main", "master"]` | أسماء الفروع التي لا يمكن الدفع إليها مباشرة. |

**مثال:**

```json
{
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "master", "release", "prod"]
    }
  }
}
```

<Tip>
للسماح بالدفع إلى جميع الفروع (تعطيل هذه السياسة بشكل فعال بدون إزالتها من `enabledPolicies`)، قم بتعيين `protectedBranches: []`.
</Tip>

---

### `block-work-on-main`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** ينكر `git commit`, `git merge`, `git rebase`, و `git cherry-pick` بينما شجرة العمل على `main` أو `master`. لا تتأثر إنشاء الفروع والتبديل (`git checkout`, `git checkout -b`, `git switch`, `git switch -c`).

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `protectedBranches` | `string[]` | `["main", "master"]` | أسماء الفروع التي يتم نكر commit/merge/rebase/cherry-pick عليها. |

---

### `block-force-push`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** ينكر `git push --force` و `git push -f`.

لا توجد معاملات خاصة بالسياسة. استخدم [`hint`](/ar/configuration#hint-cross-cutting) متقاطع لاقتراح بدائل:

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "Create a new branch from your current HEAD (e.g. `git checkout -b <new-branch>`) and push that instead."
    }
  }
}
```

---

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يوجه Claude للمتابعة بحذر عند تشغيل `git commit --amend`. لا يحجب الأمر.

لا توجد معاملات.

---

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يوجه Claude للتأكيد قبل تشغيل `git stash drop`. لا يحجب الأمر.

لا توجد معاملات.

### `warn-all-files-staged`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يوجه Claude لمراجعة ما يحضره عند تشغيل `git add -A` أو `git add .`. لا يحجب الأمر.

لا توجد معاملات.

## تحذيرات

إعطاء الوكلاء سياق إضافي قبل عمليات محتملة الخطورة لكنها غير تدميرية.

### `warn-large-file-write`

**الحدث:** PreToolUse (Write)  
**الافتراضي:** يوجه Claude للتأكيد قبل كتابة ملفات أكبر من 1024 KB.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `thresholdKb` | `number` | `1024` | حد حجم الملف بالكيلوبايت الذي يتم فوقه إصدار تحذير. |

**مثال:**

```json
{
  "policyParams": {
    "warn-large-file-write": {
      "thresholdKb": 256
    }
  }
}
```

<Note>
يفرض معالج hook حد stdin بحجم 1 MB للبيانات. لاختبار هذه السياسة بمحتوى صغير، قم بتعيين `thresholdKb` إلى قيمة أقل بكثير من 1024.
</Note>

## مديرو الحزم

فرض مديري الحزم المسموح للوكيل باستخدامهم.

### `prefer-package-manager`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** معطل. عند التفعيل، يحجب أي أمر مدير حزم لا يوجد في قائمة `allowed` ويخبر Claude بإعادة كتابة الأمر باستخدام مدير مسموح.

يكتشف: pip, pip3, python -m pip, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo.

| المعامل | النوع | الافتراضي | الوصف |
|-----------|------|---------|-------------|
| `allowed` | string[] | `[]` | أسماء مديري الحزم المسموحة. أي مدير مكتشف ليس في هذه القائمة يتم حجبه. عندما تكون فارغة، السياسة لا تفعل شيء. |
| `blocked` | string[] | `[]` | أسماء مديري إضافية لحجبها بما يتجاوز القائمة المدمجة (على سبيل المثال، `['pdm', 'pipx']`). |

تغطي قائمة الحجب المدمجة: pip, pip3, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo. استخدم `blocked` لإضافة مديرين ليسوا في هذه القائمة.

**مثال على الإعداد:**

```json
{
  "enabledPolicies": ["prefer-package-manager"],
  "policyParams": {
    "prefer-package-manager": {
      "allowed": ["uv", "bun"],
      "blocked": ["pdm", "pipx"]
    }
  }
}
```

مع هذا الإعداد، يتم نكر كل من `pip install flask` و `pdm install flask` برسالة توضح لـ Claude استخدام `uv` أو `bun` بدلاً من ذلك. أوامر مثل `uv pip install flask` مسموحة لأن `uv` في قائمة السماح ويتم فحصها أولاً.

## سير العمل

فرض سير عمل منضبط لنهاية الجلسة. تعمل هذه السياسات على حدث **Stop** وتنكر الوكيل من التوقف حتى تتحقق كل شرط. يتبعون سلسلة اعتماد طبيعية: commit → push → PR → CI. إذا نكرت سياسة، يتم تخطي السياسات اللاحقة في السلسلة (deny تقصر الدائرة).

جميع سياسات سير العمل **fail-open**: إذا لم تتوفر الأداة المطلوبة (على سبيل المثال، `gh` غير مثبت، لا توجد remote git)، تسمح السياسة برسالة إعلامية توضح السبب في تخطي الفحص.

### دلالات Stop في كل CLI

يبدو إنفاذ Stop مختلفاً قليلاً عبر سبعة CLIs المدعومة لأن كل واحد يفضح عقد hook مختلف "انتهى الوكيل". **النتيجة** هي نفسها — الوكيل لا يتمكن من التوقف بينما بوابة سير العمل تفشل — لكن **الميكانيكيات** تختلف. يلخص الجدول أدناه؛ فقط Pi لديه خصوصية تستحق الفهم قبل تفعيل سياسة `require-*-before-stop`.

| CLI | متى تنطلق البوابة | ما تراه |
|---|---|---|
| Claude Code | نفس حلقة الوكيل، فوراً | يستمر Claude في العمل — يصحح المشكلة، ثم يحاول الإنهاء مرة أخرى. لا يوجد انقطاع مرئي لك. |
| Codex | نفس حلقة الوكيل، فوراً | نفس Claude. |
| GitHub Copilot CLI | نفس حلقة الوكيل، فوراً | نفس Claude (يستخدم قناة Copilot `{decision:"block", reason}` - تحقق تجريبياً ضد Copilot CLI 1.0.41). |
| Cursor Agent | نفس حلقة الوكيل، فوراً | نفس Claude (يستخدم قناة Cursor `{followup_message}` — محدودة بـ `loop_limit`, افتراضي 5 عمليات إعادة محاولة). |
| Gemini CLI | نفس حلقة الوكيل، فوراً | نفس Claude (يستخدم قناة Gemini `{decision:"block", reason}` في `AfterAgent`). |
| OpenCode | نفس حلقة الوكيل، فوراً | نفس Claude (يستخدم استدعاء SDK `client.session.prompt(...)` الخاص بـ OpenCode موجه عبر `hookSpecificOutput.additionalContext`). |
| **Pi (pi-coding-agent)** | **المنعطف التالي للمستخدم** | **يتوقف Pi بشكل مرئي** عندما تنطلق البوابة — تخرج حلقة وكيله وتعود إليك الموجه. البوابة تنطلق بعد ذلك في المرة التالية التي تقدم فيها موجه: failproofai يضيف توجيه `MANDATORY ACTION REQUIRED` إلى system prompt تلك المنعطفة، موجهاً LLM لإكمال خطوة سير العمل (commit, push، إلخ) قبل فعل ما طلبته. |

<Note>
**قيود Pi.** `AgentEndEvent` الخاص بـ Pi (المكافئ الأعلى لـ hook Claude `Stop`) لا يحتوي على نوع Result — عندما تنطلق، حلقة وكيل Pi قد خرجت بالفعل. لا يمكن فرض Pi على إعادة محاولة نفس الحلقة بالطريقة التي يمكن بها Claude / Copilot / Cursor / Gemini / OpenCode. يحول failproofai البوابة إلى حدث Pi `before_agent_start` (الذي ينطلق بعد الموجه التالي للمستخدم) بحيث لا يزال الفحص يفرض، فقط في المنعطفة التالية بدلاً من الحالية.

**ما يعنيه هذا عملياً:**

- بعد توقف Pi، يتم التقاط سبب الرفض في الذاكرة بناءً على معرّف جلسة Pi. الموجه التالي بالضبط الذي تقدمه في نفس عملية Pi يستنزفه: يرى LLM توجيه `MANDATORY ACTION REQUIRED` أعلى system prompt الخاص به، والتزامات (أو دفع / فتح PR / انتظر CI)، وبعد ذلك فقط متابعة طلبك. سبب الرفض المقبوض هو مرة واحدة — بمجرد استنزافه، البوابة واضحة.
- البوابة محدودة بعمر عملية Pi. إذا قمت بـ `Ctrl+C` Pi أو قطعت بين المنعطفات، يتم إسقاط الإدخال في الذاكرة جنباً إلى جنب مع العملية والبوابة مفقودة. Claude, Copilot, Cursor, Gemini, و OpenCode لديهم نفس الحد (اقتل الوكيل والبوابة مفقودة) — Pi فقط يجعله أكثر وضوحاً لأن الوكيل يخرج بشكل مرئي قبل انطلاق البوابة.
- يتم مسح deny معلقة أيضاً في `session_shutdown` لأي سبب (`new` / `resume` / `fork` / `quit`)، بحيث لا يمكن لبوابة قديمة من جلسة سابقة أن تتسرب إلى جلسة جديدة تبدأ في نفس عملية Pi.

إذا كنت تحتاج إلى إعادة محاولة نفس الحلقة على نمط Claude، قم بتشغيل سياسات `Stop` تحت أي من CLIs الستة الأخرى المدعومة. نحن نتتبع Pi upstream لنوع Result مستقبلي على `AgentEndEvent` الذي سيسمح لنا بإغلاق هذه الفجوة.
</Note>

### `require-push-before-stop`

**الحدث:** Stop  
**الافتراضي:** ينكر التوقف عندما يكون هناك التزامات غير مدفوعة أو عندما لا يحتوي الفرع الحالي على فرع تتبع بعيد. يقترح `git push -u` لإنشاء فرع تتبع إذا لزم الأمر. يفشل مفتوحاً إذا لم يتم تكوين أي بعيد.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|

**مثال:**

```json
{
  "policyParams": {
    "require-push-before-stop": {
      "remote": "upstream"
    }
  }
}
```

### `require-pr-before-stop`

**الحدث:** Stop  
**الافتراضي:** ينكر التوقف عندما لا توجد pull request للفرع الحالي، أو عندما تكون PR الموجودة مغلقة دون دمج. يوجه Claude لإنشاء PR مع `gh pr create`. عندما يتم **دمج** PR، تسمح السياسة (تم شحن العمل) والرسالة توصي بالتبديل من الفرع (`git checkout main && git pull`).

لا توجد معاملات.

<Note>
تتطلب هذه السياسة [GitHub CLI](https://cli.github.com/) (`gh`) مثبتاً ومصرحاً.
قم بتشغيل `gh auth login` مع رمز الوصول الشخصي الذي يحتوي على نطاق `repo` للوصول القراءة
إلى pull requests. إذا لم تكن `gh` مثبتاً أو لم تكن مصرحة، تفشل السياسة مفتوحة وتقرير السبب إلى Claude.
</Note>

### `require-no-conflicts-before-stop`

**الحدث:** Stop  
**الافتراضي:** ينكر التوقف عندما لا يمكن للفرع الحالي الدمج بنظافة في الفرع الأساسي. تتحقق السياسة أولاً من وجود PR `OPEN` على GitHub للفرع — بدونها، لا يوجد هدف دمج لفرضه، لذلك تقصير السياسة بأكملها للسماح. بمجرد تأكيد PR `OPEN`، تعمل عمليتا فحص مستقلة:

1. **محلي** — `git merge-tree --write-tree --name-only origin/<baseBranch> HEAD`. عند حدوث تضارب، ترسل رسالة الرفض أسماء الملفات المتضاربة حتى يعرف Claude بالضبط ما يجب حله.
2. **GitHub** — يعيد استخدام نتيجة `gh pr view --json mergeable,state` التي تم جلبها بالفعل في الفحص المسبق. يكتشف التضاربات التي قد يفتقدها `origin/<baseBranch>` القديم محلياً (على سبيل المثال، وضعت شخص ما PR معارضة على `main` منذ آخر جلب). نتيجة `CONFLICTING` ترفض. نتيجة `UNKNOWN` ترفض أيضاً وتوجه Claude للانتظار ~10 ثواني وإعادة فحص قبل محاولة التوقف مرة أخرى — يمنع هذا الإيجابيات الكاذبة بينما تعيد GitHub الحساب.

تتخطى بالكامل (تسمح) عندما: `gh` غير مثبت، لا توجد PR للفرع، حالة PR ليست `OPEN` (على سبيل المثال، `MERGED`, `CLOSED`)، أو `gh pr view` تعيد مخرجات غير قابلة للتحليل. تفشل أيضاً مفتوحة عندما `origin/<baseBranch>` مفقودة محلياً أو عندما لا توجد التزامات تتقدم base — تلك النقاط البيضاء في الطبقة 1 تستشير PR المخزن مؤقتاً قابلية الدمج قبل السماح.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `baseBranch` | `string` | `"main"` | الفرع الأساسي للتحقق من التضاربات ضده. |

<Note>
GitHub CLI (`gh`) مطلوبة لهذه السياسة. تستخدم السياسة `gh pr view` لتأكيد
وجود PR `OPEN` قبل تشغيل أي فحص تضارب — بدون `gh`، تقصير السياسة
للسماح. قم بتشغيل `gh auth login` مع رمز وصول شخصي يحتوي على
نطاق `repo` للوصول القراءة إلى pull requests.
</Note>

## تعطيل السياسات الفردية

أزل سياسة معينة من `enabledPolicies` في ملف الإعدادات الخاص بك، أو قم بإيقافها في تبويب السياسات بلوحة التحكم.

```json
{
  "enabledPolicies": [
    "block-rm-rf",
    "sanitize-api-keys"
  ]
}
```

السياسات غير المدرجة في `enabledPolicies` لا تعمل، حتى إذا كانت إدخالات `policyParams` موجودة لها.
