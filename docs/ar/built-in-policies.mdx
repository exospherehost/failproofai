يأتي failproofai مع 39 سياسة مدمجة تعترض أنماط الفشل الشائعة للوكلاء. تنطلق كل سياسة على نوع حدث خطاف معين واسم أداة محددة. تقبل تسع عشرة سياسة معاملات تتيح لك ضبط سلوكها دون كتابة أكواد. وتفرض خمس سياسات سير عمل خط أنابيب commit → push → PR → CI قبل أن يتوقف Claude.

---

## نظرة عامة

يتم تجميع السياسات في فئات:

| الفئة | السياسات | نوع الخطاف |
|----------|----------|-----------|
| [الأوامر الخطيرة](#dangerous-commands) | block-sudo, block-rm-rf, block-curl-pipe-sh, block-failproofai-commands | PreToolUse |
| [أوامر البنية الأساسية](#infra-commands) | block-kubectl, block-terraform, block-aws-cli, block-gcloud, block-az-cli, block-helm, block-gh-pipeline | PreToolUse |
| [الأسرار (المعقمات)](#secrets-sanitizers) | sanitize-jwt, sanitize-api-keys, sanitize-connection-strings, sanitize-private-key-content, sanitize-bearer-tokens | PostToolUse |
| [البيئة](#environment) | block-env-files, protect-env-vars | PreToolUse |
| [الوصول إلى الملفات](#file-access) | block-read-outside-cwd, block-secrets-write | PreToolUse |
| [Git](#git) | block-push-master, block-work-on-main, block-force-push, warn-git-amend, warn-git-stash-drop, warn-all-files-staged | PreToolUse |
| [قاعدة البيانات](#database) | warn-destructive-sql, warn-schema-alteration | PreToolUse |
| [التحذيرات](#warnings) | warn-large-file-write, warn-package-publish, warn-background-process, warn-global-package-install | PreToolUse |
| [مديري الحزم](#package-managers) | prefer-package-manager | PreToolUse |
| [سير العمل](#workflow) | require-commit-before-stop, require-push-before-stop, require-pr-before-stop, require-no-conflicts-before-stop, require-ci-green-before-stop | Stop |

- **`block-`** — إيقاف الوكيل عن المتابعة.
- **`warn-`** — منح الوكيل سياقاً إضافياً حتى يتمكن من التصحيح الذاتي.
- **`sanitize-`** — مسح البيانات الحساسة من مخرجات الأداة قبل أن يراها الوكيل.

### الفراغات الاسمية

تعيش كل سياسة في فتحة `<namespace>/<name>`. تنتمي السياسات المدمجة إلى
الفراغ الاسمي **`exospherehost/`** — على سبيل المثال، `exospherehost/sanitize-jwt`. يمنع الفراغ الاسمي
التصادمات عندما تحمل أيضاً سياسات مخصصة أو من طرف ثالث
بأسماء قصيرة مشابهة.

في الإعدادات الخاصة بك، يمكنك الإشارة إلى سياسة مدمجة باسمها القصير أو اسمها
الكامل؛ يتم حل كلا النموذجين إلى نفس السياسة:

```json
{
  "enabledPolicies": [
    "sanitize-jwt",
    "exospherehost/block-rm-rf"
  ]
}
```

إذا لم يكن لاسم أي `/`، يتعامل failproofai معه كأنه ينتمي إلى الفراغ الاسمي الافتراضي
`exospherehost`. الأسماء التي تحتوي بالفعل على `/` (مثل `myorg/foo`,
`custom/my-hook`) تُحتفظ بها كما هي.
- **`require-`** — إيقاف حدث Stop حتى تتحقق الشروط.

---

<Tip>
تدعم كل سياسة حقل اختياري `hint` في `policyParams`. يُضاف التلميح إلى رسالة deny أو instruct التي يراها Claude، مما يوفر إرشادات عملية دون تعديل كود السياسة. يعمل مع السياسات المدمجة والمخصصة وسياسات الاتفاقية. انظر [التكوين → hint](/ar/configuration#hint-cross-cutting) للتفاصيل.
</Tip>

## الأوامر الخطيرة

منع الوكلاء من تنفيذ عمليات يصعب التراجع عنها أو قد تضر نظام المضيف.

### `block-sudo`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** ينكر أي أمر `sudo`.

يحظر الاستدعاءات التي تتضمن كلمة مفتاحية `sudo`. يتم مطابقة النمط على رموز الأوامر المحللة، وليس السلسلة الأولية، لمنع تجاوز طريق عبر حقن عامل الصدفة.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | بادئات أوامر دقيقة مسموح بها. يتم مطابقة كل إدخال مقابل رموز argv المحللة. |

**مثال:**

```json
{
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    }
  }
}
```

مع هذا الإعداد، `sudo systemctl status nginx` مسموح به، لكن `sudo rm /etc/hosts` ينكر.

<Note>
يتم مطابقة الأنماط مقابل الرموز المحللة، وليس السلسلة الأولية للأمر. يمنع هذا التجاوز عبر عوامل الصدفة المرفقة (مثل `sudo systemctl status x; rm -rf /` لا يطابق `sudo systemctl status *`).
</Note>

## أوامر البنية الأساسية

منع وكلاء الترميز من تشغيل CLIs البنية الأساسية أو تشغيل خطوط الأنابيب CI/CD. جميع السياسات في هذه الفئة هي **اختياري-في** (`defaultEnabled: false`) — الوكلاء الذين يحتاجون بشكل حقيقي إلى استدعاء `kubectl`, `terraform`، إلخ. لن يتعطلوا ما لم تقم بتفعيل السياسة. عند التفعيل، يتم إنكار كل استدعاء للـ CLI المطابق ما لم تطابق الأمر إدخالاً في `allowPatterns`.

قواعد النمط هي نفس [`block-sudo`](#block-sudo): يتم مطابقة الرموز مقابل argv المحلل، `*` عام لرمز واحد، وأي أمر يحتوي على عامل صدفة مستقل (`&&`, `||`, `|`, `;`) أو رمز يحتوي على أحرف صدفة مدمجة يتم رفضه قبل مطابقة قائمة السماح لمنع تجاوزات الحقن.

### `block-kubectl`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** ينكر أي استدعاء `kubectl`.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | بادئات أوامر kubectl مسموح بها. |

**مثال:**

```json
{
  "policyParams": {
    "block-kubectl": {
      "allowPatterns": ["kubectl get *", "kubectl describe *", "kubectl logs *"]
    }
  }
}
```

مع هذا الإعداد، `kubectl get pods` مسموح به لكن `kubectl apply -f deploy.yaml` ينكر.

### `block-gh-pipeline`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** ينكر الأوامر الفرعية `gh` CLI التالية التي تغير الحالة أو تشغل خطوط الأنابيب:

- `gh workflow run`, `gh workflow enable`, `gh workflow disable`
- `gh run rerun`, `gh run cancel`
- `gh pr merge`
- `gh release create`, `gh release delete`
- `gh cache delete`
- `gh secret set`, `gh secret delete`

أوامر `gh` الفرعية للقراءة فقط مثل `gh pr view`, `gh pr list`, `gh run list`, `gh release view`، و `gh api repos/.../...` **لا** تطابقها هذه السياسة — فهي مطلوبة بشكل روتيني لفحوصات سير العمل (بما في ذلك `require-ci-green-before-stop` الخاص بـ failproofai).

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | استدعاءات سيناريو محددة للسماح بها حتى لو كانت ستُنكر بخلاف ذلك. |

**مثال:**

```json
{
  "policyParams": {
    "block-gh-pipeline": {
      "allowPatterns": ["gh run rerun *"]
    }
  }
}
```

## الأسرار (المعقمات)

منع الوكلاء من تسرب بيانات الاعتماد إلى سياقهم أو مخرجاتهم. تنطلق سياسات المعقمات على أحداث **PostToolUse**. عندما يقوم Claude بتشغيل أمر Bash أو قراءة ملف أو استدعاء أي أداة، تفحص هذه السياسات المخرجات قبل إرجاعها إلى Claude. إذا تم اكتشاف نمط سري، تعيد السياسة قرار deny يمنع المخرجات من إعادتها.

### `sanitize-jwt`

**الحدث:** PostToolUse (جميع الأدوات)  
**الافتراضي:** يصحح رموز JWT (ثلاث أجزاء base64url مفصولة بـ `.`).

لا توجد معاملات.

---

### `sanitize-api-keys`

**الحدث:** PostToolUse (جميع الأدوات)  
**الافتراضي:** يصحح تنسيقات مفاتيح API الشائعة: Anthropic (`sk-ant-`), OpenAI (`sk-`), GitHub PATs (`ghp_`), AWS access keys (`AKIA`), Stripe keys (`sk_live_`, `sk_test_`), و Google API keys (`AIza`).

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `additionalPatterns` | `{ regex: string; label: string }[]` | `[]` | أنماط regex إضافية للتعامل معها كأسرار. |

**مثال:**

```json
{
  "policyParams": {
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo internal API key" },
        { "regex": "pat_[0-9a-f]{40}", "label": "Internal PAT" }
      ]
    }
  }
}
```

---

### `sanitize-connection-strings`

**الحدث:** PostToolUse (جميع الأدوات)  
**الافتراضي:** يصحح سلاسل اتصال قاعدة البيانات التي تحتوي على بيانات اعتماد مدمجة (مثل `postgresql://user:password@host/db`).

لا توجد معاملات.

---

### `sanitize-private-key-content`

**الحدث:** PostToolUse (جميع الأدوات)  
**الافتراضي:** يصحح كتل PEM (`-----BEGIN PRIVATE KEY-----`, `-----BEGIN RSA PRIVATE KEY-----`، إلخ.).

لا توجد معاملات.

---

### `sanitize-bearer-tokens`

**الحدث:** PostToolUse (جميع الأدوات)  
**الافتراضي:** يصحح رؤوس `Authorization: Bearer <token>` حيث يكون الرمز 20 حرفاً أو أكثر.

لا توجد معاملات.

## البيئة

حماية إعدادات البيئة الحساسة من القراءة أو الكشف عنها من قبل الوكلاء.

### `block-env-files`

**الحدث:** PreToolUse (Bash, Read)  
**الافتراضي:** ينكر قراءة ملفات `.env` عبر `cat .env`, استدعاءات أداة Read بـ `.env` كمسار الملف، إلخ.

لا يحظر `.envrc` أو ملفات بيئية أخرى — فقط الملفات المسماة بدقة `.env`.

لا توجد معاملات.

---

### `protect-env-vars`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** ينكر الأوامر التي تطبع متغيرات البيئة: `printenv`, `env`, `echo $VAR`.

لا توجد معاملات.

---

## الوصول إلى الملفات

جعل الوكلاء يعملون داخل حدود المشروع وبعيداً عن الملفات الحساسة.

### `block-read-outside-cwd`

**الحدث:** PreToolUse (Read, Bash)  
**الافتراضي:** ينكر قراءة الملفات خارج جذر المشروع. الحدود هي `CLAUDE_PROJECT_DIR` (مضبوط مرة واحدة لكل جلسة بواسطة Claude Code)، مع الرجوع إلى دليل العمل الحالي للجلسة عندما يكون ذلك المتغير غير محدد. استخدام جذر المشروع بدلاً من `cwd` الحي يعني أن الحد البيني يبقى مستقراً حتى بعد Claude `cd`s إلى دليل فرعي.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `allowPaths` | `string[]` | `[]` | بادئات المسارات المطلقة المسموح بها حتى لو كانت خارج جذر المشروع. |

**مثال:**

```json
{
  "policyParams": {
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company/config"]
    }
  }
}
```

---

### `block-secrets-write`

**الحدث:** PreToolUse (Write, Edit)  
**الافتراضي:** ينكر الكتابة إلى ملفات شائعة الاستخدام للمفاتيح الخاصة والشهادات: `id_rsa`, `id_ed25519`, `*.key`, `*.pem`, `*.p12`, `*.pfx`.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `additionalPatterns` | `string[]` | `[]` | أنماط اسم ملف إضافية (أسلوب glob) للحظر. |

**مثال:**

```json
{
  "policyParams": {
    "block-secrets-write": {
      "additionalPatterns": [".token", ".secret"]
    }
  }
}
```

## Git

منع الدفع العرضي والدفع القسري وأخطاء الفروع التي يصعب التراجع عنها.

### `block-push-master`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** ينكر `git push origin main` و `git push origin master`.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `protectedBranches` | `string[]` | `["main", "master"]` | أسماء الفروع التي لا يمكن دفعها مباشرة. |

**مثال:**

```json
{
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "master", "release", "prod"]
    }
  }
}
```

<Tip>
للسماح بالدفع إلى جميع الفروع (تعطيل هذه السياسة بشكل فعلي دون إزالتها من `enabledPolicies`)، اضبط `protectedBranches: []`.
</Tip>

---

### `block-work-on-main`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** ينكر `git commit`, `git merge`, `git rebase`، و `git cherry-pick` بينما تكون شجرة العمل على `main` أو `master`. إنشاء الفروع والتبديل (`git checkout`, `git checkout -b`, `git switch`, `git switch -c`) غير متأثرة.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `protectedBranches` | `string[]` | `["main", "master"]` | أسماء الفروع التي يتم فيها إنكار commit/merge/rebase/cherry-pick. |

---

### `block-force-push`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** ينكر `git push --force` و `git push -f`.

لا توجد معاملات محددة للسياسة. استخدم التلميح الشامل [`hint`](/ar/configuration#hint-cross-cutting) للاقتراح بدائل:

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "Create a new branch from your current HEAD (e.g. `git checkout -b <new-branch>`) and push that instead."
    }
  }
}
```

---

### `warn-git-amend`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يرشد Claude للمتابعة بحذر عند تشغيل `git commit --amend`. لا يحظر الأمر.

لا توجد معاملات.

---

### `warn-git-stash-drop`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يرشد Claude للتأكيد قبل تشغيل `git stash drop`. لا يحظر الأمر.

لا توجد معاملات.

---

### `warn-all-files-staged`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يرشد Claude لمراجعة ما يقوم بتجميعه عند تشغيل `git add -A` أو `git add .`. لا يحظر الأمر.

لا توجد معاملات.

## قاعدة البيانات

اعترض عمليات SQL المدمرة قبل تنفيذها على قاعدة البيانات الخاصة بك.

### `warn-destructive-sql`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يرشد Claude للتأكيد قبل تشغيل SQL يحتوي على `DROP TABLE`, `DROP DATABASE`، أو `DELETE` بدون بند `WHERE`.

لا توجد معاملات.

---

## مديري الحزم

فرض مديري الحزم المسموح للوكيل باستخدامهم.

### `prefer-package-manager`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** معطّل. عند التفعيل، يحظر أي أمر مدير حزم ليس في قائمة `allowed` ويخبر Claude بإعادة كتابة الأمر باستخدام مدير مسموح.

يكتشف: pip, pip3, python -m pip, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo.

| المعامل | النوع | الافتراضي | الوصف |
|-----------|------|---------|-------------|
| `allowed` | string[] | `[]` | أسماء مديري الحزم المسموح بها. أي مدير مكتشف ليس في هذه القائمة يتم حظره. عند ترك القائمة فارغة، السياسة لا تعمل. |
| `blocked` | string[] | `[]` | أسماء مديري إضافية للحظر بعيداً عن القائمة المدمجة (مثل `['pdm', 'pipx']`). |

تغطي قائمة الحظر المدمجة: pip, pip3, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo. استخدم `blocked` لإضافة مديري ليسوا في هذه القائمة.

**مثال الإعداد:**

```json
{
  "enabledPolicies": ["prefer-package-manager"],
  "policyParams": {
    "prefer-package-manager": {
      "allowed": ["uv", "bun"],
      "blocked": ["pdm", "pipx"]
    }
  }
}
```

مع هذا الإعداد، `pip install flask` و `pdm install flask` كلاهما منكور مع رسالة تخبر Claude بـ استخدام `uv` أو `bun` بدلاً من ذلك. أوامر مثل `uv pip install flask` مسموحة لأن `uv` في قائمة السماح ويتم فحصه أولاً.

## سير العمل

فرض سير عمل منضبط نهاية الجلسة. تنطلق هذه السياسات على حدث Stop وتنكر الوكيل من التوقف حتى يتم استيفاء كل شرط. تتبع سلسلة تبعية طبيعية: commit → push → PR → CI. إذا أنكرت السياسة، يتم تخطي السياسات اللاحقة في السلسلة (الإنكار يقطع العملية).

جميع سياسات سير العمل هي **fail-open**: إذا لم تكن الأداة المطلوبة متاحة (مثل `gh` غير مثبتة، لا توجد بعيد git)، تسمح السياسة برسالة معلوماتية توضح السبب في تخطي الفحص.

### دلالات Stop لكل CLI

يبدو تطبيق Stop مختلفاً قليلاً عبر سبعة CLIs مدعومة لأن كل منها يكشف عن عقد خطاف مختلف. النتيجة **متطابقة** — الوكيل لا يتمكن من التوقف بينما بوابة سير عمل تفشل — لكن الآليات **تختلف**. يلخص الجدول أدناه؛ فقط Pi لديها تفاصيل غريبة للمستخدم تستحق الفهم قبل تفعيل سياسة `require-*-before-stop`.

| CLI | عند انطلاق البوابة | ما تراه |
|---|---|---|
| Claude Code | نفس حلقة الوكيل، فوراً | يستمر Claude في العمل — يصحح المشكلة، ثم يحاول الانتهاء مرة أخرى. لا انقطاع مرئي لك. |
| Codex | نفس حلقة الوكيل، فوراً | نفس Claude. |
| GitHub Copilot CLI | نفس حلقة الوكيل، فوراً | نفس Claude (يستخدم قناة إعادة محاولة `{decision:"block", reason}` من Copilot — تحقق تجريبياً مقابل Copilot CLI 1.0.41). |
| Cursor Agent | نفس حلقة الوكيل، فوراً | نفس Claude (يستخدم قناة `{followup_message}` من Cursor — مسقوفة بـ `loop_limit`، الافتراضي 5 إعادات محاولة). |
| Gemini CLI | نفس حلقة الوكيل، فوراً | نفس Claude (يستخدم قناة `{decision:"block", reason}` من Gemini على `AfterAgent`). |
| OpenCode | نفس حلقة الوكيل، فوراً | نفس Claude (يستخدم استدعاء SDK `client.session.prompt(...)` من OpenCode الموجهة عبر `hookSpecificOutput.additionalContext`). |
| **Pi (pi-coding-agent)** | **بعد الإجابة التالية للمستخدم** | **يتوقف Pi بشكل مرئي** عند انطلاق البوابة — تخرج حلقة الوكيل وتُرجع إلى الطلب. ثم تنطلق البوابة في المرة التالية التي تقدم فيها استفسار: يقدم failproofai توجيهاً `MANDATORY ACTION REQUIRED` إلى موجه النظام لتلك الدورة، يرشد نموذج اللغة الكبير لإكمال خطوة سير العمل (commit، push، إلخ.) قبل فعل ما طلبته. |

<Note>
**قيد Pi.** `AgentEndEvent` من Pi (المكافئ الأعلى لخطاف Stop من Claude) ليس له نوع Result — بحلول وقت انطلاقه، تكون حلقة وكيل Pi قد خرجت بالفعل. لا يمكن فرض إعادة محاولة Pi بنفس الطريقة التي يمكن بها Claude / Copilot / Cursor / Gemini / OpenCode. يحول failproofai البوابة إلى حدث `before_agent_start` من Pi (الذي ينطلق بعد الاستفسار التالي للمستخدم) حتى يستمر فرض فحص سير العمل، فقط في الدورة التالية بدلاً من الحالية.

**ما يعنيه هذا عملياً:**

- بعد توقف Pi، يتم التقاط سبب الإنكار في الذاكرة مفتوح بـ معرف جلسة Pi. الاستفسار التالي مباشرة الذي تقدمه في نفس عملية Pi يستنزفه: يرى نموذج اللغة الكبير توجيه `MANDATORY ACTION REQUIRED` في أعلى موجه النظام الخاص به، يقوم بـ commit (أو الدفع / فتح الـ PR / انتظار CI)، وفقط ثم يستمر مع طلبك. سبب الإنكار المقبوض هو one-shot — بمجرد استنزافه، البوابة واضحة.
- البوابة محدودة بمدى حياة عملية Pi. إذا قمت بـ `Ctrl+C` Pi أو خرجت بين الدورات، يتم حذف الإدخال في الذاكرة مع العملية وتُفقد البوابة. Claude، Copilot، Cursor، Gemini، و OpenCode لها نفس الحد (اقتل الوكيل وتُفقد البوابة) — Pi فقط يجعله أكثر وضوحاً لأن الوكيل يخرج بشكل مرئي قبل انطلاق البوابة.
- يتم حذف إنكار معلق أيضاً على `session_shutdown` لأي سبب (`new` / `resume` / `fork` / `quit`)، حتى لا يمكن لبوابة قديمة من جلسة سابقة أن تتسرب إلى جلسة جديدة بدأت في نفس عملية Pi.

إذا احتجت إلى إعادة محاولة same-loop مثل Claude، قم بتشغيل سياسات `Stop` الخاصة بك تحت أي من ستة CLIs المدعومة الأخرى. نتتبع Pi upstream لنوع Result المستقبلي على `AgentEndEvent` الذي سيسمح لنا بـ سد هذا الفجوة.
</Note>

### `require-no-conflicts-before-stop`

**الحدث:** Stop  
**الافتراضي:** ينكر التوقف عندما لا يمكن للفرع الحالي الدمج بنظافة في الفرع الأساسي. تتحقق السياسة أولاً من وجود PR `OPEN` على GitHub للفرع — بدونه، لا توجد هدف دمج لفرضه، حتى السياسة بأكملها تقطع نفسها للسماح. بمجرد تأكيد PR `OPEN`، يعمل مسحان مستقلان:

1. **محلي** — `git merge-tree --write-tree --name-only origin/<baseBranch> HEAD`. عند التضارب، رسالة الإنكار تسمي الملفات المتضاربة حتى يعرف Claude بالضبط ما يجب حله.
2. **GitHub** — يعيد استخدام نتيجة `gh pr view --json mergeable,state` المجلوبة بالفعل في الفحص المسبق. يعترض التضاربات التي كان `origin/<baseBranch>` المحلي القديم قد يفوتها (مثل شخص ما هبط PR متضارب على `main` منذ آخر جلب). نتيجة `CONFLICTING` تنكر. نتيجة `UNKNOWN` أيضاً تنكر وترشد Claude للانتظار ~10 ثوانٍ وإعادة الفحص قبل محاولة التوقف مرة أخرى — هذا يمنع الإيجابيات الخاطئة بينما تعيد حسابات GitHub.

يتخطى بالكامل (يسمح) عندما: `gh` غير مثبتة، لا يوجد PR للفرع، حالة الـ PR ليست `OPEN` (مثل `MERGED`, `CLOSED`)، أو `gh pr view` تعيد مخرجات غير قابلة للتحليل. كما تفشل بشكل مفتوح عندما `origin/<baseBranch>` مفقودة محلياً أو عندما لا توجد commits قبل الأساس — تلك الاختراقات Layer 1 لا تزال تستشير mergeability المخزنة مؤقتاً للـ PR قبل السماح.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `baseBranch` | `string` | `"main"` | الفرع الأساسي للتحقق من التضاربات ضده. |

<Note>
يتطلب GitHub CLI (`gh`). تستخدم السياسة `gh pr view` لتأكيد وجود PR `OPEN` قبل تشغيل أي مسح تضارب — بدون `gh`، السياسة تقطع نفسها للسماح. قم بتشغيل `gh auth login` بـ رمز وصول شخصي بنطاق `repo` لقراءة الوصول
إلى طلبات السحب.
</Note>

## تعطيل سياسات فردية

أزل سياسة محددة من `enabledPolicies` في الإعدادات الخاصة بك، أو اقلبها في تبويب Policies في الداشبورد.

```json
{
  "enabledPolicies": [
    "block-rm-rf",
    "sanitize-api-keys"
  ]
}
```

السياسات غير المدرجة في `enabledPolicies` لا تعمل، حتى إذا كانت إدخالات `policyParams` موجودة لها.
