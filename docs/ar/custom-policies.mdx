تتيح لك السياسات المخصصة كتابة قواعد لأي سلوك من سلوكيات الوكيل: فرض أعراف المشروع، ومنع الانحراف، وتقييد العمليات الخطرة، واكتشاف الوكلاء العالقين، أو التكامل مع Slack وسير عمل الموافقات وغيرها. تستخدم نفس نظام أحداث الخطافات وقرارات `allow` و`deny` و`instruct` المتوفرة في السياسات المدمجة.

---

## مثال سريع

```js
// my-policies.js
import { customPolicies, allow, deny, instruct } from "failproofai";

customPolicies.add({
  name: "no-production-writes",
  description: "Block writes to paths containing 'production'",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Write" && ctx.toolName !== "Edit") return allow();
    const path = ctx.toolInput?.file_path ?? "";
    if (path.includes("production")) {
      return deny("Writes to production paths are blocked");
    }
    return allow();
  },
});
```

ثبّته بالأمر:

```bash
failproofai policies --install --custom ./my-policies.js
```

## التثبيت والتحديث

```bash
# تثبيت مع ملف سياسات مخصص
failproofai policies --install --custom ./my-policies.js

# استبدال مسار ملف السياسات
failproofai policies --install --custom ./new-policies.js

# إزالة مسار السياسات المخصصة من الإعدادات
failproofai policies --uninstall --custom
```

يُحفظ المسار المطلق المُحلَّل في `policies-config.json` تحت المفتاح `customPoliciesPath`. يُعاد تحميل الملف في كل حدث خطاف — لا يوجد تخزين مؤقت بين الأحداث.

## واجهة برمجة التطبيقات (API)

### الاستيراد

```js
import { customPolicies, allow, deny, instruct } from "failproofai";
```

### `customPolicies.add(hook)`

يُسجّل سياسة. استدعِه بالعدد الذي تحتاجه لإضافة سياسات متعددة في نفس الملف.

```ts
customPolicies.add({
  name: string;                         // مطلوب - معرّف فريد
  description?: string;                 // يظهر في مخرجات `failproofai policies`
  match?: { events?: HookEventType[] }; // تصفية حسب نوع الحدث؛ احذفه للمطابقة مع جميع الأحداث
  fn: (ctx: PolicyContext) => PolicyResult | Promise<PolicyResult>;
});
```

### دوال اتخاذ القرار

| الدالة | التأثير | متى تُستخدم |
|--------|---------|-------------|
| `allow()` | السماح بالعملية بصمت | الإجراء آمن ولا حاجة لرسالة |
| `deny(message)` | حظر العملية | يجب على الوكيل عدم تنفيذ هذا الإجراء |
| `instruct(message)` | إضافة سياق دون حظر | إعطاء الوكيل سياقاً إضافياً للبقاء في المسار الصحيح |

`deny(message)` - تظهر الرسالة لـ Claude مسبوقةً بـ `"Blocked by failproofai:"`. يوقف أول `deny` تقييم جميع السياسات اللاحقة فوراً.

`instruct(message)` - تُضاف الرسالة إلى سياق Claude لاستدعاء الأداة الحالي. أول `instruct` هو الذي يسري — تُتجاهل نتائج `instruct` اللاحقة من السياسات الأخرى.

### رسائل السماح الإعلامية (تجريبي)

<Note>
`allow(message)` ميزة تجريبية متاحة منذ الإصدار v0.0.2-beta.3. قد تتغير الواجهة في الإصدارات المستقبلية. الإصدارات الأقدم تدعم `allow()` فقط بدون وسائط.
</Note>

`allow(message)` يسمح بالعملية **ويرسل** رسالة إعلامية إلى Claude. تُسلَّم الرسالة كـ `additionalContext` في استجابة stdout لمعالج الخطاف — نفس الآلية التي يستخدمها `instruct`، لكنها مختلفة دلالياً: إنها تحديث حالة، وليست تحذيراً.

| الدالة | التأثير | متى تُستخدم |
|--------|---------|-------------|
| `allow(message)` | السماح وإرسال سياق إلى Claude | تأكيد نجاح فحص ما، أو شرح سبب تخطي فحص |

حالات الاستخدام:
- **تأكيدات الحالة:** `allow("All CI checks passed.")` — يُعلم Claude بأن كل شيء على ما يرام
- **شرح الفشل المفتوح:** `allow("GitHub CLI not installed, skipping CI check.")` — يُعلم Claude بسبب تخطي فحص ما لكي يكون على دراية تامة بالسياق
- **تراكم رسائل متعددة:** إذا أرجعت عدة سياسات `allow(message)` كل منها، تُجمَع جميع الرسائل بأسطر جديدة وتُسلَّم معاً

```js
customPolicies.add({
  name: "confirm-branch-status",
  match: { events: ["Stop"] },
  fn: async (ctx) => {
    const cwd = ctx.session?.cwd;
    if (!cwd) return allow("No working directory, skipping branch check.");

    // ... check branch status ...
    if (allPushed) {
      return allow("Branch is up to date with remote.");
    }
    return deny("Unpushed changes detected.");
  },
});
```

### حقول `PolicyContext`

| الحقل | النوع | الوصف |
|-------|-------|--------|
| `eventType` | `string` | `"PreToolUse"` أو `"PostToolUse"` أو `"Notification"` أو `"Stop"` |
| `toolName` | `string \| undefined` | الأداة المُستدعاة (مثل `"Bash"` أو `"Write"` أو `"Read"`) |
| `toolInput` | `Record<string, unknown> \| undefined` | معاملات إدخال الأداة |
| `payload` | `Record<string, unknown>` | حمولة الحدث الخام الكاملة من Claude Code |
| `session` | `SessionMetadata \| undefined` | سياق الجلسة (انظر أدناه) |

### حقول `SessionMetadata`

| الحقل | النوع | الوصف |
|-------|-------|--------|
| `sessionId` | `string` | معرّف جلسة Claude Code |
| `cwd` | `string` | دليل العمل لجلسة Claude Code |
| `transcriptPath` | `string` | المسار إلى ملف نص الجلسة بصيغة JSONL |

### أنواع الأحداث

| الحدث | متى يُطلَق | محتويات `toolInput` |
|-------|------------|---------------------|
| `PreToolUse` | قبل تشغيل Claude لأداة ما | إدخال الأداة (مثل `{ command: "..." }` لـ Bash) |
| `PostToolUse` | بعد اكتمال أداة ما | إدخال الأداة + `tool_result` (الناتج) |
| `Notification` | عند إرسال Claude إشعاراً | `{ message: "...", notification_type: "idle" \| "permission_prompt" \| ... }` - يجب أن تُرجع الخطافات دائماً `allow()`، ولا يمكنها حظر الإشعارات |
| `Stop` | عند انتهاء جلسة Claude | فارغ |

## ترتيب التقييم

تُقيَّم السياسات بهذا الترتيب:

1. السياسات المدمجة (بترتيب التعريف)
2. السياسات المخصصة (بترتيب استدعاء `.add()`)

<Note>
أول `deny` يوقف تقييم جميع السياسات اللاحقة فوراً. أول `instruct` هو الذي يسري — تُتجاهل نتائج `instruct` اللاحقة.
</Note>

## الاستيرادات المتعدية

يمكن لملفات السياسات المخصصة استيراد وحدات محلية باستخدام مسارات نسبية:

```js
// my-policies.js
import { isBlockedPath } from "./utils.js";
import { checkApproval } from "./approval-client.js";

customPolicies.add({
  name: "approval-gate",
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const approved = await checkApproval(ctx.toolInput?.command, ctx.session?.sessionId);
    return approved ? allow() : deny("Approval required for this command");
  },
});
```

تُحلَّل جميع الاستيرادات النسبية القابلة للوصول من ملف الدخول. يتم تنفيذ ذلك عبر إعادة كتابة استيرادات `from "failproofai"` إلى مسار dist الفعلي وإنشاء ملفات `.mjs` مؤقتة لضمان التوافق مع ESM.

## تصفية أنواع الأحداث

استخدم `match.events` للتحكم في متى تُطلَق السياسة:

```js
customPolicies.add({
  name: "require-summary-on-stop",
  match: { events: ["Stop"] },
  fn: async (ctx) => {
    // يُطلق فقط عند انتهاء الجلسة
    // ctx.session.transcriptPath يحتوي على سجل الجلسة الكامل
    return allow();
  },
});
```

احذف `match` كلياً للإطلاق عند كل نوع حدث.

## معالجة الأخطاء وأنماط الفشل

السياسات المخصصة **مفتوحة الفشل**: الأخطاء لا تحظر السياسات المدمجة أبداً ولا تُعطّل معالج الخطاف.

| حالة الفشل | السلوك |
|------------|--------|
| `customPoliciesPath` غير مضبوط | لا تعمل أي سياسات مخصصة؛ تستمر السياسات المدمجة بشكل طبيعي |
| الملف غير موجود | يُسجَّل تحذير في `~/.failproofai/hook.log`؛ تستمر السياسات المدمجة |
| خطأ في بناء الجملة أو الاستيراد | يُسجَّل الخطأ في `~/.failproofai/hook.log`؛ تُتخطى جميع السياسات المخصصة |
| `fn` يرمي خطأً وقت التشغيل | يُسجَّل الخطأ؛ يُعامَل هذا الخطاف كـ `allow`؛ تستمر الخطافات الأخرى |
| `fn` يستغرق أكثر من 10 ثوانٍ | يُسجَّل انتهاء المهلة؛ يُعامَل كـ `allow` |

<Tip>
لتصحيح أخطاء السياسات المخصصة، راقب ملف السجل:

```bash
tail -f ~/.failproofai/hook.log
```
</Tip>

## مثال كامل: سياسات متعددة

```js
// my-policies.js
import { customPolicies, allow, deny, instruct } from "failproofai";

// Prevent agent from writing to secrets/ directory
customPolicies.add({
  name: "block-secrets-dir",
  description: "Prevent agent from writing to secrets/ directory",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (!["Write", "Edit"].includes(ctx.toolName ?? "")) return allow();
    const path = ctx.toolInput?.file_path ?? "";
    if (path.includes("secrets/")) return deny("Writing to secrets/ is not permitted");
    return allow();
  },
});

// Keep the agent on track: verify tests before committing
customPolicies.add({
  name: "remind-test-before-commit",
  description: "Keep the agent on track: verify tests pass before committing",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const cmd = ctx.toolInput?.command ?? "";
    if (/git\s+commit/.test(cmd)) {
      return instruct("Verify all tests pass before committing. Run `bun test` if you haven't already.");
    }
    return allow();
  },
});

// Prevent unplanned dependency changes during freeze
customPolicies.add({
  name: "dependency-freeze",
  description: "Prevent unplanned dependency changes during freeze period",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const cmd = ctx.toolInput?.command ?? "";
    const isInstall = /^(npm install|yarn add|bun add|pnpm add)\s+\S/.test(cmd);
    if (isInstall && process.env.DEPENDENCY_FREEZE === "1") {
      return deny("Package installs are frozen. Unset DEPENDENCY_FREEZE to allow.");
    }
    return allow();
  },
});

export { customPolicies };
```
