يستخدم failproofai ملفات إعدادات JSON للتحكم في السياسات النشطة وسلوكها وموقع تحميل السياسات المخصصة. تم تصميم الإعدادات لتكون سهلة المشاركة مع فريقك - التزمها في مستودعك وسيحصل كل مطور على نفس شبكة الأمان للوكيل.

---

## نطاقات الإعدادات

هناك ثلاثة نطاقات إعدادات، يتم تقييمها بترتيب الأولوية:

| النطاق | مسار الملف | الغرض |
|-------|-----------|---------|
| **project** | `.failproofai/policies-config.json` | إعدادات لكل مستودع، مرفوعة للتحكم بالإصدارات |
| **local** | `.failproofai/policies-config.local.json` | تجاوزات شخصية لكل مستودع، مدرجة في .gitignore |
| **global** | `~/.failproofai/policies-config.json` | إعدادات افتراضية على مستوى المستخدم لجميع المشاريع |

عندما يستقبل failproofai حدث hook، يقوم بتحميل ودمج جميع الملفات الثلاثة الموجودة للمجلد الحالي.

### قواعد الدمج

**`enabledPolicies`** - اتحاد جميع النطاقات الثلاثة. السياسة المفعلة في أي مستوى تكون نشطة.

```text
project:  ["block-sudo"]
local:    ["block-rm-rf"]
global:   ["block-sudo", "sanitize-api-keys"]

resolved: ["block-sudo", "block-rm-rf", "sanitize-api-keys"]  ← deduplicated union
```

**`policyParams`** - النطاق الأول الذي يحدد معاملات سياسة معينة يفوز بالكامل. لا يوجد دمج عميق للقيم داخل معاملات السياسة.

```text
project:  block-sudo → { allowPatterns: ["sudo apt-get update"] }
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo apt-get update"] }   ← project wins, global ignored
```

```text
project:  (no block-sudo entry)
local:    (no block-sudo entry)
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo systemctl status"] }  ← falls through to global
```

**`customPoliciesPath`** - النطاق الأول الذي يحدده يفوز.

**`llm`** - النطاق الأول الذي يحدده يفوز.

## صيغة ملف الإعدادات

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "sanitize-jwt",
    "block-env-files",
    "block-read-outside-cwd"
  ],
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    },
    "block-push-master": {
      "protectedBranches": ["main", "release", "prod"]
    },
    "block-rm-rf": {
      "allowPaths": ["/tmp"]
    },
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company"]
    },
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo API key" }
      ]
    },
    "warn-large-file-write": {
      "thresholdKb": 512
    }
  },
  "customPoliciesPath": "/home/alice/myproject/my-policies.js"
}
```

## مرجع الحقول

### `enabledPolicies`

النوع: `string[]`

قائمة بأسماء السياسات المراد تفعيلها. يجب أن تتطابق الأسماء بدقة مع معرفات السياسات التي يعرضها `failproofai policies`. راجع [السياسات المدمجة](/ar/built-in-policies) للحصول على القائمة الكاملة.

السياسات غير الموجودة في `enabledPolicies` تكون غير نشطة، حتى لو كانت لديها إدخالات في `policyParams`.

### `policyParams`

النوع: `Record<string, Record<string, unknown>>`

تجاوزات معاملات لكل سياسة. المفتاح الخارجي هو اسم السياسة؛ والمفاتيح الداخلية خاصة بالسياسة. توثق كل سياسة معاملاتها المتاحة في [السياسات المدمجة](/ar/built-in-policies).

إذا كانت لدى السياسة معاملات لكنك لم تحددها، يتم استخدام الافتراضيات المدمجة للسياسة. المستخدمون الذين لا يقومون بإعداد `policyParams` على الإطلاق يحصلون على السلوك المطابق للإصدارات السابقة.

المفاتيح غير المعروفة داخل كتلة معاملات السياسة يتم تجاهلها صامتة في وقت تشغيل hook لكن يتم وضع علامة عليها كتحذيرات عند تشغيل `failproofai policies`.

#### `hint` (متعدد الاستخدام)

النوع: `string` (اختياري)

رسالة يتم إضافتها إلى السبب عندما تعيد سياسة `deny` أو `instruct`. استخدمها لإعطاء Claude إرشادات قابلة للتنفيذ دون تعديل السياسة نفسها.

يعمل مع أي نوع سياسة — مدمجة أو مخصصة (`custom/`) أو اتفاقية المشروع (`.failproofai-project/`) أو اتفاقية المستخدم (`.failproofai-user/`).

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "Try creating a fresh branch instead."
    },
    "block-sudo": {
      "allowPatterns": ["sudo apt-get"],
      "hint": "Use apt-get directly without sudo."
    },
    "custom/my-policy": {
      "hint": "Ask the user for approval first."
    }
  }
}
```

عندما يرفض `block-force-push`، يرى Claude: *"Force-pushing is blocked. Try creating a fresh branch instead."*

القيم غير النصية والسلاسل الفارغة يتم تجاهلها صامتة. إذا لم يتم تعيين `hint`، السلوك لا يتغير (متوافق للخلف).

### `customPoliciesPath`

النوع: `string` (مسار مطلق)

المسار إلى ملف JavaScript يحتوي على سياسات hook مخصصة. يتم تعيينه تلقائياً بواسطة `failproofai policies --install --custom <path>` (يتم حل المسار إلى مسار مطلق قبل التخزين).

يتم تحميل الملف جديداً على كل حدث hook - لا يوجد تخزين مؤقت. راجع [السياسات المخصصة](/ar/custom-policies) لتفاصيل الإنشاء.

### السياسات المبنية على الاتفاقية

بالإضافة إلى `customPoliciesPath` الصريح، يكتشف failproofai تلقائياً وينقل ملفات السياسة من مجلدات `.failproofai/policies/`:

| المستوى | المجلد | النطاق |
|--------|--------|--------|
| Project | `.failproofai/policies/` | مشاركة مع الفريق عبر التحكم بالإصدارات |
| User | `~/.failproofai/policies/` | شخصية، تنطبق على جميع المشاريع |

**مطابقة الملفات:** يتم تحميل الملفات المطابقة فقط `*policies.{js,mjs,ts}` (مثل `security-policies.mjs`، `workflow-policies.js`). يتم تجاهل الملفات الأخرى في المجلد.

**لا توجد حاجة للإعدادات:** السياسات الاتفاقية لا تتطلب إدخالات في `policies-config.json`. ما عليك سوى وضع الملفات في المجلد وسيتم استلامها في حدث hook التالي.

**تحميل الاتحاد:** يتم مسح مجلدات الاتفاقية للمشروع والمستخدم. يتم تحميل جميع الملفات المطابقة من كلا المستويين (على عكس `customPoliciesPath` الذي يستخدم first-scope-wins).

راجع [السياسات المخصصة](/ar/custom-policies) لمزيد من التفاصيل والأمثلة.

### `llm`

النوع: `object` (اختياري)

إعدادات عميل LLM للسياسات التي تقوم بعمليات AI. غير مطلوب لمعظم الإعدادات.

```json
{
  "llm": {
    "model": "claude-sonnet-4-6",
    "apiKey": "sk-ant-..."
  }
}
```

## إدارة الإعدادات من واجهة سطر الأوامر

تكتب أوامر `policies --install` و `policies --uninstall` إلى ملف إعدادات hook للعميل (نقاط دخول hook)، بينما `policies-config.json` هو الملف الذي تديره مباشرة. الاثنان منفصلان:

- **إعدادات عميل الوكيل** — يخبر الوكيل باستدعاء `failproofai --hook <event>` على كل استخدام أداة:
  - **Claude Code**: `~/.claude/settings.json` (user), `<cwd>/.claude/settings.json` (project), `<cwd>/.claude/settings.local.json` (local)
  - **OpenAI Codex**: `~/.codex/hooks.json` (user), `<cwd>/.codex/hooks.json` (project) — Codex ليس لديه نطاق `local`
- **`policies-config.json`** — يخبر failproofai بأي السياسات يتم تقييمها ومع أي معاملات (مشترك بين جميع عملاء الوكيل)

مرر `--cli claude|codex` لاستهداف وكيل محدد (مفصول بمسافات أو متكرر لكليهما):

```bash
failproofai policies --install --cli codex --scope project
failproofai policies --install --cli claude codex
```

عندما يتم حذف `--cli`، يكتشف `failproofai` عملاء الوكيل المثبتة (`which claude` / `which codex`):

- **عميل واحد مكتشف** — يتم الاختيار التلقائي لهذا العميل دون مطالبة.
- **كلا العميلين مكتشف** في محطة تفاعلية — يعرض مطالبة بضغطة مفتاح واحدة: تثبيت على [B]oth (افتراضي)، [C]laude فقط، أو co[D]ex فقط.
- **كلا العميلين مكتشف** في تشغيل غير تفاعلي (CI، بدون TTY) — يثبت لكليهما دون مطالبة.
- **لا يوجد عميل مكتشف** — يعود إلى `claude`، مع تحذير بعدم العثور على ملف تنفيذي وكيل في PATH؛ أمر hook لا يزال مكتوباً بحيث يتم تفعيله بمجرد تثبيتك لأحدهما.

يمكنك تحرير `policies-config.json` مباشرة في أي وقت؛ التغييرات تأخذ تأثيرها فوراً في حدث hook التالي دون الحاجة إلى إعادة تشغيل.

## مثال: إعدادات على مستوى المشروع مع افتراضيات الفريق

التزم `.failproofai/policies-config.json` في مستودعك:

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "block-env-files"
  ],
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "release", "hotfix"]
    }
  }
}
```

يمكن لكل مطور بعد ذلك إنشاء `.failproofai/policies-config.local.json` (مدرج في .gitignore) للتجاوزات الشخصية دون التأثير على الزملاء.
