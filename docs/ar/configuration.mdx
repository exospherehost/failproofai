يستخدم failproofai ملفات تكوين JSON للتحكم في السياسات النشطة، وكيفية تصرفها، ومن أين يتم تحميل السياسات المخصصة. تم تصميم التكوين ليكون سهل المشاركة مع فريقك - التزم به في مستودعك وسيحصل كل مطور على نفس شبكة الأمان للوكيل.

---

## نطاقات التكوين

هناك ثلاثة نطاقات تكوين، يتم تقييمها بترتيب الأولوية:

| النطاق | مسار الملف | الغرض |
|-------|-----------|---------|
| **المشروع** | `.failproofai/policies-config.json` | إعدادات محددة للمستودع، مرفوعة للتحكم بالإصدارات |
| **محلي** | `.failproofai/policies-config.local.json` | تجاوزات شخصية محددة للمستودع، مستثناة من git |
| **عام** | `~/.failproofai/policies-config.json` | القيم الافتراضية على مستوى المستخدم في جميع المشاريع |

عندما يتلقى failproofai حدث hook، يقوم بتحميل ودمج جميع الملفات الثلاثة الموجودة للمجلد الحالي.

### قواعد الدمج

**`enabledPolicies`** - اتحاد جميع النطاقات الثلاثة. السياسة المفعلة في أي مستوى تكون نشطة.

```text
project:  ["block-sudo"]
local:    ["block-rm-rf"]
global:   ["block-sudo", "sanitize-api-keys"]

resolved: ["block-sudo", "block-rm-rf", "sanitize-api-keys"]  ← اتحاد بدون تكرار
```

**`policyParams`** - أول نطاق يحدد المعاملات لسياسة معينة يفوز بالكامل. لا يوجد دمج عميق للقيم داخل معاملات السياسة.

```text
project:  block-sudo → { allowPatterns: ["sudo apt-get update"] }
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo apt-get update"] }   ← المشروع يفوز، العام مهمل
```

```text
project:  (no block-sudo entry)
local:    (no block-sudo entry)
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo systemctl status"] }  ← ينتقل للعام
```

**`customPoliciesPath`** - أول نطاق يحدده يفوز.

**`llm`** - أول نطاق يحدده يفوز.

## صيغة ملف التكوين

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "sanitize-jwt",
    "block-env-files",
    "block-read-outside-cwd"
  ],
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    },
    "block-push-master": {
      "protectedBranches": ["main", "release", "prod"]
    },
    "block-rm-rf": {
      "allowPaths": ["/tmp"]
    },
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company"]
    },
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo API key" }
      ]
    },
    "warn-large-file-write": {
      "thresholdKb": 512
    }
  },
  "customPoliciesPath": "/home/alice/myproject/my-policies.js"
}
```

## مرجع الحقول

### `enabledPolicies`

النوع: `string[]`

قائمة بأسماء السياسات المراد تفعيلها. يجب أن تطابق الأسماء بالضبط معرّفات السياسات المعروضة بواسطة `failproofai policies`. انظر [السياسات المدمجة](/ar/built-in-policies) للقائمة الكاملة.

السياسات التي لا تكون في `enabledPolicies` غير نشطة، حتى لو كانت لها مدخلات في `policyParams`.

### `policyParams`

النوع: `Record<string, Record<string, unknown>>`

تجاوزات المعاملات لكل سياسة. المفتاح الخارجي هو اسم السياسة؛ المفاتيح الداخلية محددة لكل سياسة. كل سياسة توثق معاملات متاحة في [السياسات المدمجة](/ar/built-in-policies).

إذا كانت لديك سياسة بمعاملات لكن لم تحددها، فيتم استخدام القيم الافتراضية المدمجة في السياسة. المستخدمون الذين لم يقوموا بتكوين `policyParams` على الإطلاق سيحصلون على نفس السلوك كما في الإصدارات السابقة.

المفاتيح غير المعروفة داخل كتلة معاملات السياسة يتم تجاهلها بسكوت عند تشغيل hook لكن يتم الإشارة إليها كتحذيرات عند تشغيل `failproofai policies`.

#### `hint` (عام)

النوع: `string` (اختياري)

رسالة تُضاف إلى السبب عندما تعيد السياسة `deny` أو `instruct`. استخدمه لإعطاء Claude إرشادات قابلة للتطبيق دون تعديل السياسة نفسها.

يعمل مع أي نوع سياسة — مدمجة أو مخصصة (`custom/`) أو اتفاقية مشروع (`.failproofai-project/`) أو اتفاقية مستخدم (`.failproofai-user/`).

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "حاول إنشاء فرع جديد بدلاً من ذلك."
    },
    "block-sudo": {
      "allowPatterns": ["sudo apt-get"],
      "hint": "استخدم apt-get مباشرة بدون sudo."
    },
    "custom/my-policy": {
      "hint": "اطلب موافقة من المستخدم أولاً."
    }
  }
}
```

عندما يرفض `block-force-push`، يرى Claude: *"Force-pushing is blocked. حاول إنشاء فرع جديد بدلاً من ذلك."*

القيم غير الحرفية والسلاسل الفارغة يتم تجاهلها بسكوت. إذا لم يتم تعيين `hint`، فإن السلوك لم يتغير (متوافق للأمام).

### `customPoliciesPath`

النوع: `string` (مسار مطلق)

المسار إلى ملف JavaScript يحتوي على سياسات hook مخصصة. يتم تعيينه تلقائياً بواسطة `failproofai policies --install --custom <path>` (يتم حل المسار للمسار المطلق قبل التخزين).

يتم تحميل الملف جديداً في كل حدث hook - لا يوجد تخزين مؤقت. انظر [السياسات المخصصة](/ar/custom-policies) لتفاصيل المأليف.

### سياسات قائمة على الاتفاقية

بالإضافة إلى `customPoliciesPath` الصريح، يكتشف failproofai تلقائياً ويحمل ملفات السياسات من مجلدات `.failproofai/policies/`:

| المستوى | المجلد | النطاق |
|-------|---------|-------|
| المشروع | `.failproofai/policies/` | مشاركة مع الفريق عبر التحكم بالإصدارات |
| المستخدم | `~/.failproofai/policies/` | شخصية، تنطبق على جميع المشاريع |

**مطابقة الملفات:** فقط الملفات المطابقة لـ `*policies.{js,mjs,ts}` يتم تحميلها (مثل `security-policies.mjs`، `workflow-policies.js`). الملفات الأخرى في المجلد يتم تجاهلها.

**لا يلزم التكوين:** سياسات الاتفاقية لا تتطلب مدخلات في `policies-config.json`. فقط ضع الملفات في المجلد وسيتم التقاطها في حدث hook التالي.

**تحميل اتحادي:** يتم مسح مجلدات الاتفاقية للمشروع والمستخدم. تُحمل جميع الملفات المطابقة من كلا المستويين (على عكس `customPoliciesPath` الذي يستخدم first-scope-wins).

انظر [السياسات المخصصة](/ar/custom-policies) لمزيد من التفاصيل والأمثلة.

### `llm`

النوع: `object` (اختياري)

تكوين عميل LLM للسياسات التي تجري استدعاءات AI. غير مطلوب لمعظم الإعدادات.

```json
{
  "llm": {
    "model": "claude-sonnet-4-6",
    "apiKey": "sk-ant-..."
  }
}
```

## إدارة التكوين من CLI

أوامر `policies --install` و `policies --uninstall` تكتب إلى ملف إعدادات hook لـ CLI الوكيل (نقاط دخول hook)، في حين أن `policies-config.json` هو الملف الذي تديره مباشرة. الاثنان منفصلان:

- **إعدادات Agent CLI** — تخبر الوكيل باستدعاء `failproofai --hook <event>` على كل استخدام أداة:
  - **Claude Code**: `~/.claude/settings.json` (مستخدم)، `<cwd>/.claude/settings.json` (مشروع)، `<cwd>/.claude/settings.local.json` (محلي)
  - **OpenAI Codex**: `~/.codex/hooks.json` (مستخدم)، `<cwd>/.codex/hooks.json` (مشروع) — Codex لا يملك نطاق `محلي`
- **`policies-config.json`** — يخبر failproofai بأي السياسات يتم تقييمها وبأي معاملات (مشترك بين جميع CLIs للوكيل)

مرر `--cli claude|codex` لاستهداف وكيل معين (مفصول بمسافة أو مكرر لكليهما):

```bash
failproofai policies --install --cli codex --scope project
failproofai policies --install --cli claude codex
```

عندما يتم حذف `--cli`، يكتشف `failproofai` أي CLIs للوكيل مثبتة (`which claude` / `which codex`):

- **CLI واحد مكتشف** — يختار تلقائياً هذا CLI بدون التماس.
- **كلا CLIs مكتشفة** في طرفية تفاعلية — تعرض موجه اختيار واحد بمفاتيح الأسهم: اختر `Both`، `Claude Code only`، أو `OpenAI Codex only` (↑↓ للتحرك، Enter للاختيار، ^C للخروج).
- **كلا CLIs مكتشفة** في تشغيل غير تفاعلي (CI، بدون TTY) — تثبت لكليهما بدون التماس.
- **لا CLIs مكتشفة** — ترجع للخلف إلى `claude`، مع تحذير بأنه لم يتم العثور على binary وكيل في PATH؛ أمر hook لا يزال مكتوباً لذلك يتم تفعيله بمجرد تثبيتك لأحدها.

يمكنك تحرير `policies-config.json` مباشرة في أي وقت؛ التغييرات تصبح فعالة فوراً في حدث hook التالي بدون الحاجة لإعادة تشغيل.

## مثال: تكوين على مستوى المشروع مع افتراضيات الفريق

التزم بـ `.failproofai/policies-config.json` في مستودعك:

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "block-env-files"
  ],
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "release", "hotfix"]
    }
  }
}
```

يمكن لكل مطور بعدها إنشاء `.failproofai/policies-config.local.json` (مستثناة من git) لتجاوزات شخصية بدون التأثير على زملائهم.
