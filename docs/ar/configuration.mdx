يستخدم failproofai ملفات تكوين JSON للتحكم في السياسات النشطة وكيفية سلوكها وحيث يتم تحميل السياسات المخصصة. تم تصميم التكوين ليكون سهل المشاركة مع فريقك - التزم به في مستودعك وكل مطور سيحصل على نفس شبكة أمان الوكيل.

---

## نطاقات التكوين

هناك ثلاثة نطاقات تكوين، يتم تقييمها بترتيب الأولوية:

| النطاق | مسار الملف | الغرض |
|-------|-----------|---------|
| **المشروع** | `.failproofai/policies-config.json` | إعدادات لكل مستودع، يتم الالتزام بها في التحكم بالإصدارات |
| **محلي** | `.failproofai/policies-config.local.json` | تجاوزات شخصية لكل مستودع، مضافة إلى gitignore |
| **عام** | `~/.failproofai/policies-config.json` | القيم الافتراضية على مستوى المستخدم في جميع المشاريع |

عندما يتلقى failproofai حدث هوك، يقوم بتحميل ودمج جميع الملفات الثلاثة الموجودة في مجلد العمل الحالي.

### قواعد الدمج

**`enabledPolicies`** - اتحاد جميع النطاقات الثلاثة. أي سياسة مفعلة في أي مستوى تكون نشطة.

```text
project:  ["block-sudo"]
local:    ["block-rm-rf"]
global:   ["block-sudo", "sanitize-api-keys"]

resolved: ["block-sudo", "block-rm-rf", "sanitize-api-keys"]  ← اتحاد مع حذف التكرارات
```

**`policyParams`** - النطاق الأول الذي يحدد المعاملات لسياسة معينة يفوز بالكامل. لا يوجد دمج عميق للقيم داخل معاملات السياسة.

```text
project:  block-sudo → { allowPatterns: ["sudo apt-get update"] }
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo apt-get update"] }   ← المشروع يفوز، العام يتم تجاهله
```

```text
project:  (لا توجد إدخالة block-sudo)
local:    (لا توجد إدخالة block-sudo)
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo systemctl status"] }  ← ينتقل إلى العام
```

**`customPoliciesPath`** - النطاق الأول الذي يحدده يفوز.

**`llm`** - النطاق الأول الذي يحدده يفوز.

## تنسيق ملف التكوين

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "sanitize-jwt",
    "block-env-files",
    "block-read-outside-cwd"
  ],
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    },
    "block-push-master": {
      "protectedBranches": ["main", "release", "prod"]
    },
    "block-rm-rf": {
      "allowPaths": ["/tmp"]
    },
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company"]
    },
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo API key" }
      ]
    },
    "warn-large-file-write": {
      "thresholdKb": 512
    }
  },
  "customPoliciesPath": "/home/alice/myproject/my-policies.js"
}
```

## مرجع الحقول

### `enabledPolicies`

النوع: `string[]`

قائمة أسماء السياسات المراد تفعيلها. يجب أن تطابق الأسماء بدقة معرفات السياسات الموضحة بواسطة `failproofai policies`. انظر [السياسات المدمجة](/ar/built-in-policies) للحصول على القائمة الكاملة.

السياسات غير الموجودة في `enabledPolicies` غير نشطة، حتى لو كانت لديها إدخالات في `policyParams`.

### `policyParams`

النوع: `Record<string, Record<string, unknown>>`

تجاوزات معاملات لكل سياسة. المفتاح الخارجي هو اسم السياسة؛ المفاتيح الداخلية خاصة بالسياسة. توثق كل سياسة معاملات متاحة في [السياسات المدمجة](/ar/built-in-policies).

إذا كانت لديك سياسة معاملات ولم تحددها، يتم استخدام القيم الافتراضية المدمجة للسياسة. المستخدمون الذين لا يقومون بتكوين `policyParams` على الإطلاق يحصلون على سلوك مطابق للإصدارات السابقة.

المفاتيح غير المعروفة داخل كتلة معاملات السياسة يتم تجاهلها بصمت في وقت تشغيل الهوك ولكن يتم الإشارة إليها كتحذيرات عند تشغيل `failproofai policies`.

#### `hint` (عام الاستخدام)

النوع: `string` (اختياري)

رسالة يتم إلحاقها بالسبب عندما تعيد السياسة `deny` أو `instruct`. استخدمه لإعطاء Claude إرشادات قابلة للتنفيذ دون تعديل السياسة نفسها.

يعمل مع أي نوع سياسة — مدمج، مخصص (`custom/`)، اتفاقية المشروع (`.failproofai-project/`)، أو اتفاقية المستخدم (`.failproofai-user/`).

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "حاول إنشاء فرع جديد بدلاً من ذلك."
    },
    "block-sudo": {
      "allowPatterns": ["sudo apt-get"],
      "hint": "استخدم apt-get مباشرة بدون sudo."
    },
    "custom/my-policy": {
      "hint": "اطلب موافقة المستخدم أولاً."
    }
  }
}
```

عند رفض `block-force-push`، يرى Claude: *"Force-pushing is blocked. Try creating a fresh branch instead."*

القيم غير النصية والسلاسل الفارغة يتم تجاهلها بصمت. إذا لم يتم تعيين `hint`، يكون السلوك بدون تغيير (متوافق مع الإصدارات السابقة).

### `customPoliciesPath`

النوع: `string` (مسار مطلق)

مسار ملف JavaScript يحتوي على سياسات هوك مخصصة. يتم تعيينه تلقائياً بواسطة `failproofai policies --install --custom <path>` (يتم حل المسار إلى مطلق قبل تخزينه).

يتم تحميل الملف بشكل جديد في كل حدث هوك - لا توجد ذاكرة تخزين مؤقت. انظر [السياسات المخصصة](/ar/custom-policies) لتفاصيل التأليف.

### سياسات قائمة على الاتفاقية

بالإضافة إلى `customPoliciesPath` الصريح، يكتشف failproofai تلقائياً ويحمل ملفات السياسة من مجلدات `.failproofai/policies/`:

| المستوى | المجلد | النطاق |
|-------|-----------|-------|
| المشروع | `.failproofai/policies/` | مشاركة مع الفريق عبر التحكم بالإصدارات |
| المستخدم | `~/.failproofai/policies/` | شخصي، ينطبق على جميع المشاريع |

**مطابقة الملفات:** يتم تحميل الملفات فقط التي تطابق `*policies.{js,mjs,ts}` (مثل `security-policies.mjs`، `workflow-policies.js`). الملفات الأخرى في المجلد يتم تجاهلها.

**بدون تكوين مطلوب:** سياسات الاتفاقية لا تتطلب إدخالات في `policies-config.json`. فقط أسقط الملفات في المجلد وسيتم اختيارها في حدث الهوك التالي.

**تحميل الاتحاد:** يتم مسح مجلدي الاتفاقية للمشروع والمستخدم. يتم تحميل جميع الملفات المطابقة من كلا المستويين (بخلاف `customPoliciesPath` الذي يستخدم النطاق الأول يفوز).

انظر [السياسات المخصصة](/ar/custom-policies) لمزيد من التفاصيل والأمثلة.

### `llm`

النوع: `object` (اختياري)

تكوين عميل LLM للسياسات التي تقوم بعمليات اتصال AI. غير مطلوب لمعظم الإعدادات.

```json
{
  "llm": {
    "model": "claude-sonnet-4-6",
    "apiKey": "sk-ant-..."
  }
}
```

## إدارة التكوين من CLI

تقوم أوامر `policies --install` و `policies --uninstall` بالكتابة إلى ملف إعدادات الهوك لـ CLI الوكيل الخاص بك (نقاط دخول الهوك)، بينما `policies-config.json` هو الملف الذي تديره مباشرة. الاثنان منفصلان:

- **إعدادات Agent CLI** — تخبر الوكيل باستدعاء `failproofai --hook <event>` في كل استخدام أداة:
  - **Claude Code**: `~/.claude/settings.json` (مستخدم)، `<cwd>/.claude/settings.json` (مشروع)، `<cwd>/.claude/settings.local.json` (محلي)
  - **OpenAI Codex**: `~/.codex/hooks.json` (مستخدم)، `<cwd>/.codex/hooks.json` (مشروع) — Codex ليس لديه نطاق `local`
  - **GitHub Copilot CLI _(beta)_**: `~/.copilot/hooks/failproofai.json` (مستخدم)، `<cwd>/.github/hooks/failproofai.json` (مشروع) — Copilot ليس لديه نطاق `local`. إدخالات الهوك تستخدم حقول أوامر `bash`/`powershell` المفتاحة حسب نظام التشغيل في Copilot مع `timeoutSec`؛ الملف يحمل علامة `version: 1` على المستوى الأعلى. دعم Copilot CLI هو **beta** بينما نتحقق من مخطط سجل `events.jsonl` (الذي لا توثقه المستندات العامة) مقابل جلسات العالم الحقيقي.
  - **Cursor Agent _(beta)_**: `~/.cursor/hooks.json` (مستخدم)، `<cwd>/.cursor/hooks.json` (مشروع) — Cursor ليس لديه نطاق `local`. إدخالات الهوك تستخدم شكل Claude `{type, command, timeout}` (بدون تقسيم `bash`/`powershell`)، لكن مخزنة تحت مفاتيح أحداث camelCase (`preToolUse`, `beforeSubmitPrompt`, …) في مصفوفة مسطحة لكل [مخطط hooks](https://cursor.com/docs/hooks) في Cursor؛ الملف يحمل علامة `version: 1` على المستوى الأعلى. المعالج يوحد camelCase → PascalCase عبر `CURSOR_EVENT_MAP` لذا السياسات المدمجة الموجودة تعمل بدون تغيير. دعم Cursor Agent هو **beta** بينما نتحقق من نص Cursor على القرص (غير محدد في المستندات العامة) مقابل المزيد من التثبيتات في العالم الحقيقي.
  - **OpenCode _(beta)_**: `~/.config/opencode/opencode.json` + `~/.config/opencode/plugins/failproofai.mjs` (مستخدم)، `<cwd>/.opencode/opencode.json` + `<cwd>/.opencode/plugins/failproofai.mjs` (مشروع) — OpenCode ليس لديه نطاق `local`. بخلاف CLIs الأربعة الأخرى، OpenCode **ليس لديه نظام هوك خارجي للأوامر**: يحمل مكونات إضافية JS/TS في العملية مسجلة بشكل صريح عبر مصفوفة `plugin: []` في `opencode.json` (الاكتشاف التلقائي من `.opencode/plugins/` **ليس** كيفية تحميل المكونات الإضافية في opencode v1.14.33). التثبيت يسقط shim مكون إضافي صغير تم إنشاؤه يستدعي ثنائي failproofai في عملية فرعية ويترجم استجابة JSON على شكل Claude للثنائي إلى دلالات المكون الإضافي (`throw new Error()` للرفض، `client.session.prompt(...)` للتعليمات، بدون عملية للسماح). الجلسات تعيش في قاعدة بيانات SQLite في OpenCode على `~/.local/share/opencode/opencode.db`؛ عارض جلسة لوحة المعلومات يقرأها عبر `opencode db --format json` و `opencode export <id>`. دعم OpenCode هو **beta** بينما نتحقق من السلوك عبر الإصدارات ومقابل جلسات العالم الحقيقي. انظر [وثائق مكونات OpenCode الإضافية](https://opencode.ai/docs/plugins/).
  - **Pi _(beta)_**: `~/.pi/agent/settings.json` (مستخدم)، `<cwd>/.pi/settings.json` (مشروع) — Pi ليس لديه نطاق `local`. يحمل Pi حزم امتدادات TypeScript عند بدء التشغيل؛ ملف الإعدادات هو مصفوفة سلسلة مسطحة `{"packages": ["./relative/path", …]}`. يكتب failproofai إدخال مصفوفة حزم واحد يشير إلى مجلد `pi-extension/` المجمع. الامتداد داخلياً يشترك في أحداث `tool_call` / `user_bash` / `input` / `session_start` في Pi ويستدعي shell إلى `failproofai --hook <Event> --cli pi`؛ المعالج يوحد underscore_lower_snake_case → PascalCase عبر `PI_EVENT_MAP` لذا السياسات المدمجة الموجودة تعمل بدون تغيير. دعم Pi هو **beta** بينما API الامتداد وتخطيط سجل الجلسة في Pi تتحول.
  - **Gemini CLI _(beta)_**: `~/.gemini/settings.json` (مستخدم)، `<cwd>/.gemini/settings.json` (مشروع) — Gemini ليس لديه نطاق `local` (توثق نطاق `system` على `/etc/gemini-cli/settings.json` الذي لا يكشفه failproofai). إدخالات الهوك تستخدم شكل Claude `{type, command, timeout}` ملفوف في مخطط مطابقة Gemini `{matcher, hooks: [...]}` مع `matcher: "*"` بشكل افتراضي. الأحداث هي PascalCase (`SessionStart`, `BeforeAgent`, `AfterAgent`, `BeforeModel`, `AfterModel`, `BeforeToolSelection`, `BeforeTool`, `AfterTool`, `PreCompress`, `Notification`, `SessionEnd`)؛ المعالج يرسم لأسماء Claude الكنسية عبر `GEMINI_EVENT_MAP`. أسماء الأدوات هي snake_case (`run_shell_command`, `read_file`, `write_file`, `replace`, …) — المعالج يوحد عبر `GEMINI_TOOL_MAP` لذا السياسات المدمجة الموجودة تعمل بدون تغيير. محيِّم السياسة ينبعث شكل Gemini المسطح `{decision: "deny", reason}` (مفضل لكل قانون Gemini الذهبي في عقد exit-0)، `{hookSpecificOutput: {hookEventName, additionalContext}}` لحقن السياق على BeforeAgent / AfterTool / SessionStart، و `{decision: "block", reason}` على AfterAgent لدلالات إعادة المحاولة القسرية. دعم Gemini CLI هو **beta** بينما نوسع التغطية في العالم الحقيقي. انظر [وثائق hooks في Gemini CLI](https://geminicli.com/docs/hooks/).
- **`policies-config.json`** — يخبر failproofai بأي سياسات تقييم ومع أي معاملات (مشترك في جميع agent CLIs)

مرر `--cli claude|codex|copilot|cursor|opencode|pi|gemini` لاستهداف وكيل معين (مفصول بمسافات أو مكرر لأي مجموعة فرعية):

```bash
failproofai policies --install --cli codex --scope project
failproofai policies --install --cli copilot --scope project
failproofai policies --install --cli cursor --scope project
failproofai policies --install --cli opencode --scope project
failproofai policies --install --cli pi --scope project
failproofai policies --install --cli gemini --scope project
failproofai policies --install --cli claude codex copilot cursor opencode pi gemini
```

عند حذف `--cli`، يكتشف `failproofai` وكيل CLIs المثبتة (`which claude` / `which codex` / `which copilot` / `which cursor-agent` / `which opencode` / `which pi` / `which gemini`):

- **CLI واحد مكتشف** — يختار هذا CLI تلقائياً بدون سؤال.
- **عدة CLIs مكتشفة** في محطة طرفية تفاعلية — يظهر موجه تحديد واحد مع مفاتيح الأسهم مجمعة في قسم `Detected (N)` (مع صف تجميع `Install for all N detected` + كل CLI مكتشف بشكل فردي) وقسم `Not installed (M) · install hooks ahead of time` يسرد كل CLI غير مكتشف مدعوم كخيار تثبيت للأمام (↑↓ للتحرك، Enter للتحديد، ^C للخروج). تدفق الإلغاء يظهر فقط قسم الـ Detected.
- **عدة CLIs مكتشفة** في تشغيل غير تفاعلي (CI، بدون TTY) — يثبت لجميع CLIs المكتشفة بدون سؤال.
- **لم يتم اكتشاف أي** — ينجرف إلى `claude`، مع تحذير بأنه لم يتم العثور على ثنائي وكيل في PATH؛ أمر الهوك لا يزال مكتوب بحيث ينشط بمجرد تثبيت واحد.

يمكنك تحرير `policies-config.json` مباشرة في أي وقت؛ التغييرات تصبح سارية المفعول فوراً في حدث الهوك التالي بدون الحاجة لإعادة تشغيل.

## مثال: تكوين على مستوى المشروع مع قيم افتراضية للفريق

التزم بـ `.failproofai/policies-config.json` في مستودعك:

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "block-env-files"
  ],
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "release", "hotfix"]
    }
  }
}
```

يمكن لكل مطور بعد ذلك إنشاء `.failproofai/policies-config.local.json` (مضافة إلى gitignore) للتجاوزات الشخصية بدون التأثير على زملاء الفريق.
