failproofai 内置了 39 条策略，用于捕获常见的 Agent 失效模式。每条策略会在特定的 Hook 事件类型和工具名称上触发。其中 19 条策略支持参数配置，让你无需编写代码即可调整其行为。5 条工作流策略在 Claude 停止之前强制执行提交 → 推送 → PR → CI 的完整流水线。

---

## 概览

策略按类别分组如下：

| 类别 | 策略 | Hook 类型 |
|----------|----------|-----------|
| [危险命令](#dangerous-commands) | block-sudo, block-rm-rf, block-curl-pipe-sh, block-failproofai-commands | PreToolUse |
| [基础设施命令](#infra-commands) | block-kubectl, block-terraform, block-aws-cli, block-gcloud, block-az-cli, block-helm, block-gh-pipeline | PreToolUse |
| [密钥（脱敏器）](#secrets-sanitizers) | sanitize-jwt, sanitize-api-keys, sanitize-connection-strings, sanitize-private-key-content, sanitize-bearer-tokens | PostToolUse |
| [环境变量](#environment) | block-env-files, protect-env-vars | PreToolUse |
| [文件访问](#file-access) | block-read-outside-cwd, block-secrets-write | PreToolUse |
| [Git](#git) | block-push-master, block-work-on-main, block-force-push, warn-git-amend, warn-git-stash-drop, warn-all-files-staged | PreToolUse |
| [数据库](#database) | warn-destructive-sql, warn-schema-alteration | PreToolUse |
| [警告](#warnings) | warn-large-file-write, warn-package-publish, warn-background-process, warn-global-package-install | PreToolUse |
| [包管理器](#package-managers) | prefer-package-manager | PreToolUse |
| [工作流](#workflow) | require-commit-before-stop, require-push-before-stop, require-pr-before-stop, require-no-conflicts-before-stop, require-ci-green-before-stop | Stop |

- **`block-`** — 阻止 Agent 继续执行。
- **`warn-`** — 向 Agent 提供额外上下文，使其能够自我纠正。
- **`sanitize-`** — 在工具输出返回给 Agent 之前清除敏感数据。

### 命名空间

每条策略都位于 `<namespace>/<name>` 槽位中。内置策略属于 **`failproofai/`** 命名空间，例如 `failproofai/sanitize-jwt`。命名空间可防止在同时加载自定义或第三方策略时发生名称冲突。

在配置中，你可以使用短名称或完整限定名来引用内置策略，两种形式解析到同一条策略：

```json
{
  "enabledPolicies": [
    "sanitize-jwt",
    "failproofai/block-rm-rf"
  ]
}
```

如果名称中不含 `/`，failproofai 会将其视为属于默认命名空间 `failproofai`。已包含 `/` 的名称（如 `myorg/foo`、`custom/my-hook`）则保持原样。
- **`require-`** — 阻止 Stop 事件，直到满足条件为止。

---

<Tip>
每条策略在 `policyParams` 中都支持可选的 `hint` 字段。该提示会附加到 Claude 收到的 deny 或 instruct 消息中，提供可操作的指导，而无需修改策略代码。适用于内置策略、自定义策略和约定策略。详见 [配置 → hint](/zh/configuration#hint-cross-cutting)。
</Tip>

## 危险命令

防止 Agent 执行难以撤销或可能损坏宿主系统的操作。

### `block-sudo`

**事件：** PreToolUse (Bash)  
**默认行为：** 拒绝任何包含 `sudo` 的命令。

对包含 `sudo` 关键字的调用进行拦截。模式匹配在解析后的命令 token 上执行，而非原始字符串，以防止通过 Shell 运算符注入绕过检测。

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | 允许执行的精确命令前缀，每条记录与解析后的 argv token 进行匹配。 |

**示例：**

```json
{
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    }
  }
}
```

使用此配置，`sudo systemctl status nginx` 会被允许，但 `sudo rm /etc/hosts` 会被拒绝。

<Note>
模式匹配针对解析后的 token，而非原始命令字符串。这可防止通过附加 Shell 运算符绕过检测（例如 `sudo systemctl status x; rm -rf /` 不会匹配 `sudo systemctl status *`）。
</Note>

---

### `block-rm-rf`

**事件：** PreToolUse (Bash)  
**默认行为：** 拒绝 `rm -rf`、`rm -fr` 及类似的递归删除形式。

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|-------|------|---------|-------------|
| `allowPaths` | `string[]` | `[]` | 允许递归删除的安全路径（如 `/tmp`）。 |

**示例：**

```json
{
  "policyParams": {
    "block-rm-rf": {
      "allowPaths": ["/tmp", "/var/cache"]
    }
  }
}
```

### `block-curl-pipe-sh`

**事件：** PreToolUse (Bash)  
**默认行为：** 拒绝 `curl <url> | bash`、`curl <url> | sh`、`wget <url> | bash` 及类似模式。

无参数。

### `block-failproofai-commands`

**事件：** PreToolUse (Bash)  
**默认行为：** 拒绝会卸载或禁用 failproofai 自身的命令（如 `npm uninstall failproofai`、`failproofai policies --uninstall`）。

无参数。

## 基础设施命令

阻止编码 Agent 运行基础设施 CLI 或触发 CI/CD 流水线。本类别中的所有策略均为**按需启用**（`defaultEnabled: false`）——确实需要调用 `kubectl`、`terraform` 等工具的 Agent 不会受到干扰，除非你显式启用该策略。启用后，被匹配的 CLI 的每次调用都会被拒绝，除非命令与 `allowPatterns` 中的某条记录匹配。

模式语法与 [`block-sudo`](#block-sudo) 相同：token 与解析后的 argv 进行匹配，`*` 是单个 token 的通配符，任何包含独立 Shell 运算符（`&&`、`||`、`|`、`;`）或嵌入 Shell 元字符的 token 的命令，在白名单匹配前都会被拒绝，以防止注入绕过。

### `block-kubectl`

**事件：** PreToolUse (Bash)  
**默认行为：** 拒绝任何 `kubectl` 调用。

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | 允许执行的 kubectl 命令前缀。 |

**示例：**

```json
{
  "policyParams": {
    "block-kubectl": {
      "allowPatterns": ["kubectl get *", "kubectl describe *", "kubectl logs *"]
    }
  }
}
```

使用此配置，`kubectl get pods` 被允许，但 `kubectl apply -f deploy.yaml` 被拒绝。

### `block-gh-pipeline`

**事件：** PreToolUse (Bash)  
**默认行为：** 拒绝以下会改变状态或触发流水线的 `gh` CLI 子命令：

- `gh workflow run`、`gh workflow enable`、`gh workflow disable`
- `gh run rerun`、`gh run cancel`
- `gh pr merge`
- `gh release create`、`gh release delete`
- `gh cache delete`
- `gh secret set`、`gh secret delete`

只读的 `gh` 子命令（如 `gh pr view`、`gh pr list`、`gh run list`、`gh release view` 和 `gh api repos/.../...`）**不在**本策略的匹配范围内——这些命令通常是工作流检查所必需的（包括 failproofai 自身的 `require-ci-green-before-stop`）。

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | 即便通常会被拒绝，也允许执行的特定脚本调用。 |

**示例：**

```json
{
  "policyParams": {
    "block-gh-pipeline": {
      "allowPatterns": ["gh run rerun *"]
    }
  }
}
```

## 密钥（脱敏器）

防止 Agent 将凭据泄露到其上下文或输出中。脱敏策略在 **PostToolUse** 事件上触发。当 Claude 运行 Bash 命令、读取文件或调用任何工具时，这些策略会在输出返回给 Claude 之前对其进行检查。如果检测到密钥模式，策略会返回拒绝决定，阻止输出被回传。

### `sanitize-jwt`

**事件：** PostToolUse（所有工具）  
**默认行为：** 脱敏 JWT token（由 `.` 分隔的三段 base64url 结构）。

无参数。

---

### `sanitize-api-keys`

**事件：** PostToolUse（所有工具）  
**默认行为：** 脱敏常见 API 密钥格式：Anthropic（`sk-ant-`）、OpenAI（`sk-`）、GitHub PAT（`ghp_`）、AWS 访问密钥（`AKIA`）、Stripe 密钥（`sk_live_`、`sk_test_`）以及 Google API 密钥（`AIza`）。

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|-------|------|---------|-------------|
| `additionalPatterns` | `{ regex: string; label: string }[]` | `[]` | 额外需要视为密钥的正则表达式模式。 |

**示例：**

```json
{
  "policyParams": {
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo internal API key" },
        { "regex": "pat_[0-9a-f]{40}", "label": "Internal PAT" }
      ]
    }
  }
}
```

---

### `sanitize-connection-strings`

**事件：** PostToolUse（所有工具）  
**默认行为：** 脱敏包含嵌入凭据的数据库连接字符串（如 `postgresql://user:password@host/db`）。

无参数。

---

### `sanitize-private-key-content`

**事件：** PostToolUse（所有工具）  
**默认行为：** 脱敏 PEM 块（`-----BEGIN PRIVATE KEY-----`、`-----BEGIN RSA PRIVATE KEY-----` 等）。

无参数。

---

### `sanitize-bearer-tokens`

**事件：** PostToolUse（所有工具）  
**默认行为：** 脱敏 `Authorization: Bearer <token>` 请求头中 token 长度不少于 20 个字符的内容。

无参数。

---

## 环境变量

保护敏感的环境配置，防止其被 Agent 读取或暴露。

### `block-env-files`

**事件：** PreToolUse (Bash, Read)  
**默认行为：** 拒绝通过 `cat .env`、文件路径为 `.env` 的 Read 工具调用等方式读取 `.env` 文件。

不会拦截 `.envrc` 或其他与环境相关的文件，仅针对名称恰好为 `.env` 的文件。

无参数。

---

### `protect-env-vars`

**事件：** PreToolUse (Bash)  
**默认行为：** 拒绝打印环境变量的命令：`printenv`、`env`、`echo $VAR`。

无参数。

---

## 文件访问

将 Agent 限制在项目边界内，并远离敏感文件。

### `block-read-outside-cwd`

**事件：** PreToolUse (Read, Bash)  
**默认行为：** 拒绝读取项目根目录以外的文件。边界由 `CLAUDE_PROJECT_DIR`（由 Claude Code 在每次会话开始时设置一次）决定，若该变量未设置，则回退到会话的当前工作目录。使用项目根目录而非实时的 `cwd`，意味着即使 Claude `cd` 进入子目录后，边界依然保持稳定。

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|-------|------|---------|-------------|
| `allowPaths` | `string[]` | `[]` | 即便位于项目根目录之外也允许访问的绝对路径前缀。 |

**示例：**

```json
{
  "policyParams": {
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company/config"]
    }
  }
}
```

---

### `block-secrets-write`

**事件：** PreToolUse (Write, Edit)  
**默认行为：** 拒绝向常用于存储私钥和证书的文件写入内容：`id_rsa`、`id_ed25519`、`*.key`、`*.pem`、`*.p12`、`*.pfx`。

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|-------|------|---------|-------------|
| `additionalPatterns` | `string[]` | `[]` | 额外需要拦截的文件名模式（glob 风格）。 |

**示例：**

```json
{
  "policyParams": {
    "block-secrets-write": {
      "additionalPatterns": [".token", ".secret"]
    }
  }
}
```

## Git

防止意外推送、强制推送以及难以撤销的分支误操作。

### `block-push-master`

**事件：** PreToolUse (Bash)  
**默认行为：** 拒绝 `git push origin main` 和 `git push origin master`。

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|-------|------|---------|-------------|
| `protectedBranches` | `string[]` | `["main", "master"]` | 不允许直接推送的分支名称。 |

**示例：**

```json
{
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "master", "release", "prod"]
    }
  }
}
```

<Tip>
如果想允许推送到所有分支（实际上禁用此策略但不将其从 `enabledPolicies` 中移除），可将 `protectedBranches` 设置为 `[]`。
</Tip>

---

### `block-work-on-main`

**事件：** PreToolUse (Bash)  
**默认行为：** 当工作树位于 `main` 或 `master` 分支时，拒绝执行 `git commit`、`git merge`、`git rebase` 和 `git cherry-pick`。分支创建和切换（`git checkout`、`git checkout -b`、`git switch`、`git switch -c`）不受影响。

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|-------|------|---------|-------------|
| `protectedBranches` | `string[]` | `["main", "master"]` | 禁止执行 commit/merge/rebase/cherry-pick 的分支名称。 |

---

### `block-force-push`

**事件：** PreToolUse (Bash)  
**默认行为：** 拒绝 `git push --force` 和 `git push -f`。

无策略专属参数。可使用通用的 [`hint`](/zh/configuration#hint-cross-cutting) 建议替代方案：

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "Create a new branch from your current HEAD (e.g. `git checkout -b <new-branch>`) and push that instead."
    }
  }
}
```

---

### `warn-git-amend`

**事件：** PreToolUse (Bash)  
**默认行为：** 在 Claude 运行 `git commit --amend` 时，指示其谨慎执行。不阻止该命令。

无参数。

---

### `warn-git-stash-drop`

**事件：** PreToolUse (Bash)  
**默认行为：** 在 Claude 运行 `git stash drop` 前，指示其进行确认。不阻止该命令。

无参数。

---

### `warn-all-files-staged`

**事件：** PreToolUse (Bash)  
**默认行为：** 在 Claude 运行 `git add -A` 或 `git add .` 时，指示其检查将要暂存的内容。不阻止该命令。

无参数。

## 警告

在执行潜在风险但非破坏性的操作之前，为 Agent 提供额外上下文。

### `warn-large-file-write`

**事件：** PreToolUse (Write)  
**默认行为：** 在 Claude 写入超过 1024 KB 的文件前，指示其进行确认。

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|-------|------|---------|-------------|
| `thresholdKb` | `number` | `1024` | 触发警告的文件大小阈值（单位：KB）。 |

**示例：**

```json
{
  "policyParams": {
    "warn-large-file-write": {
      "thresholdKb": 256
    }
  }
}
```

<Note>
Hook 处理器对 payload 强制执行 1 MB 的 stdin 限制。如需用小内容测试此策略，请将 `thresholdKb` 设置为远低于 1024 的值。
</Note>

## 包管理器

强制规定 Agent 允许使用的包管理器。

### `prefer-package-manager`

**事件：** PreToolUse (Bash)  
**默认行为：** 禁用状态。启用后，会拦截不在 `allowed` 列表中的包管理器命令，并告知 Claude 使用允许的管理器重写该命令。

可检测：pip, pip3, python -m pip, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo。

| 参数 | 类型 | 默认值 | 说明 |
|-----------|------|---------|-------------|
| `allowed` | string[] | `[]` | 允许使用的包管理器名称。任何检测到的管理器若不在此列表中则会被拦截。列表为空时，该策略不执行任何操作。 |
| `blocked` | string[] | `[]` | 在内置拦截列表之外额外需要拦截的管理器名称（如 `['pdm', 'pipx']`）。 |

内置拦截列表涵盖：pip, pip3, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo。使用 `blocked` 可追加不在此列表中的管理器。

**配置示例：**

```json
{
  "enabledPolicies": ["prefer-package-manager"],
  "policyParams": {
    "prefer-package-manager": {
      "allowed": ["uv", "bun"],
      "blocked": ["pdm", "pipx"]
    }
  }
}
```

使用此配置，`pip install flask` 和 `pdm install flask` 都会被拒绝，并提示 Claude 改用 `uv` 或 `bun`。而 `uv pip install flask` 则会被允许，因为 `uv` 在白名单中且会被优先检查。

## 工作流

强制执行规范的会话结束工作流。这些策略在 **Stop** 事件上触发，阻止 Agent 在每项条件满足之前停止。它们遵循自然的依赖链：提交 → 推送 → PR → CI。如果某条策略拒绝，则链中后续策略会被跳过（拒绝即短路）。

所有工作流策略均为**开放失效（fail-open）**：如果所需工具不可用（如未安装 `gh`、无 git 远程仓库），策略将允许通过，并附上说明为何跳过检查的提示信息。

### 各 CLI 的 Stop 语义

由于各 CLI 暴露的"Agent 完成"Hook 合约不同，Stop 强制执行在七种受支持的 CLI 中的表现略有差异。**结果**是相同的——Agent 无法在工作流门控失败时停止——但**机制**有所不同。下表做了汇总；只有 Pi 存在一个值得在启用 `require-*-before-stop` 策略前了解的用户可见行为差异。

| CLI | 门控触发时机 | 你看到的效果 |
|---|---|---|
| Claude Code | 同一 Agent 循环，立即触发 | Claude 继续工作——修复问题后再次尝试结束。对你而言无任何中断。 |
| Codex | 同一 Agent 循环，立即触发 | 与 Claude 相同。 |
| GitHub Copilot CLI | 同一 Agent 循环，立即触发 | 与 Claude 相同（使用 Copilot 的 `{decision:"block", reason}` 重试通道——已针对 Copilot CLI 1.0.41 进行实证验证）。 |
| Cursor Agent | 同一 Agent 循环，立即触发 | 与 Claude 相同（使用 Cursor 的 `{followup_message}` 通道——受 `loop_limit` 限制，默认最多重试 5 次）。 |
| Gemini CLI | 同一 Agent 循环，立即触发 | 与 Claude 相同（在 `AfterAgent` 上使用 Gemini 的 `{decision:"block", reason}` 通道）。 |
| OpenCode | 同一 Agent 循环，立即触发 | 与 Claude 相同（使用 OpenCode 的 `client.session.prompt(...)` SDK 调用，通过 `hookSpecificOutput.additionalContext` 路由）。 |
| **Pi (pi-coding-agent)** | **下一个用户轮次** | **门控触发时 Pi 会可见地停止**——其 Agent 循环退出，控制权返还给提示符。门控会在下次你提交提示时再次触发：failproofai 会在该轮的系统提示前添加 `MANDATORY ACTION REQUIRED` 指令，要求 LLM 在执行你的请求之前先完成工作流步骤（提交、推送等）。 |

<Note>
**Pi 的局限性。** Pi 的 `AgentEndEvent`（上游等同于 Claude 的 `Stop` Hook）没有 Result 类型——当它触发时，Pi 的 Agent 循环已经退出。Pi 无法像 Claude / Copilot / Cursor / Gemini / OpenCode 那样被强制重试同一循环。failproofai 将门控转移到 Pi 的 `before_agent_start` 事件（在下一个用户提示后触发），使工作流检查仍能生效，只是在下一轮而非当前轮执行。

**实际影响：**

- Pi 停止后，拒绝原因会以 Pi 会话 id 为键存储在内存中。你在同一 Pi 进程中提交的下一个提示将消耗它：LLM 会在系统提示顶部看到 `MANDATORY ACTION REQUIRED` 指令，先完成提交（或推送/开 PR/等待 CI），然后再处理你的请求。已消耗的拒绝原因是一次性的——消耗后门控即清除。
- 门控受 Pi 进程生命周期约束。如果你在两轮之间 `Ctrl+C` 或退出 Pi，内存中的记录会随进程一同丢失，门控也就错过了。Claude、Copilot、Cursor、Gemini 和 OpenCode 有相同的约束（杀死 Agent 则门控错过）——Pi 只是因为 Agent 在门控触发前可见地退出，使这一点更加明显。
- 待处理的拒绝记录也会在任何原因导致的 `session_shutdown`（`new` / `resume` / `fork` / `quit`）时被清除，因此来自上一个会话的过期门控不会泄漏到同一 Pi 进程中新启动的会话。

如果你需要 Claude 风格的同循环重试，请在其他六种受支持的 CLI 下运行 `Stop` 策略。我们正在跟踪 Pi 上游，等待 `AgentEndEvent` 未来的 Result 类型，届时即可填补这一差距。
</Note>

### `require-commit-before-stop`

**事件：** Stop  
**默认行为：** 当存在未提交的更改（已修改、已暂存或未跟踪的文件）时，拒绝停止。工作目录干净时返回提示信息。

无参数。

### `require-push-before-stop`

**事件：** Stop  
**默认行为：** 当存在未推送的提交，或当前分支没有远程跟踪分支时，拒绝停止。如需要，建议使用 `git push -u` 创建跟踪分支。若未配置远程仓库则开放失效。

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|-------|------|---------|-------------|
| `remote` | `string` | `"origin"` | 推送目标的远程仓库名称。 |

**示例：**

```json
{
  "policyParams": {
    "require-push-before-stop": {
      "remote": "upstream"
    }
  }
}
```

### `require-pr-before-stop`

**事件：** Stop  
**默认行为：** 当当前分支不存在 Pull Request，或现有 PR 已关闭且未合并时，拒绝停止。指示 Claude 使用 `gh pr create` 创建 PR。当 PR 被**合并**后，策略允许通过（工作已交付），并提示切换回分支（`git checkout main && git pull`）。

无参数。

<Note>
此策略需要安装并完成身份验证的 [GitHub CLI](https://cli.github.com/)（`gh`）。
使用具有 `repo` 权限范围的个人访问 token 运行 `gh auth login` 以获取 Pull Request 读取权限。若 `gh` 未安装或未完成身份验证，策略将开放失效并向 Claude 报告原因。
</Note>

---

### `require-no-conflicts-before-stop`

**事件：** Stop  
**默认行为：** 当当前分支无法干净地合并到基础分支时，拒绝停止。策略首先确认 GitHub 上是否存在该分支的 `OPEN` PR——如果没有，则没有可强制执行的合并目标，整个策略直接短路为允许。一旦确认存在 `OPEN` PR，将独立运行两项探测：

1. **本地探测** — `git merge-tree --write-tree --name-only origin/<baseBranch> HEAD`。发生冲突时，拒绝消息中会列出冲突文件名，让 Claude 清楚地知道需要解决哪些冲突。
2. **GitHub 探测** — 复用在预检阶段已获取的 `gh pr view --json mergeable,state` 结果。可捕获因本地 `origin/<baseBranch>` 过期而遗漏的冲突（例如，自上次 fetch 以来已有冲突的 PR 合并到了 `main`）。`CONFLICTING` 结果会导致拒绝。`UNKNOWN` 结果同样会导致拒绝，并指示 Claude 等待约 10 秒后重新检查再尝试停止——这可防止在 GitHub 重新计算合并性时出现误判。

在以下情况下完全跳过（允许通过）：`gh` 未安装、该分支不存在 PR、PR 状态不为 `OPEN`（如 `MERGED`、`CLOSED`），或 `gh pr view` 返回无法解析的输出。当本地缺少 `origin/<baseBranch>` 或当前分支没有领先于基础分支的提交时同样开放失效——这些第一层回退仍会在允许前查询缓存的 PR 合并性。

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|-------|------|---------|-------------|
| `baseBranch` | `string` | `"main"` | 用于检查冲突的基础分支。 |

<Note>
此策略需要 GitHub CLI（`gh`）。策略使用 `gh pr view` 确认是否存在 `OPEN` PR，然后才运行冲突探测——如果没有 `gh`，策略将短路为允许。使用具有 `repo` 权限范围的个人访问 token 运行 `gh auth login` 以获取 Pull Request 读取权限。
</Note>

---

### `require-ci-green-before-stop`

**事件：** Stop  
**默认行为：** 当当前分支的 CI 检查正在失败或仍在运行时，拒绝停止。同时检查 GitHub Actions 工作流运行和第三方机器人检查（如 CodeRabbit、SonarCloud、Codecov）。将 `skipped` 和 `cancelled` 结论视为成功。所有检查通过时返回提示信息。

无参数。

<Note>
此策略需要安装并完成身份验证的 [GitHub CLI](https://cli.github.com/)（`gh`）。
使用具有 `repo` 权限范围的个人访问 token 运行 `gh auth login` 以获取 Actions 工作流运行和 Checks API 的读取权限。若 `gh` 未安装或未完成身份验证，策略将开放失效并向 Claude 报告原因。
</Note>

## 禁用单条策略

从配置中的 `enabledPolicies` 里移除特定策略，或在控制台的"策略"标签页中将其关闭。

```json
{
  "enabledPolicies": [
    "block-rm-rf",
    "sanitize-api-keys"
  ]
}
```

未列入 `enabledPolicies` 的策略不会运行，即使 `policyParams` 中存在对应的配置项。
