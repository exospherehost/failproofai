failproofai 内置了 39 条策略，用于拦截常见的 Agent 故障模式。每条策略针对特定的 Hook 事件类型和工具名称触发。其中 19 条策略支持参数配置，无需编写代码即可调整行为。5 条工作流策略在 Claude 停止之前强制执行 commit → push → PR → CI 流水线。

---

## 概览

策略按类别分组：

| 类别 | 策略 | Hook 类型 |
|------|------|-----------|
| [危险命令](#dangerous-commands) | block-sudo, block-rm-rf, block-curl-pipe-sh, block-failproofai-commands | PreToolUse |
| [基础设施命令](#infra-commands) | block-kubectl, block-terraform, block-aws-cli, block-gcloud, block-az-cli, block-helm, block-gh-pipeline | PreToolUse |
| [密钥（脱敏）](#secrets-sanitizers) | sanitize-jwt, sanitize-api-keys, sanitize-connection-strings, sanitize-private-key-content, sanitize-bearer-tokens | PostToolUse |
| [环境变量](#environment) | block-env-files, protect-env-vars | PreToolUse |
| [文件访问](#file-access) | block-read-outside-cwd, block-secrets-write | PreToolUse |
| [Git](#git) | block-push-master, block-work-on-main, block-force-push, warn-git-amend, warn-git-stash-drop, warn-all-files-staged | PreToolUse |
| [数据库](#database) | warn-destructive-sql, warn-schema-alteration | PreToolUse |
| [警告](#warnings) | warn-large-file-write, warn-package-publish, warn-background-process, warn-global-package-install | PreToolUse |
| [包管理器](#package-managers) | prefer-package-manager | PreToolUse |
| [工作流](#workflow) | require-commit-before-stop, require-push-before-stop, require-pr-before-stop, require-no-conflicts-before-stop, require-ci-green-before-stop | Stop |

- **`block-`** — 阻止 Agent 继续执行。
- **`warn-`** — 为 Agent 提供额外上下文，使其能够自我纠正。
- **`sanitize-`** — 在 Agent 看到工具输出之前，将其中的敏感数据脱敏处理。

### 命名空间

每条策略都位于 `<namespace>/<name>` 插槽中。内置策略属于 **`exospherehost/`** 命名空间，例如 `exospherehost/sanitize-jwt`。命名空间可防止在同时加载自定义策略或第三方策略时出现短名称冲突。

在配置中，既可以使用短名称，也可以使用完整限定名引用内置策略，两种形式解析到同一条策略：

```json
{
  "enabledPolicies": [
    "sanitize-jwt",
    "exospherehost/block-rm-rf"
  ]
}
```

如果名称中不含 `/`，failproofai 会将其视为属于默认命名空间 `exospherehost`。已包含 `/` 的名称（例如 `myorg/foo`、`custom/my-hook`）则保持原样。
- **`require-`** — 在满足条件之前阻止 Stop 事件。

---

<Tip>
每条策略在 `policyParams` 中都支持可选的 `hint` 字段。该提示会追加到 Claude 看到的 deny 或 instruct 消息中，提供可操作的指导，而无需修改策略代码。适用于内置策略、自定义策略和约定策略。详见 [配置 → hint](/zh/configuration#hint-cross-cutting)。
</Tip>

## 危险命令

防止 Agent 执行难以撤销或可能损坏宿主系统的操作。

### `block-sudo`

**事件：** PreToolUse (Bash)  
**默认行为：** 拒绝任何包含 `sudo` 的命令。

对解析后的命令 token 进行模式匹配（而非原始字符串），以防止通过 Shell 操作符注入绕过拦截。

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `allowPatterns` | `string[]` | `[]` | 允许执行的命令前缀。每条记录与解析后的 argv token 进行匹配。 |

**示例：**

```json
{
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    }
  }
}
```

使用此配置后，`sudo systemctl status nginx` 将被允许，但 `sudo rm /etc/hosts` 将被拒绝。

<Note>
模式匹配针对解析后的 token，而非原始命令字符串。这可防止通过附加 Shell 操作符绕过（例如 `sudo systemctl status x; rm -rf /` 不会匹配 `sudo systemctl status *`）。
</Note>

---

### `block-rm-rf`

**事件：** PreToolUse (Bash)  
**默认行为：** 拒绝 `rm -rf`、`rm -fr` 及类似的递归删除形式。

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `allowPaths` | `string[]` | `[]` | 允许递归删除的路径（例如 `/tmp`）。 |

**示例：**

```json
{
  "policyParams": {
    "block-rm-rf": {
      "allowPaths": ["/tmp", "/var/cache"]
    }
  }
}
```

---

### `block-curl-pipe-sh`

**事件：** PreToolUse (Bash)  
**默认行为：** 拒绝 `curl <url> | bash`、`curl <url> | sh`、`wget <url> | bash` 及类似模式。

无参数。

---

### `block-failproofai-commands`

**事件：** PreToolUse (Bash)  
**默认行为：** 拒绝会卸载或禁用 failproofai 自身的命令（例如 `npm uninstall failproofai`、`failproofai policies --uninstall`）。

无参数。

## 基础设施命令

防止编码 Agent 运行基础设施 CLI 或触发 CI/CD 流水线。此类别中的所有策略均为**按需启用**（`defaultEnabled: false`）——合理需要调用 `kubectl`、`terraform` 等工具的 Agent 不会受到干扰，除非显式启用该策略。启用后，除非命令匹配 `allowPatterns` 中的条目，否则所有匹配 CLI 的调用均会被拒绝。

模式语法与 [`block-sudo`](#block-sudo) 相同：token 与解析后的 argv 进行匹配，`*` 为单个 token 的通配符，任何包含独立 Shell 操作符（`&&`、`||`、`|`、`;`）或 token 中嵌入 Shell 元字符的命令，均会在白名单匹配前被拒绝，以防止注入绕过。

### `block-kubectl`

**事件：** PreToolUse (Bash)  
**默认行为：** 拒绝任何 `kubectl` 调用。

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `allowPatterns` | `string[]` | `[]` | 允许执行的 kubectl 命令前缀。 |

**示例：**

```json
{
  "policyParams": {
    "block-kubectl": {
      "allowPatterns": ["kubectl get *", "kubectl describe *", "kubectl logs *"]
    }
  }
}
```

使用此配置后，`kubectl get pods` 将被允许，但 `kubectl apply -f deploy.yaml` 将被拒绝。

### `block-gh-pipeline`

**事件：** PreToolUse (Bash)  
**默认行为：** 拒绝以下会改变状态或触发流水线的 `gh` CLI 子命令：

- `gh workflow run`、`gh workflow enable`、`gh workflow disable`
- `gh run rerun`、`gh run cancel`
- `gh pr merge`
- `gh release create`、`gh release delete`
- `gh cache delete`
- `gh secret set`、`gh secret delete`

只读的 `gh` 子命令（如 `gh pr view`、`gh pr list`、`gh run list`、`gh release view`、`gh api repos/.../...`）**不在**此策略的匹配范围内——它们在工作流检查中（包括 failproofai 自身的 `require-ci-green-before-stop`）是常规需求。

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `allowPatterns` | `string[]` | `[]` | 即使原本会被拒绝，也允许执行的特定脚本化调用。 |

**示例：**

```json
{
  "policyParams": {
    "block-gh-pipeline": {
      "allowPatterns": ["gh run rerun *"]
    }
  }
}
```

## 密钥（脱敏）

防止 Agent 将凭证泄露到其上下文或输出中。脱敏策略在 **PostToolUse** 事件上触发。当 Claude 执行 Bash 命令、读取文件或调用任何工具时，这些策略会在输出返回给 Claude 之前检查内容。如果检测到密钥模式，策略将返回拒绝决策，阻止输出被传回。

### `sanitize-jwt`

**事件：** PostToolUse（所有工具）  
**默认行为：** 脱敏 JWT token（由 `.` 分隔的三段 base64url 字符串）。

无参数。

---

### `sanitize-api-keys`

**事件：** PostToolUse（所有工具）  
**默认行为：** 脱敏常见 API Key 格式：Anthropic（`sk-ant-`）、OpenAI（`sk-`）、GitHub PAT（`ghp_`）、AWS 访问密钥（`AKIA`）、Stripe 密钥（`sk_live_`、`sk_test_`）以及 Google API 密钥（`AIza`）。

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `additionalPatterns` | `{ regex: string; label: string }[]` | `[]` | 额外需要视为密钥的正则表达式模式。 |

**示例：**

```json
{
  "policyParams": {
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo internal API key" },
        { "regex": "pat_[0-9a-f]{40}", "label": "Internal PAT" }
      ]
    }
  }
}
```

---

### `sanitize-connection-strings`

**事件：** PostToolUse（所有工具）  
**默认行为：** 脱敏包含内嵌凭证的数据库连接字符串（例如 `postgresql://user:password@host/db`）。

无参数。

---

### `sanitize-private-key-content`

**事件：** PostToolUse（所有工具）  
**默认行为：** 脱敏 PEM 块（`-----BEGIN PRIVATE KEY-----`、`-----BEGIN RSA PRIVATE KEY-----` 等）。

无参数。

---

### `sanitize-bearer-tokens`

**事件：** PostToolUse（所有工具）  
**默认行为：** 脱敏长度不少于 20 个字符的 `Authorization: Bearer <token>` 请求头。

无参数。

---

## 环境变量

防止 Agent 读取或暴露敏感的环境配置。

### `block-env-files`

**事件：** PreToolUse (Bash, Read)  
**默认行为：** 拒绝通过 `cat .env`、以 `.env` 为文件路径的 `Read` 工具调用等方式读取 `.env` 文件。

不拦截 `.envrc` 或其他相关环境文件，仅拦截名称完全为 `.env` 的文件。

无参数。

---

### `protect-env-vars`

**事件：** PreToolUse (Bash)  
**默认行为：** 拒绝打印环境变量的命令：`printenv`、`env`、`echo $VAR`。

无参数。

---

## 文件访问

将 Agent 的操作限制在项目目录内，并阻止其访问敏感文件。

### `block-read-outside-cwd`

**事件：** PreToolUse (Read, Bash)  
**默认行为：** 拒绝读取项目根目录以外的文件。边界由 `CLAUDE_PROJECT_DIR`（由 Claude Code 在每个会话开始时设置一次）确定，当该变量未设置时，回退到会话的当前工作目录。使用项目根目录而非实时 `cwd` 意味着边界保持稳定，即使 Claude `cd` 进入子目录也不会改变。

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `allowPaths` | `string[]` | `[]` | 即使在项目根目录之外也允许访问的绝对路径前缀。 |

**示例：**

```json
{
  "policyParams": {
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company/config"]
    }
  }
}
```

---

### `block-secrets-write`

**事件：** PreToolUse (Write, Edit)  
**默认行为：** 拒绝写入常用于私钥和证书的文件：`id_rsa`、`id_ed25519`、`*.key`、`*.pem`、`*.p12`、`*.pfx`。

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `additionalPatterns` | `string[]` | `[]` | 额外需要拦截的文件名模式（glob 风格）。 |

**示例：**

```json
{
  "policyParams": {
    "block-secrets-write": {
      "additionalPatterns": [".token", ".secret"]
    }
  }
}
```

## Git

防止意外推送、强制推送和难以撤销的分支操作失误。

### `block-push-master`

**事件：** PreToolUse (Bash)  
**默认行为：** 拒绝 `git push origin main` 和 `git push origin master`。

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `protectedBranches` | `string[]` | `["main", "master"]` | 不允许直接推送的分支名称。 |

**示例：**

```json
{
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "master", "release", "prod"]
    }
  }
}
```

<Tip>
若要允许推送到所有分支（即在不从 `enabledPolicies` 中移除该策略的情况下禁用其效果），可将 `protectedBranches` 设置为 `[]`。
</Tip>

---

### `block-work-on-main`

**事件：** PreToolUse (Bash)  
**默认行为：** 当工作树处于 `main` 或 `master` 分支时，拒绝执行 `git commit`、`git merge`、`git rebase` 和 `git cherry-pick`。创建分支和切换分支（`git checkout`、`git checkout -b`、`git switch`、`git switch -c`）不受影响。

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `protectedBranches` | `string[]` | `["main", "master"]` | 禁止执行 commit/merge/rebase/cherry-pick 的分支名称。 |

---

### `block-force-push`

**事件：** PreToolUse (Bash)  
**默认行为：** 拒绝 `git push --force` 和 `git push -f`。

无策略专属参数。可使用通用 [`hint`](/zh/configuration#hint-cross-cutting) 建议替代方案：

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "Create a new branch from your current HEAD (e.g. `git checkout -b <new-branch>`) and push that instead."
    }
  }
}
```

---

### `warn-git-amend`

**事件：** PreToolUse (Bash)  
**默认行为：** 当执行 `git commit --amend` 时，提示 Claude 谨慎操作。不阻止命令执行。

无参数。

---

### `warn-git-stash-drop`

**事件：** PreToolUse (Bash)  
**默认行为：** 在执行 `git stash drop` 前，提示 Claude 进行确认。不阻止命令执行。

无参数。

---

### `warn-all-files-staged`

**事件：** PreToolUse (Bash)  
**默认行为：** 当执行 `git add -A` 或 `git add .` 时，提示 Claude 检查暂存内容。不阻止命令执行。

无参数。

## 警告

在可能存在风险但非破坏性操作之前，为 Agent 提供额外上下文。

### `warn-large-file-write`

**事件：** PreToolUse (Write)  
**默认行为：** 在写入大于 1024 KB 的文件之前，提示 Claude 进行确认。

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `thresholdKb` | `number` | `1024` | 触发警告的文件大小阈值（单位：KB）。 |

**示例：**

```json
{
  "policyParams": {
    "warn-large-file-write": {
      "thresholdKb": 256
    }
  }
}
```

<Note>
Hook 处理器对 stdin 载荷强制限制为 1 MB。若要使用较小内容测试此策略，请将 `thresholdKb` 设置为远低于 1024 的值。
</Note>

## 包管理器

强制 Agent 只使用指定的包管理器。

### `prefer-package-manager`

**事件：** PreToolUse (Bash)  
**默认行为：** 禁用。启用后，拦截不在 `allowed` 列表中的包管理器命令，并告知 Claude 改用允许的包管理器重写该命令。

可检测的包管理器：pip, pip3, python -m pip, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo。

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `allowed` | string[] | `[]` | 允许使用的包管理器名称。检测到的包管理器若不在此列表中则被拦截。为空时，策略不生效。 |
| `blocked` | string[] | `[]` | 在内置列表之外额外需要拦截的包管理器名称（例如 `['pdm', 'pipx']`）。 |

内置拦截列表包含：pip, pip3, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo。使用 `blocked` 追加不在此列表中的包管理器。

**配置示例：**

```json
{
  "enabledPolicies": ["prefer-package-manager"],
  "policyParams": {
    "prefer-package-manager": {
      "allowed": ["uv", "bun"],
      "blocked": ["pdm", "pipx"]
    }
  }
}
```

使用此配置后，`pip install flask` 和 `pdm install flask` 均会被拒绝，并告知 Claude 改用 `uv` 或 `bun`。`uv pip install flask` 这类命令则被允许，因为 `uv` 在允许列表中且优先被检查。

## 工作流

强制执行规范的会话结束工作流。这些策略在 **Stop** 事件上触发，并在每个条件满足之前阻止 Claude 停止。它们遵循自然的依赖链：commit → push → PR → CI。若某条策略拒绝，则后续策略将被跳过（拒绝短路）。

所有工作流策略均为**失败放行**：如果所需工具不可用（例如未安装 `gh`、无 git remote），策略将放行并返回信息性消息说明跳过检查的原因。

### `require-commit-before-stop`

**事件：** Stop  
**默认行为：** 当存在未提交的变更（已修改、已暂存或未跟踪的文件）时，拒绝停止。工作目录干净时返回信息性消息。

无参数。

---

### `require-push-before-stop`

**事件：** Stop  
**默认行为：** 当存在未推送的提交，或当前分支没有远程跟踪分支时，拒绝停止。如有需要，建议使用 `git push -u` 创建跟踪分支。未配置 remote 时失败放行。

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `remote` | `string` | `"origin"` | 推送目标的 remote 名称。 |

**示例：**

```json
{
  "policyParams": {
    "require-push-before-stop": {
      "remote": "upstream"
    }
  }
}
```

---

### `require-pr-before-stop`

**事件：** Stop  
**默认行为：** 当当前分支不存在 Pull Request，或已有的 PR 未合并即关闭时，拒绝停止。提示 Claude 使用 `gh pr create` 创建 PR。当 PR 已**合并**时，策略放行（工作已交付），并提示切换回主分支（`git checkout main && git pull`）。

无参数。

<Note>
此策略需要安装并完成认证的 [GitHub CLI](https://cli.github.com/)（`gh`）。
使用具有 `repo` 作用域的个人访问令牌运行 `gh auth login`，以获取 Pull Request 的读取权限。如果未安装 `gh` 或未完成认证，策略将失败放行并向 Claude 报告原因。
</Note>

---

### `require-no-conflicts-before-stop`

**事件：** Stop  
**默认行为：** 当当前分支无法干净地合并到基础分支时，拒绝停止。策略首先确认该分支在 GitHub 上存在 `OPEN` 状态的 PR——若不存在，则没有需要强制执行的合并目标，整条策略短路放行。确认存在 `OPEN` PR 后，将独立执行两项检查：

1. **本地检查** — `git merge-tree --write-tree --name-only origin/<baseBranch> HEAD`。发生冲突时，拒绝消息会列出冲突文件，让 Claude 明确知道需要解决哪些问题。
2. **GitHub 检查** — 复用前置检查中已获取的 `gh pr view --json mergeable,state` 结果。可捕获本地过期 `origin/<baseBranch>` 会遗漏的冲突（例如自上次 fetch 后，有人将存在冲突的 PR 合并到 `main`）。结果为 `CONFLICTING` 则拒绝；结果为 `UNKNOWN` 也拒绝，并指示 Claude 等待约 10 秒后重新检查再尝试停止——防止 GitHub 重新计算期间出现假阴性。

以下情况会完全跳过此策略（放行）：未安装 `gh`、该分支不存在 PR、PR 状态不是 `OPEN`（例如 `MERGED`、`CLOSED`），或 `gh pr view` 返回无法解析的输出。当本地缺少 `origin/<baseBranch>` 或没有领先于基础分支的提交时也会失败放行——这些第一层回退仍会在放行前查询缓存的 PR 可合并性。

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `baseBranch` | `string` | `"main"` | 用于检查冲突的基础分支。 |

<Note>
此策略需要 GitHub CLI（`gh`）。策略使用 `gh pr view` 确认存在 `OPEN` PR 后才运行冲突检查——未安装 `gh` 时策略短路放行。使用具有 `repo` 作用域的个人访问令牌运行 `gh auth login`，以获取 Pull Request 的读取权限。
</Note>

---

### `require-ci-green-before-stop`

**事件：** Stop  
**默认行为：** 当当前分支的 CI 检查失败或仍在运行时，拒绝停止。同时检查 GitHub Actions 工作流运行状态和第三方机器人检查（例如 CodeRabbit、SonarCloud、Codecov）。将 `skipped` 和 `cancelled` 结论视为成功。所有检查通过时返回信息性消息。

无参数。

<Note>
此策略需要安装并完成认证的 [GitHub CLI](https://cli.github.com/)（`gh`）。
使用具有 `repo` 作用域的个人访问令牌运行 `gh auth login`，以获取 Actions 工作流运行状态和 Checks API 的读取权限。如果未安装 `gh` 或未完成认证，策略将失败放行并向 Claude 报告原因。
</Note>

## 禁用单条策略

在配置文件的 `enabledPolicies` 中移除特定策略，或在控制台的"策略"标签页中将其关闭。

```json
{
  "enabledPolicies": [
    "block-rm-rf",
    "sanitize-api-keys"
  ]
}
```

未列在 `enabledPolicies` 中的策略不会运行，即使 `policyParams` 中存在对应条目也不例外。
