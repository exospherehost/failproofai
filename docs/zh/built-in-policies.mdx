failproofai 内置 39 条策略，用于捕获常见的 Agent 故障模式。每条策略针对特定的 Hook 事件类型和工具名称触发。其中 19 条策略支持参数配置，无需编写代码即可调整行为。5 条工作流策略在 Claude 停止前强制执行提交 → 推送 → PR → CI 的流水线。

---

## 概览

策略按类别分组：

| 类别 | 策略 | Hook 类型 |
|------|------|-----------|
| [危险命令](#dangerous-commands) | block-sudo, block-rm-rf, block-curl-pipe-sh, block-failproofai-commands | PreToolUse |
| [基础设施命令](#infra-commands) | block-kubectl, block-terraform, block-aws-cli, block-gcloud, block-az-cli, block-helm, block-gh-pipeline | PreToolUse |
| [密钥（清洗器）](#secrets-sanitizers) | sanitize-jwt, sanitize-api-keys, sanitize-connection-strings, sanitize-private-key-content, sanitize-bearer-tokens | PostToolUse |
| [环境变量](#environment) | block-env-files, protect-env-vars | PreToolUse |
| [文件访问](#file-access) | block-read-outside-cwd, block-secrets-write | PreToolUse |
| [Git](#git) | block-push-master, block-work-on-main, block-force-push, warn-git-amend, warn-git-stash-drop, warn-all-files-staged | PreToolUse |
| [数据库](#database) | warn-destructive-sql, warn-schema-alteration | PreToolUse |
| [警告](#warnings) | warn-large-file-write, warn-package-publish, warn-background-process, warn-global-package-install | PreToolUse |
| [包管理器](#package-managers) | prefer-package-manager | PreToolUse |
| [工作流](#workflow) | require-commit-before-stop, require-push-before-stop, require-pr-before-stop, require-no-conflicts-before-stop, require-ci-green-before-stop | Stop |

- **`block-`** — 阻止 Agent 继续执行。
- **`warn-`** — 为 Agent 提供额外上下文，使其能够自我纠正。
- **`sanitize-`** — 在 Agent 看到工具输出前，清除其中的敏感数据。

### 命名空间

每条策略都位于 `<namespace>/<name>` 槽中。内置策略属于 **`exospherehost/`** 命名空间——例如 `exospherehost/sanitize-jwt`。命名空间可防止与短名称相似的自定义或第三方策略发生冲突。

在配置中，您可以使用短名称或完整限定名引用内置策略，两种形式解析结果相同：

```json
{
  "enabledPolicies": [
    "sanitize-jwt",
    "exospherehost/block-rm-rf"
  ]
}
```

若名称中不含 `/`，failproofai 会将其视为属于默认命名空间 `exospherehost`。已包含 `/` 的名称（如 `myorg/foo`、`custom/my-hook`）保持原样不变。
- **`require-`** — 阻止 Stop 事件，直到满足条件为止。

---

<Tip>
每条策略在 `policyParams` 中都支持可选的 `hint` 字段。该提示会追加到 Claude 收到的 deny 或 instruct 消息中，提供可操作的指导，无需修改策略代码。适用于内置策略、自定义策略和约定策略。详见[配置 → hint](/zh/configuration#hint-cross-cutting)。
</Tip>

## 危险命令

阻止 Agent 执行难以撤销或可能损害宿主系统的操作。

### `block-sudo`

**事件：** PreToolUse (Bash)  
**默认行为：** 拒绝任何包含 `sudo` 的命令。

对包含 `sudo` 关键字的调用进行拦截。匹配基于解析后的命令词元（token），而非原始字符串，以防止通过 Shell 操作符注入绕过检测。

**参数：**

| 参数 | 类型 | 默认值 | 描述 |
|------|------|--------|------|
| `allowPatterns` | `string[]` | `[]` | 允许执行的命令前缀（精确匹配）。每条记录与解析后的 argv 词元进行匹配。 |

**示例：**

```json
{
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    }
  }
}
```

使用此配置，`sudo systemctl status nginx` 将被允许，但 `sudo rm /etc/hosts` 将被拒绝。

<Note>
模式与解析后的词元匹配，而非原始命令字符串。这可防止通过追加 Shell 操作符绕过（例如 `sudo systemctl status x; rm -rf /` 不会匹配 `sudo systemctl status *`）。
</Note>

### `block-rm-rf`

**事件：** PreToolUse (Bash)  
**默认行为：** 拒绝 `rm -rf`、`rm -fr` 及类似的递归删除形式。

**参数：**

| 参数 | 类型 | 默认值 | 描述 |
|------|------|--------|------|
| `allowPaths` | `string[]` | `[]` | 允许递归删除的安全路径（如 `/tmp`）。 |

**示例：**

```json
{
  "policyParams": {
    "block-rm-rf": {
      "allowPaths": ["/tmp", "/var/cache"]
    }
  }
}
```

**默认行为：** 拒绝 `curl <url> | bash`、`curl <url> | sh`、`wget <url> | bash` 及类似模式。

无参数。

## 基础设施命令

阻止编码 Agent 运行基础设施 CLI 或触发 CI/CD 流水线。此类别中的所有策略均为**按需启用**（`defaultEnabled: false`）——合理需要调用 `kubectl`、`terraform` 等命令的 Agent 不会受到干扰，除非您手动启用该策略。启用后，所有匹配 CLI 的调用均会被拒绝，除非命令与 `allowPatterns` 中的条目匹配。

模式语法与 [`block-sudo`](#block-sudo) 相同：词元与解析后的 argv 匹配，`*` 为单个词元通配符，任何包含独立 Shell 操作符（`&&`、`||`、`|`、`;`）或含有内嵌 Shell 元字符词元的命令，在白名单匹配之前即会被拒绝，以防止注入绕过。

### `block-kubectl`

**事件：** PreToolUse (Bash)  
**默认行为：** 拒绝任何 `kubectl` 调用。

**参数：**

| 参数 | 类型 | 默认值 | 描述 |
|------|------|--------|------|
| `allowPatterns` | `string[]` | `[]` | 允许执行的 kubectl 命令前缀。 |

**示例：**

```json
{
  "policyParams": {
    "block-kubectl": {
      "allowPatterns": ["kubectl get *", "kubectl describe *", "kubectl logs *"]
    }
  }
}
```

使用此配置，`kubectl get pods` 将被允许，但 `kubectl apply -f deploy.yaml` 将被拒绝。

### `block-gh-pipeline`

**事件：** PreToolUse (Bash)  
**默认行为：** 拒绝以下会改变状态或触发流水线的 `gh` CLI 子命令：

- `gh workflow run`、`gh workflow enable`、`gh workflow disable`
- `gh run rerun`、`gh run cancel`
- `gh pr merge`
- `gh release create`、`gh release delete`
- `gh cache delete`
- `gh secret set`、`gh secret delete`

只读的 `gh` 子命令（如 `gh pr view`、`gh pr list`、`gh run list`、`gh release view` 和 `gh api repos/.../...`）**不在**此策略的匹配范围内——这些命令通常用于工作流检查（包括 failproofai 自身的 `require-ci-green-before-stop`）。

**参数：**

| 参数 | 类型 | 默认值 | 描述 |
|------|------|--------|------|
| `allowPatterns` | `string[]` | `[]` | 即使原本会被拒绝，也允许执行的特定脚本调用。 |

**示例：**

```json
{
  "policyParams": {
    "block-gh-pipeline": {
      "allowPatterns": ["gh run rerun *"]
    }
  }
}
```

## 密钥（清洗器）

阻止 Agent 将凭据泄露到其上下文或输出中。清洗器策略在 **PostToolUse** 事件上触发。当 Claude 运行 Bash 命令、读取文件或调用任何工具时，这些策略会在输出返回给 Claude 之前对其进行检查。如果检测到密钥模式，策略将返回拒绝决定，阻止输出被传回。

### `sanitize-jwt`

**事件：** PostToolUse（所有工具）  
**默认行为：** 清除 JWT 令牌（由 `.` 分隔的三段 base64url 字符串）。

无参数。

---

### `sanitize-api-keys`

**事件：** PostToolUse（所有工具）  
**默认行为：** 清除常见的 API 密钥格式：Anthropic（`sk-ant-`）、OpenAI（`sk-`）、GitHub PAT（`ghp_`）、AWS 访问密钥（`AKIA`）、Stripe 密钥（`sk_live_`、`sk_test_`）以及 Google API 密钥（`AIza`）。

**参数：**

| 参数 | 类型 | 默认值 | 描述 |
|------|------|--------|------|
| `additionalPatterns` | `{ regex: string; label: string }[]` | `[]` | 额外的正则表达式模式，用于标识需要清除的密钥。 |

**示例：**

```json
{
  "policyParams": {
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo internal API key" },
        { "regex": "pat_[0-9a-f]{40}", "label": "Internal PAT" }
      ]
    }
  }
}
```

---

### `sanitize-connection-strings`

**事件：** PostToolUse（所有工具）  
**默认行为：** 清除包含内嵌凭据的数据库连接字符串（如 `postgresql://user:password@host/db`）。

无参数。

---

### `sanitize-private-key-content`

**事件：** PostToolUse（所有工具）  
**默认行为：** 清除 PEM 块（`-----BEGIN PRIVATE KEY-----`、`-----BEGIN RSA PRIVATE KEY-----` 等）。

无参数。

---

### `sanitize-bearer-tokens`

**事件：** PostToolUse（所有工具）  
**默认行为：** 清除 `Authorization: Bearer <token>` 标头中长度不少于 20 个字符的令牌。

无参数。

---

## 环境变量

保护敏感的环境配置，防止 Agent 读取或暴露这些信息。

### `block-env-files`

**事件：** PreToolUse (Bash, Read)  
**默认行为：** 拒绝通过 `cat .env`、以 `.env` 为文件路径的 `Read` 工具调用等方式读取 `.env` 文件。

不会阻止 `.envrc` 或其他与环境相关的文件，仅针对名称完全为 `.env` 的文件。

无参数。

---

### `protect-env-vars`

**事件：** PreToolUse (Bash)  
**默认行为：** 拒绝打印环境变量的命令：`printenv`、`env`、`echo $VAR`。

无参数。

---

## 文件访问

将 Agent 的操作范围限制在项目目录内，避免其访问敏感文件。

### `block-read-outside-cwd`

**事件：** PreToolUse (Read, Bash)  
**默认行为：** 拒绝读取项目根目录以外的文件。边界为 `CLAUDE_PROJECT_DIR`（由 Claude Code 在每次会话开始时设置），若该变量未设置，则回退到会话的当前工作目录。使用项目根目录而非实时 `cwd` 可确保边界稳定，即使 Claude `cd` 进入子目录后也不会改变。

**参数：**

| 参数 | 类型 | 默认值 | 描述 |
|------|------|--------|------|
| `allowPaths` | `string[]` | `[]` | 即使位于项目根目录之外，也允许访问的绝对路径前缀。 |

**示例：**

```json
{
  "policyParams": {
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company/config"]
    }
  }
}
```

---

### `block-secrets-write`

**事件：** PreToolUse (Write, Edit)  
**默认行为：** 拒绝写入常用于私钥和证书的文件：`id_rsa`、`id_ed25519`、`*.key`、`*.pem`、`*.p12`、`*.pfx`。

**参数：**

| 参数 | 类型 | 默认值 | 描述 |
|------|------|--------|------|
| `additionalPatterns` | `string[]` | `[]` | 额外的文件名模式（glob 风格）用于阻止写入。 |

**示例：**

```json
{
  "policyParams": {
    "block-secrets-write": {
      "additionalPatterns": [".token", ".secret"]
    }
  }
}
```

## Git

防止意外推送、强制推送及难以撤销的分支操作失误。

### `block-push-master`

**事件：** PreToolUse (Bash)  
**默认行为：** 拒绝 `git push origin main` 和 `git push origin master`。

**参数：**

| 参数 | 类型 | 默认值 | 描述 |
|------|------|--------|------|
| `protectedBranches` | `string[]` | `["main", "master"]` | 不允许直接推送的分支名称。 |

**示例：**

```json
{
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "master", "release", "prod"]
    }
  }
}
```

<Tip>
若要允许推送到所有分支（即在不从 `enabledPolicies` 中移除该策略的情况下禁用它），请将 `protectedBranches` 设置为 `[]`。
</Tip>

---

### `block-work-on-main`

**事件：** PreToolUse (Bash)  
**默认行为：** 当工作树位于 `main` 或 `master` 分支时，拒绝执行 `git commit`、`git merge`、`git rebase` 和 `git cherry-pick`。创建分支和切换分支（`git checkout`、`git checkout -b`、`git switch`、`git switch -c`）不受影响。

**参数：**

| 参数 | 类型 | 默认值 | 描述 |
|------|------|--------|------|
| `protectedBranches` | `string[]` | `["main", "master"]` | 禁止在其上执行 commit/merge/rebase/cherry-pick 的分支名称。 |

---

### `block-force-push`

**事件：** PreToolUse (Bash)  
**默认行为：** 拒绝 `git push --force` 和 `git push -f`。

无策略专属参数。可使用跨策略的 [`hint`](/zh/configuration#hint-cross-cutting) 建议替代方案：

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "Create a new branch from your current HEAD (e.g. `git checkout -b <new-branch>`) and push that instead."
    }
  }
}
```

---

### `warn-git-amend`

**事件：** PreToolUse (Bash)  
**默认行为：** 在运行 `git commit --amend` 时，指示 Claude 谨慎操作。不会阻止该命令执行。

无参数。

---

### `warn-git-stash-drop`

**事件：** PreToolUse (Bash)  
**默认行为：** 在运行 `git stash drop` 前，指示 Claude 进行确认。不会阻止该命令执行。

无参数。

---

### `warn-all-files-staged`

**事件：** PreToolUse (Bash)  
**默认行为：** 在运行 `git add -A` 或 `git add .` 时，指示 Claude 审查将要暂存的内容。不会阻止该命令执行。

无参数。

## 警告

在 Agent 执行潜在风险但非破坏性的操作之前，为其提供额外的上下文信息。

### `warn-large-file-write`

**事件：** PreToolUse (Write)  
**默认行为：** 在写入大于 1024 KB 的文件之前，指示 Claude 进行确认。

**参数：**

| 参数 | 类型 | 默认值 | 描述 |
|------|------|--------|------|
| `thresholdKb` | `number` | `1024` | 触发警告的文件大小阈值（单位：KB）。 |

**示例：**

```json
{
  "policyParams": {
    "warn-large-file-write": {
      "thresholdKb": 256
    }
  }
}
```

<Note>
Hook 处理程序对 stdin 负载强制执行 1 MB 的限制。若要使用小内容测试此策略，请将 `thresholdKb` 设置为远低于 1024 的值。
</Note>

## 包管理器

强制规定 Agent 可以使用的包管理器。

### `prefer-package-manager`

**事件：** PreToolUse (Bash)  
**默认行为：** 已禁用。启用后，将阻止使用不在 `allowed` 列表中的包管理器命令，并告知 Claude 使用允许的包管理器重写该命令。

检测范围：pip, pip3, python -m pip, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo。

| 参数 | 类型 | 默认值 | 描述 |
|------|------|--------|------|
| `allowed` | string[] | `[]` | 允许使用的包管理器名称列表。未在此列表中的已检测到的包管理器将被阻止。为空时，该策略不执行任何操作。 |
| `blocked` | string[] | `[]` | 除内置列表外，额外需要阻止的包管理器名称（如 `['pdm', 'pipx']`）。 |

内置阻止列表涵盖：pip, pip3, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo。使用 `blocked` 追加不在此列表中的包管理器。

**配置示例：**

```json
{
  "enabledPolicies": ["prefer-package-manager"],
  "policyParams": {
    "prefer-package-manager": {
      "allowed": ["uv", "bun"],
      "blocked": ["pdm", "pipx"]
    }
  }
}
```

使用此配置，`pip install flask` 和 `pdm install flask` 都将被拒绝，并提示 Claude 改用 `uv` 或 `bun`。`uv pip install flask` 之类的命令则被允许，因为 `uv` 在允许列表中且会被优先检查。

## 工作流

强制执行规范的会话结束工作流。这些策略在 **Stop** 事件上触发，阻止 Agent 停止，直到满足每个条件为止。它们遵循自然的依赖链：提交 → 推送 → PR → CI。若某条策略拒绝，链中后续策略将被跳过（拒绝会短路后续检查）。

所有工作流策略均为**失败开放（fail-open）**：如果所需工具不可用（如未安装 `gh`、无 git 远程仓库），策略将允许通过并附带说明信息，解释检查为何被跳过。

### 各 CLI 的 Stop 语义

Stop 强制执行在七种支持的 CLI 中表现略有不同，因为每种 CLI 暴露的"Agent 完成"Hook 合约不同。**结果**是相同的——Agent 在工作流门控失败时无法停止——但**机制**有所差异。下表做了总结，只有 Pi 有一个用户可见的特殊情况，在启用 `require-*-before-stop` 策略前值得了解。

| CLI | 门控触发时机 | 用户所见 |
|---|---|---|
| Claude Code | 同一 Agent 循环，立即触发 | Claude 继续工作——修复问题后再次尝试完成。用户无感知中断。 |
| Codex | 同一 Agent 循环，立即触发 | 与 Claude 相同。 |
| GitHub Copilot CLI | 同一 Agent 循环，立即触发 | 与 Claude 相同（使用 Copilot 的 `{decision:"block", reason}` 重试通道——已针对 Copilot CLI 1.0.41 进行实证验证）。 |
| Cursor Agent | 同一 Agent 循环，立即触发 | 与 Claude 相同（使用 Cursor 的 `{followup_message}` 通道——上限为 `loop_limit`，默认 5 次重试）。 |
| Gemini CLI | 同一 Agent 循环，立即触发 | 与 Claude 相同（在 `AfterAgent` 上使用 Gemini 的 `{decision:"block", reason}` 通道）。 |
| OpenCode | 同一 Agent 循环，立即触发 | 与 Claude 相同（通过 `hookSpecificOutput.additionalContext` 路由使用 OpenCode 的 `client.session.prompt(...)` SDK 调用）。 |
| **Pi (pi-coding-agent)** | **下一次用户轮次** | **Pi 会明显停止**——门控触发时其 Agent 循环退出，用户被返回到提示符。下一次提交提示时门控触发：failproofai 在该轮次的系统提示前添加 `MANDATORY ACTION REQUIRED` 指令，要求 LLM 在执行您的请求之前完成工作流步骤（提交、推送等）。 |

<Note>
**Pi 的限制。** Pi 的 `AgentEndEvent`（上游等同于 Claude 的 `Stop` Hook）没有 Result 类型——触发时 Pi 的 Agent 循环已经退出。Pi 无法像 Claude / Copilot / Cursor / Gemini / OpenCode 那样被强制重试同一循环。failproofai 将门控转移到 Pi 的 `before_agent_start` 事件（在下一次用户提示后触发），使工作流检查仍能生效，只是推迟到下一轮而非当前轮。

**实际影响：**

- Pi 停止后，拒绝原因以 Pi 会话 ID 为键保存在内存中。您在同一 Pi 进程中提交的下一个提示会触发消耗：LLM 会在系统提示顶部看到 `MANDATORY ACTION REQUIRED` 指令，完成提交（或推送/开启 PR/等待 CI），然后再继续执行您的请求。这个捕获的拒绝原因是一次性的——消耗后门控即清除。
- 门控受 Pi 进程生命周期的限制。如果您在两轮之间 `Ctrl+C` 退出 Pi，内存中的条目会随进程一起丢失，门控也就错过了。Claude、Copilot、Cursor、Gemini 和 OpenCode 也有同样的限制（杀死 Agent 则门控错过）——Pi 只是因为 Agent 在门控触发前明显退出而使这一点更加显眼。
- 待处理的拒绝在任何原因的 `session_shutdown` 时也会被清除（`new` / `resume` / `fork` / `quit`），因此来自先前会话的过期门控不会泄漏到同一 Pi 进程中启动的新会话。

如果您需要类似 Claude 的同循环重试，请在其他六种支持的 CLI 下运行您的 `Stop` 策略。我们正在跟踪 Pi 上游，等待 `AgentEndEvent` 未来增加 Result 类型，届时可以弥合这一差距。
</Note>

### `require-commit-before-stop`

**事件：** Stop  
**默认行为：** 当存在未提交的更改（已修改、已暂存或未跟踪的文件）时，拒绝停止。工作目录干净时返回信息提示。

无参数。

### `require-push-before-stop`

**事件：** Stop  
**默认行为：** 当存在未推送的提交，或当前分支没有远程跟踪分支时，拒绝停止。如需要，建议使用 `git push -u` 创建跟踪分支。若未配置远程仓库，则失败开放。

**参数：**

| 参数 | 类型 | 默认值 | 描述 |
|------|------|--------|------|
| `remote` | `string` | `"origin"` | 推送的目标远程仓库名称。 |

**示例：**

```json
{
  "policyParams": {
    "require-push-before-stop": {
      "remote": "upstream"
    }
  }
}
```

### `require-pr-before-stop`

**事件：** Stop  
**默认行为：** 当当前分支不存在 Pull Request，或现有 PR 已关闭且未合并时，拒绝停止。指示 Claude 使用 `gh pr create` 创建 PR。当 PR **已合并**时，策略允许通过（工作已发布），并提示切换离开该分支（`git checkout main && git pull`）。

无参数。

<Note>
此策略需要安装并认证 [GitHub CLI](https://cli.github.com/)（`gh`）。
请使用具有 `repo` 范围（用于读取 Pull Request）的个人访问令牌运行 `gh auth login`。若未安装 `gh` 或未完成认证，策略将失败开放并将原因告知 Claude。
</Note>

---

### `require-no-conflicts-before-stop`

**事件：** Stop  
**默认行为：** 当当前分支无法与基础分支进行干净合并时，拒绝停止。策略首先确认该分支在 GitHub 上是否存在 `OPEN` 状态的 PR——若无，则没有需要强制执行的合并目标，整个策略短路为允许。一旦确认存在 `OPEN` PR，将运行两项独立探测：

1. **本地探测** — `git merge-tree --write-tree --name-only origin/<baseBranch> HEAD`。发生冲突时，拒绝消息会列出冲突文件，让 Claude 清楚地知道需要解决哪些问题。
2. **GitHub 探测** — 复用预检中已获取的 `gh pr view --json mergeable,state` 结果。可捕获本地过期的 `origin/<baseBranch>` 会遗漏的冲突（例如，自上次 fetch 以来有人在 `main` 上合并了一个冲突 PR）。`CONFLICTING` 结果将被拒绝。`UNKNOWN` 结果也会被拒绝，并指示 Claude 等待约 10 秒后再次检查，然后再尝试停止——这可以防止在 GitHub 重新计算合并状态时产生误判。

以下情况策略将完全跳过（允许通过）：未安装 `gh`、分支不存在 PR、PR 状态非 `OPEN`（如 `MERGED`、`CLOSED`），或 `gh pr view` 返回无法解析的输出。当 `origin/<baseBranch>` 在本地不存在，或没有领先于基础分支的提交时，同样失败开放——这些 Layer 1 的回退情况在允许前仍会查阅缓存的 PR 可合并性。

**参数：**

| 参数 | 类型 | 默认值 | 描述 |
|------|------|--------|------|
| `baseBranch` | `string` | `"main"` | 用于检查冲突的基础分支。 |

<Note>
此策略需要 GitHub CLI（`gh`）。策略使用 `gh pr view` 确认存在 `OPEN` PR 后才会运行任何冲突探测——若无 `gh`，策略将短路为允许。请使用具有 `repo` 范围（用于读取 Pull Request）的个人访问令牌运行 `gh auth login`。
</Note>

---

### `require-ci-green-before-stop`

**事件：** Stop  
**默认行为：** 当当前分支的 CI 检查失败或仍在运行时，拒绝停止。同时检查 GitHub Actions 工作流运行和第三方机器人检查（如 CodeRabbit、SonarCloud、Codecov）。将 `skipped` 和 `cancelled` 结论视为成功。所有检查通过时返回信息提示。

无参数。

<Note>
此策略需要安装并认证 [GitHub CLI](https://cli.github.com/)（`gh`）。
请使用具有 `repo` 范围（用于读取 Actions 工作流运行和 Checks API）的个人访问令牌运行 `gh auth login`。若未安装 `gh` 或未完成认证，策略将失败开放并将原因告知 Claude。
</Note>

## 禁用单个策略

在配置的 `enabledPolicies` 中移除特定策略，或在控制台的策略选项卡中将其关闭。

```json
{
  "enabledPolicies": [
    "block-rm-rf",
    "sanitize-api-keys"
  ]
}
```

未列入 `enabledPolicies` 的策略不会运行，即使 `policyParams` 中存在对应配置项也不例外。
