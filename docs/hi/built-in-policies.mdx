failproofai 39 बिल्ट-इन पॉलिसीज़ के साथ आता है जो एजेंट की आम विफलताओं को पकड़ते हैं। प्रत्येक पॉलिसी एक विशिष्ट हुक इवेंट टाइप और टूल नाम पर फायर करती है। उन्नीस पॉलिसीज़ पैरामीटर स्वीकार करती हैं जो आपको बिना कोड लिखे उनके व्यवहार को ट्यून करने देते हैं। पाँच वर्कफ़्लो पॉलिसीज़ एक commit → push → PR → CI पाइपलाइन को लागू करती हैं जिससे Claude रुकने से पहले इसे पूरा करे।

---

## अवलोकन

पॉलिसीज़ को श्रेणियों में बाँटा गया है:

| श्रेणी | पॉलिसीज़ | हुक प्रकार |
|----------|----------|-----------|
| [खतरनाक कमांड्स](#dangerous-commands) | block-sudo, block-rm-rf, block-curl-pipe-sh, block-failproofai-commands | PreToolUse |
| [इन्फ्रा कमांड्स](#infra-commands) | block-kubectl, block-terraform, block-aws-cli, block-gcloud, block-az-cli, block-helm, block-gh-pipeline | PreToolUse |
| [सीक्रेट्स (सैनिटाइज़र्स)](#secrets-sanitizers) | sanitize-jwt, sanitize-api-keys, sanitize-connection-strings, sanitize-private-key-content, sanitize-bearer-tokens | PostToolUse |
| [वातावरण](#environment) | block-env-files, protect-env-vars | PreToolUse |
| [फाइल एक्सेस](#file-access) | block-read-outside-cwd, block-secrets-write | PreToolUse |
| [गिट](#git) | block-push-master, block-work-on-main, block-force-push, warn-git-amend, warn-git-stash-drop, warn-all-files-staged | PreToolUse |
| [डेटाबेस](#database) | warn-destructive-sql, warn-schema-alteration | PreToolUse |
| [चेतावनियाँ](#warnings) | warn-large-file-write, warn-package-publish, warn-background-process, warn-global-package-install | PreToolUse |
| [पैकेज मैनेजर्स](#package-managers) | prefer-package-manager | PreToolUse |
| [वर्कफ़्लो](#workflow) | require-commit-before-stop, require-push-before-stop, require-pr-before-stop, require-no-conflicts-before-stop, require-ci-green-before-stop | Stop |

- **`block-`** — एजेंट को आगे बढ़ने से रोकता है।
- **`warn-`** — एजेंट को अतिरिक्त संदर्भ देता है ताकि वह स्वयं को सुधार सके।
- **`sanitize-`** — टूल आउटपुट से संवेदनशील डेटा को स्क्रब करता है जिससे पहले एजेंट इसे न देखे।

### नेमस्पेस

प्रत्येक पॉलिसी `<namespace>/<name>` स्लॉट में रहती है। बिल्ट-इन पॉलिसीज़ **`failproofai/`** नेमस्पेस से संबंधित होती हैं — उदाहरण के लिए, `failproofai/sanitize-jwt`। नेमस्पेस तब टकराव को रोकता है जब आप कस्टम या थर्ड-पार्टी पॉलिसीज़ को समान छोटे नामों के साथ लोड करते हैं।

अपने कॉन्फ़िग में आप एक बिल्ट-इन को उसके छोटे नाम या योग्य नाम से संदर्भित कर सकते हैं; दोनों रूप एक ही पॉलिसी को हल करते हैं:

```json
{
  "enabledPolicies": [
    "sanitize-jwt",
    "failproofai/block-rm-rf"
  ]
}
```

यदि किसी नाम में `/` नहीं है, तो failproofai इसे डिफ़ॉल्ट नेमस्पेस `failproofai` से संबंधित मानता है। ऐसे नाम जिनमें पहले से `/` है (जैसे `myorg/foo`, `custom/my-hook`) जैसे-जैसे हैं।

- **`require-`** — शर्तें पूरी होने तक Stop इवेंट को ब्लॉक करता है।

---

<Tip>
प्रत्येक पॉलिसी `policyParams` में एक वैकल्पिक `hint` फील्ड को समर्थन करती है। hint को deny या instruct संदेश में जोड़ा जाता है जो Claude देखता है, जिससे पॉलिसी कोड को संशोधित किए बिना कार्रवाई योग्य मार्गदर्शन मिलता है। बिल्ट-इन, कस्टम और कन्वेंशन पॉलिसीज़ के साथ काम करता है। विवरण के लिए [Configuration → hint](/hi/configuration#hint-cross-cutting) देखें।
</Tip>

## खतरनाक कमांड्स

एजेंट्स को ऐसी ऑपरेशन चलाने से रोकें जो वापस लाना मुश्किल है या होस्ट सिस्टम को नुकसान पहुंचा सकती है।

### `block-sudo`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** किसी भी `sudo` कमांड को नकारता है।

ऐसी कॉल्स को ब्लॉक करता है जिनमें `sudo` कीवर्ड शामिल है। पैटर्न मिलान raw स्ट्रिंग पर नहीं बल्कि पार्स की गई कमांड टोकन्स पर किया जाता है, जिससे शेल ऑपरेटर इंजेक्शन के माध्यम से बाईपास को रोका जा सके।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | सटीक कमांड प्रीफिक्स जो अनुमति प्राप्त हैं। प्रत्येक प्रविष्टि को पार्स किए गए argv टोकन्स के विरुद्ध मिलाया जाता है। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    }
  }
}
```

इस कॉन्फ़िग के साथ, `sudo systemctl status nginx` की अनुमति है, लेकिन `sudo rm /etc/hosts` नहीं।

<Note>
पैटर्न्स को raw कमांड स्ट्रिंग पर नहीं बल्कि पार्स किए गए टोकन्स के विरुद्ध मिलाया जाता है। यह appended शेल ऑपरेटर्स के माध्यम से बाईपास को रोकता है (जैसे `sudo systemctl status x; rm -rf /` को `sudo systemctl status *` से मेल नहीं खाता)।
</Note>

---

### `block-rm-rf`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** `rm -rf`, `rm -fr` और इसी तरह के रिकर्सिव डिलीशन फॉर्म्स को नकारता है।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `allowPaths` | `string[]` | `[]` | पाथ्स जो रिकर्सिव्ली डिलीट करने के लिए सुरक्षित हैं (जैसे `/tmp`)। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-rm-rf": {
      "allowPaths": ["/tmp", "/var/cache"]
    }
  }
}
```

कोई पैरामीटर नहीं।

## इन्फ्रा कमांड्स

कोडिंग एजेंट्स को इन्फ्रास्ट्रक्चर CLIs चलाने या CI/CD पाइपलाइन्स को ट्रिगर करने से रोकें। इस श्रेणी में सभी पॉलिसीज़ **opt-in** हैं (`defaultEnabled: false`) — एजेंट्स जिन्हें वास्तव में `kubectl`, `terraform`, आदि को कॉल करने की आवश्यकता है, वे बाधित नहीं होंगे जब तक आप पॉलिसी को सक्षम न करें। जब सक्षम हो, तो मिलान किए गए CLI के प्रत्येक आह्वान को नकार दिया जाता है जब तक कि कमांड `allowPatterns` में किसी प्रविष्टि से मेल न खाए।

पैटर्न ग्रामर [`block-sudo`](#block-sudo) जैसा ही है: टोकन्स को पार्स किए गए argv के विरुद्ध मिलाया जाता है, `*` एक टोकन के लिए वाइल्डकार्ड है, और कोई भी कमांड जिसमें स्टैंडअलोन शेल ऑपरेटर (`&&`, `||`, `|`, `;`) या embedded शेल मेटाकैरेक्टर्स वाला टोकन है, इंजेक्शन बाईपास को रोकने के लिए allowlist मिलान से पहले अस्वीकृत किया जाता है।

इस कॉन्फ़िग के साथ, `kubectl get pods` की अनुमति है लेकिन `kubectl apply -f deploy.yaml` नहीं।

- `gh release create`, `gh release delete`

**उदाहरण:**

```json

    }
  }
}
```

## सीक्रेट्स (सैनिटाइज़र्स)

एजेंट्स को क्रेडेंशियल्स को अपने संदर्भ या आउटपुट में लीक करने से रोकें। सैनिटाइज़र पॉलिसीज़ **PostToolUse** इवेंट्स पर फायर करती हैं। जब Claude Bash कमांड चलाता है, फाइल पढ़ता है, या किसी भी टूल को कॉल करता है, तो ये पॉलिसीज़ आउटपुट को निरीक्षण करती हैं इससे पहले कि इसे Claude को वापस किया जाए। यदि कोई सीक्रेट पैटर्न पाया जाता है, तो पॉलिसी एक नकार निर्णय लौटाती है जो आउटपुट को वापस पास किए जाने से रोकता है।

### `sanitize-jwt`

**इवेंट:** PostToolUse (सभी टूल्स)  
**डिफ़ॉल्ट:** JWT टोकन्स को रिडैक्ट करता है (`.` से अलग किए गए तीन base64url सेगमेंट्स)।

कोई पैरामीटर नहीं।

---

### `sanitize-api-keys`

**इवेंट:** PostToolUse (सभी टूल्स)  
**डिफ़ॉल्ट:** सामान्य API की फॉर्मेट्स को रिडैक्ट करता है: Anthropic (`sk-ant-`), OpenAI (`sk-`), GitHub PATs (`ghp_`), AWS एक्सेस कीज़ (`AKIA`), Stripe कीज़ (`sk_live_`, `sk_test_`), और Google API कीज़ (`AIza`)।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `additionalPatterns` | `{ regex: string; label: string }[]` | `[]` | अतिरिक्त regex पैटर्न्स सीक्रेट्स के रूप में ट्रीट करने के लिए। |

**उदाहरण:**

```json
{
  "policyParams": {
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo internal API key" },
        { "regex": "pat_[0-9a-f]{40}", "label": "Internal PAT" }
      ]
    }
  }
}
```

---

### `sanitize-connection-strings`

**इवेंट:** PostToolUse (सभी टूल्स)  
**डिफ़ॉल्ट:** डेटाबेस कनेक्शन स्ट्रिंग्स को रिडैक्ट करता है जिनमें embedded क्रेडेंशियल्स होते हैं (जैसे `postgresql://user:password@host/db`)।

कोई पैरामीटर नहीं।

---

### `sanitize-private-key-content`

**इवेंट:** PostToolUse (सभी टूल्स)

## पैकेज मैनेजर्स

लागू करें कि एजेंट किन पैकेज मैनेजर्स का उपयोग करने की अनुमति दी जाती है।

### `prefer-package-manager`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** अक्षम। जब सक्षम हो, `allowed` सूची में कोई भी पैकेज मैनेजर कमांड को ब्लॉक करता है और Claude को अनुमत मैनेजर का उपयोग करके कमांड को फिर से लिखने के लिए कहता है।

पहचानता है: pip, pip3, python -m pip, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo।

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-----------|------|---------|-------------|
| `allowed` | string[] | `[]` | अनुमत पैकेज मैनेजर नाम। इस सूची में न आने वाला कोई भी पहचाना हुआ मैनेजर ब्लॉक किया जाता है। जब empty हो, तो पॉलिसी एक no-op है। |
| `blocked` | string[] | `[]` | बिल्ट-इन सूची के अलावा ब्लॉक करने के लिए अतिरिक्त मैनेजर नाम (जैसे `['pdm', 'pipx']`)। |

बिल्ट-इन ब्लॉक सूची शामिल है: pip, pip3, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo। इस सूची में न आने वाले मैनेजर्स को जोड़ने के लिए `blocked` का उपयोग करें।

**उदाहरण कॉन्फ़िगरेशन:**

```json
{
  "enabledPolicies": ["prefer-package-manager"],
  "policyParams": {
    "prefer-package-manager": {
      "allowed": ["uv", "bun"],
      "blocked": ["pdm", "pipx"]
    }
  }
}
```

इस कॉन्फ़िग के साथ, `pip install flask` और `pdm install flask` दोनों को नकार दिया जाता है जिसमें Claude को `uv` या `bun` का उपयोग करने का संदेश दिया जाता है। `uv pip install flask` जैसी कमांड्स की अनुमति है क्योंकि `uv` allowlist में है और पहले चेक किया जाता है।

## वर्कफ़्लो

अनुशासित end-of-session वर्कफ़्लो को लागू करें। ये पॉलिसीज़ **Stop** इवेंट पर फायर करती हैं और एजेंट को रोकने से इनकार करती हैं जब तक प्रत्येक शर्त पूरी न हो जाए। वे एक प्राकृतिक dependency chain का अनुसरण करते हैं: commit → push → PR → CI। यदि कोई पॉलिसी नकारती है, तो chain में बाद की पॉलिसीज़ को छोड़ दिया जाता है (deny short-circuits)।

सभी वर्कफ़्लो पॉलिसीज़ **fail-open** हैं: यदि आवश्यक टूल उपलब्ध नहीं है (जैसे `gh` installed नहीं, कोई git remote नहीं), तो पॉलिसी एक informational संदेश के साथ अनुमति देती है जिसमें बताया जाता है कि चेक को क्यों छोड़ा गया।

### प्रति-CLI Stop सिमेंटिक्स

Stop enforcement सात समर्थित CLIs में थोड़ा अलग दिखता है क्योंकि प्रत्येक एक अलग agent finished hook contract को expose करता है। **परिणाम** समान है — एजेंट workflow gate विफल रहते हुए stop नहीं होता — लेकिन **mechanics** अलग होते हैं। नीचे दी गई तालिका एक summary प्रदान करती है; केवल Pi के पास एक user-visible quirk है जो आप एक `require-*-before-stop` पॉलिसी को enable करने से पहले समझना चाहते हैं।

| CLI | gate कब फायर करता है | आप क्या देखते हैं |
|---|---|---|
| Claude Code | एक ही agent loop, तुरंत | Claude काम करना जारी रखता है — समस्या को ठीक करता है, फिर फिर से finish करने का प्रयास करता है। आपके लिए कोई व्यवधान दृश्यमान नहीं। |
| Codex | एक ही agent loop, तुरंत | Claude जैसा। |
| GitHub Copilot CLI | एक ही agent loop, तुरंत | Claude जैसा (Copilot के `{decision:"block", reason}` retry channel का उपयोग करता है — Copilot CLI 1.0.41 के विरुद्ध empirically verified)। |
| Cursor Agent | एक ही agent loop, तुरंत | Claude जैसा (Cursor के `{followup_message}` channel का उपयोग करता है — `loop_limit` पर capped, डिफ़ॉल्ट 5 retries)। |
| Gemini CLI | एक ही agent loop, तुरंत | Claude जैसा (Gemini के `{decision:"block", reason}` channel का उपयोग करता है `AfterAgent` पर)। |
| OpenCode | एक ही agent loop, तुरंत | Claude जैसा (OpenCode के `client.session.prompt(...)` SDK call का उपयोग करता है `hookSpecificOutput.additionalContext` के माध्यम से routed)। |
| **Pi (pi-coding-agent)** | **अगला user turn** | **Pi visibly stop होता है** जब gate फायर करता है — इसका agent loop exit हो जाता है और आप prompt में वापस आ जाते हैं। gate फिर अगली बार फायर करता है जब आप एक prompt submit करते हैं: failproofai एक `MANDATORY ACTION REQUIRED` directive को उस turn के system prompt में prepend करता है, LLM को workflow step (commit, push, आदि) को पूरा करने का निर्देश देता है जो आप माँग रहे हैं उससे पहले। |

<Note>
**Pi सीमा।** Pi के `AgentEndEvent` (Claude के `Stop` hook का upstream समतुल्य) के पास कोई Result type नहीं है — जब तक यह फायर करता है, Pi का agent loop पहले ही exit हो चुका होता है। Pi को Claude / Copilot / Cursor / Gemini / OpenCode की तरह एक ही loop को retry करने के लिए force नहीं किया जा सकता। failproofai gate को Pi के `before_agent_start` event में shift करता है (जो अगले user prompt के बाद फायर करता है) ताकि workflow check अभी भी लागू हो, बस वर्तमान के बजाय अगले turn पर।

**इसका व्यावहारिक अर्थ क्या है:**

- Pi के stop करने के बाद, deny reason को Pi session id द्वारा keyed memory में captured किया जाता है। अगली prompt जो आप एक ही Pi process में submit करते हैं इसे drain करता है: LLM अपने system prompt के top पर `MANDATORY ACTION REQUIRED` directive देखता है, commit करता है (या push / PR खोलता है / CI के लिए प्रतीक्षा करता है), और फिर आपके request के साथ continue करता है। captured deny reason one-shot है — एक बार drain होने के बाद, gate clear है।
- Gate Pi के process lifetime द्वारा bounded है। यदि आप Pi के बीच `Ctrl+C` करते हैं या quit करते हैं, तो in-memory entry process के साथ dropped हो जाता है और gate miss हो जाता है। Claude, Copilot, Cursor, Gemini, और OpenCode का एक ही bound है (agent को kill करें और gate miss हो जाता है) — Pi बस इसे अधिक दृश्यमान बनाता है क्योंकि agent visibly exit होता है gate फायर करने से पहले।
- एक pending deny को भी `session_shutdown` पर किसी भी कारण (`new` / `resume` / `fork` / `quit`) के लिए cleared किया जाता है, इसलिए एक prior session से stale gate एक fresh session में leak नहीं हो सकता जो एक ही Pi process में शुरू किया जाता है।

यदि आप Claude-style same-loop retry की आवश्यकता है, तो अपनी `Stop` पॉलिसीज़ को अन्य छः समर्थित CLIs में से किसी में भी चलाएं। हम Pi upstream को `AgentEndEvent` पर एक future Result type के लिए ट्रैक कर रहे हैं जो हमें इस gap को close करने देगा।
</Note>

### `require-commit-before-stop`

**इवेंट:** Stop  
**डिफ़ॉल्ट:** uncommitted changes (modified, staged, या untracked files) होने पर stop करने से नकारता है। working directory clean होने पर एक informational message return करता है।

### `require-push-before-stop`

**इवेंट:** Stop  
**डिफ़ॉल्ट:** unpushed commits होने पर या जब current branch के पास कोई remote tracking branch नहीं है तो stop करने से नकारता है। यदि आवश्यक हो तो `git push -u` का सुझाव देकर एक tracking branch create करता है। यदि कोई remote configured नहीं है तो fail open करता है।

      "remote": "upstream"
    }

### `require-pr-before-stop`

**इवेंट:** Stop  
**डिफ़ॉल्ट:** जब current branch के लिए कोई pull request नहीं है या जब existing PR बिना merge किए close हो गया है तो stop करने से नकारता है। Claude को `gh pr create` के साथ PR create करने का निर्देश देता है। जब PR **merged** है, तो पॉलिसी अनुमति देती है (काम ship हो चुका है) और संदेश hint देता है कि branch से switch करें (`git checkout main && git pull`)।

कोई पैरामीटर नहीं।

<Note>
यह पॉलिसी [GitHub CLI](https://cli.github.com/) (`gh`) को installed और authenticated होने की आवश्यकता है।
pull requests के लिए read access के साथ `repo` scope के साथ एक personal access token के साथ `gh auth login` चलाएं। यदि `gh` installed नहीं है या authenticated नहीं है, तो पॉलिसी fail open करती है और कारण को Claude को report करती है।
</Note>

### `require-no-conflicts-before-stop`

**इवेंट:** Stop  
**डिफ़ॉल्ट:** जब current branch cleanly base branch में merge नहीं हो सकता है तो stop करने से नकारता है। पॉलिसी पहले confirm करती है कि branch के लिए GitHub पर एक `OPEN` PR है — बिना एक के, कोई merge target नहीं है enforcement के लिए, इसलिए पूरी पॉलिसी short-circuits allow करने के लिए। एक बार `OPEN` PR confirm हो जाने के बाद, दो independent probes run करते हैं:

1. **Local** — `git merge-tree --write-tree --name-only origin/<baseBranch> HEAD`। Conflict पर, deny message conflicted files का नाम देता है ताकि Claude को पता हो कि क्या resolve करना है।
2. **GitHub** — precheck में पहले से fetch किए गए `gh pr view --json mergeable,state` result को reuse करता है। Conflicts को पकड़ता है जो एक stale local `origin/<baseBranch>` को miss करेगा (जैसे किसी ने `main` पर एक conflicting PR land किया है since last fetch)। एक `CONFLICTING` result नकारता है। एक `UNKNOWN` result भी नकारता है और Claude को ~10 सेकंड प्रतीक्षा करने और फिर से check करने के लिए निर्देश देता है stop attempt से पहले — यह false negatives को prevent करता है जबकि GitHub recomputes।

निम्न में skip करता है (allows): `gh` installed नहीं, branch के लिए कोई PR नहीं, PR की state `OPEN` नहीं है (जैसे `MERGED`, `CLOSED`), या `gh pr view` unparseable output return करता है। जब `origin/<baseBranch>` locally missing है या कोई commits base से ahead नहीं हैं तब भी Layer 1 fall-throughs cached PR mergeability को consult करते हैं allow से पहले।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `baseBranch` | `string` | `"main"` | Base branch conflicts के विरुद्ध check करने के लिए। |

<Note>
यह पॉलिसी [GitHub CLI](https://cli.github.com/) (`gh`) की आवश्यकता है। पॉलिसी किसी भी conflict probe को run करने से पहले एक `OPEN` PR exist करने को confirm करने के लिए `gh pr view` का उपयोग करती है — `gh` बिना, पॉलिसी short-circuits allow करती है। pull requests के लिए read access के साथ `repo` scope के साथ `gh auth login` चलाएं।
</Note>

## व्यक्तिगत पॉलिसीज़ को अक्षम करना

अपने config में `enabledPolicies` से एक specific पॉलिसी को remove करें, या dashboard के Policies tab में इसे toggle off करें।

```json
{
  "enabledPolicies": [
    "block-rm-rf",
    "sanitize-api-keys"
  ]
}
```

`enabledPolicies` में listed नहीं की गई पॉलिसीज़ run नहीं होती हैं, भले ही उनके लिए `policyParams` entries exist करें।
