failproofai 39 निर्मित नीतियों के साथ आता है जो एजेंट की सामान्य विफलता के तरीकों को पकड़ता है। प्रत्येक नीति एक विशिष्ट हुक इवेंट प्रकार और टूल नाम पर लागू होती है। उन्नीस नीतियां पैरामीटर स्वीकार करती हैं जो आपको कोड लिखे बिना उनके व्यवहार को ट्यून करने देते हैं। पांच वर्कफ़्लो नीतियां Claude के रुकने से पहले एक कमिट → पुश → PR → CI पाइपलाइन को लागू करते हैं।

---

## अवलोकन

नीतियां श्रेणियों में समूहीकृत हैं:

| श्रेणी | नीतियां | हुक प्रकार |
|----------|----------|-----------|
| [खतरनाक कमांड](#dangerous-commands) | block-sudo, block-rm-rf, block-curl-pipe-sh, block-failproofai-commands | PreToolUse |
| [इंफ्रा कमांड](#infra-commands) | block-kubectl, block-terraform, block-aws-cli, block-gcloud, block-az-cli, block-helm, block-gh-pipeline | PreToolUse |
| [सीक्रेट (सैनिटाइज़र)](#secrets-sanitizers) | sanitize-jwt, sanitize-api-keys, sanitize-connection-strings, sanitize-private-key-content, sanitize-bearer-tokens | PostToolUse |
| [पर्यावरण](#environment) | block-env-files, protect-env-vars | PreToolUse |
| [फ़ाइल एक्सेस](#file-access) | block-read-outside-cwd, block-secrets-write | PreToolUse |
| [गिट](#git) | block-push-master, block-work-on-main, block-force-push, warn-git-amend, warn-git-stash-drop, warn-all-files-staged | PreToolUse |
| [डेटाबेस](#database) | warn-destructive-sql, warn-schema-alteration | PreToolUse |
| [चेतावनियां](#warnings) | warn-large-file-write, warn-package-publish, warn-background-process, warn-global-package-install | PreToolUse |
| [पैकेज मैनेजर](#package-managers) | prefer-package-manager | PreToolUse |
| [वर्कफ़्लो](#workflow) | require-commit-before-stop, require-push-before-stop, require-pr-before-stop, require-no-conflicts-before-stop, require-ci-green-before-stop | Stop |

- **`block-`** — एजेंट को आगे बढ़ने से रोकता है।
- **`warn-`** — एजेंट को अतिरिक्त संदर्भ देता है ताकि वह स्वयं को सही कर सके।
- **`sanitize-`** — एजेंट को देखने से पहले टूल आउटपुट से संवेदनशील डेटा को स्क्रब करता है।

### नेमस्पेस

प्रत्येक नीति एक `<namespace>/<name>` स्लॉट में रहती है। निर्मित नीतियां **`failproofai/`** नेमस्पेस से संबंधित हैं — उदाहरण के लिए, `failproofai/sanitize-jwt`। नेमस्पेस उस समय टकराव को रोकता है जब आप समान छोटे नामों वाली कस्टम या तीसरे पक्ष की नीतियों को भी लोड करते हैं।

आपके कॉन्फ़िगरेशन में आप एक निर्मित को या तो इसके छोटे नाम से या इसके योग्य नाम से संदर्भित कर सकते हैं; दोनों रूप एक ही नीति को हल करते हैं:

```json
{
  "enabledPolicies": [
    "sanitize-jwt",
    "failproofai/block-rm-rf"
  ]
}
```

यदि किसी नाम में कोई `/` नहीं है, तो failproofai इसे डिफ़ॉल्ट नेमस्पेस `failproofai` से संबंधित मानता है। ऐसे नाम जिनमें पहले से ही `/` है (उदाहरण के लिए `myorg/foo`, `custom/my-hook`) जैसे हैं वैसे ही रखे जाते हैं।
- **`require-`** — Stop ईवेंट को तब तक ब्लॉक करता है जब तक शर्तें पूरी न हों।

---

<Tip>
प्रत्येक नीति `policyParams` में एक वैकल्पिक `hint` फ़ील्ड का समर्थन करती है। संकेत को deny या instruct संदेश में जोड़ा जाता है जो Claude देखता है, जिससे नीति कोड को संशोधित किए बिना कार्यकारी निर्देश दिए जाते हैं। निर्मित, कस्टम और परंपरा नीतियों के साथ काम करता है। विवरण के लिए [कॉन्फ़िगरेशन → hint](/hi/configuration#hint-cross-cutting) देखें।
</Tip>

## खतरनाक कमांड

एजेंट को ऐसे ऑपरेशन चलाने से रोकें जो पूर्ववत करना कठिन हो या जो होस्ट सिस्टम को नुकसान पहुंचा सकते हों।

### `block-sudo`

**ईवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** किसी भी `sudo` कमांड को अस्वीकार करता है।

`sudo` कीवर्ड युक्त आह्वानों को ब्लॉक करता है। पैटर्न मिलान कच्ची स्ट्रिंग पर नहीं बल्कि पार्स की गई कमांड टोकन पर किया जाता है, ताकि शेल ऑपरेटर इंजेक्शन के माध्यम से बाईपास को रोका जा सके।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | सटीक कमांड उपसर्ग जो अनुमत हैं। प्रत्येक प्रविष्टि को पार्स किए गए argv टोकन के विरुद्ध मिलाया जाता है। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    }
  }
}
```

इस कॉन्फ़िगरेशन के साथ, `sudo systemctl status nginx` अनुमत है, लेकिन `sudo rm /etc/hosts` अस्वीकृत है।

<Note>
पैटर्न को कच्ची कमांड स्ट्रिंग के विरुद्ध नहीं बल्कि पार्स किए गए टोकन के विरुद्ध मिलाया जाता है। यह अपेंडेड शेल ऑपरेटर के माध्यम से बाईपास को रोकता है (उदाहरण के लिए `sudo systemctl status x; rm -rf /` `sudo systemctl status *` से मेल नहीं खाता)।
</Note>

---

### `block-rm-rf`

**ईवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** `rm -rf`, `rm -fr` और समान पुनरावर्ती विलोपन रूपों को अस्वीकार करता है।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `allowPaths` | `string[]` | `[]` | पथ जो पुनरावर्ती विलोपन के लिए सुरक्षित हैं (उदाहरण के लिए `/tmp`)। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-rm-rf": {
      "allowPaths": ["/tmp", "/var/cache"]
    }
  }
}
```

कोई पैरामीटर नहीं।

कोई पैरामीटर नहीं।

## इंफ्रा कमांड

कोडिंग एजेंट को बुनियादी ढांचे CLIs चलाने या CI/CD पाइपलाइनों को ट्रिगर करने से रोकें। इस श्रेणी में सभी नीतियां **ऑप्ट-इन** हैं (`defaultEnabled: false`) — एजेंट जिन्हें वास्तव में `kubectl`, `terraform` आदि को कॉल करने की आवश्यकता है, वे बाधित नहीं होंगे जब तक आप नीति को सक्षम न करें। जब सक्षम हो, तो मिलान किए गए CLI का प्रत्येक आह्वान अस्वीकृत हो जाता है जब तक कि कमांड `allowPatterns` में एक प्रविष्टि से मेल न खाए।

पैटर्न व्याकरण [`block-sudo`](#block-sudo) के समान है: टोकन को पार्स किए गए argv के विरुद्ध मिलाया जाता है, `*` एक टोकन के लिए एक वाइल्डकार्ड है, और कोई भी कमांड जिसमें एक अकेला शेल ऑपरेटर (`&&`, `||`, `|`, `;`) या एम्बेडेड शेल मेटाकैरेक्टर वाला टोकन है, इंजेक्शन बाईपास को रोकने के लिए अनुमतिसूची मिलान से पहले अस्वीकार किया जाता है।

```json
    "block-kubectl": {
```

इस कॉन्फ़िगरेशन के साथ, `kubectl get pods` अनुमत है लेकिन `kubectl apply -f deploy.yaml` अस्वीकृत है।


## सीक्रेट (सैनिटाइज़र)

एजेंट को क्रेडेंशियल को अपने संदर्भ या आउटपुट में लीक करने से रोकें। सैनिटाइज़र नीतियां **PostToolUse** ईवेंट पर लागू होती हैं। जब Claude एक Bash कमांड चलाता है, एक फ़ाइल पढ़ता है, या कोई भी टूल कॉल करता है, तो ये नीतियां आउटपुट का निरीक्षण करती हैं इससे पहले कि यह Claude को वापस किया जाए। यदि एक सीक्रेट पैटर्न का पता चलता है, तो नीति एक अस्वीकार निर्णय लौटाती है जो आउटपुट को पास किए जाने से रोकता है।

### `sanitize-jwt`

**ईवेंट:** PostToolUse (सभी टूल)  
**डिफ़ॉल्ट:** JWT टोकन को रीडैक्ट करता है (तीन base64url सेगमेंट `.` द्वारा अलग किए गए)।

कोई पैरामीटर नहीं।

---

### `sanitize-api-keys`

**ईवेंट:** PostToolUse (सभी टूल)  
**डिफ़ॉल्ट:** सामान्य API कुंजी प्रारूप को रीडैक्ट करता है: Anthropic (`sk-ant-`), OpenAI (`sk-`), GitHub PATs (`ghp_`), AWS एक्सेस कुंजियां (`AKIA`), Stripe कुंजियां (`sk_live_`, `sk_test_`), और Google API कुंजियां (`AIza`)।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `additionalPatterns` | `{ regex: string; label: string }[]` | `[]` | सीक्रेट के रूप में व्यवहार करने के लिए अतिरिक्त regex पैटर्न। |

**उदाहरण:**

```json
{
  "policyParams": {
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo internal API key" },
        { "regex": "pat_[0-9a-f]{40}", "label": "Internal PAT" }
      ]
    }
  }
}
```

---

**ईवेंट:** PostToolUse (सभी टूल)  
**डिफ़ॉल्ट:** डेटाबेस कनेक्शन स्ट्रिंग्स को रीडैक्ट करता है जिनमें एम्बेडेड क्रेडेंशियल हैं (उदाहरण के लिए `postgresql://user:password@host/db`)।

कोई पैरामीटर नहीं।

### `sanitize-private-key-content`

**ईवेंट:** PostToolUse (सभी टूल)

## फ़ाइल एक्सेस

### `block-read-outside-cwd`

**ईवेंट:** PreToolUse (Read, Bash)  
**डिफ़ॉल्ट:** प्रोजेक्ट रूट के बाहर फ़ाइलों को पढ़ने को अस्वीकार करता है। सीमा `CLAUDE_PROJECT_DIR` है (Claude Code द्वारा सत्र के अनुसार सेट), जब वह चर अनसेट हो तो सत्र की वर्तमान कार्य निर्देशिका में fallback होता है। लाइव `cwd` के बजाय प्रोजेक्ट रूट का उपयोग करने का मतलब है कि Claude के `cd` करने के बाद भी सीमा स्थिर रहती है।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `allowPaths` | `string[]` | `[]` | निरपेक्ष पथ उपसर्ग जो प्रोजेक्ट रूट के बाहर होने पर भी अनुमत हैं। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company/config"]
    }
  }
}
```

**ईवेंट:** PreToolUse (Write, Edit)  
**डिफ़ॉल्ट:** निजी कुंजियों और प्रमाणपत्रों के लिए आमतौर पर उपयोग की जाने वाली फ़ाइलों में लिखने को अस्वीकार करता है: `id_rsa`, `id_ed25519`, `*.key`, `*.pem`, `*.p12`, `*.pfx`।

## पैकेज मैनेजर

निर्दिष्ट करें कि एजेंट किस पैकेज मैनेजर का उपयोग करने की अनुमति है।

### `prefer-package-manager`

**ईवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** अक्षम। जब सक्षम हो, तो `allowed` सूची में न होने वाली किसी भी पैकेज मैनेजर कमांड को ब्लॉक करता है और Claude को अनुमत मैनेजर का उपयोग करके कमांड को पुनः लिखने के लिए कहता है।

पता लगाता है: pip, pip3, python -m pip, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo।

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-----------|------|---------|-------------|
| `allowed` | string[] | `[]` | अनुमत पैकेज मैनेजर नाम। इस सूची में न होने वाला कोई भी पता लगाया गया मैनेजर ब्लॉक किया जाता है। जब खाली हो, तो नीति एक no-op है। |
| `blocked` | string[] | `[]` | अतिरिक्त मैनेजर नाम जो निर्मित सूची के परे ब्लॉक करने हैं (उदाहरण के लिए `['pdm', 'pipx']`)। |

निर्मित ब्लॉक सूची कवर करती है: pip, pip3, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo। इस सूची में न होने वाले मैनेजर को जोड़ने के लिए `blocked` का उपयोग करें।

**उदाहरण कॉन्फ़िगरेशन:**

```json
{
  "enabledPolicies": ["prefer-package-manager"],
  "policyParams": {
    "prefer-package-manager": {
      "allowed": ["uv", "bun"],
      "blocked": ["pdm", "pipx"]
    }
  }
}
```

इस कॉन्फ़िगरेशन के साथ, `pip install flask` और `pdm install flask` दोनों अस्वीकृत हैं और Claude को `uv` या `bun` का उपयोग करने का संदेश दिया जाता है। `uv pip install flask` जैसी कमांड अनुमत हैं क्योंकि `uv` अनुमतिसूची में है और पहले जांचा जाता है।

## वर्कफ़्लो

अनुशासित अंत-सत्र वर्कफ़्लो लागू करें। ये नीतियां **Stop** ईवेंट पर लागू होती हैं और एजेंट को तब तक रोकने से अस्वीकार करती हैं जब तक प्रत्येक शर्त पूरी न हो। वे एक प्राकृतिक निर्भरता श्रृंखला का पालन करते हैं: commit → push → PR → CI। यदि कोई नीति अस्वीकार करती है, तो श्रृंखला में बाद की नीतियां छोड़ दी जाती हैं (अस्वीकार शॉर्ट-सर्किट करता है)।

सभी वर्कफ़्लो नीतियां **fail-open** हैं: यदि आवश्यक टूल उपलब्ध नहीं है (उदाहरण के लिए `gh` स्थापित नहीं, कोई git दूरस्थ नहीं), तो नीति एक सूचना संदेश के साथ अनुमति देती है जो समझाता है कि जांच को क्यों छोड़ दिया गया।

### प्रति-CLI Stop शब्दार्थ

Stop प्रवर्तन सात समर्थित CLIs में थोड़ा भिन्न दिखता है क्योंकि प्रत्येक एक अलग "एजेंट समाप्त" हुक अनुबंध को उजागर करता है। **परिणाम** समान है — एजेंट वर्कफ़्लो गेट विफल होने के दौरान रुकने से नहीं बचता है — लेकिन **यांत्रिकी** अलग होती है। नीचे दी गई तालिका सारांश देती है; केवल Pi में एक उपयोगकर्ता-दृश्यमान विषमता है जो आप एक `require-*-before-stop` नीति को सक्षम करने से पहले समझना चाह सकते हैं।

| CLI | गेट कब लागू होता है | आप क्या देखते हैं |
|---|---|---|
| Claude Code | समान एजेंट लूप, तुरंत | Claude काम करना जारी रखता है — समस्या को ठीक करता है, फिर फिर से समाप्त करने का प्रयास करता है। आपको कोई व्यवधान दिखाई नहीं देता है। |
| Codex | समान एजेंट लूप, तुरंत | Claude के समान। |
| GitHub Copilot CLI | समान एजेंट लूप, तुरंत | Claude के समान (Copilot के `{decision:"block", reason}` retry चैनल का उपयोग करता है — Copilot CLI 1.0.41 के विरुद्ध अनुभवजन्य रूप से सत्यापित)। |
| Cursor Agent | समान एजेंट लूप, तुरंत | Claude के समान (Cursor के `{followup_message}` चैनल का उपयोग करता है — `loop_limit` सीमित, डिफ़ॉल्ट 5 retry)। |
| Gemini CLI | समान एजेंट लूप, तुरंत | Claude के समान (Gemini के `{decision:"block", reason}` चैनल का उपयोग करता है `AfterAgent`)। |
| OpenCode | समान एजेंट लूप, तुरंत | Claude के समान (OpenCode के `client.session.prompt(...)` SDK कॉल का उपयोग करता है `hookSpecificOutput.additionalContext` के माध्यम से रूट किया गया)। |
| **Pi (pi-coding-agent)** | **अगली उपयोगकर्ता बारी** | **Pi दृश्यमान रूप से रुकता है** जब गेट लागू होता है — इसका एजेंट लूप बाहर निकलता है और आपको प्रॉम्प्ट पर वापस किया जाता है। गेट तब अगली बार लागू होता है जब आप एक प्रॉम्प्ट जमा करते हैं: failproofai उस मोड़ के सिस्टम प्रॉम्प्ट के लिए एक `MANDATORY ACTION REQUIRED` निर्देश को प्रीपेंड करता है, LLM को वर्कफ़्लो चरण को पूरा करने का निर्देश देता है (कमिट, पुश, आदि) आप जो मांग रहे हैं उससे पहले। |

<Note>
**Pi सीमा।** Pi का `AgentEndEvent` (Claude के `Stop` हुक का अपस्ट्रीम समतुल्य) का कोई Result प्रकार नहीं है — जब तक यह लागू होता है, Pi का एजेंट लूप पहले से बाहर निकल गया है। Pi को उसी तरह retry के लिए मजबूर नहीं किया जा सकता जैसे Claude / Copilot / Cursor / Gemini / OpenCode कर सकते हैं। failproofai गेट को Pi के `before_agent_start` ईवेंट में स्थानांतरित करता है (जो अगले उपयोगकर्ता प्रॉम्प्ट के बाद लागू होता है) ताकि वर्कफ़्लो जांच अभी भी लागू हो, बस वर्तमान मोड़ के बजाय अगले मोड़ पर।

**व्यवहार में इसका अर्थ क्या है:**

- Pi रुकने के बाद, निषेध कारण को Pi सेशन ID द्वारा कुंजीकृत इन-मेमोरी में कैप्चर किया जाता है। आप उसी Pi प्रक्रिया में जो अगला प्रॉम्प्ट जमा करते हैं वह इसे ड्रेन करता है: LLM अपने सिस्टम प्रॉम्प्ट के शीर्ष पर `MANDATORY ACTION REQUIRED` निर्देश देखता है, कमिट (या पुश / PR खोलता है / CI के लिए प्रतीक्षा करता है), और केवल तब ही आपके अनुरोध के साथ जारी रहता है। कैप्चर किया गया निषेध कारण एक-शॉट है — एक बार ड्रेन किया गया, गेट स्पष्ट है।
- गेट Pi की प्रक्रिया आयु द्वारा सीमित है। यदि आप मोड़ के बीच Pi को `Ctrl+C` करते हैं या बाहर निकलते हैं, तो इन-मेमोरी प्रविष्टि प्रक्रिया के साथ गिरा दी जाती है और गेट मिस हो जाता है। Claude, Copilot, Cursor, Gemini, और OpenCode की एक ही बाध्यता है (एजेंट को मार डालो और गेट मिस हो जाता है) — Pi इसे अधिक दृश्यमान बनाता है क्योंकि एजेंट गेट लागू होने से पहले दृश्यमान रूप से बाहर निकलता है।
- एक लंबित निषेध भी किसी भी कारण से `session_shutdown` पर साफ़ किया जाता है (`new` / `resume` / `fork` / `quit`), इसलिए पूर्व सेशन से एक स्टेल गेट उसी Pi प्रक्रिया में शुरू किए गए ताज़े सेशन में लीक नहीं हो सकता है।

यदि आपको Claude-शैली समान-लूप retry की आवश्यकता है, तो अन्य छह समर्थित CLIs में से किसी के तहत अपनी `Stop` नीतियां चलाएं। हम Pi अपस्ट्रीम को ट्रैक कर रहे हैं `AgentEndEvent` पर एक भविष्य के लिए Result प्रकार के लिए जो हमें इस अंतर को बंद करने दे।
</Note>

### `require-no-conflicts-before-stop`

**ईवेंट:** Stop  
**डिफ़ॉल्ट:** रुकने को अस्वीकार करता है जब वर्तमान ब्रांच आधार ब्रांच में स्वच्छ रूप से merge नहीं हो सकता। नीति पहले पुष्टि करती है कि GitHub पर ब्रांच के लिए एक `OPEN` PR है — बिना एक के, कोई merge लक्ष्य नहीं है जिसे लागू करना है, इसलिए पूरी नीति short-circuit करके allow करती है। एक बार `OPEN` PR की पुष्टि हो जाने के बाद, दो स्वतंत्र जांचें चलती हैं:

1. **स्थानीय** — `git merge-tree --write-tree --name-only origin/<baseBranch> HEAD`। conflict पर, निषेध संदेश conflicted फ़ाइलों को नाम देता है ताकि Claude को पता चले कि वास्तव में क्या resolve करना है।
2. **GitHub** — पूर्वाभास में पहले से fetch किए गए `gh pr view --json mergeable,state` परिणाम को पुनः उपयोग करता है। Conflicts को पकड़ता है जो एक स्टेल स्थानीय `origin/<baseBranch>` को मिस करेगा (उदाहरण के लिए कोई `main` पर एक conflicting PR लैंड करता है जब से अंतिम fetch हुआ)। एक `CONFLICTING` परिणाम अस्वीकार करता है। एक `UNKNOWN` परिणाम भी अस्वीकार करता है और Claude को ~10 सेकंड प्रतीक्षा करने और फिर से जांचने से पहले रुकने का प्रयास करने का निर्देश देता है — यह GitHub recompute करते समय गलत नकारात्मक को रोकता है।

छोड़ता है पूरी तरह (अनुमति देता है) जब: `gh` स्थापित नहीं है, ब्रांच के लिए कोई PR मौजूद नहीं है, PR की स्थिति `OPEN` नहीं है (उदाहरण के लिए `MERGED`, `CLOSED`), या `gh pr view` unparseable आउटपुट लौटाता है। स्थानीय रूप से `origin/<baseBranch>` गायब होने या कोई कमिट आगे न होने पर भी fail open करता है — वे Layer 1 fall-through अभी भी allow करने से पहले कैश किए गए PR mergeability से परामर्श करते हैं।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `baseBranch` | `string` | `"main"` | Conflicts के लिए जांचने के लिए आधार ब्रांच। |

<Note>
यह नीति के लिए GitHub CLI (`gh`) आवश्यक है। नीति किसी भी conflict probe को चलाने से पहले पुष्टि करने के लिए `gh pr view` का उपयोग करता है कि एक `OPEN` PR मौजूद है — बिना `gh` के, नीति short-circuit करती है। एक व्यक्तिगत एक्सेस टोकन के साथ `gh auth login` चलाएं जिसमें pull requests और Checks API को पढ़ने के लिए एक्सेस के लिए `repo` स्कोप हो।
</Note>

## व्यक्तिगत नीतियों को अक्षम करना

अपने कॉन्फ़िगरेशन में `enabledPolicies` से एक विशिष्ट नीति को हटाएं, या डैशबोर्ड के Policies टैब में इसे बंद करें।

```json
{
  "enabledPolicies": [
    "block-rm-rf",
    "sanitize-api-keys"
  ]
}
```

`enabledPolicies` में सूचीबद्ध नहीं की गई नीतियां नहीं चलती हैं, भले ही उनके लिए `policyParams` प्रविष्टि मौजूद हों।
