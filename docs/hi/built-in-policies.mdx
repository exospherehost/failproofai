failproofai के साथ 39 बिल्ट-इन नीतियाँ आती हैं जो एजेंट की सामान्य विफलता मोड को पकड़ती हैं। प्रत्येक नीति एक विशिष्ट हुक इवेंट प्रकार और टूल नाम पर सक्रिय होती है। उन्नीस नीतियाँ पैरामीटर स्वीकार करती हैं जो आपको कोड लिखे बिना उनके व्यवहार को ट्यून करने देती हैं। पाँच वर्कफ़्लो नीतियाँ Claude के रुकने से पहले एक commit → push → PR → CI पाइपलाइन को लागू करती हैं।

---

## अवलोकन

नीतियों को श्रेणियों में विभाजित किया गया है:

| श्रेणी | नीतियाँ | हुक प्रकार |
|----------|----------|-----------|
| [खतरनाक कमांड](#dangerous-commands) | block-sudo, block-rm-rf, block-curl-pipe-sh, block-failproofai-commands | PreToolUse |
| [इन्फ्रा कमांड](#infra-commands) | block-kubectl, block-terraform, block-aws-cli, block-gcloud, block-az-cli, block-helm, block-gh-pipeline | PreToolUse |
| [सीक्रेट (सैनिटाइज़र)](#secrets-sanitizers) | sanitize-jwt, sanitize-api-keys, sanitize-connection-strings, sanitize-private-key-content, sanitize-bearer-tokens | PostToolUse |
| [पर्यावरण](#environment) | block-env-files, protect-env-vars | PreToolUse |
| [फाइल एक्सेस](#file-access) | block-read-outside-cwd, block-secrets-write | PreToolUse |
| [Git](#git) | block-push-master, block-work-on-main, block-force-push, warn-git-amend, warn-git-stash-drop, warn-all-files-staged | PreToolUse |
| [डेटाबेस](#database) | warn-destructive-sql, warn-schema-alteration | PreToolUse |
| [चेतावनियाँ](#warnings) | warn-large-file-write, warn-package-publish, warn-background-process, warn-global-package-install | PreToolUse |
| [पैकेज मैनेजर](#package-managers) | prefer-package-manager | PreToolUse |
| [वर्कफ़्लो](#workflow) | require-commit-before-stop, require-push-before-stop, require-pr-before-stop, require-no-conflicts-before-stop, require-ci-green-before-stop | Stop |

- **`block-`** — एजेंट को आगे बढ़ने से रोकता है।
- **`warn-`** — एजेंट को अतिरिक्त संदर्भ देता है ताकि वह स्वयं को ठीक कर सके।
- **`sanitize-`** — एजेंट द्वारा देखने से पहले टूल आउटपुट से संवेदनशील डेटा को हटा देता है।

### नेमस्पेस

प्रत्येक नीति एक `<namespace>/<name>` स्लॉट में रहती है। बिल्ट-इन नीतियाँ **`exospherehost/`** नेमस्पेस से संबंधित हैं — उदाहरण के लिए, `exospherehost/sanitize-jwt`। नेमस्पेस तब टकराव को रोकता है जब आप कस्टम या तीसरे पक्ष की नीतियों को समान छोटे नामों के साथ लोड करते हैं।

आपकी कॉन्फ़िगरेशन में आप एक बिल्ट-इन को या तो इसके छोटे नाम से या इसके योग्य नाम से संदर्भित कर सकते हैं; दोनों रूप एक ही नीति को हल करते हैं:

```json
{
  "enabledPolicies": [
    "sanitize-jwt",
    "exospherehost/block-rm-rf"
  ]
}
```

यदि किसी नाम में कोई `/` नहीं है, failproofai इसे डिफ़ॉल्ट नेमस्पेस `exospherehost` से संबंधित मानता है। नाम जिनमें पहले से `/` है (जैसे `myorg/foo`, `custom/my-hook`) जैसे-जैसे रखे जाते हैं।
- **`require-`** — Stop इवेंट को तब तक ब्लॉक करता है जब तक शर्तें पूरी न हों।

---

<Tip>
हर नीति `policyParams` में एक वैकल्पिक `hint` फील्ड का समर्थन करती है। संकेत deny या instruct संदेश में जोड़ा जाता है जो Claude देखता है, नीति कोड को संशोधित किए बिना कार्रवाई योग्य मार्गदर्शन देता है। बिल्ट-इन, कस्टम, और कन्वेंशन नीतियों के साथ काम करता है। विस्तार के लिए [कॉन्फ़िगरेशन → hint](/hi/configuration#hint-cross-cutting) देखें।
</Tip>

## खतरनाक कमांड

एजेंटों को ऐसी ऑपरेशन चलाने से रोकें जो पूर्ववत करना कठिन हो या जो होस्ट सिस्टम को नुकसान पहुँचा सकती हो।

### `block-sudo`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** कोई भी `sudo` कमांड को अस्वीकार करता है।

`sudo` कीवर्ड को शामिल करने वाले आह्वानों को अवरुद्ध करता है। पैटर्न मिलान कच्ची स्ट्रिंग पर नहीं बल्कि पार्स किए गए कमांड टोकन पर किया जाता है, शेल ऑपरेटर इंजेक्शन के माध्यम से बाईपास को रोकने के लिए।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | सटीक कमांड प्रीफिक्स जो अनुमत हैं। प्रत्येक प्रविष्टि पार्स किए गए argv टोकन के विरुद्ध मेल खाई जाती है। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    }
  }
}
```

इस कॉन्फ़िग के साथ, `sudo systemctl status nginx` की अनुमति है, लेकिन `sudo rm /etc/hosts` को अस्वीकार किया जाता है।

<Note>
पैटर्न को कच्ची कमांड स्ट्रिंग पर नहीं बल्कि पार्स किए गए टोकन के विरुद्ध मेल खाया जाता है। यह जोड़े गए शेल ऑपरेटर के माध्यम से बाईपास को रोकता है (उदाहरण के लिए `sudo systemctl status x; rm -rf /` `sudo systemctl status *` से मेल नहीं खाता)।
</Note>

---

### `block-rm-rf`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** `rm -rf`, `rm -fr`, और समान पुनरावर्ती विलोपन रूपों को अस्वीकार करता है।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `allowPaths` | `string[]` | `[]` | पथ जो पुनरावर्ती हटाने के लिए सुरक्षित हैं (जैसे `/tmp`)। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-rm-rf": {
      "allowPaths": ["/tmp", "/var/cache"]
    }
  }
}
```

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** `curl <url> | bash`, `curl <url> | sh`, `wget <url> | bash`, और समान पैटर्न को अस्वीकार करता है।

कोई पैरामीटर नहीं।

### `block-failproofai-commands`

**इवेंट:** PreToolUse (Bash)

कोई पैरामीटर नहीं।

## इन्फ्रा कमांड

कोडिंग एजेंटों को इंफ्रास्ट्रक्चर CLI चलाने या CI/CD पाइपलाइन को ट्रिगर करने से रोकें। इस श्रेणी में सभी नीतियाँ **ऑप्ट-इन** हैं (`defaultEnabled: false`) — एजेंट जिन्हें वास्तव में `kubectl`, `terraform`, आदि को कॉल करना है वे बाधित नहीं होंगे जब तक आप नीति को सक्षम न करें। जब सक्षम किया जाता है, तो मेल की गई CLI के प्रत्येक आह्वान को अस्वीकार किया जाता है जब तक कि कमांड `allowPatterns` में एक प्रविष्टि से मेल न खाए।

पैटर्न व्याकरण [`block-sudo`](#block-sudo) के समान है: टोकन पार्स किए गए argv के विरुद्ध मेल खाए जाते हैं, `*` एक टोकन के लिए एक वाइल्डकार्ड है, और स्टैंडअलोन शेल ऑपरेटर (`&&`, `||`, `|`, `;`) या एम्बेडेड शेल मेटाकैरेक्टर वाले किसी भी कमांड को इंजेक्शन बाईपास को रोकने के लिए अनुमति सूची मिलान से पहले अस्वीकार किया जाता है।

### `block-kubectl`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** किसी भी `kubectl` आह्वान को अस्वीकार करता है।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | kubectl कमांड प्रीफिक्स जो अनुमत हैं। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-kubectl": {
      "allowPatterns": ["kubectl get *", "kubectl describe *", "kubectl logs *"]
```

इस कॉन्फ़िग के साथ, `kubectl get pods` की अनुमति है लेकिन `kubectl apply -f deploy.yaml` को अस्वीकार किया जाता है।

**इवेंट:** PreToolUse (Bash)

- `gh workflow run`, `gh workflow enable`, `gh workflow disable`
- `gh run rerun`, `gh run cancel`
- `gh pr merge`
- `gh release create`, `gh release delete`
- `gh cache delete`

**पैरामीटर:**

**उदाहरण:**

```json
{
  "policyParams": {
    "block-gh-pipeline": {
      "allowPatterns": ["gh run rerun *"]
    }
  }
}
```

## सीक्रेट (सैनिटाइज़र)

एजेंटों को साख को अपने संदर्भ या आउटपुट में लीक करने से रोकें। सैनिटाइज़र नीतियाँ **PostToolUse** इवेंट पर सक्रिय होती हैं। जब Claude एक Bash कमांड चलाता है, एक फाइल पढ़ता है, या किसी भी टूल को कॉल करता है, ये नीतियाँ आउटपुट को Claude को वापस किए जाने से पहले निरीक्षण करती हैं। यदि एक सीक्रेट पैटर्न का पता लगाया जाता है, तो नीति एक अस्वीकार निर्णय लौटाती है जो आउटपुट को वापस किए जाने से रोकता है।

### `sanitize-jwt`

**इवेंट:** PostToolUse (सभी टूल)  
**डिफ़ॉल्ट:** JWT टोकन को रीडैक्ट करता है (तीन base64url सेगमेंट `.` से अलग)।

कोई पैरामीटर नहीं।

---

### `sanitize-api-keys`

**इवेंट:** PostToolUse (सभी टूल)  
**डिफ़ॉल्ट:** सामान्य API कुंजी प्रारूप को रीडैक्ट करता है: Anthropic (`sk-ant-`), OpenAI (`sk-`), GitHub PATs (`ghp_`), AWS एक्सेस कुंजी (`AKIA`), Stripe कुंजी (`sk_live_`, `sk_test_`), और Google API कुंजी (`AIza`)।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `additionalPatterns` | `{ regex: string; label: string }[]` | `[]` | अतिरिक्त regex पैटर्न जिन्हें सीक्रेट के रूप में मानना है। |

**उदाहरण:**

```json
{
  "policyParams": {
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo internal API key" },
        { "regex": "pat_[0-9a-f]{40}", "label": "Internal PAT" }
      ]
    }
  }
}
```

---

कोई पैरामीटर नहीं।

## पैकेज मैनेजर

एजेंट को किस पैकेज मैनेजर का उपयोग करने की अनुमति है यह निर्धारित करें।

### `prefer-package-manager`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** अक्षम। जब सक्षम किया जाता है, तो अनुमत सूची में न होने वाली पैकेज मैनेजर कमांड को अवरुद्ध करता है और Claude को अनुमत मैनेजर का उपयोग करके कमांड को फिर से लिखने का निर्देश देता है।

पता लगाता है: pip, pip3, python -m pip, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo।

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-----------|------|---------|-------------|
| `allowed` | string[] | `[]` | अनुमत पैकेज मैनेजर नाम। इस सूची में न होने वाला कोई भी पता लगाया गया मैनेजर अवरुद्ध है। जब खाली हो, नीति एक no-op है। |
| `blocked` | string[] | `[]` | अतिरिक्त मैनेजर नाम जिन्हें अंतर्निहित सूची के अतिरिक्त अवरुद्ध करना है (जैसे `['pdm', 'pipx']`)। |

अंतर्निहित ब्लॉक सूची कवर करती है: pip, pip3, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo। इस सूची में न होने वाले मैनेजर को जोड़ने के लिए `blocked` का उपयोग करें।

**उदाहरण कॉन्फ़िगरेशन:**

```json
{
  "enabledPolicies": ["prefer-package-manager"],
  "policyParams": {
    "prefer-package-manager": {
      "allowed": ["uv", "bun"],
      "blocked": ["pdm", "pipx"]
    }
  }
}
```

इस कॉन्फ़िग के साथ, `pip install flask` और `pdm install flask` दोनों को अस्वीकार किया जाता है जिसमें एक संदेश है जो Claude को `uv` या `bun` का उपयोग करने का निर्देश देता है। `uv pip install flask` जैसी कमांड की अनुमति है क्योंकि `uv` अनुमति सूची में है और पहले जाँचा जाता है।

## वर्कफ़्लो

अनुशासित अंत-सेशन वर्कफ़्लो लागू करें। ये नीतियाँ **Stop** इवेंट पर सक्रिय होती हैं और Claude को तब तक रुकने से अस्वीकार करती हैं जब तक प्रत्येक शर्त पूरी न हो। वे एक प्राकृतिक निर्भरता श्रृंखला का पालन करते हैं: commit → push → PR → CI। यदि कोई नीति अस्वीकार करती है, श्रृंखला में बाद की नीतियों को छोड़ दिया जाता है (अस्वीकार शॉर्ट-सर्किट करता है)।

सभी वर्कफ़्लो नीतियाँ **fail-open** हैं: यदि आवश्यक टूल उपलब्ध नहीं है (जैसे `gh` स्थापित नहीं है, कोई git remote नहीं है), तो नीति जानकारीपूर्ण संदेश के साथ अनुमति देती है जो समझाता है कि जाँच को क्यों छोड़ा गया।

### `require-commit-before-stop`

**इवेंट:** Stop  
**डिफ़ॉल्ट:** जब कोई अपरिवर्तित परिवर्तन हो (संशोधित, स्टेज किए गए, या अनट्रैक किए गए फाइलें) तो रुकने से अस्वीकार करता है। कार्य निर्देशिका स्वच्छ होने पर जानकारीपूर्ण संदेश लौटाता है।

कोई पैरामीटर नहीं।

---

### `require-push-before-stop`

**इवेंट:** Stop  
**डिफ़ॉल्ट:** जब अनपुश किए गए प्रतिबद्ध हों या जब वर्तमान शाखा का कोई दूरस्थ ट्रैकिंग शाखा न हो तो रुकने से अस्वीकार करता है। यदि आवश्यक हो तो ट्रैकिंग शाखा बनाने के लिए `git push -u` का सुझाव देता है। यदि कोई remote कॉन्फ़िगर नहीं है तो fail open।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `remote` | `string` | `"origin"` | Remote का नाम जिसमें धकेलना है। |

**उदाहरण:**

```json
{
  "policyParams": {
    "require-push-before-stop": {
      "remote": "upstream"
    }
  }
}
```

---

### `require-pr-before-stop`

**इवेंट:** Stop  
**डिफ़ॉल्ट:** जब वर्तमान शाखा के लिए कोई pull request मौजूद न हो, या मौजूदा PR विलय के बिना बंद हो, तो रुकने से अस्वीकार करता है। Claude को `gh pr create` के साथ एक PR बनाने का निर्देश देता है। जब PR **विलय** किया जाता है, तो नीति अनुमति देती है (काम भेज दिया गया है) और संदेश शाखा को बंद करने का संकेत देता है (`git checkout main && git pull`)।

कोई पैरामीटर नहीं।

<Note>
इस नीति के लिए [GitHub CLI](https://cli.github.com/) (`gh`) को स्थापित और प्रमाणित किया जाना आवश्यक है।
एक व्यक्तिगत एक्सेस टोकन के साथ `gh auth login` चलाएँ जिसमें pull requests के पढ़ने की एक्सेस के लिए `repo` स्कोप हो। यदि `gh` स्थापित नहीं है या प्रमाणित नहीं है, तो नीति fail open होती है और कारण की रिपोर्ट Claude को देती है।
</Note>

---

### `require-no-conflicts-before-stop`

**इवेंट:** Stop  
**डिफ़ॉल्ट:** जब वर्तमान शाखा आधार शाखा में स्वच्छ रूप से विलय नहीं कर सकती तो रुकने से अस्वीकार करता है। नीति पहले पुष्टि करती है कि शाखा के लिए GitHub पर एक `OPEN` PR है — एक के बिना, कोई विलय लक्ष्य नहीं है जिसे लागू किया जाए, इसलिए पूरी नीति अनुमति देने के लिए short-circuit करती है। एक `OPEN` PR की पुष्टि करने के बाद, दो स्वतंत्र जाँचें चलती हैं:

1. **स्थानीय** — `git merge-tree --write-tree --name-only origin/<baseBranch> HEAD`। विरोध पर, deny संदेश विरोधाभासी फाइलों का नाम देता है ताकि Claude जानता हो कि बिल्कुल क्या हल करना है।
2. **GitHub** — पहले से की गई `gh pr view --json mergeable,state` परिणाम को फिर से उपयोग करता है। विरोधाभास को पकड़ता है जिन्हें एक पुरानी स्थानीय `origin/<baseBranch>` मिस करेगी (जैसे कोई `main` पर एक विरोधाभासी PR लैंड करता है पिछली fetch के बाद)। एक `CONFLICTING` परिणाम अस्वीकार करता है। एक `UNKNOWN` परिणाम भी अस्वीकार करता है और Claude को ~10 सेकंड प्रतीक्षा करने और फिर से जाँचने का निर्देश देता है फिर से रुकने का प्रयास करने से पहले — यह false negative को रोकता है जबकि GitHub फिर से कंप्यूट करता है।

पूरी तरह छोड़ देता है (अनुमति देता है) जब: `gh` स्थापित नहीं है, शाखा के लिए कोई PR नहीं है, PR की स्थिति `OPEN` नहीं है (जैसे `MERGED`, `CLOSED`), या `gh pr view` अपार्सेबल आउटपुट लौटाता है। जब `origin/<baseBranch>` स्थानीय रूप से गायब हो या जब कोई प्रतिबद्ध आधार से आगे न हो तो भी fail open होता है — वे Layer 1 fall-through अभी भी अनुमति देने से पहले कैश किए गए PR mergeability को परामर्श देते हैं।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `baseBranch` | `string` | `"main"` | आधार शाखा को विरोध के लिए जाँचना है। |

<Note>
इस नीति के लिए GitHub CLI (`gh`) आवश्यक है। नीति कोई भी conflict probe चलाने से पहले एक `OPEN` PR मौजूद है यह पुष्टि करने के लिए `gh pr view` का उपयोग करती है — `gh` के बिना, नीति अनुमति देने के लिए short-circuit करती है। एक व्यक्तिगत एक्सेस टोकन के साथ `gh auth login` चलाएँ जिसमें pull requests के पढ़ने की एक्सेस के लिए `repo` स्कोप हो।
</Note>

### `require-ci-green-before-stop`

**इवेंट:** Stop  
**डिफ़ॉल्ट:** जब CI जाँच वर्तमान शाखा पर विफल हो रही हों या अभी भी चल रही हों तो रुकने से अस्वीकार करता है। GitHub Actions वर्कफ़्लो रन और तीसरे पक्ष के bot जाँच (जैसे CodeRabbit, SonarCloud, Codecov) दोनों को जाँचता है। `skipped` और `cancelled` निष्कर्षों को सफलता मानता है। सभी जाँचें पास होने पर जानकारीपूर्ण संदेश लौटाता है।

कोई पैरामीटर नहीं।

<Note>
इस नीति के लिए [GitHub CLI](https://cli.github.com/) (`gh`) को स्थापित और प्रमाणित किया जाना आवश्यक है।
एक व्यक्तिगत एक्सेस टोकन के साथ `gh auth login` चलाएँ जिसमें Actions वर्कफ़्लो रन और Checks API के पढ़ने की एक्सेस के लिए `repo` स्कोप हो। यदि `gh` स्थापित नहीं है या प्रमाणित नहीं है, तो नीति fail open होती है और कारण की रिपोर्ट Claude को देती है।
</Note>

## व्यक्तिगत नीतियों को अक्षम करना

आपकी कॉन्फ़िगरेशन में `enabledPolicies` से एक विशिष्ट नीति को निकालें, या डैशबोर्ड के Policies टैब में इसे बंद करें।

```json
{
  "enabledPolicies": [
    "block-rm-rf",
    "sanitize-api-keys"
  ]
}
```

`enabledPolicies` में सूचीबद्ध न होने वाली नीतियाँ नहीं चलती हैं, भले ही उनके लिए `policyParams` प्रविष्टियाँ मौजूद हों।
