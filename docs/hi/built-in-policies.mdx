failproofai 32 बिल्ट-इन नीतियों के साथ आता है जो एजेंट की सामान्य विफलता के तरीकों को पकड़ते हैं। प्रत्येक नीति एक विशिष्ट हुक इवेंट प्रकार और टूल नाम पर काम करती है। बारह नीतियां पैरामीटर स्वीकार करती हैं जो आपको कोड लिखे बिना उनके व्यवहार को ट्यून करने देते हैं। पांच वर्कफ़्लो नीतियां एक commit → push → PR → CI पाइपलाइन को लागू करती हैं इससे पहले कि Claude रुके।

---

## अवलोकन

नीतियां श्रेणियों में विभाजित हैं:

| श्रेणी | नीतियां | हुक प्रकार |
|----------|----------|-----------|
| [खतरनाक आदेश](#dangerous-commands) | block-sudo, block-rm-rf, block-curl-pipe-sh, block-failproofai-commands | PreToolUse |
| [रहस्य (सेनिटाइजर)](#secrets-sanitizers) | sanitize-jwt, sanitize-api-keys, sanitize-connection-strings, sanitize-private-key-content, sanitize-bearer-tokens | PostToolUse |
| [पर्यावरण](#environment) | block-env-files, protect-env-vars | PreToolUse |
| [फ़ाइल एक्सेस](#file-access) | block-read-outside-cwd, block-secrets-write | PreToolUse |
| [Git](#git) | block-push-master, block-work-on-main, block-force-push, warn-git-amend, warn-git-stash-drop, warn-all-files-staged | PreToolUse |
| [डेटाबेस](#database) | warn-destructive-sql, warn-schema-alteration | PreToolUse |
| [चेतावनियां](#warnings) | warn-large-file-write, warn-package-publish, warn-background-process, warn-global-package-install | PreToolUse |
| [पैकेज मैनेजर](#package-managers) | prefer-package-manager | PreToolUse |
| [वर्कफ़्लो](#workflow) | require-commit-before-stop, require-push-before-stop, require-pr-before-stop, require-no-conflicts-before-stop, require-ci-green-before-stop | Stop |

- **`block-`** — एजेंट को आगे बढ़ने से रोकें।
- **`warn-`** — एजेंट को अतिरिक्त संदर्भ दें ताकि वह स्वयं को सही कर सके।
- **`sanitize-`** — एजेंट के देखने से पहले टूल आउटपुट से संवेदनशील डेटा को साफ करें।

### नेमस्पेस

प्रत्येक नीति एक `<namespace>/<name>` स्लॉट में रहती है। बिल्ट-इन नीतियां **`exospherehost/`** नेमस्पेस से संबंधित हैं — उदाहरण के लिए, `exospherehost/sanitize-jwt`। नेमस्पेस तब टकराव को रोकता है जब आप समान छोटे नामों वाली कस्टम या तीसरे पक्ष की नीतियों को भी लोड करते हैं।

आपके कॉन्फ़िगरेशन में आप किसी बिल्ट-इन को उसके छोटे नाम या योग्य नाम से संदर्भित कर सकते हैं; दोनों फॉर्म समान नीति को हल करते हैं:

```json
{
  "enabledPolicies": [
    "sanitize-jwt",
    "exospherehost/block-rm-rf"
  ]
}
```

यदि किसी नाम में `/` नहीं है, तो failproofai इसे डिफ़ॉल्ट नेमस्पेस `exospherehost` से संबंधित मानता है। जो नाम पहले से ही `/` शामिल करते हैं (जैसे `myorg/foo`, `custom/my-hook`) उन्हें जैसे हैं वैसे ही रखा जाता है।
- **`require-`** — जब तक शर्तें पूरी न हों तब तक Stop इवेंट को ब्लॉक करें।

---

<Tip>
प्रत्येक नीति `policyParams` में एक वैकल्पिक `hint` फील्ड का समर्थन करती है। हिंट को deny या instruct संदेश में जोड़ा जाता है जो Claude देखता है, नीति कोड को संशोधित किए बिना कार्रवाई योग्य मार्गदर्शन देता है। बिल्ट-इन, कस्टम और कन्वेंशन नीतियों के साथ काम करता है। विवरण के लिए [कॉन्फ़िगरेशन → hint](/hi/configuration#hint-cross-cutting) देखें।
</Tip>

## खतरनाक आदेश

एजेंटों को ऐसे संचालन चलाने से रोकें जो पूर्ववत करना कठिन हैं या जो होस्ट सिस्टम को नुकसान पहुंचा सकते हैं।

### `block-sudo`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** किसी भी `sudo` कमांड को अस्वीकार करता है।

जिस आदेश में `sudo` कीवर्ड शामिल है उसे ब्लॉक करता है। पैटर्न मिलान कच्ची स्ट्रिंग के बजाय पार्स किए गए कमांड टोकन पर किया जाता है, शेल ऑपरेटर इंजेक्शन के माध्यम से बाईपास को रोकने के लिए।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | सटीक कमांड उपसर्ग जो अनुमत हैं। प्रत्येक प्रविष्टि को पार्स किए गए argv टोकन के विरुद्ध मिलाया जाता है। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    }
  }
}
```

इस कॉन्फ़िगरेशन के साथ, `sudo systemctl status nginx` की अनुमति है, लेकिन `sudo rm /etc/hosts` को अस्वीकार कर दिया जाता है।

<Note>
पैटर्न को कच्ची कमांड स्ट्रिंग के बजाय पार्स किए गए टोकन के विरुद्ध मिलाया जाता है। यह जोड़े गए शेल ऑपरेटर के माध्यम से बाईपास को रोकता है (जैसे `sudo systemctl status x; rm -rf /` `sudo systemctl status *` से मेल नहीं खाता)।
</Note>

---

### `block-rm-rf`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** `rm -rf`, `rm -fr` और समान पुनरावर्ती हटाने के फॉर्म को अस्वीकार करता है।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `allowPaths` | `string[]` | `[]` | पथ जो पुनरावर्ती रूप से हटाने के लिए सुरक्षित हैं (जैसे `/tmp`)। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-rm-rf": {
      "allowPaths": ["/tmp", "/var/cache"]
    }
  }
}
```

---

### `block-curl-pipe-sh`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** `curl <url> | bash`, `curl <url> | sh`, `wget <url> | bash` और समान पैटर्न को अस्वीकार करता है।

कोई पैरामीटर नहीं।

---

### `block-failproofai-commands`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** ऐसी कमांड को अस्वीकार करता है जो failproofai को स्वयं को अनइंस्टॉल या अक्षम करेगी (जैसे `npm uninstall failproofai`, `failproofai policies --uninstall`)।

कोई पैरामीटर नहीं।

## रहस्य (सेनिटाइजर)

एजेंटों को अपने संदर्भ या आउटपुट में क्रेडेंशियल लीक करने से रोकें। सेनिटाइजर नीतियां **PostToolUse** इवेंट पर काम करती हैं। जब Claude Bash कमांड चलाता है, फ़ाइल पढ़ता है, या कोई भी टूल कॉल करता है, ये नीतियां आउटपुट का निरीक्षण करती हैं इससे पहले कि यह Claude को वापस किया जाए। यदि कोई गुप्त पैटर्न पाया जाता है, तो नीति एक deny निर्णय देता है जो आउटपुट को वापस पास होने से रोकता है।

### `sanitize-jwt`

**इवेंट:** PostToolUse (सभी टूल)  
**डिफ़ॉल्ट:** JWT टोकन को संपादित करता है (तीन base64url खंड `.` से अलग)।

कोई पैरामीटर नहीं।

---

### `sanitize-api-keys`

**इवेंट:** PostToolUse (सभी टूल)  
**डिफ़ॉल्ट:** सामान्य API कुंजी प्रारूपों को संपादित करता है: Anthropic (`sk-ant-`), OpenAI (`sk-`), GitHub PATs (`ghp_`), AWS एक्सेस कुंजियां (`AKIA`), Stripe कुंजियां (`sk_live_`, `sk_test_`), और Google API कुंजियां (`AIza`)।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `additionalPatterns` | `{ regex: string; label: string }[]` | `[]` | गुप्त के रूप में मानने के लिए अतिरिक्त regex पैटर्न। |

**उदाहरण:**

```json
{
  "policyParams": {
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo internal API key" },
        { "regex": "pat_[0-9a-f]{40}", "label": "Internal PAT" }
      ]
    }
  }
}
```

---

### `sanitize-connection-strings`

**इवेंट:** PostToolUse (सभी टूल)  
**डिफ़ॉल्ट:** डेटाबेस कनेक्शन स्ट्रिंग्स को संपादित करता है जो एम्बेडेड क्रेडेंशियल शामिल करते हैं (जैसे `postgresql://user:password@host/db`)।

कोई पैरामीटर नहीं।

---

### `sanitize-private-key-content`

**इवेंट:** PostToolUse (सभी टूल)  
**डिफ़ॉल्ट:** PEM ब्लॉक्स को संपादित करता है (`-----BEGIN PRIVATE KEY-----`, `-----BEGIN RSA PRIVATE KEY-----`, आदि)।

कोई पैरामीटर नहीं।

---

### `sanitize-bearer-tokens`

**इवेंट:** PostToolUse (सभी टूल)  
**डिफ़ॉल्ट:** `Authorization: Bearer <token>` हेडर को संपादित करता है जहां टोकन 20 या अधिक वर्ण हैं।

कोई पैरामीटर नहीं।

## पर्यावरण

संवेदनशील पर्यावरण कॉन्फ़िगरेशन को एजेंटों द्वारा पढ़े या प्रकट किए जाने से सुरक्षित रखें।

### `block-env-files`

**इवेंट:** PreToolUse (Bash, Read)  
**डिफ़ॉल्ट:** `.env` फ़ाइलों को `cat .env`, `.env` को फ़ाइल पथ के रूप में पढ़ने के Tool कॉल आदि के माध्यम से पढ़ने से इनकार करता है।

`.envrc` या अन्य पर्यावरण-संबंधित फ़ाइलों को ब्लॉक नहीं करता है - केवल ठीक `.env` नामित फ़ाइलें।

कोई पैरामीटर नहीं।

---

### `protect-env-vars`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** ऐसी कमांड को अस्वीकार करता है जो पर्यावरण चर प्रिंट करते हैं: `printenv`, `env`, `echo $VAR`।

कोई पैरामीटर नहीं।

---

## फ़ाइल एक्सेस

एजेंटों को प्रोजेक्ट सीमाओं के अंदर काम करते रहने और संवेदनशील फ़ाइलों से दूर रखें।

### `block-read-outside-cwd`

**इवेंट:** PreToolUse (Read, Bash)  
**डिफ़ॉल्ट:** प्रोजेक्ट रूट के बाहर फ़ाइलों को पढ़ने से इनकार करता है। सीमा `CLAUDE_PROJECT_DIR` है (Claude Code द्वारा प्रति सेशन एक बार सेट), उस वेरिएबल के अनसेट होने पर सेशन की वर्तमान कार्यशील निर्देशिका में fallback के साथ। लाइव `cwd` के बजाय प्रोजेक्ट रूट का उपयोग करने का मतलब है कि सीमा तब भी स्थिर रहती है जब Claude किसी सबडायरेक्टरी में `cd` करता है।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `allowPaths` | `string[]` | `[]` | निरपेक्ष पथ उपसर्ग जो अनुमत हैं भले ही प्रोजेक्ट रूट के बाहर हों। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company/config"]
    }
  }
}
```

---

### `block-secrets-write`

**इवेंट:** PreToolUse (Write, Edit)  
**डिफ़ॉल्ट:** निजी कुंजियों और प्रमाणपत्रों के लिए आमतौर पर उपयोग की जाने वाली फ़ाइलों में लिखने से इनकार करता है: `id_rsa`, `id_ed25519`, `*.key`, `*.pem`, `*.p12`, `*.pfx`।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `additionalPatterns` | `string[]` | `[]` | ब्लॉक करने के लिए अतिरिक्त फ़ाइल नाम पैटर्न (glob-शैली)। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-secrets-write": {
      "additionalPatterns": [".token", ".secret"]
    }
  }
}
```

## Git

आकस्मिक पुशे, force-push, और ब्रांच गलतियों को रोकें जो पूर्ववत करना कठिन है।

### `block-push-master`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** `git push origin main` और `git push origin master` को अस्वीकार करता है।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `protectedBranches` | `string[]` | `["main", "master"]` | ब्रांच नाम जिन्हें सीधे पुश नहीं किया जा सकता। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "master", "release", "prod"]
    }
  }
}
```

<Tip>
सभी ब्रांचों को पुश करने की अनुमति देने के लिए (प्रभावी रूप से `enabledPolicies` से निकाले बिना इस नीति को अक्षम करना), `protectedBranches: []` सेट करें।
</Tip>

---

### `block-work-on-main`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** `main` या `master` ब्रांचों को सीधे चेक आउट करने से इनकार करता है।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `protectedBranches` | `string[]` | `["main", "master"]` | ब्रांच नाम जिन्हें सीधे चेक आउट नहीं किया जा सकता। |

---

### `block-force-push`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** `git push --force` और `git push -f` को अस्वीकार करता है।

कोई नीति-विशिष्ट पैरामीटर नहीं। विकल्प सुझाने के लिए क्रॉस-कटिंग [`hint`](/hi/configuration#hint-cross-cutting) का उपयोग करें:

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "अपने वर्तमान HEAD से एक नई ब्रांच बनाएं (जैसे `git checkout -b <new-branch>`) और उसके बजाय वह पुश करें।"
    }
  }
}
```

---

### `warn-git-amend`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** `git commit --amend` चलाते समय Claude को सावधानी से आगे बढ़ने के लिए निर्दिष्ट करता है। कमांड को ब्लॉक नहीं करता।

कोई पैरामीटर नहीं।

---

### `warn-git-stash-drop`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** `git stash drop` चलाने से पहले Claude को पुष्टि करने के लिए निर्दिष्ट करता है। कमांड को ब्लॉक नहीं करता।

कोई पैरामीटर नहीं।

---

### `warn-all-files-staged`

**इवेंट:** PreToolUse (Bash)

## पैकेज मैनेजर

यह लागू करें कि एजेंट कौन से पैकेज मैनेजर का उपयोग करने की अनुमति है।

### `prefer-package-manager`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** अक्षम। जब सक्षम किया जाता है, तो `allowed` सूची में नहीं है ऐसी किसी भी पैकेज मैनेजर कमांड को ब्लॉक करता है और Claude को अनुमत मैनेजर का उपयोग करके कमांड को फिर से लिखने के लिए कहता है।

पहचानता है: pip, pip3, python -m pip, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo।

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-----------|------|---------|-------------|
| `allowed` | string[] | `[]` | अनुमत पैकेज मैनेजर नाम। इस सूची में नहीं है ऐसा कोई भी पहचाना गया मैनेजर ब्लॉक किया जाता है। जब खाली हो, तो नीति कोई-op है। |
| `blocked` | string[] | `[]` | अतिरिक्त मैनेजर नाम बिल्ट-इन सूची के अलावा ब्लॉक करें (जैसे `['pdm', 'pipx']`)। |

बिल्ट-इन ब्लॉक सूची शामिल करती है: pip, pip3, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo। इस सूची में न होने वाले मैनेजर को जोड़ने के लिए `blocked` का उपयोग करें।

**उदाहरण कॉन्फ़िगरेशन:**

```json
{
  "enabledPolicies": ["prefer-package-manager"],
  "policyParams": {
    "prefer-package-manager": {
      "allowed": ["uv", "bun"],
      "blocked": ["pdm", "pipx"]
    }
  }
}
```

इस कॉन्फ़िगरेशन के साथ, `pip install flask` और `pdm install flask` दोनों को अस्वीकार कर दिया जाता है और Claude को `uv` या `bun` का उपयोग करने के लिए कहा जाता है। `uv pip install flask` जैसी कमांड की अनुमति है क्योंकि `uv` अनुमत सूची में है और पहले जांचा जाता है।

## वर्कफ़्लो

एक अनुशासित सेशन के अंत वाला वर्कफ़्लो लागू करें। ये नीतियां **Stop** इवेंट पर काम करती हैं और Claude को रुकने से तब तक रोकती हैं जब तक प्रत्येक शर्त पूरी न हो। वे एक प्राकृतिक निर्भरता श्रृंखला का अनुसरण करते हैं: commit → push → PR → CI। यदि कोई नीति अस्वीकार करती है, तो श्रृंखला में बाद की नीतियां छोड़ दी जाती हैं (अस्वीकार short-circuits करता है)।

सभी वर्कफ़्लो नीतियां **fail-open** हैं: यदि आवश्यक टूल उपलब्ध नहीं है (जैसे `gh` स्थापित नहीं है, कोई git remote नहीं), तो नीति एक सूचनात्मक संदेश के साथ अनुमति देता है जो समझाता है कि जांच को क्यों छोड़ा गया था।

### `require-commit-before-stop`

**इवेंट:** Stop  
**डिफ़ॉल्ट:** जब अपरिवर्तित परिवर्तन (संशोधित, स्टेज किए गए या अनट्रैक किए गए फ़ाइलें) हों तो रुकने से इनकार करता है। कार्यशील निर्देशिका स्वच्छ होने पर एक सूचनात्मक संदेश देता है।

कोई पैरामीटर नहीं।

---

### `require-push-before-stop`

**इवेंट:** Stop  
**डिफ़ॉल्ट:** जब अपुष्ट commits हों या वर्तमान branch के पास कोई remote tracking branch नहीं हो तो रुकने से इनकार करता है। आवश्यक होने पर `git push -u` का सुझाव देता है tracking branch बनाने के लिए। कोई remote कॉन्फ़िगर नहीं होने पर fail open करता है।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `remote` | `string` | `"origin"` | पुश करने के लिए remote नाम। |

**उदाहरण:**

```json
{
  "policyParams": {
    "require-push-before-stop": {
      "remote": "upstream"
    }
  }
}
```

<Note>
इस नीति के लिए [GitHub CLI](https://cli.github.com/) (`gh`) को स्थापित और प्रमाणित होना आवश्यक है।
`gh auth login` को एक personal access token के साथ चलाएं जिसके पास pull request को पढ़ने की एक्सेस के लिए `repo` scope हो। यदि `gh` स्थापित नहीं है या प्रमाणित नहीं है, तो नीति fail open करता है और कारण को Claude को रिपोर्ट करता है।
</Note>

### `require-no-conflicts-before-stop`

**इवेंट:** Stop  
**डिफ़ॉल्ट:** जब वर्तमान branch base branch में स्वच्छता से merge नहीं हो सकता तो रुकने से इनकार करता है। नीति पहले यह पुष्टि करता है कि GitHub पर branch के लिए एक `OPEN` PR है — बिना इसके, merge target को लागू करने के लिए कुछ नहीं है, तो पूरी नीति allow के लिए short-circuits होती है। एक बार `OPEN` PR की पुष्टि हो जाने के बाद, दो स्वतंत्र probes चलते हैं:

1. **Local** — `git merge-tree --write-tree --name-only origin/<baseBranch> HEAD`। Conflict होने पर, deny संदेश conflicted फ़ाइलों को नाम देता है ताकि Claude को बिल्कुल पता हो कि क्या हल करना है।
2. **GitHub** — पहले से ही precheck में लाए गए `gh pr view --json mergeable,state` परिणाम का पुन:उपयोग करता है। ऐसे conflicts को पकड़ता है जो stale local `origin/<baseBranch>` miss करते (जैसे कोई `main` पर conflicting PR लाया है पिछले fetch के बाद)। एक `CONFLICTING` परिणाम को अस्वीकार करता है। एक `UNKNOWN` परिणाम भी अस्वीकार करता है और Claude को ~10 सेकंड प्रतीक्षा करने और फिर से जांच करने से पहले फिर से रुकने का प्रयास करने के लिए निर्दिष्ट करता है — यह गलत negatives को रोकते हुए GitHub को पुनः गणना करने देता है।

Entirely skip करता है (allows) जब: `gh` स्थापित नहीं है, branch के लिए कोई PR नहीं है, PR का state `OPEN` नहीं है (जैसे `MERGED`, `CLOSED`), या `gh pr view` unparseable आउटपुट देता है। जब `origin/<baseBranch>` स्थानीय रूप से missing है या कोई commits base के आगे नहीं हैं तब भी fail open करता है — वे Layer 1 fall-throughs अभी भी allowed करने से पहले cached PR mergeability सलाह लेते हैं।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `baseBranch` | `string` | `"main"` | Conflicts के विरुद्ध जांचने के लिए base branch। |

<Note>
इस नीति के लिए GitHub CLI (`gh`) आवश्यक है। नीति किसी भी conflict probe से पहले एक `OPEN` PR की पुष्टि करने के लिए `gh pr view` का उपयोग करता है — बिना `gh` के, नीति allow के लिए short-circuits होता है। `gh auth login` को एक personal access token के साथ चलाएं जिसके पास pull requests को पढ़ने की एक्सेस के लिए `repo` scope हो।
</Note>

## व्यक्तिगत नीतियों को अक्षम करना

अपने कॉन्फ़िगरेशन से `enabledPolicies` से एक विशिष्ट नीति को निकालें, या डैशबोर्ड के Policies टैब में इसे अक्षम करें।

```json
{
  "enabledPolicies": [
    "block-rm-rf",
    "sanitize-api-keys"
  ]
}
```

`enabledPolicies` में सूचीबद्ध नहीं हैं ऐसी नीतियां नहीं चलती हैं, भले ही `policyParams` प्रविष्टियां उनके लिए मौजूद हों।
