failproofai 30 built-in policies के साथ आता है जो common agent failure modes को catch करती हैं। प्रत्येक policy एक specific hook event type और tool name पर fire होती है। नौ policies ऐसे parameters स्वीकार करती हैं जो आपको code लिखे बिना उनके behavior को tune करने देते हैं। चार workflow policies एक commit → push → PR → CI pipeline enforce करती हैं जब तक Claude stop न हो जाए।

---

## Overview

Policies को categories में grouped किया गया है:

| Category | Policies | Hook type |
|----------|----------|-----------|
| [Dangerous commands](#dangerous-commands) | block-sudo, block-rm-rf, block-curl-pipe-sh, block-failproofai-commands | PreToolUse |
| [Secrets (sanitizers)](#secrets-sanitizers) | sanitize-jwt, sanitize-api-keys, sanitize-connection-strings, sanitize-private-key-content, sanitize-bearer-tokens | PostToolUse |
| [Environment](#environment) | block-env-files, protect-env-vars | PreToolUse |
| [File access](#file-access) | block-read-outside-cwd, block-secrets-write | PreToolUse |
| [Git](#git) | block-push-master, block-work-on-main, block-force-push, warn-git-amend, warn-git-stash-drop, warn-all-files-staged | PreToolUse |
| [Database](#database) | warn-destructive-sql, warn-schema-alteration | PreToolUse |
| [Warnings](#warnings) | warn-large-file-write, warn-package-publish, warn-background-process, warn-global-package-install | PreToolUse |
| [Package managers](#package-managers) | prefer-package-manager | PreToolUse |
| [Workflow](#workflow) | require-commit-before-stop, require-push-before-stop, require-pr-before-stop, require-no-conflicts-before-stop, require-ci-green-before-stop | Stop |

- **`block-`** — agent को आगे बढ़ने से रोकता है।
- **`warn-`** — agent को additional context देता है ताकि यह self-correct कर सके।
- **`sanitize-`** — agent को देखने से पहले tool output से sensitive data को हटाता है।
- **`require-`** — Stop event को तब तक block करता है जब तक conditions पूरी न हो जाएं।

---

<Tip>
हर policy `policyParams` में एक optional `hint` field को support करती है। Hint को deny या instruct message में append किया जाता है जो Claude देखता है, जिससे policy code को modify किए बिना actionable guidance मिलती है। Built-in, custom, और convention policies के साथ काम करता है। विवरण के लिए [Configuration → hint](/hi/configuration#hint-cross-cutting) देखें।
</Tip>

## Dangerous commands

Agents को ऐसे operations चलाने से रोकें जो undo करना मुश्किल हों या जो host system को damage कर सकते हों।

### `block-sudo`

**Event:** PreToolUse (Bash)  
**Default:** किसी भी `sudo` command को deny करता है।

`sudo` keyword वाले invocations को block करता है। Pattern matching raw string के बजाय parsed command tokens पर की जाती है, ताकि shell operator injection से bypass को रोका जा सके।

**Parameters:**

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | Exact command prefixes जो permitted हैं। प्रत्येक entry parsed argv tokens के विरुद्ध match किया जाता है। |

**Example:**

```json
{
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    }
  }
}
```

इस config के साथ, `sudo systemctl status nginx` को allow किया जाता है, लेकिन `sudo rm /etc/hosts` को deny किया जाता है।

<Note>
Patterns को raw command string के बजाय parsed tokens के विरुद्ध match किया जाता है। यह appended shell operators से bypass को रोकता है (उदाहरण के लिए `sudo systemctl status x; rm -rf /` `sudo systemctl status *` से match नहीं करता)।
</Note>

---

### `block-rm-rf`

**Event:** PreToolUse (Bash)  
**Default:** `rm -rf`, `rm -fr`, और similar recursive deletion forms को deny करता है।

**Parameters:**

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `allowPaths` | `string[]` | `[]` | Paths जो recursively delete करने के लिए safe हैं (जैसे `/tmp`)। |

**Example:**

```json
{
  "policyParams": {
    "block-rm-rf": {
      "allowPaths": ["/tmp", "/var/cache"]
    }
  }
}
```

---

**Event:** PreToolUse (Bash)  
**Default:** `curl <url> | bash`, `curl <url> | sh`, `wget <url> | bash`, और similar patterns को deny करता है।

कोई parameters नहीं।

कोई parameters नहीं।

## Secrets (sanitizers)

Agents को credentials को अपने context या output में leak करने से रोकें। Sanitizer policies **PostToolUse** events पर fire होती हैं। जब Claude एक Bash command चलाता है, एक file पढ़ता है, या कोई भी tool call करता है, ये policies output को inspect करती हैं इससे पहले कि वह Claude को return किया जाए। अगर एक secret pattern detect होता है, तो policy एक deny decision return करती है जो output को pass back होने से रोकता है।

### `sanitize-jwt`

**Event:** PostToolUse (all tools)  
**Default:** JWT tokens को redact करता है (तीन base64url segments जो `.` से separated हों)।

कोई parameters नहीं।

---

### `sanitize-api-keys`

**Event:** PostToolUse (all tools)  
**Default:** Common API key formats को redact करता है: Anthropic (`sk-ant-`), OpenAI (`sk-`), GitHub PATs (`ghp_`), AWS access keys (`AKIA`), Stripe keys (`sk_live_`, `sk_test_`), और Google API keys (`AIza`)।

**Parameters:**

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `additionalPatterns` | `{ regex: string; label: string }[]` | `[]` | Additional regex patterns को secrets के रूप में treat करने के लिए। |

**Example:**

```json
{
  "policyParams": {
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo internal API key" },
        { "regex": "pat_[0-9a-f]{40}", "label": "Internal PAT" }
      ]
    }
  }
}
```

---

### `sanitize-connection-strings`

**Event:** PostToolUse (all tools)  
**Default:** Database connection strings को redact करता है जिनमें embedded credentials हों (जैसे `postgresql://user:password@host/db`)।

कोई parameters नहीं।

---

### `sanitize-private-key-content`

**Event:** PostToolUse (all tools)  
**Default:** PEM blocks को redact करता है (`-----BEGIN PRIVATE KEY-----`, `-----BEGIN RSA PRIVATE KEY-----`, आदि)।

कोई parameters नहीं।

---

### `sanitize-bearer-tokens`

**Event:** PostToolUse (all tools)  
**Default:** `Authorization: Bearer <token>` headers को redact करता है जहाँ token 20 या अधिक characters हो।

कोई parameters नहीं।

## Environment

Sensitive environment configuration को agents द्वारा read या expose किए जाने से protect करें।

### `block-env-files`

**Event:** PreToolUse (Bash, Read)  
**Default:** `.env` files को `cat .env`, Read tool calls with `.env` आदि के माध्यम से read करने को deny करता है।

`.envrc` या अन्य environment-adjacent files को block नहीं करता — केवल exactly `.env` नाम की files को।

कोई parameters नहीं।

---

### `protect-env-vars`

**Event:** PreToolUse (Bash)  
**Default:** उन commands को deny करता है जो environment variables को print करते हैं: `printenv`, `env`, `echo $VAR`।

कोई parameters नहीं।

## File access

Agents को project boundaries के अंदर काम करने और sensitive files से दूर रहने के लिए रखें।

### `block-read-outside-cwd`

**Event:** PreToolUse (Read, Bash)  
**Default:** Project root के बाहर files को read करने को deny करता है। Boundary `CLAUDE_PROJECT_DIR` है (Claude Code द्वारा प्रति session एक बार set किया जाता है), जब वह variable unset हो तो session के current working directory को fallback के रूप में use करता है। Live `cwd` के बजाय project root को use करने का मतलब है कि boundary stable रहता है यहाँ तक कि Claude के `cd` करने के बाद भी।

**Parameters:**

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `allowPaths` | `string[]` | `[]` | Absolute path prefixes जो permitted हैं यहाँ तक कि project root के बाहर हों। |

**Example:**

```json
{
  "policyParams": {
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company/config"]
    }
  }
}
```

---

### `block-secrets-write`

**Event:** PreToolUse (Write, Edit)  
**Default:** Private keys और certificates के लिए commonly used files को write करने को deny करता है: `id_rsa`, `id_ed25519`, `*.key`, `*.pem`, `*.p12`, `*.pfx`।

**Parameters:**

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `additionalPatterns` | `string[]` | `[]` | Additional filename patterns (glob-style) को block करने के लिए। |

**Example:**

```json
{
  "policyParams": {
    "block-secrets-write": {
      "additionalPatterns": [".token", ".secret"]
    }
  }
}
```

## Git

Accidental pushes, force-pushes, और branch mistakes को रोकें जो undo करना मुश्किल हों।

### `block-push-master`

**Event:** PreToolUse (Bash)  
**Default:** `git push origin main` और `git push origin master` को deny करता है।

**Parameters:**

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `protectedBranches` | `string[]` | `["main", "master"]` | Branch names जिन्हें directly push नहीं किया जा सकता। |

**Example:**

```json
{
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "master", "release", "prod"]
    }
  }
}
```

<Tip>
सभी branches को push करने की अनुमति देने के लिए (इस policy को `enabledPolicies` से हटाए बिना effectively disable करने के लिए), `protectedBranches: []` set करें।
</Tip>

---

### `block-work-on-main`

**Event:** PreToolUse (Bash)  
**Default:** `main` या `master` branches को directly checkout करने को deny करता है।

**Parameters:**

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `protectedBranches` | `string[]` | `["main", "master"]` | Branch names जिन्हें directly checkout नहीं किया जा सकता। |

---

### `block-force-push`

**Event:** PreToolUse (Bash)  
**Default:** `git push --force` और `git push -f` को deny करता है।

कोई policy-specific parameters नहीं। Alternatives को suggest करने के लिए cross-cutting [`hint`](/hi/configuration#hint-cross-cutting) use करें:

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "अपने current HEAD से एक new branch बनाएँ (जैसे `git checkout -b <new-branch>`) और उसे push करें।"
    }
  }
}
```

---

## Warnings

Agents को potentially risky लेकिन non-destructive operations से पहले extra context दें।

### `warn-large-file-write`

**Event:** PreToolUse (Write)  
**Default:** Claude को 1024 KB से बड़ी files write करने से पहले confirm करने के लिए instruct करता है।

**Parameters:**

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `thresholdKb` | `number` | `1024` | File size threshold किlobytes में जिसके ऊपर एक warning issue किया जाता है। |

**Example:**

```json
{
  "policyParams": {
    "warn-large-file-write": {
      "thresholdKb": 256
    }
  }
}
```

<Note>
Hook handler payloads पर 1 MB stdin limit enforce करता है। इस policy को small content के साथ test करने के लिए, `thresholdKb` को 1024 के नीचे एक value पर set करें।
</Note>

## Package managers

Enforce करें कि agent किन package managers को use करने की अनुमति है।

### `prefer-package-manager`

**Event:** PreToolUse (Bash)  
**Default:** Disabled। जब enabled हो, तो `allowed` list में न आने वाले किसी भी package manager command को block करता है और Claude को command को एक allowed manager के साथ rewrite करने के लिए कहता है।

Detects: pip, pip3, python -m pip, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo।

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `allowed` | string[] | `[]` | Allowed package manager names। कोई भी detected manager जो इस list में न हो block हो जाता है। जब empty हो, तो policy एक no-op है। |
| `blocked` | string[] | `[]` | Additional manager names को built-in list से परे block करने के लिए (जैसे `['pdm', 'pipx']`)। |

Built-in block list covers: pip, pip3, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo। इस list में न आने वाले managers को append करने के लिए `blocked` use करें।

**Example configuration:**

```json
{
  "enabledPolicies": ["prefer-package-manager"],
  "policyParams": {
    "prefer-package-manager": {
      "allowed": ["uv", "bun"],
      "blocked": ["pdm", "pipx"]
    }
  }
}
```

इस config के साथ, `pip install flask` और `pdm install flask` दोनों को deny किया जाता है एक message के साथ जो Claude को `uv` या `bun` use करने के लिए कहता है। `uv pip install flask` जैसे commands को allow किया जाता है क्योंकि `uv` allowlist में है और पहले check किया जाता है।

## Workflow

एक disciplined end-of-session workflow enforce करें। ये policies **Stop** event पर fire होती हैं और Claude को stop करने से deny करती हैं जब तक हर condition पूरी न हो जाए। ये एक natural dependency chain को follow करती हैं: commit → push → PR → CI। अगर कोई policy deny करती है, तो chain में बाद की policies को skip किया जाता है (deny short-circuits)।

सभी workflow policies **fail-open** हैं: अगर required tool available नहीं है (जैसे `gh` installed नहीं है, no git remote), तो policy allow करती है एक informational message के साथ जो explain करता है कि check क्यों skip किया गया।

### `require-commit-before-stop`

**Event:** Stop  
**Default:** जब uncommitted changes हों (modified, staged, या untracked files) तो stop करने को deny करता है। Working directory clean होने पर एक informational message return करता है।

कोई parameters नहीं।

---

### `require-push-before-stop`

**Event:** Stop  
**Default:** जब unpushed commits हों या current branch के पास कोई remote tracking branch न हो तो stop करने को deny करता है। अगर needed हो तो `git push -u` suggest करता है एक tracking branch बनाने के लिए। अगर कोई remote configured नहीं है तो fail open करता है।

**Parameters:**

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `remote` | `string` | `"origin"` | Remote name जिसे push करना है। |

**Example:**

```json
{
  "policyParams": {
    "require-push-before-stop": {
      "remote": "upstream"
    }
  }
}
```

---

### `require-pr-before-stop`

**Event:** Stop  
**Default:** जब current branch के लिए कोई pull request exist न करे, या existing PR closed/merged हो तो stop करने को deny करता है। Claude को `gh pr create` के साथ एक PR बनाने के लिए instruct करता है।

कोई parameters नहीं।

<Note>
इस policy के लिए [GitHub CLI](https://cli.github.com/) (`gh`) को installed और authenticated होना आवश्यक है।
`gh auth login` को एक personal access token के साथ run करें जिसके पास pull requests को read करने के लिए `repo` scope हो। अगर `gh` installed या authenticated नहीं है, तो policy fail open करती है और reason को Claude को report करती है।
</Note>

---

### `require-no-conflicts-before-stop`

**Event:** Stop  
**Default:** जब current branch base branch में cleanly merge न कर सके तो stop करने को deny करता है। दो independent probes चलाता है:

1. **Local** — `git merge-tree --write-tree --name-only origin/<baseBranch> HEAD`। Conflict पर, deny message conflicted files को name करता है ताकि Claude को exactly पता चले कि क्या resolve करना है।
2. **GitHub** — `gh pr view --json mergeable`। उन conflicts को catch करता है जो एक stale local `origin/<baseBranch>` would miss करे (जैसे कोई conflicting PR को `main` पर land कर दे since last fetch)। एक `CONFLICTING` result deny करता है। एक `UNKNOWN` result (GitHub still computing) भी deny करता है और Claude को instruct करता है कि ~10 seconds wait करे और फिर से re-check करे stop attempt करने से पहले — यह false negatives को prevent करता है जबकि GitHub recomputes करता है।

Fails open जब `origin/<baseBranch>` locally missing हो, जब कोई commits base से ahead न हों, जब `gh` installed न हो, या जब branch के लिए कोई PR exist न करे।

**Parameters:**

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `baseBranch` | `string` | `"main"` | Base branch जिसके विरुद्ध conflicts को check करना है। |

<Note>
GitHub CLI (`gh`) इस policy के लिए optional लेकिन recommended है — यह दूसरा detection layer provide करता है जो एक stale local `origin/<baseBranch>` को miss करने वाले conflicts को catch करता है।
`gh` के बिना, केवल local `git merge-tree` check चलता है।
</Note>

---

### `require-ci-green-before-stop`

**Event:** Stop  
**Default:** जब CI checks current branch पर failing या still running हों तो stop करने को deny करता है। GitHub Actions workflow runs और third-party bot checks दोनों को check करता है (जैसे CodeRabbit, SonarCloud, Codecov)। `skipped` और `cancelled` conclusions को success के रूप में treat करता है। जब सभी checks pass करें तो एक informational message return करता है।

कोई parameters नहीं।

<Note>
इस policy के लिए [GitHub CLI](https://cli.github.com/) (`gh`) को installed और authenticated होना आवश्यक है।
`gh auth login` को एक personal access token के साथ run करें जिसके पास Actions workflow runs और Checks API को read करने के लिए `repo` scope हो। अगर `gh` installed या authenticated नहीं है, तो policy fail open करती है और reason को Claude को report करती है।
</Note>

## Individual policies को disable करना

अपने config में `enabledPolicies` से एक specific policy को remove करें, या dashboard के Policies tab में इसे toggle off करें।

```json
{
  "enabledPolicies": [
    "block-rm-rf",
    "sanitize-api-keys"
  ]
}
```

`enabledPolicies` में list न की गई policies run नहीं होती हैं, यहाँ तक कि अगर `policyParams` entries उनके लिए exist करें भी।
