failproofai के साथ 39 बिल्ट-इन पॉलिसीज़ शामिल हैं जो एजेंट की सामान्य विफलता मोड को पकड़ती हैं। प्रत्येक पॉलिसी एक विशिष्ट हुक इवेंट टाइप और टूल नाम पर काम करती है। उन्नीस पॉलिसीज़ पैरामीटर स्वीकार करती हैं जो आपको कोड लिखे बिना उनके व्यवहार को ट्यून करने देते हैं। पाँच वर्कफ़्लो पॉलिसीज़ एक कमिट → पुश → PR → CI पाइपलाइन लागू करती हैं जो Claude को रोकने से पहले चलनी चाहिए।

---

## अवलोकन

पॉलिसीज़ को श्रेणियों में समूहीकृत किया गया है:

| श्रेणी | पॉलिसीज़ | हुक प्रकार |
|----------|----------|-----------|
| [खतरनाक कमांड](#dangerous-commands) | block-sudo, block-rm-rf, block-curl-pipe-sh, block-failproofai-commands | PreToolUse |
| [इन्फ्रा कमांड](#infra-commands) | block-kubectl, block-terraform, block-aws-cli, block-gcloud, block-az-cli, block-helm, block-gh-pipeline | PreToolUse |
| [सीक्रेट्स (सैनिटाइज़र)](#secrets-sanitizers) | sanitize-jwt, sanitize-api-keys, sanitize-connection-strings, sanitize-private-key-content, sanitize-bearer-tokens | PostToolUse |
| [पर्यावरण](#environment) | block-env-files, protect-env-vars | PreToolUse |
| [फ़ाइल एक्सेस](#file-access) | block-read-outside-cwd, block-secrets-write | PreToolUse |
| [गिट](#git) | block-push-master, block-work-on-main, block-force-push, warn-git-amend, warn-git-stash-drop, warn-all-files-staged | PreToolUse |
| [डेटाबेस](#database) | warn-destructive-sql, warn-schema-alteration | PreToolUse |
| [चेतावनियाँ](#warnings) | warn-large-file-write, warn-package-publish, warn-background-process, warn-global-package-install | PreToolUse |
| [पैकेज मैनेजर](#package-managers) | prefer-package-manager | PreToolUse |
| [वर्कफ़्लो](#workflow) | require-commit-before-stop, require-push-before-stop, require-pr-before-stop, require-no-conflicts-before-stop, require-ci-green-before-stop | Stop |

- **`block-`** — एजेंट को आगे बढ़ने से रोकें।
- **`warn-`** — एजेंट को अतिरिक्त संदर्भ दें ताकि वह स्व-सुधार कर सके।
- **`sanitize-`** — टूल आउटपुट से संवेदनशील डेटा साफ़ करें इससे पहले कि एजेंट इसे देखे।

### नेमस्पेस

प्रत्येक पॉलिसी एक `<namespace>/<name>` स्लॉट में रहती है। बिल्ट-इन पॉलिसीज़ **`exospherehost/`** नेमस्पेस से संबंधित हैं — उदाहरण के लिए, `exospherehost/sanitize-jwt`। नेमस्पेस तब टकराव से बचाता है जब आप कस्टम या तीसरे पक्ष की पॉलिसीज़ को समान छोटे नामों के साथ लोड करते हैं।

आपके कॉन्फ़िग में आप एक बिल्ट-इन को छोटे नाम या योग्य नाम दोनों से संदर्भित कर सकते हैं; दोनों रूप एक ही पॉलिसी को हल करते हैं:

```json
{
  "enabledPolicies": [
    "sanitize-jwt",
    "exospherehost/block-rm-rf"
  ]
}
```

यदि किसी नाम में कोई `/` नहीं है, तो failproofai इसे डिफ़ॉल्ट नेमस्पेस `exospherehost` से संबंधित मानता है। ऐसे नाम जिनमें पहले से `/` है (जैसे `myorg/foo`, `custom/my-hook`) जैसे-जैसे रखे जाते हैं।
- **`require-`** — स्टॉप इवेंट को तब तक ब्लॉक करें जब तक शर्तें पूरी न हों।

---

<Tip>
प्रत्येक पॉलिसी `policyParams` में एक वैकल्पिक `hint` फ़ील्ड का समर्थन करती है। हिंट को deny या instruct संदेश में जोड़ा जाता है जो Claude देखता है, पॉलिसी कोड को संशोधित किए बिना कार्यशील मार्गदर्शन देता है। बिल्ट-इन, कस्टम और कन्वेंशन पॉलिसीज़ के साथ काम करता है। विवरण के लिए [कॉन्फ़िगरेशन → hint](/hi/configuration#hint-cross-cutting) देखें।
</Tip>

## खतरनाक कमांड

एजेंट्स को ऐसी ऑपरेशन चलाने से रोकें जो आने वाले को मुश्किल हो या होस्ट सिस्टम को नुकसान पहुँचा सकें।

### `block-sudo`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** किसी भी `sudo` कमांड को अस्वीकार करता है।

`sudo` कीवर्ड वाली इनवोकेशन को ब्लॉक करता है। पैटर्न मिलान कच्ची स्ट्रिंग पर नहीं, पार्स किए गए कमांड टोकन पर किया जाता है, ताकि शेल ऑपरेटर इंजेक्शन के माध्यम से बाईपास को रोका जा सके।

**पैरामीटर:**

| पैरम | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | सटीक कमांड प्रीफिक्स जो अनुमत हैं। प्रत्येक प्रविष्टि पार्स किए गए argv टोकन के विरुद्ध मेल खाई जाती है। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    }
  }
}
```

इस कॉन्फ़िग के साथ, `sudo systemctl status nginx` अनुमत है, लेकिन `sudo rm /etc/hosts` अस्वीकृत है।

<Note>
पैटर्न को कच्ची कमांड स्ट्रिंग के विरुद्ध नहीं, पार्स किए गए टोकन के विरुद्ध मेल खाया जाता है। यह संलग्न शेल ऑपरेटर के माध्यम से बाईपास को रोकता है (उदाहरण के लिए `sudo systemctl status x; rm -rf /` `sudo systemctl status *` से मेल नहीं खाता)।
</Note>

### `block-rm-rf`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** `rm -rf`, `rm -fr`, और समान रिकर्सिव डिलीशन फॉर्म को अस्वीकार करता है।

**पैरामीटर:**

| पैरम | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `allowPaths` | `string[]` | `[]` | पाथ जो रिकर्सिवली डिलीट करने के लिए सुरक्षित हैं (जैसे `/tmp`)। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-rm-rf": {
      "allowPaths": ["/tmp", "/var/cache"]
    }
  }
}
```

## इन्फ्रा कमांड

कोडिंग एजेंट्स को इन्फ्रास्ट्रक्चर CLI चलाने या CI/CD पाइपलाइन ट्रिगर करने से रोकें। इस श्रेणी में सभी पॉलिसीज़ **ऑप्ट-इन** हैं (`defaultEnabled: false`) — एजेंट्स जिन्हें वास्तव में `kubectl`, `terraform`, आदि को कॉल करने की आवश्यकता है, वे विचलित नहीं होंगे जब तक आप पॉलिसी सक्षम नहीं करते। जब सक्षम हो, तो मिली कमांड की प्रत्येक इनवोकेशन को अस्वीकार कर दिया जाता है जब तक कि कमांड `allowPatterns` में एक प्रविष्टि से मेल नहीं खाता।

पैटर्न व्याकरण [`block-sudo`](#block-sudo) के समान है: टोकन को पार्स किए गए argv के विरुद्ध मेल खाया जाता है, `*` एक टोकन के लिए वाइल्डकार्ड है, और कोई भी कमांड जिसमें एक स्टैंडअलोन शेल ऑपरेटर (`&&`, `||`, `|`, `;`) या एम्बेडेड शेल मेटाकैरेक्टर वाला टोकन है, इंजेक्शन बाईपास को रोकने के लिए अलाउलिस्ट मिलान से पहले खारिज किया जाता है।

## सीक्रेट्स (सैनिटाइज़र)

एजेंट्स को अपने संदर्भ या आउटपुट में क्रेडेंशियल लीक करने से रोकें। सैनिटाइज़र पॉलिसीज़ **PostToolUse** इवेंट पर काम करती हैं। जब Claude एक Bash कमांड चलाता है, एक फ़ाइल पढ़ता है, या कोई भी टूल कॉल करता है, तो ये पॉलिसीज़ Claude को वापस किए जाने से पहले आउटपुट का निरीक्षण करती हैं। यदि एक सीक्रेट पैटर्न पाया जाता है, तो पॉलिसी एक अस्वीकार निर्णय देती है जो आउटपुट को वापस पास करने से रोकता है।

### `sanitize-jwt`

**इवेंट:** PostToolUse (सभी टूल्स)  
**डिफ़ॉल्ट:** JWT टोकन को रिडैक्ट करता है (`.` द्वारा अलग किए गए तीन base64url सेगमेंट)।

कोई पैरामीटर नहीं।

---

### `sanitize-api-keys`

**इवेंट:** PostToolUse (सभी टूल्स)  
**डिफ़ॉल्ट:** सामान्य API कुंजी प्रारूपों को रिडैक्ट करता है: Anthropic (`sk-ant-`), OpenAI (`sk-`), GitHub PATs (`ghp_`), AWS एक्सेस कुंजी (`AKIA`), Stripe कुंजी (`sk_live_`, `sk_test_`), और Google API कुंजी (`AIza`)।

**पैरामीटर:**

| पैरम | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `additionalPatterns` | `{ regex: string; label: string }[]` | `[]` | सीक्रेट के रूप में मानने के लिए अतिरिक्त regex पैटर्न। |

**उदाहरण:**

```json
{
  "policyParams": {
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo internal API key" },
        { "regex": "pat_[0-9a-f]{40}", "label": "Internal PAT" }
      ]
    }
  }
}
```

---

कोई पैरामीटर नहीं।

### `sanitize-private-key-content`

## फ़ाइल एक्सेस

### `block-read-outside-cwd`

**इवेंट:** PreToolUse (Read, Bash)  
**डिफ़ॉल्ट:** प्रोजेक्ट रूट के बाहर फ़ाइलों को पढ़ने से इनकार करता है। सीमा `CLAUDE_PROJECT_DIR` है (प्रति सेशन Claude Code द्वारा सेट किया जाता है), जब वह वेरिएबल अनसेट हो तो सेशन की वर्तमान कार्य निर्देशिका में फॉलबैक होता है। लाइव `cwd` के बजाय प्रोजेक्ट रूट का उपयोग करने का मतलब है कि सीमा स्थिर रहती है यहाँ तक कि Claude के `cd` करने के बाद भी।

**पैरामीटर:**

| पैरम | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `allowPaths` | `string[]` | `[]` | पूर्ण पथ प्रीफिक्स जो अनुमत हैं यहाँ तक कि प्रोजेक्ट रूट के बाहर हों। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company/config"]
    }
  }
}
```

## पैकेज मैनेजर

यह लागू करें कि एजेंट किस पैकेज मैनेजर का उपयोग करने की अनुमति है।

### `prefer-package-manager`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** अक्षम। जब सक्षम हो, तो `allowed` सूची में न होने वाली किसी भी पैकेज मैनेजर कमांड को ब्लॉक करता है और Claude को अनुमत मैनेजर का उपयोग करके कमांड को फिर से लिखने के लिए कहता है।

पहचानता है: pip, pip3, python -m pip, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo।

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-----------|------|---------|-------------|
| `allowed` | string[] | `[]` | अनुमत पैकेज मैनेजर नाम। इस सूची में न होने वाली कोई भी पहचानी गई मैनेजर को ब्लॉक किया जाता है। जब खाली हो, तो पॉलिसी एक निरर्थक है। |
| `blocked` | string[] | `[]` | बिल्ट-इन सूची के साथ अतिरिक्त मैनेजर नाम को ब्लॉक करने के लिए (जैसे `['pdm', 'pipx']`)। |

बिल्ट-इन ब्लॉक सूची को कवर करता है: pip, pip3, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo। इस सूची में न होने वाली मैनेजर को जोड़ने के लिए `blocked` का उपयोग करें।

**कॉन्फ़िगरेशन उदाहरण:**

```json
{
  "enabledPolicies": ["prefer-package-manager"],
  "policyParams": {
    "prefer-package-manager": {
      "allowed": ["uv", "bun"],
      "blocked": ["pdm", "pipx"]
    }
  }
}
```

इस कॉन्फ़िग के साथ, `pip install flask` और `pdm install flask` दोनों को Claude को `uv` या `bun` का उपयोग करने के लिए कहते हुए एक संदेश के साथ अस्वीकार किया जाता है। `uv pip install flask` जैसी कमांड्स अनुमत हैं क्योंकि `uv` अलाउलिस्ट में है और पहले जाँचा जाता है।

## वर्कफ़्लो

एक अनुशासित सेशन-अंत वर्कफ़्लो लागू करें। ये पॉलिसीज़ **Stop** इवेंट पर काम करती हैं और Claude को रोकने से तब तक इनकार करती हैं जब तक प्रत्येक शर्त पूरी न हो। वे एक प्राकृतिक निर्भरता श्रृंखला का पालन करते हैं: कमिट → पुश → PR → CI। यदि एक पॉलिसी इनकार करती है, तो श्रृंखला में बाद की पॉलिसीज़ छोड़ दी जाती हैं (इनकार शॉर्ट-सर्किट करता है)।

सभी वर्कफ़्लो पॉलिसीज़ **fail-open** हैं: यदि आवश्यक टूल उपलब्ध नहीं है (जैसे `gh` स्थापित नहीं है, कोई git रिमोट नहीं), तो पॉलिसी एक सूचनात्मक संदेश के साथ अनुमति देती है जो समझाता है कि चेक क्यों छोड़ दिया गया।

### `require-commit-before-stop`

**इवेंट:** Stop  
**डिफ़ॉल्ट:** रोकने से इनकार करता है जब असंचित परिवर्तन हों (संशोधित, स्टेज किए गए, या अट्रैक न किए गए फ़ाइलें)। कार्य निर्देशिका स्वच्छ होने पर एक सूचनात्मक संदेश देता है।

कोई पैरामीटर नहीं।

---

### `require-push-before-stop`

**इवेंट:** Stop  
**डिफ़ॉल्ट:** रोकने से इनकार करता है जब अ-पुश किए गए कमिट हों या वर्तमान ब्रांच का कोई रिमोट ट्रैकिंग ब्रांच नहीं है। यदि आवश्यक हो तो `git push -u` का सुझाव देता है एक ट्रैकिंग ब्रांच बनाने के लिए। कोई रिमोट कॉन्फ़िगर नहीं होने पर fail open होता है।

**पैरामीटर:**

| पैरम | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `remote` | `string` | `"origin"` | पुश करने के लिए रिमोट नाम। |

**उदाहरण:**

```json
{
  "policyParams": {
    "require-push-before-stop": {

    }
  }
}
```

<Note>
इस पॉलिसी को [GitHub CLI](https://cli.github.com/) (`gh`) स्थापित और प्रमाणित होने की आवश्यकता है।

pull request तक पढ़ने की पहुँच के लिए। यदि `gh` स्थापित नहीं है या प्रमाणित नहीं है, तो पॉलिसी fail open होती है और कारण को Claude को रिपोर्ट करती है।
</Note>

### `require-no-conflicts-before-stop`

**इवेंट:** Stop  
**डिफ़ॉल्ट:** रोकने से इनकार करता है जब वर्तमान ब्रांच को आधार ब्रांच में स्वच्छ रूप से मर्ज नहीं किया जा सकता। पॉलिसी पहले पुष्टि करती है कि ब्रांच के लिए GitHub पर एक `OPEN` PR है — बिना एक के, कोई मर्ज टार्गेट नहीं है अनुमति देने के लिए, तो संपूर्ण पॉलिसी अनुमति देने के लिए शॉर्ट-सर्किट करती है। एक `OPEN` PR की पुष्टि किए जाने के बाद, दो स्वतंत्र जाँच चलाई जाती हैं:

1. **स्थानीय** — `git merge-tree --write-tree --name-only origin/<baseBranch> HEAD`। संघर्ष पर, अस्वीकार संदेश विरोधी फ़ाइलों को नाम देता है ताकि Claude को बिल्कुल पता हो कि क्या हल करना है।
2. **GitHub** — पहले से प्राप्त `gh pr view --json mergeable,state` परिणाम को फिर से उपयोग करता है। संघर्षों को पकड़ता है जो एक पुरानी स्थानीय `origin/<baseBranch>` से चूक जाते (जैसे कोई `main` पर एक विरोधी PR पहुँचाता है पिछली fetch के बाद)। एक `CONFLICTING` परिणाम अस्वीकार करता है। एक `UNKNOWN` परिणाम भी अस्वीकार करता है और Claude को ~10 सेकंड प्रतीक्षा करने और फिर से जाँचने का निर्देश देता है रुकने का प्रयास करने से पहले — यह झूठी नकारात्मकता को रोकते हुए GitHub को पुनः गणना करने देता है।

पूरी तरह छोड़ देता है (अनुमति देता है) जब: `gh` स्थापित नहीं है, ब्रांच के लिए कोई PR नहीं है, PR की स्थिति `OPEN` नहीं है (उदाहरण के लिए `MERGED`, `CLOSED`), या `gh pr view` अविश्लेषणीय आउटपुट देता है। `origin/<baseBranch>` स्थानीय रूप से गायब होने या आधार से आगे कोई कमिट न होने पर भी fail open होता है — वे Layer 1 fall-through अनुमति देने से पहले अभी भी कैश किए गए PR मर्जेबिलिटी से परामर्श करते हैं।

**पैरामीटर:**

| पैरम | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `baseBranch` | `string` | `"main"` | संघर्षों के लिए जाँचने के लिए आधार ब्रांच। |

<Note>
इस पॉलिसी के लिए GitHub CLI (`gh`) आवश्यक है। पॉलिसी किसी भी संघर्ष जाँच को चलाने से पहले एक `OPEN` PR की पुष्टि करने के लिए `gh pr view` का उपयोग करती है — बिना `gh` के, पॉलिसी अनुमति देने के लिए शॉर्ट-सर्किट करती है। `gh auth login` को `repo` स्कोप वाले व्यक्तिगत एक्सेस टोकन के साथ चलाएँ
pull request तक पढ़ने की पहुँच के लिए।
</Note>

### `require-ci-green-before-stop`

**इवेंट:** Stop  
**डिफ़ॉल्ट:** रोकने से इनकार करता है जब CI चेक वर्तमान ब्रांच पर विफल या अभी चल रहे हों। GitHub Actions वर्कफ़्लो रन और तीसरे पक्ष के बॉट चेक दोनों को जाँचता है (उदाहरण के लिए CodeRabbit, SonarCloud, Codecov)। `skipped` और `cancelled` निष्कर्षों को सफलता मानता है। सभी चेक पास होने पर एक सूचनात्मक संदेश देता है।

कोई पैरामीटर नहीं।

<Note>
इस पॉलिसी के लिए [GitHub CLI](https://cli.github.com/) (`gh`) स्थापित और प्रमाणित होने की आवश्यकता है।
`gh auth login` को `repo` स्कोप वाले व्यक्तिगत एक्सेस टोकन के साथ चलाएँ
Actions वर्कफ़्लो रन और Checks API तक पढ़ने की पहुँच के लिए। यदि `gh` स्थापित नहीं है या प्रमाणित नहीं है, तो पॉलिसी fail open होती है और कारण को Claude को रिपोर्ट करती है।
</Note>

## व्यक्तिगत पॉलिसीज़ को अक्षम करना

अपने कॉन्फ़िग में `enabledPolicies` से एक विशिष्ट पॉलिसी को हटाएँ, या डैशबोर्ड के Policies टैब में इसे बंद करें।

```json
{
  "enabledPolicies": [
    "block-rm-rf",
    "sanitize-api-keys"
  ]
}
```

`enabledPolicies` में सूचीबद्ध न की जाने वाली पॉलिसीज़ नहीं चलती हैं, यहाँ तक कि उनके लिए `policyParams` प्रविष्टियाँ मौजूद हों।
