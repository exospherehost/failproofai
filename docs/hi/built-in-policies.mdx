failproofai 32 बिल्ट-इन पॉलिसीज़ के साथ आता है जो एजेंट की सामान्य विफलता मोड को पकड़ते हैं। प्रत्येक पॉलिसी एक विशेष हुक इवेंट प्रकार और टूल नाम पर फायर करती है। बारह पॉलिसीज़ पैरामीटर स्वीकार करती हैं जो आपको कोड लिखे बिना उनके व्यवहार को ट्यून करने देते हैं। पांच वर्कफ़्लो पॉलिसीज़ Claude को रोकने से पहले एक commit → push → PR → CI पाइपलाइन को लागू करते हैं।

---

## अवलोकन

पॉलिसीज़ को श्रेणियों में विभाजित किया जाता है:

| श्रेणी | पॉलिसीज़ | हुक प्रकार |
|----------|----------|-----------|
| [खतरनाक कमांड](#dangerous-commands) | block-sudo, block-rm-rf, block-curl-pipe-sh, block-failproofai-commands | PreToolUse |
| [सीक्रेट्स (सैनिटाइज़र)](#secrets-sanitizers) | sanitize-jwt, sanitize-api-keys, sanitize-connection-strings, sanitize-private-key-content, sanitize-bearer-tokens | PostToolUse |
| [पर्यावरण](#environment) | block-env-files, protect-env-vars | PreToolUse |
| [फ़ाइल एक्सेस](#file-access) | block-read-outside-cwd, block-secrets-write | PreToolUse |
| [Git](#git) | block-push-master, block-work-on-main, block-force-push, warn-git-amend, warn-git-stash-drop, warn-all-files-staged | PreToolUse |
| [डेटाबेस](#database) | warn-destructive-sql, warn-schema-alteration | PreToolUse |
| [चेतावनियां](#warnings) | warn-large-file-write, warn-package-publish, warn-background-process, warn-global-package-install | PreToolUse |
| [पैकेज मैनेजर](#package-managers) | prefer-package-manager | PreToolUse |
| [वर्कफ़्लो](#workflow) | require-commit-before-stop, require-push-before-stop, require-pr-before-stop, require-no-conflicts-before-stop, require-ci-green-before-stop | Stop |

- **`block-`** — एजेंट को आगे बढ़ने से रोकता है।
- **`warn-`** — एजेंट को अतिरिक्त संदर्भ देता है ताकि वह स्वयं को सुधार सके।
- **`sanitize-`** — एजेंट को दिखाई देने से पहले टूल आउटपुट से संवेदनशील डेटा को हटा देता है।

### नेमस्पेस

प्रत्येक पॉलिसी एक `<namespace>/<name>` स्लॉट में रहती है। बिल्ट-इन पॉलिसीज़ **`exospherehost/`** नेमस्पेस से संबंधित हैं — उदाहरण के लिए, `exospherehost/sanitize-jwt`। नेमस्पेस तब टकराव को रोकता है जब आप कस्टम या तीसरे पक्ष की पॉलिसीज़ को समान नामों के साथ लोड करते हैं।

आपके कॉन्फ़िग में आप एक बिल्ट-इन को या तो इसके छोटे नाम से या योग्य नाम से संदर्भित कर सकते हैं; दोनों रूप एक ही पॉलिसी को हल करते हैं:

```json
{
  "enabledPolicies": [
    "sanitize-jwt",
    "exospherehost/block-rm-rf"
  ]
}
```

यदि किसी नाम में `/` नहीं है, तो failproofai इसे डिफ़ॉल्ट नेमस्पेस `exospherehost` से संबंधित मानता है। नाम जिनमें पहले से `/` है (जैसे `myorg/foo`, `custom/my-hook`) जैसे हैं रहते हैं।
- **`require-`** — जब तक शर्तें पूरी न हो जाएं तब तक Stop इवेंट को ब्लॉक करता है।

---

<Tip>
प्रत्येक पॉलिसी `policyParams` में एक वैकल्पिक `hint` फ़ील्ड को सपोर्ट करती है। हिंट deny या instruct संदेश में जोड़ा जाता है जो Claude देखता है, पॉलिसी कोड को संशोधित किए बिना कार्रवाई योग्य मार्गदर्शन प्रदान करता है। बिल्ट-इन, कस्टम और कन्वेंशन पॉलिसीज़ के साथ काम करता है। विवरण के लिए [Configuration → hint](/hi/configuration#hint-cross-cutting) देखें।
</Tip>

## खतरनाक कमांड

एजेंट्स को ऐसे ऑपरेशन चलाने से रोकें जो पूर्ववत करने के लिए कठिन हों या जो होस्ट सिस्टम को नुकसान पहुंचा सकें।

### `block-sudo`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** किसी भी `sudo` कमांड को अस्वीकार करता है।

`sudo` कीवर्ड वाली इन्वोकेशन को ब्लॉक करता है। पैटर्न मिलान को कच्ची स्ट्रिंग के बजाय पार्स किए गए कमांड टोकन पर किया जाता है, शेल ऑपरेटर इंजेक्शन के माध्यम से बाईपास को रोकने के लिए।

**पैरामीटर:**

| पैरामीटर | टाइप | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | सटीक कमांड प्रीफिक्स जो अनुमत हैं। प्रत्येक प्रविष्टि को पार्स किए गए argv टोकन के विरुद्ध मिलान किया जाता है। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    }
  }
}
```

इस कॉन्फ़िग के साथ, `sudo systemctl status nginx` को अनुमति दी जाती है, लेकिन `sudo rm /etc/hosts` को अस्वीकार किया जाता है।

<Note>
पैटर्न को कच्ची कमांड स्ट्रिंग के बजाय पार्स किए गए टोकन के विरुद्ध मिलान किया जाता है। यह अनुलग्न शेल ऑपरेटर के माध्यम से बाईपास को रोकता है (जैसे `sudo systemctl status x; rm -rf /` `sudo systemctl status *` से मेल नहीं खाता)।
</Note>

---

### `block-rm-rf`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** `rm -rf`, `rm -fr` और समान रिकर्सिव डिलीशन फॉर्म को अस्वीकार करता है।

**पैरामीटर:**

| पैरामीटर | टाइप | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `allowPaths` | `string[]` | `[]` | पाथ जो पुनरावर्ती रूप से हटाने के लिए सुरक्षित हैं (जैसे `/tmp`)। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-rm-rf": {
      "allowPaths": ["/tmp", "/var/cache"]
    }
  }
}
```

---

### `block-curl-pipe-sh`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** `curl <url> | bash`, `curl <url> | sh`, `wget <url> | bash` और समान पैटर्न को अस्वीकार करता है।

कोई पैरामीटर नहीं।

---

### `block-failproofai-commands`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** ऐसी कमांड को अस्वीकार करता है जो failproofai को अनइंस्टॉल या अक्षम करेगी (जैसे `npm uninstall failproofai`, `failproofai policies --uninstall`)।

कोई पैरामीटर नहीं।

## सीक्रेट्स (सैनिटाइज़र)

एजेंट्स को अपने संदर्भ या आउटपुट में क्रेडेंशियल लीक करने से रोकें। सैनिटाइज़र पॉलिसीज़ **PostToolUse** इवेंट पर फायर करती हैं। जब Claude एक Bash कमांड चलाता है, एक फ़ाइल पढ़ता है, या किसी भी टूल को कॉल करता है, ये पॉलिसीज़ आउटपुट को Claude को वापस किए जाने से पहले निरीक्षण करती हैं। यदि एक सीक्रेट पैटर्न का पता चलता है, तो पॉलिसी एक अस्वीकार निर्णय देती है जो आउटपुट को पास होने से रोकता है।

### `sanitize-jwt`

**इवेंट:** PostToolUse (सभी टूल्स)  
**डिफ़ॉल्ट:** JWT टोकन को रिडैक्ट करता है (`.` से अलग तीन base64url सेगमेंट)।

कोई पैरामीटर नहीं।

---

### `sanitize-api-keys`

**इवेंट:** PostToolUse (सभी टूल्स)  
**डिफ़ॉल्ट:** सामान्य API कुंजी फॉर्मेट को रिडैक्ट करता है: Anthropic (`sk-ant-`), OpenAI (`sk-`), GitHub PAT्स (`ghp_`), AWS एक्सेस कुंजियां (`AKIA`), Stripe कुंजियां (`sk_live_`, `sk_test_`), और Google API कुंजियां (`AIza`)।

**पैरामीटर:**

| पैरामीटर | टाइप | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `additionalPatterns` | `{ regex: string; label: string }[]` | `[]` | अतिरिक्त regex पैटर्न जो सीक्रेट मानने हैं। |

**उदाहरण:**

```json
{
  "policyParams": {
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo internal API key" },
        { "regex": "pat_[0-9a-f]{40}", "label": "Internal PAT" }
      ]
    }
  }
}
```

---

### `sanitize-connection-strings`

**इवेंट:** PostToolUse (सभी टूल्स)  
**डिफ़ॉल्ट:** डेटाबेस कनेक्शन स्ट्रिंग्स को रिडैक्ट करता है जिनमें एम्बेडेड क्रेडेंशियल हैं (जैसे `postgresql://user:password@host/db`)।

कोई पैरामीटर नहीं।

---

### `sanitize-private-key-content`

**इवेंट:** PostToolUse (सभी टूल्स)  
**डिफ़ॉल्ट:** PEM ब्लॉक को रिडैक्ट करता है (`-----BEGIN PRIVATE KEY-----`, `-----BEGIN RSA PRIVATE KEY-----`, आदि)।

कोई पैरामीटर नहीं।

---

### `sanitize-bearer-tokens`

**इवेंट:** PostToolUse (सभी टूल्स)  
**डिफ़ॉल्ट:** `Authorization: Bearer <token>` हेडर्स को रिडैक्ट करता है जहां टोकन 20 या अधिक वर्ण है।

कोई पैरामीटर नहीं।

## पर्यावरण

संवेदनशील पर्यावरण कॉन्फ़िगरेशन को एजेंट्स द्वारा पढ़े या उजागर होने से सुरक्षित रखें।

### `block-env-files`

**इवेंट:** PreToolUse (Bash, Read)  
**डिफ़ॉल्ट:** `.env` फ़ाइलों को `cat .env` के माध्यम से, `.env` को फ़ाइल पाथ के रूप में Read टूल कॉल के साथ आदि पढ़ने से अस्वीकार करता है।

`.envrc` या अन्य पर्यावरण-संबंधित फ़ाइलों को ब्लॉक नहीं करता है - केवल बिल्कुल `.env` नामित फ़ाइलें।

कोई पैरामीटर नहीं।

---

### `protect-env-vars`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** पर्यावरण वेरिएबल्स को प्रिंट करने वाली कमांड्स को अस्वीकार करता है: `printenv`, `env`, `echo $VAR`।

कोई पैरामीटर नहीं।

---

## फ़ाइल एक्सेस

एजेंट्स को प्रोजेक्ट सीमाओं के अंदर काम करने और संवेदनशील फ़ाइलों से दूर रहने दें।

### `block-read-outside-cwd`

**इवेंट:** PreToolUse (Read, Bash)  
**डिफ़ॉल्ट:** प्रोजेक्ट रूट के बाहर फ़ाइलों को पढ़ने से अस्वीकार करता है। सीमा `CLAUDE_PROJECT_DIR` है (Claude Code द्वारा प्रति सेशन सेट किया जाता है), जब वह वेरिएबल अनसेट होता है तो सेशन की वर्तमान कार्यिंग डायरेक्टरी में फॉलबैक होता है। लाइव `cwd` के बजाय प्रोजेक्ट रूट का उपयोग करने का अर्थ है कि सीमा स्थिर रहती है भले ही Claude `cd` को एक उप-डायरेक्टरी में करे।

**पैरामीटर:**

| पैरामीटर | टाइप | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `allowPaths` | `string[]` | `[]` | निरपेक्ष पाथ प्रीफिक्स जो अनुमत हैं भले ही प्रोजेक्ट रूट के बाहर हों। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company/config"]
    }
  }
}
```

---

### `block-secrets-write`

**इवेंट:** PreToolUse (Write, Edit)  
**डिफ़ॉल्ट:** निजी कुंजियों और प्रमाणपत्रों के लिए आमतौर पर उपयोग की जाने वाली फ़ाइलों को लिखने से अस्वीकार करता है: `id_rsa`, `id_ed25519`, `*.key`, `*.pem`, `*.p12`, `*.pfx`।

**पैरामीटर:**

| पैरामीटर | टाइप | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `additionalPatterns` | `string[]` | `[]` | अतिरिक्त फ़ाइलनाम पैटर्न (glob-स्टाइल) को ब्लॉक करते हैं। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-secrets-write": {
      "additionalPatterns": [".token", ".secret"]
    }
  }
}
```

## Git

आकस्मिक पुश, फोर्स-पुश और ब्रांच गलतियों को रोकें जो पूर्ववत करने के लिए कठिन हैं।

### `block-push-master`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** `git push origin main` और `git push origin master` को अस्वीकार करता है।

**पैरामीटर:**

| पैरामीटर | टाइप | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `protectedBranches` | `string[]` | `["main", "master"]` | ब्रांच नाम जो सीधे पुश नहीं किए जा सकते। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "master", "release", "prod"]
    }
  }
}
```

<Tip>
सभी ब्रांच को पुश करने की अनुमति देने के लिए (प्रभावी रूप से इस पॉलिसी को `enabledPolicies` से हटाए बिना अक्षम करने के लिए), `protectedBranches: []` सेट करें।
</Tip>

---

### `block-work-on-main`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** `main` या `master` ब्रांच को सीधे चेक आउट करने से अस्वीकार करता है।

**पैरामीटर:**

| पैरामीटर | टाइप | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `protectedBranches` | `string[]` | `["main", "master"]` | ब्रांच नाम जो सीधे चेक आउट नहीं किए जा सकते। |

---

### `block-force-push`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** `git push --force` और `git push -f` को अस्वीकार करता है।

कोई पॉलिसी-विशिष्ट पैरामीटर नहीं। क्रॉस-कटिंग [`hint`](/hi/configuration#hint-cross-cutting) का उपयोग विकल्प सुझाने के लिए करें:

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "अपने वर्तमान HEAD से एक नई शाखा बनाएं (जैसे `git checkout -b <new-branch>`) और उसे पुश करें।"
    }
  }
}
```

---

### `warn-git-amend`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** Claude को सावधानीपूर्वक आगे बढ़ने के लिए निर्देशित करता है जब `git commit --amend` चला रहा हो। कमांड को ब्लॉक नहीं करता है।

कोई पैरामीटर नहीं।

---

### `warn-git-stash-drop`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** Claude को `git stash drop` चलाने से पहले पुष्टि करने के लिए निर्देशित करता है। कमांड को ब्लॉक नहीं करता है।

## पैकेज मैनेजर

लागू करें कि एजेंट किस पैकेज मैनेजर को उपयोग करने की अनुमति दी जाती है।

### `prefer-package-manager`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** अक्षम। जब सक्षम किया जाता है, तो `allowed` सूची में नहीं होने वाली किसी भी पैकेज मैनेजर कमांड को ब्लॉक करता है और Claude को अनुमत मैनेजर का उपयोग करके कमांड को फिर से लिखने के लिए बताता है।

पहचानता है: pip, pip3, python -m pip, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo।

| पैरामीटर | टाइप | डिफ़ॉल्ट | विवरण |
|-----------|------|---------|-------------|
| `allowed` | string[] | `[]` | अनुमत पैकेज मैनेजर नाम। इस सूची में नहीं पाया गया कोई पहचाना गया मैनेजर ब्लॉक किया जाता है। जब खाली हो, तो पॉलिसी एक no-op है। |
| `blocked` | string[] | `[]` | बिल्ट-इन सूची से परे अतिरिक्त मैनेजर नाम को ब्लॉक करें (जैसे `['pdm', 'pipx']`)। |

बिल्ट-इन ब्लॉक सूची शामिल है: pip, pip3, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo। इस सूची में नहीं होने वाले मैनेजर को जोड़ने के लिए `blocked` का उपयोग करें।

**उदाहरण कॉन्फ़िगरेशन:**

```json
{
  "enabledPolicies": ["prefer-package-manager"],
  "policyParams": {
    "prefer-package-manager": {
      "allowed": ["uv", "bun"],
      "blocked": ["pdm", "pipx"]
    }
  }
}
```

इस कॉन्फ़िग के साथ, `pip install flask` और `pdm install flask` दोनों को अस्वीकार किया जाता है जिसमें एक संदेश Claude को `uv` या `bun` का उपयोग करने के लिए बताता है। `uv pip install flask` जैसी कमांड अनुमत हैं क्योंकि `uv` allowlist में है और पहले चेक किया जाता है।

## वर्कफ़्लो

अनुशासित एंड-ऑफ-सेशन वर्कफ़्लो को लागू करें। ये पॉलिसीज़ **Stop** इवेंट पर फायर करते हैं और Claude को तब तक रोकने से अस्वीकार करते हैं जब तक प्रत्येक शर्त पूरी न हो जाए। वे एक प्राकृतिक निर्भरता श्रृंखला का पालन करते हैं: commit → push → PR → CI। यदि कोई पॉलिसी अस्वीकार करती है, तो श्रृंखला में बाद की पॉलिसीज़ को छोड़ दिया जाता है (अस्वीकार शॉर्ट-सर्किट करता है)।

सभी वर्कफ़्लो पॉलिसीज़ **फेल-ओपन** हैं: यदि आवश्यक टूल उपलब्ध नहीं है (जैसे `gh` इंस्टॉल नहीं है, कोई git remote नहीं), पॉलिसी एक सूचनात्मक संदेश के साथ अनुमति देती है जो समझाता है कि चेक को क्यों छोड़ दिया गया।

### `require-commit-before-stop`

**इवेंट:** Stop  
**डिफ़ॉल्ट:** अपरिवर्तित परिवर्तन (संशोधित, स्टेज किए गए, या अट्रैक न किए गए फ़ाइलें) होने पर रोकने से अस्वीकार करता है। जब कार्यिंग डायरेक्टरी स्वच्छ हो तो सूचनात्मक संदेश देता है।

कोई पैरामीटर नहीं।

---

### `require-push-before-stop`

**इवेंट:** Stop  
**डिफ़ॉल्ट:** अप्रकाशित commits होने पर या जब वर्तमान ब्रांच के पास कोई remote tracking branch नहीं होता है तो रोकने से अस्वीकार करता है। यदि आवश्यक हो तो `git push -u` का सुझाव देता है। यदि कोई remote कॉन्फ़िगर नहीं है तो फेल ओपन करता है।

**पैरामीटर:**

| पैरामीटर | टाइप | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `remote` | `string` | `"origin"` | Push करने के लिए remote नाम। |

**उदाहरण:**

```json
{
  "policyParams": {
    "require-push-before-stop": {
      "remote": "upstream"
    }
  }
}
```

**इवेंट:** Stop  
**डिफ़ॉल्ट:** जब वर्तमान ब्रांच के लिए कोई pull request नहीं होता है, या जब मौजूदा PR बंद/मर्ज होता है तो रोकने से अस्वीकार करता है। Claude को `gh pr create` के साथ एक PR बनाने के लिए निर्देशित करता है।

कोई पैरामीटर नहीं।

<Note>
यह पॉलिसी [GitHub CLI](https://cli.github.com/) (`gh`) को इंस्टॉल और प्रामाणीकृत करने की आवश्यकता है।
`gh auth login` को एक personal access token के साथ चलाएं जिसमें pull request को पढ़ने की एक्सेस के लिए `repo` scope हो।
यदि `gh` इंस्टॉल या प्रामाणीकृत नहीं है, तो पॉलिसी फेल ओपन करता है और कारण को Claude को रिपोर्ट करता है।
</Note>

### `require-no-conflicts-before-stop`

**इवेंट:** Stop  
**डिफ़ॉल्ट:** जब वर्तमान ब्रांच को बेस ब्रांच में स्वच्छता से मर्ज नहीं किया जा सकता है तो रोकने से अस्वीकार करता है। दो स्वतंत्र जांच चलाता है:

1. **स्थानीय** — `git merge-tree --write-tree --name-only origin/<baseBranch> HEAD`। संघर्ष पर, अस्वीकार संदेश टकराव वाली फ़ाइलों को नाम देता है ताकि Claude को पता चले कि क्या हल करना है।
2. **GitHub** — `gh pr view --json mergeable,state`। टकराव को पकड़ता है जो एक मृत स्थानीय `origin/<baseBranch>` से चूक जाता (जैसे कोई व्यक्ति एक टकराव वाला PR को `main` पर लैंड करता है क्योंकि अंतिम फेच के बाद)। एक `CONFLICTING` परिणाम अस्वीकार करता है। एक `UNKNOWN` परिणाम एक `OPEN` PR पर (GitHub अभी भी computing कर रहा है) भी अस्वीकार करता है और Claude को ~10 सेकंड प्रतीक्षा करने और फिर से चेक करने के लिए निर्देशित करता है फिर रोकने का प्रयास करने से पहले — यह GitHub recomputes के दौरान false negatives को रोकता है। चेक को पूरी तरह से छोड़ दिया जाता है किसी भी PR के लिए जिसकी स्थिति `OPEN` नहीं है (जैसे `MERGED`, `CLOSED`), क्योंकि GitHub non-open PRs के लिए mergeability computing बंद करता है।

फेल ओपन करता है जब `origin/<baseBranch>` स्थानीय रूप से missing है, जब base के आगे कोई commits नहीं हैं, जब `gh` इंस्टॉल नहीं है, जब ब्रांच के लिए कोई PR नहीं है, या जब PR OPEN नहीं है।

**पैरामीटर:**

| पैरामीटर | टाइप | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `baseBranch` | `string` | `"main"` | टकराव के विरुद्ध जांच करने के लिए बेस ब्रांच। |

<Note>
GitHub CLI (`gh`) इस पॉलिसी के लिए वैकल्पिक लेकिन अनुशंसित है — यह दूसरी पहचान परत प्रदान करता है जो एक मृत स्थानीय `origin/<baseBranch>` को चूक जाने वाले टकराव को पकड़ता है।
`gh` के बिना, केवल स्थानीय `git merge-tree` चेक चलता है।
</Note>

## व्यक्तिगत पॉलिसीज़ को अक्षम करना

अपने कॉन्फ़िग में `enabledPolicies` से एक विशेष पॉलिसी को हटाएं, या डैशबोर्ड के Policies टैब में इसे बंद करें।

```json
{
  "enabledPolicies": [
    "block-rm-rf",
    "sanitize-api-keys"
  ]
}
```

`enabledPolicies` में सूचीबद्ध नहीं की गई पॉलिसीज़ नहीं चलती हैं, भले ही `policyParams` प्रविष्टियां उनके लिए मौजूद हों।
