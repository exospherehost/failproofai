failproofai 39 built-in policies के साथ आता है जो common agent failure modes को पकड़ते हैं। प्रत्येक policy एक विशिष्ट hook event type और tool name पर कार्य करता है। उन्नीस policies parameters स्वीकार करती हैं जो आपको कोड लिखे बिना उनके behavior को tune करने देते हैं। पाँच workflow policies एक commit → push → PR → CI pipeline को लागू करते हैं जब तक Claude रुक न जाए।

---

## Overview

Policies को categories में समूहीकृत किया जाता है:

| Category | Policies | Hook type |
|----------|----------|-----------|
| [Dangerous commands](#dangerous-commands) | block-sudo, block-rm-rf, block-curl-pipe-sh, block-failproofai-commands | PreToolUse |
| [Infra commands](#infra-commands) | block-kubectl, block-terraform, block-aws-cli, block-gcloud, block-az-cli, block-helm, block-gh-pipeline | PreToolUse |
| [Secrets (sanitizers)](#secrets-sanitizers) | sanitize-jwt, sanitize-api-keys, sanitize-connection-strings, sanitize-private-key-content, sanitize-bearer-tokens | PostToolUse |
| [Environment](#environment) | block-env-files, protect-env-vars | PreToolUse |
| [File access](#file-access) | block-read-outside-cwd, block-secrets-write | PreToolUse |
| [Git](#git) | block-push-master, block-work-on-main, block-force-push, warn-git-amend, warn-git-stash-drop, warn-all-files-staged | PreToolUse |
| [Database](#database) | warn-destructive-sql, warn-schema-alteration | PreToolUse |
| [Warnings](#warnings) | warn-large-file-write, warn-package-publish, warn-background-process, warn-global-package-install | PreToolUse |
| [Package managers](#package-managers) | prefer-package-manager | PreToolUse |
| [Workflow](#workflow) | require-commit-before-stop, require-push-before-stop, require-pr-before-stop, require-no-conflicts-before-stop, require-ci-green-before-stop | Stop |

- **`block-`** — agent को आगे बढ़ने से रोकता है।
- **`warn-`** — agent को अतिरिक्त context देता है ताकि वह self-correct कर सके।
- **`sanitize-`** — tool output से sensitive data को हटाता है इससे पहले कि agent इसे देखे।

### Namespaces

प्रत्येक policy एक `<namespace>/<name>` स्लॉट में रहता है। Built-in policies **`exospherehost/`** namespace में होती हैं — उदाहरण के लिए, `exospherehost/sanitize-jwt`। यह namespace collisions को रोकता है जब आप custom या third-party policies को भी load करते हैं जिनके समान short names हो सकते हैं।

आप अपनी config में एक built-in policy को उसके short name या qualified name से refer कर सकते हैं; दोनों forms एक ही policy को resolve करते हैं:

```json
{
  "enabledPolicies": [
    "sanitize-jwt",
    "exospherehost/block-rm-rf"
  ]
}
```

अगर किसी नाम में `/` नहीं है, तो failproofai इसे default namespace `exospherehost` से संबंधित मानता है। ऐसे नाम जिनमें पहले से `/` है (जैसे `myorg/foo`, `custom/my-hook`) को as-is रखा जाता है।
- **`require-`** — Stop event को तब तक block करता है जब तक conditions पूरी न हो जाएँ।

---

<Tip>
प्रत्येक policy `policyParams` में एक optional `hint` field को support करता है। यह hint deny या instruct message में append होता है जो Claude देखता है, actionable guidance देता है बिना policy code को modify किए। Built-in, custom, और convention policies के साथ काम करता है। विवरण के लिए [Configuration → hint](/hi/configuration#hint-cross-cutting) देखें।
</Tip>

## Dangerous commands

Agents को ऐसे operations से रोकें जो undo करना मुश्किल हो या host system को नुकसान पहुँचा सकते हों।

### `block-sudo`

**Event:** PreToolUse (Bash)  
**Default:** किसी भी `sudo` command को deny करता है।

`sudo` keyword को शामिल करने वाली invocations को block करता है। Pattern matching को raw string पर नहीं बल्कि parsed command tokens पर किया जाता है, ताकि shell operator injection के माध्यम से bypass को रोका जा सके।

**Parameters:**

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | Exact command prefixes जो permitted हैं। प्रत्येक entry को parsed argv tokens के विरुद्ध match किया जाता है। |

**Example:**

```json
{
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    }
  }
}
```

इस config के साथ, `sudo systemctl status nginx` को allow किया जाता है, लेकिन `sudo rm /etc/hosts` को deny किया जाता है।

<Note>
Patterns को raw command string पर नहीं बल्कि parsed tokens के विरुद्ध match किया जाता है। यह appended shell operators के माध्यम से bypass को रोकता है (जैसे `sudo systemctl status x; rm -rf /` को `sudo systemctl status *` के विरुद्ध match नहीं किया जाता)।
</Note>

## Infra commands

Coding agents को infrastructure CLIs चलाने या CI/CD pipelines को trigger करने से रोकें। इस category में सभी policies **opt-in** हैं (`defaultEnabled: false`) — agents जिन्हें legitimately `kubectl`, `terraform`, आदि को call करना है, वे disrupted नहीं होंगे जब तक आप policy को enable न करें। जब enabled हों, तो matched CLI की हर invocation को deny किया जाता है जब तक command `allowPatterns` में entry से match न करे।

Pattern grammar [`block-sudo`](#block-sudo) के समान है: tokens को parsed argv के विरुद्ध match किया जाता है, `*` एक token के लिए wildcard है, और कोई भी command जिसमें standalone shell operator (`&&`, `||`, `|`, `;`) या embedded shell metacharacters वाला token हो, injection bypasses को रोकने के लिए allowlist matching से पहले reject किया जाता है।

### `block-kubectl`

**Event:** PreToolUse (Bash)  
**Default:** किसी भी `kubectl` invocation को deny करता है।

**Parameters:**

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | kubectl command prefixes जो permitted हैं। |

**Example:**

```json
{
  "policyParams": {
    "block-kubectl": {
      "allowPatterns": ["kubectl get *", "kubectl describe *", "kubectl logs *"]
    }
  }
}
```

इस config के साथ, `kubectl get pods` को allow किया जाता है लेकिन `kubectl apply -f deploy.yaml` को deny किया जाता है।

### `block-gh-pipeline`

**Event:** PreToolUse (Bash)  
**Default:** निम्नलिखित `gh` CLI subcommands को deny करता है जो state को mutate करते हैं या pipelines को trigger करते हैं:

- `gh workflow run`, `gh workflow enable`, `gh workflow disable`
- `gh run rerun`, `gh run cancel`
- `gh pr merge`
- `gh release create`, `gh release delete`
- `gh cache delete`
- `gh secret set`, `gh secret delete`

Read-only `gh` subcommands जैसे `gh pr view`, `gh pr list`, `gh run list`, `gh release view`, और `gh api repos/.../...` इस policy से **match नहीं** होते हैं — वे workflow checks के लिए regularly जरूरी होते हैं (failproofai के अपने `require-ci-green-before-stop` को शामिल करते हुए)।

**Parameters:**

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | Specific scripted invocations जिन्हें allow किया जाए भले ही वे अन्यथा deny हों। |

**Example:**

```json
{
  "policyParams": {
    "block-gh-pipeline": {
      "allowPatterns": ["gh run rerun *"]
    }
  }
}
```

## Secrets (sanitizers)

Agents को credentials को अपने context या output में leak करने से रोकें। Sanitizer policies **PostToolUse** events पर fire होती हैं। जब Claude एक Bash command चलाता है, एक file पढ़ता है, या कोई tool call करता है, तो ये policies को output को inspect करते हैं इससे पहले कि यह Claude को return किया जाए। अगर एक secret pattern detect होता है, तो policy एक deny decision return करता है जो output को pass back होने से रोकता है।

### `sanitize-jwt`

**Event:** PostToolUse (all tools)  
**Default:** JWT tokens को redact करता है (तीन base64url segments जो `.` से separated हों)।

कोई parameters नहीं।

---

### `sanitize-api-keys`

**Event:** PostToolUse (all tools)  
**Default:** Common API key formats को redact करता है: Anthropic (`sk-ant-`), OpenAI (`sk-`), GitHub PATs (`ghp_`), AWS access keys (`AKIA`), Stripe keys (`sk_live_`, `sk_test_`), और Google API keys (`AIza`)।

**Parameters:**

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `additionalPatterns` | `{ regex: string; label: string }[]` | `[]` | Additional regex patterns जो secrets के रूप में treat किए जाएँ। |

**Example:**

```json
{
  "policyParams": {
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo internal API key" },
        { "regex": "pat_[0-9a-f]{40}", "label": "Internal PAT" }
      ]
    }
  }
}
```

---

### `sanitize-connection-strings`

**Event:** PostToolUse (all tools)  
**Default:** Database connection strings को redact करता है जिनमें embedded credentials हों (जैसे `postgresql://user:password@host/db`)।

कोई parameters नहीं।

---

### `sanitize-private-key-content`

**Event:** PostToolUse (all tools)  
**Default:** PEM blocks को redact करता है (`-----BEGIN PRIVATE KEY-----`, `-----BEGIN RSA PRIVATE KEY-----`, आदि)।

कोई parameters नहीं।

---

### `sanitize-bearer-tokens`

**Event:** PostToolUse (all tools)  
**Default:** `Authorization: Bearer <token>` headers को redact करता है जहाँ token 20 या अधिक characters हो।

कोई parameters नहीं।

## Environment

Agents से sensitive environment configuration को पढ़े जाने या expose होने से बचाएँ।

### `block-env-files`

**Event:** PreToolUse (Bash, Read)  
**Default:** `.env` files को पढ़ने से deny करता है `cat .env` के माध्यम से, Read tool calls जहाँ `.env` file path हो, आदि।

`.envrc` या अन्य environment-adjacent files को block नहीं करता है — केवल exactly `.env` नाम वाली files को।

कोई parameters नहीं।

---

### `protect-env-vars`

**Event:** PreToolUse (Bash)  
**Default:** ऐसे commands को deny करता है जो environment variables को print करते हैं: `printenv`, `env`, `echo $VAR`।

कोई parameters नहीं।

---

## File access

Agents को project boundaries के अंदर काम करने देते हुए sensitive files से दूर रखें।

### `block-read-outside-cwd`

**Event:** PreToolUse (Read, Bash)  
**Default:** Project root के बाहर files को पढ़ने से deny करता है। Boundary `CLAUDE_PROJECT_DIR` है (session per एक बार Claude Code द्वारा set किया गया), जब वह variable unset हो तो session के current working directory को fallback के रूप में। Project root को live `cwd` के बजाय use करने का मतलब है कि boundary stable रहता है भले ही Claude किसी subdirectory में `cd` करे।

**Parameters:**

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `allowPaths` | `string[]` | `[]` | Absolute path prefixes जो permitted हैं भले ही project root के बाहर हों। |

**Example:**

```json
{
  "policyParams": {
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company/config"]
    }
  }
}
```

---

### `block-secrets-write`

**Event:** PreToolUse (Write, Edit)  
**Default:** Commonly used files में writes को deny करता है private keys और certificates के लिए: `id_rsa`, `id_ed25519`, `*.key`, `*.pem`, `*.p12`, `*.pfx`।

**Parameters:**

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `additionalPatterns` | `string[]` | `[]` | Additional filename patterns (glob-style) जो block किए जाएँ। |

**Example:**

```json
{
  "policyParams": {
    "block-secrets-write": {
      "additionalPatterns": [".token", ".secret"]
    }
  }
}
```

## Git

Accidental pushes, force-pushes, और branch mistakes को रोकें जो undo करना मुश्किल हो।

### `block-push-master`

**Event:** PreToolUse (Bash)  
**Default:** `git push origin main` और `git push origin master` को deny करता है।

**Parameters:**

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `protectedBranches` | `string[]` | `["main", "master"]` | Branch names जो directly push नहीं किए जा सकते। |

**Example:**

```json
{
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "master", "release", "prod"]
    }
  }
}
```

<Tip>
सभी branches में push करने की अनुमति देने के लिए (बिना `enabledPolicies` से इसे हटाए इस policy को effectively disable करने के लिए), `protectedBranches: []` को set करें।
</Tip>

---

### `block-work-on-main`

**Event:** PreToolUse (Bash)  
**Default:** `git commit`, `git merge`, `git rebase`, और `git cherry-pick` को deny करता है जब working tree `main` या `master` पर हो। Branch creation और switching (`git checkout`, `git checkout -b`, `git switch`, `git switch -c`) affected नहीं होते हैं।

**Parameters:**

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `protectedBranches` | `string[]` | `["main", "master"]` | Branch names जहाँ commit/merge/rebase/cherry-pick को deny किया जाता है। |

---

### `block-force-push`

**Event:** PreToolUse (Bash)  
**Default:** `git push --force` और `git push -f` को deny करता है।

कोई policy-specific parameters नहीं। Cross-cutting [`hint`](/hi/configuration#hint-cross-cutting) को use करें alternatives suggest करने के लिए:

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "अपने current HEAD से एक नई branch बनाएँ (जैसे `git checkout -b <new-branch>`) और उसे push करें।"
    }
  }
}
```

---

### `warn-git-amend`

**Event:** PreToolUse (Bash)  
**Default:** Claude को सावधानी से proceed करने के लिए instruct करता है जब `git commit --amend` चलाया जाता है। Command को block नहीं करता।

कोई parameters नहीं।

---

### `warn-git-stash-drop`

**Event:** PreToolUse (Bash)  
**Default:** Claude को `git stash drop` चलाने से पहले confirm करने के लिए instruct करता है। Command को block नहीं करता।

कोई parameters नहीं।

---

### `warn-all-files-staged`

**Event:** PreToolUse (Bash)  
**Default:** Claude को review करने के लिए instruct करता है कि यह क्या stage कर रहा है जब `git add -A` या `git add .` चलाया जाता है। Command को block नहीं करता।

कोई parameters नहीं।

## Warnings

Agents को potentially risky लेकिन non-destructive operations से पहले extra context दें।

### `warn-large-file-write`

**Event:** PreToolUse (Write)  
**Default:** Claude को 1024 KB से बड़ी files लिखने से पहले confirm करने के लिए instruct करता है।

**Parameters:**

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `thresholdKb` | `number` | `1024` | File size threshold kilobytes में जहाँ से एक warning issue किया जाता है। |

**Example:**

```json
{
  "policyParams": {
    "warn-large-file-write": {
      "thresholdKb": 256
    }
  }
}
```

<Note>
Hook handler payloads पर 1 MB stdin limit enforce करता है। इस policy को small content के साथ test करने के लिए, `thresholdKb` को 1024 से बहुत नीचे की value पर set करें।
</Note>

---

## Package managers

Enforce करें कि agent कौन से package managers use कर सकता है।

### `prefer-package-manager`

**Event:** PreToolUse (Bash)  
**Default:** Disabled। जब enabled हो, तो कोई भी package manager command को block करता है जो `allowed` list में न हो और Claude को command को एक allowed manager का use करके rewrite करने के लिए बताता है।

Detects: pip, pip3, python -m pip, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo।

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `allowed` | string[] | `[]` | Allowed package manager names। कोई भी detected manager जो इस list में न हो, block किया जाता है। जब empty हो, तो policy एक no-op है। |
| `blocked` | string[] | `[]` | Additional manager names जो built-in list के beyond block किए जाएँ (जैसे `['pdm', 'pipx']`)। |

Built-in block list covers: pip, pip3, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo। इस list में न होने वाले managers को append करने के लिए `blocked` को use करें।

**Example configuration:**

```json
{
  "enabledPolicies": ["prefer-package-manager"],
  "policyParams": {
    "prefer-package-manager": {
      "allowed": ["uv", "bun"],
      "blocked": ["pdm", "pipx"]
    }
  }
}
```

इस config के साथ, `pip install flask` और `pdm install flask` दोनों को deny किया जाता है जिसमें Claude को `uv` या `bun` का use करने के लिए कहा जाता है। `uv pip install flask` जैसे commands को allow किया जाता है क्योंकि `uv` allowlist में है और पहले check किया जाता है।

## Workflow

एक disciplined end-of-session workflow को enforce करें। ये policies **Stop** event पर fire होती हैं और agent को तब तक stop करने से deny करती हैं जब तक प्रत्येक condition पूरी न हो जाए। वे एक natural dependency chain follow करती हैं: commit → push → PR → CI। अगर एक policy deny करता है, तो chain में later policies skip किए जाते हैं (deny short-circuits)।

सभी workflow policies **fail-open** हैं: अगर required tool available नहीं है (जैसे `gh` installed नहीं है, कोई git remote नहीं है), तो policy allow करता है एक informational message के साथ जो explain करता है कि check को क्यों skip किया गया।

### Per-CLI Stop semantics

Stop enforcement हर एक supported CLI में slightly अलग दिखता है क्योंकि प्रत्येक एक अलग expose करता है agent finished hook contract। **Outcome** same है — agent stop करते समय एक workflow gate failing पर दूर नहीं हो जाता — लेकिन **mechanics** अलग हैं। नीचे दी गई table summarize करती है; केवल Pi के पास एक user-visible quirk है जो समझने लायक है इससे पहले कि आप एक `require-*-before-stop` policy को enable करें।

| CLI | जब gate fire होता है | आप क्या देखते हैं |
|---|---|---|
| Claude Code | Same agent loop, immediately | Claude काम करना जारी रखता है — issue को fix करता है, फिर finish करने का attempt करता है। आपके लिए कोई visible interruption नहीं। |
| Codex | Same agent loop, immediately | Claude के समान। |
| GitHub Copilot CLI | Same agent loop, immediately | Claude के समान (Copilot की `{decision:"block", reason}` retry channel को use करता है — empirically Copilot CLI 1.0.41 के विरुद्ध verified)। |
| Cursor Agent | Same agent loop, immediately | Claude के समान (Cursor की `{followup_message}` channel को use करता है — `loop_limit` तक capped, default 5 retries)। |
| Gemini CLI | Same agent loop, immediately | Claude के समान (Gemini की `{decision:"block", reason}` channel को `AfterAgent` पर use करता है)। |
| OpenCode | Same agent loop, immediately | Claude के समान (OpenCode की `client.session.prompt(...)` SDK call को `hookSpecificOutput.additionalContext` के माध्यम से routed)। |
| **Pi (pi-coding-agent)** | **Next user turn** | **Pi visibly stops** जब gate fire होता है — इसका agent loop exit होता है और आप prompt पर return होते हैं। Gate फिर अगली बार fire होता है जब आप prompt submit करते हैं: failproofai एक `MANDATORY ACTION REQUIRED` directive को उस turn के system prompt में prepend करता है, instruction देता है कि LLM को workflow step (commit, push, आदि) को complete करना है आप जो पूछा है उससे पहले। |

<Note>
**Pi limitation।** Pi की `AgentEndEvent` (Claude की `Stop` hook के upstream equivalent) का कोई Result type नहीं है — जब तक यह fire होता है, Pi का agent loop पहले से ही exit हो चुका होता है। Pi को same loop को retry करने के लिए force नहीं किया जा सकता Claude / Copilot / Cursor / Gemini / OpenCode तरीके से। failproofai gate को Pi की `before_agent_start` event में shift करता है (जो अगली user prompt के बाद fire होती है) ताकि workflow check अभी भी enforce हो, बस current one के बजाय next turn पर।

**व्यावहारिक रूप से इसका क्या मतलब है:**

- Pi के stop होने के बाद, deny reason को in-memory में Pi session id द्वारा keyed capture किया जाता है। बिल्कुल अगला prompt जो आप same Pi process में submit करते हैं इसे drain करता है: LLM अपने system prompt के top पर `MANDATORY ACTION REQUIRED` directive को देखता है, commit करता है (या push / PR खोलता है / CI के लिए wait करता है), और केवल तब आपके request के साथ continue करता है। Captured deny reason one-shot है — एक बार drain होने पर, gate clear है।

- Gate को Pi की process lifetime से bound किया जाता है। अगर आप Pi को `Ctrl+C` करते हैं या turns के बीच quit करते हैं, तो in-memory entry को process के साथ drop किया जाता है और gate miss होता है। Claude, Copilot, Cursor, Gemini, और OpenCode का same bound है (agent को kill करो और gate miss होता है) — Pi बस इसे अधिक visible बनाता है क्योंकि agent visibly exit होता है gate fire होने से पहले।

- एक pending deny भी किसी भी कारण से `session_shutdown` पर clear होता है (`new` / `resume` / `fork` / `quit`), तो एक stale gate एक prior session से fresh session में leak नहीं हो सकता same Pi process में start किया गया।

अगर आपको Claude-style same-loop retry चाहिए, अपने `Stop` policies को अन्य छह supported CLIs में से किसी के अंतर्गत run करें। हम Pi upstream को track कर रहे हैं एक future Result type के लिए `AgentEndEvent` पर जो हमें इस gap को close करने देगा।
</Note>

### `require-pr-before-stop`

**Event:** Stop  
**Default:** जब current branch के लिए कोई pull request exist न हो या existing PR को merge के बिना closed हो तो stop करने से deny करता है। Claude को `gh pr create` के साथ PR बनाने के लिए instruct करता है। जब PR **merged** हो, तो policy allow करता है (work ship हो गया है) और message hint देता है branch से switch करने के लिए (`git checkout main && git pull`)।

कोई parameters नहीं।

<Note>
यह policy [GitHub CLI](https://cli.github.com/) (`gh`) को installed और authenticated होना require करता है।
`gh auth login` को एक personal access token के साथ run करें जिसके पास pull requests को read access के लिए `repo` scope हो। अगर `gh` installed नहीं है या authenticated नहीं है, तो policy fail open करता है और reason को Claude को report करता है।
</Note>

### `require-no-conflicts-before-stop`

**Event:** Stop  
**Default:** जब current branch को cleanly base branch में merge नहीं किया जा सकता तो stop करने से deny करता है। Policy पहले confirm करता है कि current branch के लिए GitHub पर एक `OPEN` PR है — उसके बिना, कोई merge target enforce करने के लिए नहीं है, तो entire policy short-circuit होकर allow करता है। एक बार `OPEN` PR confirm होने पर, दो independent probes run होते हैं:

1. **Local** — `git merge-tree --write-tree --name-only origin/<baseBranch> HEAD`। Conflict पर, deny message conflicted files को name करता है ताकि Claude को पता चले कि क्या resolve करना है।
2. **GitHub** — `gh pr view --json mergeable,state` result को reuse करता है जो पहले से ही precheck में fetch किया गया है। Conflicts को catch करता है जो एक stale local `origin/<baseBranch>` miss करेगा (जैसे कोई conflicting PR को `main` पर land किया है last fetch के बाद)। एक `CONFLICTING` result deny करता है। एक `UNKNOWN` result भी deny करता है और Claude को instruct करता है ~10 seconds wait करने और फिर से check करने के लिए stop करने का attempt करने से पहले — यह false negatives को prevent करता है जबकि GitHub recompute करता है।

Entirely skip करता है (allow करता है) जब: `gh` installed नहीं है, branch के लिए कोई PR exist नहीं है, PR की state `OPEN` नहीं है (जैसे `MERGED`, `CLOSED`), या `gh pr view` unparseable output return करता है। भी fail open करता है जब `origin/<baseBranch>` locally missing है या जब कोई commits base के ahead में नहीं हैं — वे Layer 1 fall-throughs भी cached PR mergeability को consult करते हैं allow करने से पहले।

**Parameters:**

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `baseBranch` | `string` | `"main"` | Base branch जो conflicts के लिए check किया जाता है। |

<Note>
GitHub CLI (`gh`) इस policy के लिए required है। Policy merge होने से पहले एक `OPEN` PR exist करने को confirm करने के लिए `gh pr view` को use करता है — `gh` के बिना, policy short-circuit होकर allow करता है। `gh auth login` को एक personal access token के साथ run करें जिसके पास pull requests को read access के लिए `repo` scope हो।
</Note>

## Individual policies को disable करना

अपनी config में `enabledPolicies` से एक specific policy को remove करें, या dashboard के Policies tab में इसे toggle off करें।

```json
{
  "enabledPolicies": [
    "block-rm-rf",
    "sanitize-api-keys"
  ]
}
```

Policies जो `enabledPolicies` में listed नहीं हैं, run नहीं होती हैं, भले ही `policyParams` entries उनके लिए exist करें।
