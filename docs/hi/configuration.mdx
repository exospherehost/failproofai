failproofai JSON configuration files का उपयोग करके यह नियंत्रित करता है कि कौन सी policies सक्रिय हैं, वे कैसे व्यवहार करती हैं, और custom policies कहाँ से लोड होती हैं। Configuration को आपकी team के साथ साझा करना आसान है - इसे अपने repo में commit करें और हर developer को same agent safety net मिलेगा।

---

## Configuration scopes

तीन configuration scopes हैं, जिनका मूल्यांकन प्राथमिकता क्रम में किया जाता है:

| Scope | File path | Purpose |
|-------|-----------|---------|
| **project** | `.failproofai/policies-config.json` | Per-repo settings, version control में commit किए जाते हैं |
| **local** | `.failproofai/policies-config.local.json` | Personal per-repo overrides, gitignored |
| **global** | `~/.failproofai/policies-config.json` | User-level defaults सभी projects में |

जब failproofai को hook event प्राप्त होता है, तो यह वर्तमान working directory के लिए सभी तीन files को लोड और merge करता है।

### Merge rules

**`enabledPolicies`** - सभी तीन scopes का union। किसी भी level पर सक्षम की गई policy सक्रिय है।

```text
project:  ["block-sudo"]
local:    ["block-rm-rf"]
global:   ["block-sudo", "sanitize-api-keys"]

resolved: ["block-sudo", "block-rm-rf", "sanitize-api-keys"]  ← deduplicated union
```

**`policyParams`** - पहला scope जो किसी दिए गए policy के params को परिभाषित करता है, पूरी तरह जीत जाता है। policy के params के भीतर values की कोई deep merging नहीं होती है।

```text
project:  block-sudo → { allowPatterns: ["sudo apt-get update"] }
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo apt-get update"] }   ← project wins, global ignored
```

```text
project:  (no block-sudo entry)
local:    (no block-sudo entry)
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo systemctl status"] }  ← falls through to global
```

**`customPoliciesPath`** - पहला scope जो इसे परिभाषित करता है, जीत जाता है।

**`llm`** - पहला scope जो इसे परिभाषित करता है, जीत जाता है।

## Config file format

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "sanitize-jwt",
    "block-env-files",
    "block-read-outside-cwd"
  ],
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    },
    "block-push-master": {
      "protectedBranches": ["main", "release", "prod"]
    },
    "block-rm-rf": {
      "allowPaths": ["/tmp"]
    },
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company"]
    },
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo API key" }
      ]
    },
    "warn-large-file-write": {
      "thresholdKb": 512
    }
  },
  "customPoliciesPath": "/home/alice/myproject/my-policies.js"
}
```

## Field reference

### `enabledPolicies`

Type: `string[]`

सक्षम करने के लिए policy names की list। Names को `failproofai policies` द्वारा दिखाए गए policy identifiers से बिल्कुल मेल खाना चाहिए। पूरी list के लिए [Built-in Policies](/hi/built-in-policies) देखें।

`enabledPolicies` में नहीं होने वाली policies निष्क्रिय होती हैं, भले ही `policyParams` में उनकी entries हों।

### `policyParams`

Type: `Record<string, Record<string, unknown>>`

Per-policy parameter overrides। बाहरी key policy name है; inner keys policy-specific हैं। प्रत्येक policy अपने available parameters को [Built-in Policies](/hi/built-in-policies) में दस्तावेज़ित करता है।

यदि किसी policy के पास parameters हैं लेकिन आप उन्हें निर्दिष्ट नहीं करते हैं, तो policy की built-in defaults का उपयोग किया जाता है। जो users `policyParams` को कॉन्फ़िगर नहीं करते हैं, उन्हें पिछले versions के समान व्यवहार मिलता है।

किसी policy के params block के अंदर अज्ञात keys hook-fire time पर चुप रहते हैं लेकिन जब आप `failproofai policies` चलाते हैं तो warnings के रूप में flagged होते हैं।

#### `hint` (cross-cutting)

Type: `string` (optional)

एक संदेश जो policy के `deny` या `instruct` return करने पर reason में जोड़ा जाता है। इसका उपयोग Claude को actionable guidance देने के लिए करें बिना policy को स्वयं संशोधित किए।

किसी भी policy type के साथ काम करता है — built-in, custom (`custom/`), project convention (`.failproofai-project/`), या user convention (`.failproofai-user/`)।

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "Try creating a fresh branch instead."
    },
    "block-sudo": {
      "allowPatterns": ["sudo apt-get"],
      "hint": "Use apt-get directly without sudo."
    },
    "custom/my-policy": {
      "hint": "Ask the user for approval first."
    }
  }
}
```

जब `block-force-push` deny करता है, Claude देखता है: *"Force-pushing is blocked. Try creating a fresh branch instead."*

Non-string values और empty strings चुप रहते हैं। यदि `hint` सेट नहीं है, तो व्यवहार unchanged है (backward-compatible)।

### `customPoliciesPath`

Type: `string` (absolute path)

एक JavaScript file का path जिसमें custom hook policies हैं। यह `failproofai policies --install --custom <path>` द्वारा स्वचालित रूप से सेट किया जाता है (path को store करने से पहले absolute के रूप में resolve किया जाता है)।

File को हर hook event पर fresh load किया जाता है - कोई caching नहीं है। Authoring details के लिए [Custom Policies](/hi/custom-policies) देखें।

### Convention-based policies

Explicit `customPoliciesPath` के अतिरिक्त, failproofai `.failproofai/policies/` directories से policy files को स्वचालित रूप से discover और load करता है:

| Level | Directory | Scope |
|-------|-----------|-------|
| Project | `.failproofai/policies/` | Team के साथ version control के माध्यम से साझा किया जाता है |
| User | `~/.failproofai/policies/` | Personal, सभी projects पर लागू होता है |

**File matching:** केवल `*policies.{js,mjs,ts}` से मेल खाने वाली files load होती हैं (जैसे `security-policies.mjs`, `workflow-policies.js`)। Directory में अन्य files को ignore किया जाता है।

**कोई config की आवश्यकता नहीं:** Convention policies को `policies-config.json` में entries की आवश्यकता नहीं है। बस files को directory में डालें और वे अगली hook event पर pick up हो जाएंगी।

**Union loading:** Project और user दोनों convention directories को scan किया जाता है। दोनों levels से सभी मेल खाने वाली files load होती हैं (unlike `customPoliciesPath` जो first-scope-wins का उपयोग करता है)।

अधिक विवरण और examples के लिए [Custom Policies](/hi/custom-policies) देखें।

### `llm`

Type: `object` (optional)

LLM client configuration policies के लिए जो AI calls करती हैं। अधिकांश setups के लिए आवश्यक नहीं है।

```json
{
  "llm": {
    "model": "claude-sonnet-4-6",
    "apiKey": "sk-ant-..."
  }
}
```

## CLI से configuration प्रबंधित करना

`policies --install` और `policies --uninstall` commands आपके agent CLI की hook settings file (hook entry points) में लिखते हैं, जबकि `policies-config.json` वह file है जिसे आप सीधे प्रबंधित करते हैं। दोनों अलग हैं:

- **Agent CLI settings** — agent को हर tool use पर `failproofai --hook <event>` call करने के लिए कहता है:
  - **Claude Code**: `~/.claude/settings.json` (user), `<cwd>/.claude/settings.json` (project), `<cwd>/.claude/settings.local.json` (local)
  - **OpenAI Codex**: `~/.codex/hooks.json` (user), `<cwd>/.codex/hooks.json` (project) — Codex के पास `local` scope नहीं है
- **`policies-config.json`** — failproofai को बताता है कि कौन सी policies का मूल्यांकन करना है और किन params के साथ (सभी agent CLIs में साझा)

किसी विशिष्ट agent को target करने के लिए `--cli claude|codex` pass करें (space-separated या दोनों के लिए repeated):

```bash
failproofai policies --install --cli codex --scope project
failproofai policies --install --cli claude codex
```

जब `--cli` छोड़ा जाता है, तो `failproofai` detect करता है कि कौन से agent CLIs installed हैं (`which claude` / `which codex`):

- **एक CLI detected** — prompt किए बिना उस CLI को auto-select करता है।
- **दोनों CLIs detected** interactive terminal में — एक arrow-key single-select prompt दिखाता है: `Both`, `Claude Code only`, या `OpenAI Codex only` चुनें (↑↓ को move करने के लिए, Enter करने के लिए, ^C to quit)।
- **दोनों CLIs detected** non-interactive run में (CI, no TTY) — prompt किए बिना दोनों के लिए install करता है।
- **कोई भी CLI detected नहीं** — `claude` पर fallback करता है, एक warning के साथ कि कोई agent binary PATH में नहीं मिला; hook command फिर भी लिखा जाता है इसलिए यह सक्रिय हो जाता है जैसे ही आप एक install करते हैं।

आप किसी भी समय `policies-config.json` को सीधे edit कर सकते हैं; changes अगली hook event पर तुरंत प्रभावी होते हैं, restart की कोई आवश्यकता नहीं है।

## Example: project-level config with team defaults

`.failproofai/policies-config.json` को अपने repo में commit करें:

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "block-env-files"
  ],
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "release", "hotfix"]
    }
  }
}
```

प्रत्येक developer फिर `.failproofai/policies-config.local.json` (gitignored) बना सकते हैं personal overrides के लिए teammates को प्रभावित किए बिना।
