failproofai JSON कॉन्फ़िगरेशन फ़ाइलों का उपयोग करता है यह नियंत्रित करने के लिए कि कौन सी नीतियां सक्रिय हैं, वे कैसे व्यवहार करती हैं, और कस्टम नीतियां कहां से लोड की जाती हैं। कॉन्फ़िगरेशन आपकी टीम के साथ साझा करना आसान बनाया गया है - इसे अपने रिपो में कमिट करें और हर डेवलपर को एक जैसी एजेंट सेफ्टी नेट मिलेगी।

---

## कॉन्फ़िगरेशन स्कोप

तीन कॉन्फ़िगरेशन स्कोप हैं, जिन्हें प्राथमिकता क्रम में मूल्यांकन किया जाता है:

| स्कोप | फ़ाइल पाथ | उद्देश्य |
|-------|-----------|---------|
| **project** | `.failproofai/policies-config.json` | प्रति-रिपो सेटिंग्स, वर्जन कंट्रोल में कमिट की गई |
| **local** | `.failproofai/policies-config.local.json` | व्यक्तिगत प्रति-रिपो ओवरराइड, gitignored |
| **global** | `~/.failproofai/policies-config.json` | सभी प्रोजेक्ट्स में उपयोगकर्ता-स्तरीय डिफ़ॉल्ट |

जब failproofai को कोई हुक इवेंट मिलता है, तो यह मौजूदा कार्यशील निर्देशिका के लिए सभी तीन फ़ाइलों को लोड और मर्ज करता है।

### मर्ज नियम

**`enabledPolicies`** - सभी तीन स्कोप का यूनियन। किसी भी स्तर पर सक्षम की गई नीति सक्रिय है।

```text
project:  ["block-sudo"]
local:    ["block-rm-rf"]
global:   ["block-sudo", "sanitize-api-keys"]

resolved: ["block-sudo", "block-rm-rf", "sanitize-api-keys"]  ← deduplicated union
```

**`policyParams`** - पहला स्कोप जो किसी दी गई नीति के लिए पैरामीटर परिभाषित करता है पूरी तरह जीत जाता है। नीति के पैरामीटर के अंदर कोई गहरी मर्जिंग नहीं होती है।

```text
project:  block-sudo → { allowPatterns: ["sudo apt-get update"] }
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo apt-get update"] }   ← project जीतता है, global अनदेखा किया जाता है
```

```text
project:  (no block-sudo entry)
local:    (no block-sudo entry)
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo systemctl status"] }  ← global तक फॉल-थ्रू होता है
```

**`customPoliciesPath`** - पहला स्कोप जो इसे परिभाषित करता है जीतता है।

**`llm`** - पहला स्कोप जो इसे परिभाषित करता है जीतता है।

## कॉन्फ़िग फ़ाइल फॉर्मेट

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "sanitize-jwt",
    "block-env-files",
    "block-read-outside-cwd"
  ],
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    },
    "block-push-master": {
      "protectedBranches": ["main", "release", "prod"]
    },
    "block-rm-rf": {
      "allowPaths": ["/tmp"]
    },
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company"]
    },
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo API key" }
      ]
    },
    "warn-large-file-write": {
      "thresholdKb": 512
    }
  },
  "customPoliciesPath": "/home/alice/myproject/my-policies.js"
}
```

## फ़ील्ड संदर्भ

### `enabledPolicies`

प्रकार: `string[]`

सक्षम करने के लिए नीति नामों की सूची। नाम बिल्कुल `failproofai policies` द्वारा दिखाए गए नीति पहचानकर्ताओं से मेल खाने चाहिए। पूरी सूची के लिए [बिल्ट-इन पॉलिसीज](/hi/built-in-policies) देखें।

`enabledPolicies` में नहीं होने वाली नीतियां निष्क्रिय होती हैं, भले ही उनके पास `policyParams` में प्रविष्टियां हों।

### `policyParams`

प्रकार: `Record<string, Record<string, unknown>>`

प्रति-नीति पैरामीटर ओवरराइड। बाहरी कुंजी नीति का नाम है; आंतरिक कुंजियां नीति-विशिष्ट हैं। प्रत्येक नीति अपने उपलब्ध पैरामीटर को [बिल्ट-इन पॉलिसीज](/hi/built-in-policies) में दस्तावेज करती है।

यदि किसी नीति के पास पैरामीटर हैं लेकिन आप उन्हें निर्दिष्ट नहीं करते हैं, तो नीति के बिल्ट-इन डिफ़ॉल्ट का उपयोग किया जाता है। जो उपयोगकर्ता `policyParams` को बिल्कुल कॉन्फ़िगर नहीं करते हैं वे पिछले संस्करणों के समान व्यवहार प्राप्त करते हैं।

किसी नीति के पैरामीटर ब्लॉक के अंदर अज्ञात कुंजियों को हुक-फायर समय पर चुप्पी से अनदेखा किया जाता है लेकिन जब आप `failproofai policies` चलाते हैं तो चेतावनियों के रूप में फ्लैग किया जाता है।

#### `hint` (क्रॉस-कटिंग)

प्रकार: `string` (वैकल्पिक)

जब कोई नीति `deny` या `instruct` रिटर्न करती है तो कारण में जोड़ा जाने वाला संदेश। Claude को नीति को संशोधित किए बिना कार्रवाई योग्य मार्गदर्शन देने के लिए इसका उपयोग करें।

किसी भी नीति प्रकार के साथ काम करता है - बिल्ट-इन, कस्टम (`custom/`), प्रोजेक्ट कन्वेंशन (`.failproofai-project/`), या उपयोगकर्ता कन्वेंशन (`.failproofai-user/`)।

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "इसके बजाय एक नई ब्रांच बनाने का प्रयास करें।"
    },
    "block-sudo": {
      "allowPatterns": ["sudo apt-get"],
      "hint": "sudo के बिना सीधे apt-get का उपयोग करें।"
    },
    "custom/my-policy": {
      "hint": "पहले उपयोगकर्ता से अनुमति मांगें।"
    }
  }
}
```

जब `block-force-push` deny करता है, तो Claude देखता है: *"Force-pushing अवरुद्ध है। इसके बजाय एक नई ब्रांच बनाने का प्रयास करें।"*

गैर-स्ट्रिंग मान और खाली स्ट्रिंग्स को चुप्पी से अनदेखा किया जाता है। यदि `hint` सेट नहीं है, तो व्यवहार अपरिवर्तित रहता है (पिछड़े-संगत)।

### `customPoliciesPath`

प्रकार: `string` (पूर्ण पाथ)

कस्टम हुक नीतियों वाली JavaScript फ़ाइल का पाथ। यह `failproofai policies --install --custom <path>` द्वारा स्वचालित रूप से सेट किया जाता है (पाथ को संग्रहीत करने से पहले पूर्ण में हल किया जाता है)।

फ़ाइल हर हुक इवेंट पर नई लोड की जाती है - कोई कैशिंग नहीं होती है। विवरण के लिए [कस्टम पॉलिसीज](/hi/custom-policies) देखें।

### कन्वेंशन-आधारित नीतियां

स्पष्ट `customPoliciesPath` के अलावा, failproofai स्वचालित रूप से `.failproofai/policies/` निर्देशिकाओं से नीति फ़ाइलें खोजता है और लोड करता है:

| स्तर | निर्देशिका | स्कोप |
|-------|-----------|-------|
| Project | `.failproofai/policies/` | टीम के साथ वर्जन कंट्रोल के माध्यम से साझा किया गया |
| User | `~/.failproofai/policies/` | व्यक्तिगत, सभी प्रोजेक्ट्स पर लागू होता है |

**फ़ाइल मिलान:** केवल `*policies.{js,mjs,ts}` से मेल खाने वाली फ़ाइलें लोड की जाती हैं (उदाहरण के लिए `security-policies.mjs`, `workflow-policies.js`)। निर्देशिका में अन्य फ़ाइलें अनदेखी की जाती हैं।

**कॉन्फ़िग की आवश्यकता नहीं:** कन्वेंशन नीतियों के लिए `policies-config.json` में प्रविष्टियों की आवश्यकता नहीं होती है। बस फ़ाइलों को निर्देशिका में ड्रॉप करें और वे अगले हुक इवेंट पर उठाई जाती हैं।

**यूनियन लोडिंग:** प्रोजेक्ट और उपयोगकर्ता दोनों कन्वेंशन निर्देशिकाएं स्कैन की जाती हैं। दोनों स्तरों से सभी मेल खाने वाली फ़ाइलें लोड की जाती हैं (`customPoliciesPath` के विपरीत जो पहले-स्कोप-जीतता-है का उपयोग करता है)।

अधिक विवरण और उदाहरणों के लिए [कस्टम पॉलिसीज](/hi/custom-policies) देखें।

### `llm`

प्रकार: `object` (वैकल्पिक)

AI कॉल करने वाली नीतियों के लिए LLM क्लाइंट कॉन्फ़िगरेशन। अधिकांश सेटअप के लिए आवश्यक नहीं है।

```json
{
  "llm": {
    "model": "claude-sonnet-4-6",
    "apiKey": "sk-ant-..."
  }
}
```

## CLI से कॉन्फ़िगरेशन प्रबंधित करना

`policies --install` और `policies --uninstall` कमांड आपके एजेंट CLI की हुक सेटिंग्स फ़ाइल में लिखते हैं (हुक प्रवेश बिंदु), जबकि `policies-config.json` वह फ़ाइल है जिसे आप सीधे प्रबंधित करते हैं। ये दोनों अलग हैं:

- **एजेंट CLI सेटिंग्स** — एजेंट को प्रत्येक टूल उपयोग पर `failproofai --hook <event>` कॉल करने के लिए बताता है:
  - **Claude Code**: `~/.claude/settings.json` (उपयोगकर्ता), `<cwd>/.claude/settings.json` (प्रोजेक्ट), `<cwd>/.claude/settings.local.json` (स्थानीय)
  - **OpenAI Codex**: `~/.codex/hooks.json` (उपयोगकर्ता), `<cwd>/.codex/hooks.json` (प्रोजेक्ट) — Codex के पास `local` स्कोप नहीं है
  - **GitHub Copilot CLI _(beta)_**: `~/.copilot/hooks/failproofai.json` (उपयोगकर्ता), `<cwd>/.github/hooks/failproofai.json` (प्रोजेक्ट) — Copilot के पास कोई `local` स्कोप नहीं है। हुक प्रविष्टियां Copilot के OS-कुंजी `bash`/`powershell` कमांड फ़ील्ड का उपयोग करती हैं `timeoutSec` के साथ; फ़ाइल में शीर्ष-स्तरीय `version: 1` मार्कर होता है। Copilot CLI सपोर्ट **beta** है जबकि हम `events.jsonl` रिकॉर्ड स्कीमा को सत्यापित करते हैं (जिसे सार्वजनिक दस्तावेज निर्दिष्ट नहीं करते) अधिक वास्तविक-दुनिया सत्रों के विरुद्ध।
  - **Cursor Agent _(beta)_**: `~/.cursor/hooks.json` (उपयोगकर्ता), `<cwd>/.cursor/hooks.json` (प्रोजेक्ट) — Cursor के पास कोई `local` स्कोप नहीं है। हुक प्रविष्टियां Claude-आकार `{type, command, timeout}` फॉर्म का उपयोग करती हैं (कोई `bash`/`powershell` विभाजन नहीं), लेकिन camelCase इवेंट कुंजी (`preToolUse`, `beforeSubmitPrompt`, …) के तहत एक फ्लैट सरणी में संग्रहीत की जाती हैं Cursor के [hooks स्कीमा](https://cursor.com/docs/hooks) के अनुसार; फ़ाइल में शीर्ष-स्तरीय `version: 1` मार्कर होता है। हैंडलर camelCase → PascalCase को `CURSOR_EVENT_MAP` के माध्यम से सामान्य करता है ताकि मौजूदा बिल्ट-इन नीतियां अपरिवर्तित रूप से फायर हों। Cursor Agent सपोर्ट **beta** है जबकि हम Cursor के डिस्क-पर-ट्रांसक्रिप्ट प्रारूप को सत्यापित करते हैं (सार्वजनिक दस्तावेजों में निर्दिष्ट नहीं) अधिक वास्तविक-दुनिया इंस्टॉल के विरुद्ध।
  - **OpenCode _(beta)_**: `~/.config/opencode/opencode.json` + `~/.config/opencode/plugins/failproofai.mjs` (उपयोगकर्ता), `<cwd>/.opencode/opencode.json` + `<cwd>/.opencode/plugins/failproofai.mjs` (प्रोजेक्ट) — OpenCode के पास कोई `local` स्कोप नहीं है। अन्य छह CLIs के विपरीत, OpenCode के पास **कोई बाहरी-कमांड हुक सिस्टम नहीं है**: यह `opencode.json` में `plugin: []` सरणी के माध्यम से स्पष्ट रूप से पंजीकृत इन-प्रोसेस JS/TS प्लगइन लोड करता है (`.opencode/plugins/` से स्वचालित खोज **नहीं है** कि प्लगइन opencode v1.14.33 पर कैसे लोड होते हैं)। इंस्टॉल एक छोटी जेनरेट की गई प्लगइन शिम ड्रॉप करता है जो failproofai बाइनरी को सबप्रोसेस-कॉल करता है और बाइनरी के Claude-आकार JSON प्रतिक्रिया को प्लगइन सिमेंटिक्स में वापस अनुवादित करता है: tool-event deny के लिए `throw new Error()` (टूल कॉल रद्द करता है), instruct और `Stop` / `SubagentStop` deny दोनों के लिए `client.session.prompt(...)` (deny कारण को अगले उपयोगकर्ता संदेश के रूप में जमा करता है — एकमात्र force-retry चैनल चूंकि `session.idle` केवल notification है और इससे फेंकना एक no-op है), और allow के लिए no-op। शिम टूल नामों (lowercase → PascalCase via `OPENCODE_TOOL_MAP`) और टूल-इनपुट arg कुंजियों (camelCase → snake_case via `OPENCODE_TOOL_INPUT_MAP` for `Read` / `Write` / `Edit`, जैसे `filePath` → `file_path`, `oldString` → `old_string`) दोनों को सामान्य करता है बाइनरी को अग्रेषित करने से पहले, इसलिए पाथ-चेकिंग बिल्ट-इन जैसे `block-read-outside-cwd`, `block-env-files`, और `block-secrets-write` OpenCode टूल कॉल पर अपरिवर्तित रूप से फायर होते हैं। सत्र `~/.local/share/opencode/opencode.db` पर opencode के SQLite DB में रहते हैं; dashboard का सत्र दर्शक उन्हें `opencode db --format json` और `opencode export <id>` के माध्यम से पढ़ता है। OpenCode सपोर्ट **beta** है जबकि हम संस्करणों में और अधिक वास्तविक-दुनिया सत्रों के विरुद्ध व्यवहार सत्यापित करते हैं। [OpenCode प्लगइन दस्तावेज](https://opencode.ai/docs/plugins/) देखें।
  - **Pi _(beta)_**: `~/.pi/agent/settings.json` (उपयोगकर्ता), `<cwd>/.pi/settings.json` (प्रोजेक्ट) — Pi के पास कोई `local` स्कोप नहीं है। Pi स्टार्टअप पर TypeScript एक्सटेंशन पैकेज लोड करता है; सेटिंग्स फ़ाइल एक फ्लैट स्ट्रिंग सरणी `{"packages": ["./relative/path", …]}` है। failproofai अपनी बंडल की गई `pi-extension/` निर्देशिका की ओर इशारा करने वाली एक packages-array प्रविष्टि लिखता है। एक्सटेंशन आंतरिक रूप से Pi की `tool_call` / `user_bash` / `input` / `session_start` इवेंट्स को सबस्क्राइब करता है और `failproofai --hook <Event> --cli pi` के लिए शेल आउट करता है; हैंडलर underscore_lower_snake_case → PascalCase को `PI_EVENT_MAP` के माध्यम से सामान्य करता है ताकि मौजूदा बिल्ट-इन नीतियां अपरिवर्तित रूप से फायर हों। टूल इनपुट args भी `PI_TOOL_INPUT_MAP` के माध्यम से सामान्य किए जाते हैं (Pi का Read / Write / Edit `file_path` के बजाय `path` प्रदान करता है; शीर्ष-स्तरीय कुंजी को मैप करना `block-env-files` और `block-secrets-write` को फायर करने देता है — `block-read-outside-cwd` के पास पहले से ही एक `path` फॉलबैक है)। Pi सपोर्ट **beta** है जबकि Pi का एक्सटेंशन API और सत्र-log लेआउट स्थिर होता है।
  - **Gemini CLI _(beta)_**: `~/.gemini/settings.json` (उपयोगकर्ता), `<cwd>/.gemini/settings.json` (प्रोजेक्ट) — Gemini के पास कोई `local` स्कोप नहीं है (यह `/etc/gemini-cli/settings.json` पर एक `system` स्कोप का दस्तावेज करता है जिसे failproofai उजागर नहीं करता है)। हुक प्रविष्टियां Claude के `{type, command, timeout}` फॉर्म का उपयोग करती हैं Gemini के `{matcher, hooks: [...]}` मैचर स्कीमा में लपेटी गई `matcher: "*"` डिफ़ॉल्ट के साथ। इवेंट्स PascalCase हैं (`SessionStart`, `BeforeAgent`, `AfterAgent`, `BeforeModel`, `AfterModel`, `BeforeToolSelection`, `BeforeTool`, `AfterTool`, `PreCompress`, `Notification`, `SessionEnd`); हैंडलर `GEMINI_EVENT_MAP` के माध्यम से Claude कैनोनिकल नामों के लिए मैप करता है। टूल नामें snake_case हैं (`run_shell_command`, `read_file`, `write_file`, `replace`, …) — हैंडलर `GEMINI_TOOL_MAP` के माध्यम से सामान्य करता है ताकि मौजूदा बिल्ट-इन नीतियां अपरिवर्तित रूप से फायर हों। नीति मूल्यांकनकर्ता Gemini के फ्लैट `{decision: "deny", reason}` आकार को उत्सर्जित करता है (Gemini के Golden Rule exit-0 अनुबंध प्रति पसंद), `{hookSpecificOutput: {hookEventName, additionalContext}}` BeforeAgent / AfterTool / SessionStart पर संदर्भ इंजेक्शन के लिए, और AfterAgent पर `{decision: "block", reason}` force-retry सिमेंटिक्स के लिए। Gemini CLI सपोर्ट **beta** है जबकि हम वास्तविक-दुनिया कवरेज को चौड़ा करते हैं। [Gemini CLI हुक दस्तावेज](https://geminicli.com/docs/hooks/) देखें।
- **`policies-config.json`** — failproofai को बताता है कि कौन सी नीतियां मूल्यांकन करनी हैं और किस पैरामीटर के साथ (सभी एजेंट CLIs में साझा की जाती हैं)

किसी विशिष्ट एजेंट को लक्षित करने के लिए `--cli claude|codex|copilot|cursor|opencode|pi|gemini` पास करें (स्पेस-अलग या किसी भी सबसेट के लिए दोहराया गया):

```bash
failproofai policies --install --cli codex --scope project
failproofai policies --install --cli copilot --scope project
failproofai policies --install --cli cursor --scope project
failproofai policies --install --cli opencode --scope project
failproofai policies --install --cli pi --scope project
failproofai policies --install --cli gemini --scope project
failproofai policies --install --cli claude codex copilot cursor opencode pi gemini
```

जब `--cli` को छोड़ा जाता है, तो `failproofai` पहचानता है कि कौन से एजेंट CLIs इंस्टॉल किए गए हैं (`which claude` / `which codex` / `which copilot` / `which cursor-agent` / `which opencode` / `which pi` / `which gemini`):

- **एक CLI पहचाना गया** — बिना प्रॉम्पट किए उस CLI को ऑटो-सिलेक्ट करता है।
- **एक इंटरैक्टिव टर्मिनल में कई CLIs पहचाने गए** — एक arrow-key सिंगल-सिलेक्ट प्रॉम्प्ट दिखाता है जिसे एक `Detected (N)` सेक्शन में समूहीकृत किया जाता है (एक `Install for all N detected` एग्रीगेट पंक्ति + प्रत्येक पहचाने गए CLI व्यक्तिगत रूप से) और एक `Not installed (M) · install hooks ahead of time` सेक्शन प्रत्येक अनडिटेक्टेड समर्थित CLI को फॉरवर्ड-इंस्टॉल विकल्प के रूप में सूचीबद्ध करता है (↑↓ को स्थानांतरित करने के लिए, Enter को सिलेक्ट करने के लिए, ^C को छोड़ने के लिए)। अनइंस्टॉल प्रवाह केवल Detected सेक्शन दिखाता है।
- **एक गैर-इंटरैक्टिव रन में कई CLIs पहचाने गए** (CI, no TTY) — बिना प्रॉम्पट किए सभी पहचाने गए CLIs के लिए इंस्टॉल करता है।
- **कोई नहीं पहचाना गया** — `claude` में फॉलबैक करता है, एक चेतावनी के साथ कि PATH में कोई एजेंट बाइनरी नहीं मिला; हुक कमांड अभी भी लिखा जाता है इसलिए यह सक्रिय हो जाता है जैसे ही आप एक इंस्टॉल करते हैं।

आप किसी भी समय `policies-config.json` को सीधे संपादित कर सकते हैं; परिवर्तन अगले हुक इवेंट पर तुरंत प्रभाव डालते हैं पुनरारंभ की कोई आवश्यकता नहीं है।

## उदाहरण: टीम डिफ़ॉल्ट के साथ प्रोजेक्ट-स्तरीय कॉन्फ़िग

`.failproofai/policies-config.json` को अपने रिपो में कमिट करें:

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "block-env-files"
  ],
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "release", "hotfix"]
    }
  }
}
```

प्रत्येक डेवलपर फिर व्यक्तिगत ओवरराइड के लिए `.failproofai/policies-config.local.json` (gitignored) बना सकता है टीममेट्स को प्रभावित किए बिना।
