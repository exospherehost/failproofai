failproofai sử dụng file cấu hình JSON để kiểm soát các policy nào đang hoạt động, cách chúng hoạt động và nơi tải custom policy từ đó. Cấu hình được thiết kế để dễ dàng chia sẻ với nhóm — commit vào repo và mọi lập trình viên đều được áp dụng cùng một lớp bảo vệ an toàn cho agent.

---

## Phạm vi cấu hình

Có ba phạm vi cấu hình, được đánh giá theo thứ tự ưu tiên:

| Phạm vi | Đường dẫn file | Mục đích |
|---------|----------------|----------|
| **project** | `.failproofai/policies-config.json` | Cài đặt theo từng repo, được commit vào version control |
| **local** | `.failproofai/policies-config.local.json` | Ghi đè cá nhân theo từng repo, đã được gitignore |
| **global** | `~/.failproofai/policies-config.json` | Cài đặt mặc định ở cấp người dùng cho tất cả dự án |

Khi failproofai nhận một hook event, nó sẽ tải và hợp nhất cả ba file tồn tại trong thư mục làm việc hiện tại.

### Quy tắc hợp nhất

**`enabledPolicies`** — hợp nhất theo kiểu hợp (union) của cả ba phạm vi. Policy được bật ở bất kỳ cấp nào đều được kích hoạt.

```text
project:  ["block-sudo"]
local:    ["block-rm-rf"]
global:   ["block-sudo", "sanitize-api-keys"]

resolved: ["block-sudo", "block-rm-rf", "sanitize-api-keys"]  ← hợp nhất đã loại trùng lặp
```

**`policyParams`** — phạm vi đầu tiên định nghĩa params cho một policy nhất định sẽ thắng hoàn toàn. Không có việc hợp nhất sâu các giá trị bên trong params của một policy.

```text
project:  block-sudo → { allowPatterns: ["sudo apt-get update"] }
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo apt-get update"] }   ← project thắng, global bị bỏ qua
```

```text
project:  (không có mục block-sudo)
local:    (không có mục block-sudo)
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo systemctl status"] }  ← rơi xuống global
```

**`customPoliciesPath`** — phạm vi đầu tiên định nghĩa giá trị này sẽ thắng.

**`llm`** — phạm vi đầu tiên định nghĩa giá trị này sẽ thắng.

## Định dạng file cấu hình

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "sanitize-jwt",
    "block-env-files",
    "block-read-outside-cwd"
  ],
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    },
    "block-push-master": {
      "protectedBranches": ["main", "release", "prod"]
    },
    "block-rm-rf": {
      "allowPaths": ["/tmp"]
    },
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company"]
    },
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo API key" }
      ]
    },
    "warn-large-file-write": {
      "thresholdKb": 512
    }
  },
  "customPoliciesPath": "/home/alice/myproject/my-policies.js"
}
```

## Tham chiếu các trường

### `enabledPolicies`

Kiểu: `string[]`

Danh sách tên policy cần bật. Tên phải khớp chính xác với các định danh policy được hiển thị bởi `failproofai policies`. Xem [Built-in Policies](/vi/built-in-policies) để biết danh sách đầy đủ.

Các policy không có trong `enabledPolicies` sẽ không hoạt động, ngay cả khi chúng có mục trong `policyParams`.

### `policyParams`

Kiểu: `Record<string, Record<string, unknown>>`

Ghi đè tham số theo từng policy. Khóa ngoài là tên policy; các khóa bên trong là đặc thù của từng policy. Mỗi policy ghi lại các tham số khả dụng của nó trong [Built-in Policies](/vi/built-in-policies).

Nếu một policy có tham số nhưng bạn không chỉ định chúng, các giá trị mặc định tích hợp của policy sẽ được sử dụng. Người dùng không cấu hình `policyParams` sẽ có hành vi giống hệt các phiên bản trước.

Các khóa không xác định bên trong block params của một policy sẽ bị bỏ qua âm thầm khi hook kích hoạt, nhưng sẽ được gắn cờ cảnh báo khi bạn chạy `failproofai policies`.

### `customPoliciesPath`

Kiểu: `string` (đường dẫn tuyệt đối)

Đường dẫn đến file JavaScript chứa các custom hook policy. Giá trị này được đặt tự động bởi `failproofai policies --install --custom <path>` (đường dẫn được chuyển thành tuyệt đối trước khi lưu).

File được tải mới trên mỗi hook event — không có bộ nhớ đệm. Xem [Custom Policies](/vi/custom-policies) để biết chi tiết về cách viết.

### `llm`

Kiểu: `object` (tùy chọn)

Cấu hình LLM client cho các policy thực hiện lời gọi AI. Không bắt buộc đối với hầu hết các thiết lập.

```json
{
  "llm": {
    "model": "claude-sonnet-4-6",
    "apiKey": "sk-ant-..."
  }
}
```

## Quản lý cấu hình từ CLI

Các lệnh `policies --install` và `policies --uninstall` ghi vào `settings.json` của Claude Code (các điểm vào hook), trong khi `policies-config.json` là file bạn quản lý trực tiếp. Hai file này là riêng biệt:

- **`settings.json`** — báo cho Claude Code gọi `failproofai --hook <event>` trên mỗi lần sử dụng công cụ
- **`policies-config.json`** — báo cho failproofai biết policy nào cần đánh giá và với tham số gì

Bạn có thể chỉnh sửa `policies-config.json` trực tiếp bất cứ lúc nào; các thay đổi có hiệu lực ngay lập tức ở hook event tiếp theo mà không cần khởi động lại.

## Ví dụ: cấu hình cấp project với mặc định cho nhóm

Commit `.failproofai/policies-config.json` vào repo của bạn:

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "block-env-files"
  ],
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "release", "hotfix"]
    }
  }
}
```

Mỗi lập trình viên sau đó có thể tạo `.failproofai/policies-config.local.json` (đã được gitignore) để ghi đè cá nhân mà không ảnh hưởng đến đồng nghiệp.
