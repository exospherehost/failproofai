failproofai sử dụng các file cấu hình JSON để kiểm soát những chính sách nào hoạt động, cách chúng hoạt động, và nơi tải các chính sách tùy chỉnh. Cấu hình được thiết kế để dễ dàng chia sẻ với nhóm của bạn - commit nó vào kho lưu trữ và mọi nhà phát triển sẽ nhận được cùng một lưới bảo vệ agent.

---

## Phạm vi cấu hình

Có ba phạm vi cấu hình, được đánh giá theo thứ tự ưu tiên:

| Phạm vi | Đường dẫn file | Mục đích |
|-------|-----------|---------|
| **project** | `.failproofai/policies-config.json` | Cài đặt cho mỗi repo, được commit vào kiểm soát phiên bản |
| **local** | `.failproofai/policies-config.local.json` | Ghi đè cá nhân cho mỗi repo, được gitignore |
| **global** | `~/.failproofai/policies-config.json` | Giá trị mặc định cấp người dùng trên tất cả các dự án |

Khi failproofai nhận được một sự kiện hook, nó tải và hợp nhất tất cả ba file tồn tại cho thư mục làm việc hiện tại.

### Quy tắc hợp nhất

**`enabledPolicies`** - hợp của tất cả ba phạm vi. Một chính sách được bật ở bất kỳ cấp độ nào đều hoạt động.

```text
project:  ["block-sudo"]
local:    ["block-rm-rf"]
global:   ["block-sudo", "sanitize-api-keys"]

resolved: ["block-sudo", "block-rm-rf", "sanitize-api-keys"]  ← hợp đã loại bỏ trùng lặp
```

**`policyParams`** - phạm vi đầu tiên định nghĩa tham số cho một chính sách nhất định sẽ thắng hoàn toàn. Không có hợp nhất sâu của các giá trị trong tham số của một chính sách.

```text
project:  block-sudo → { allowPatterns: ["sudo apt-get update"] }
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo apt-get update"] }   ← project thắng, global bị bỏ qua
```

```text
project:  (không có mục block-sudo)
local:    (không có mục block-sudo)
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo systemctl status"] }  ← rơi xuống global
```

**`customPoliciesPath`** - phạm vi đầu tiên định nghĩa nó sẽ thắng.

**`llm`** - phạm vi đầu tiên định nghĩa nó sẽ thắng.

## Định dạng file cấu hình

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "sanitize-jwt",
    "block-env-files",
    "block-read-outside-cwd"
  ],
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    },
    "block-push-master": {
      "protectedBranches": ["main", "release", "prod"]
    },
    "block-rm-rf": {
      "allowPaths": ["/tmp"]
    },
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company"]
    },
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo API key" }
      ]
    },
    "warn-large-file-write": {
      "thresholdKb": 512
    }
  },
  "customPoliciesPath": "/home/alice/myproject/my-policies.js"
}
```

## Tham khảo trường

### `enabledPolicies`

Loại: `string[]`

Danh sách tên chính sách để bật. Tên phải khớp chính xác với các mã định danh chính sách được hiển thị bởi `failproofai policies`. Xem [Chính sách tích hợp sẵn](/vi/built-in-policies) để xem danh sách đầy đủ.

Các chính sách không có trong `enabledPolicies` không hoạt động, ngay cả khi chúng có các mục trong `policyParams`.

### `policyParams`

Loại: `Record<string, Record<string, unknown>>`

Ghi đè tham số cho mỗi chính sách. Khóa bên ngoài là tên chính sách; các khóa bên trong là cụ thể từng chính sách. Mỗi chính sách ghi lại các tham số có sẵn của nó trong [Chính sách tích hợp sẵn](/vi/built-in-policies).

Nếu một chính sách có tham số nhưng bạn không chỉ định chúng, các giá trị mặc định tích hợp của chính sách sẽ được sử dụng. Người dùng không cấu hình `policyParams` cũng sẽ nhận được hành vi giống hệt như các phiên bản trước.

Các khóa không xác định bên trong khối tham số của một chính sách sẽ được im lặng bỏ qua tại thời điểm kích hoạt hook nhưng được gắn cờ là cảnh báo khi bạn chạy `failproofai policies`.

#### `hint` (vượt qua ranh giới)

Loại: `string` (tùy chọn)

Một tin nhắn được thêm vào lý do khi một chính sách trả về `deny` hoặc `instruct`. Sử dụng nó để cung cấp Claude hướng dẫn có thể thực hiện được mà không cần chỉnh sửa chính sách.

Hoạt động với bất kỳ loại chính sách nào — tích hợp sẵn, tùy chỉnh (`custom/`), quy ước dự án (`.failproofai-project/`), hoặc quy ước người dùng (`.failproofai-user/`).

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "Thử tạo một nhánh mới thay vào đó."
    },
    "block-sudo": {
      "allowPatterns": ["sudo apt-get"],
      "hint": "Sử dụng apt-get trực tiếp mà không cần sudo."
    },
    "custom/my-policy": {
      "hint": "Hãy hỏi người dùng xin phép trước."
    }
  }
}
```

Khi `block-force-push` từ chối, Claude thấy: *"Force-pushing bị chặn. Thử tạo một nhánh mới thay vào đó."*

Các giá trị không phải chuỗi và chuỗi trống sẽ được im lặng bỏ qua. Nếu `hint` không được đặt, hành vi không thay đổi (tương thích ngược).

### `customPoliciesPath`

Loại: `string` (đường dẫn tuyệt đối)

Đường dẫn đến file JavaScript chứa các chính sách hook tùy chỉnh. Điều này được đặt tự động bởi `failproofai policies --install --custom <path>` (đường dẫn được giải quyết thành tuyệt đối trước khi được lưu trữ).

File được tải mới trên mỗi sự kiện hook - không có bộ nhớ đệm. Xem [Chính sách tùy chỉnh](/vi/custom-policies) để biết chi tiết về tác giả.

### Chính sách dựa trên quy ước

Ngoài `customPoliciesPath` rõ ràng, failproofai tự động khám phá và tải các file chính sách từ các thư mục `.failproofai/policies/`:

| Cấp độ | Thư mục | Phạm vi |
|-------|-----------|-------|
| Dự án | `.failproofai/policies/` | Chia sẻ với nhóm thông qua kiểm soát phiên bản |
| Người dùng | `~/.failproofai/policies/` | Cá nhân, áp dụng cho tất cả các dự án |

**Khớp file:** Chỉ các file khớp với `*policies.{js,mjs,ts}` mới được tải (ví dụ `security-policies.mjs`, `workflow-policies.js`). Các file khác trong thư mục sẽ bị bỏ qua.

**Không cần cấu hình:** Các chính sách quy ước không yêu cầu các mục trong `policies-config.json`. Chỉ cần thả các file vào thư mục và chúng sẽ được nhận trên sự kiện hook tiếp theo.

**Tải hợp:** Cả thư mục quy ước dự án và người dùng đều được quét. Tất cả các file khớp từ cả hai cấp độ được tải (không giống `customPoliciesPath` sử dụng first-scope-wins).

Xem [Chính sách tùy chỉnh](/vi/custom-policies) để biết thêm chi tiết và ví dụ.

### `llm`

Loại: `object` (tùy chọn)

Cấu hình client LLM cho các chính sách thực hiện các lệnh gọi AI. Không bắt buộc đối với hầu hết các thiết lập.

```json
{
  "llm": {
    "model": "claude-sonnet-4-6",
    "apiKey": "sk-ant-..."
  }
}
```

## Quản lý cấu hình từ CLI

Các lệnh `policies --install` và `policies --uninstall` ghi vào file cài đặt hook của CLI agent của bạn (các điểm vào hook), trong khi `policies-config.json` là file bạn quản lý trực tiếp. Hai file này là riêng biệt:

- **Cài đặt Agent CLI** — cho agent biết gọi `failproofai --hook <event>` trên mỗi lần sử dụng công cụ:
  - **Claude Code**: `~/.claude/settings.json` (người dùng), `<cwd>/.claude/settings.json` (dự án), `<cwd>/.claude/settings.local.json` (cục bộ)
  - **OpenAI Codex**: `~/.codex/hooks.json` (người dùng), `<cwd>/.codex/hooks.json` (dự án) — Codex không có phạm vi `local`
  - **GitHub Copilot CLI _(beta)_**: `~/.copilot/hooks/failproofai.json` (người dùng), `<cwd>/.github/hooks/failproofai.json` (dự án) — Copilot không có phạm vi `local`. Các mục hook sử dụng các trường lệnh `bash`/`powershell` được khóa bằng HĐH của Copilot với `timeoutSec`; file mang một đánh dấu `version: 1` cấp cao nhất. Hỗ trợ GitHub Copilot CLI là **beta** trong khi chúng tôi xác minh lược đồ bản ghi `events.jsonl` (mà các tài liệu công khai không chỉ định) so với các phiên làm việc thực tế hơn.
  - **Cursor Agent _(beta)_**: `~/.cursor/hooks.json` (người dùng), `<cwd>/.cursor/hooks.json` (dự án) — Cursor không có phạm vi `local`. Các mục hook sử dụng hình thức hình dạng Claude `{type, command, timeout}` (không chia `bash`/`powershell`), nhưng được lưu trữ dưới các khóa sự kiện camelCase (`preToolUse`, `beforeSubmitPrompt`, …) trong một mảng phẳng theo [lược đồ hook](https://cursor.com/docs/hooks) của Cursor; file mang một đánh dấu `version: 1` cấp cao nhất. Trình xử lý chuẩn hóa camelCase → PascalCase thông qua `CURSOR_EVENT_MAP` để các chính sách tích hợp sẵn hiện có kích hoạt không thay đổi. Hỗ trợ Cursor Agent là **beta** trong khi chúng tôi xác minh định dạng đoạn trích trên đĩa của Cursor (không được chỉ định trong các tài liệu công khai) so với nhiều cài đặt thực tế hơn.
  - **OpenCode _(beta)_**: `~/.config/opencode/opencode.json` + `~/.config/opencode/plugins/failproofai.mjs` (người dùng), `<cwd>/.opencode/opencode.json` + `<cwd>/.opencode/plugins/failproofai.mjs` (dự án) — OpenCode không có phạm vi `local`. Không giống như sáu CLI khác, OpenCode **không có hệ thống hook lệnh bên ngoài**: nó tải các plugin JS/TS trong quá trình đăng ký rõ ràng thông qua mảng `plugin: []` trong `opencode.json` (tự động khám phá từ `.opencode/plugins/` **không phải** là cách plugin tải trên opencode v1.14.33). Install thả một shim plugin tạo nhỏ gọi subprocess của nhị phân failproofai và dịch lại phản hồi JSON hình dạng Claude của nhị phân thành ngữ nghĩa plugin: `throw new Error()` cho tool-event deny (hủy bỏ lệnh gọi công cụ), `client.session.prompt(...)` cho instruct AND cho `Stop` / `SubagentStop` deny (gửi lý do từ chối dưới dạng tin nhắn người dùng tiếp theo — kênh thử lại bắt buộc duy nhất vì `session.idle` chỉ là thông báo và ném từ nó là no-op), và no-op cho allow. Shim chuẩn hóa cả tên công cụ (chữ thường → PascalCase thông qua `OPENCODE_TOOL_MAP`) và khóa đối số đầu vào công cụ (camelCase → snake_case thông qua `OPENCODE_TOOL_INPUT_MAP` cho `Read` / `Write` / `Edit`, ví dụ `filePath` → `file_path`, `oldString` → `old_string`) trước khi chuyển tiếp đến nhị phân, do đó các builtin kiểm tra đường dẫn như `block-read-outside-cwd`, `block-env-files`, và `block-secrets-write` kích hoạt không thay đổi trên các lệnh gọi công cụ OpenCode. Các phiên sống trong cơ sở dữ liệu SQLite của opencode tại `~/.local/share/opencode/opencode.db`; trình xem phiên của bảng điều khiển đọc chúng thông qua `opencode db --format json` và `opencode export <id>`. Hỗ trợ OpenCode là **beta** trong khi chúng tôi xác minh hành vi trên các phiên bản và so với nhiều phiên làm việc thực tế hơn. Xem [tài liệu plugin OpenCode](https://opencode.ai/docs/plugins/).
  - **Pi _(beta)_**: `~/.pi/agent/settings.json` (người dùng), `<cwd>/.pi/settings.json` (dự án) — Pi không có phạm vi `local`. Pi tải các gói phần mở rộng TypeScript tại khởi động; file cài đặt là một mảng chuỗi phẳng `{"packages": ["./relative/path", …]}`. failproofai ghi một mục mảng package duy nhất trỏ vào thư mục `pi-extension/` được đóng gói của nó. Phần mở rộng bên trong đăng ký các sự kiện `tool_call` / `user_bash` / `input` / `session_start` của Pi và shell ra `failproofai --hook <Event> --cli pi`; trình xử lý chuẩn hóa underscore_lower_snake_case → PascalCase thông qua `PI_EVENT_MAP` để các chính sách tích hợp sẵn hiện có kích hoạt không thay đổi. Các đối số đầu vào công cụ cũng được chuẩn hóa thông qua `PI_TOOL_INPUT_MAP` (Pi's Read / Write / Edit cung cấp `path` thay vì `file_path`; ánh xạ khóa cấp cao nhất cho phép `block-env-files` và `block-secrets-write` kích hoạt — `block-read-outside-cwd` đã có một fallback `path`). Hỗ trợ Pi là **beta** trong khi API phần mở rộng Pi và bố cục nhật ký phiên ổn định.
  - **Gemini CLI _(beta)_**: `~/.gemini/settings.json` (người dùng), `<cwd>/.gemini/settings.json` (dự án) — Gemini không có phạm vi `local` (nó ghi lại một phạm vi `system` tại `/etc/gemini-cli/settings.json` mà failproofai không hỗ trợ). Các mục hook sử dụng hình thức Claude `{type, command, timeout}` được bao bọc trong lược đồ matcher `{matcher, hooks: [...]}` của Gemini với `matcher: "*"` theo mặc định. Các sự kiện là PascalCase (`SessionStart`, `BeforeAgent`, `AfterAgent`, `BeforeModel`, `AfterModel`, `BeforeToolSelection`, `BeforeTool`, `AfterTool`, `PreCompress`, `Notification`, `SessionEnd`); trình xử lý ánh xạ tới các tên chính tắc Claude thông qua `GEMINI_EVENT_MAP`. Tên công cụ là snake_case (`run_shell_command`, `read_file`, `write_file`, `replace`, …) — trình xử lý chuẩn hóa thông qua `GEMINI_TOOL_MAP` để các chính sách tích hợp sẵn hiện có kích hoạt không thay đổi. Trình đánh giá chính sách phát ra hình dạng phẳng Gemini `{decision: "deny", reason}` (ưa thích theo Gemini's "Golden Rule" hợp đồng exit-0), `{hookSpecificOutput: {hookEventName, additionalContext}}` cho việc tiêm bối cảnh trên BeforeAgent / AfterTool / SessionStart, và `{decision: "block", reason}` trên AfterAgent cho ngữ nghĩa thử lại bắt buộc. Hỗ trợ Gemini CLI là **beta** trong khi chúng tôi mở rộng phạm vi thực tế. Xem [tài liệu hook Gemini CLI](https://geminicli.com/docs/hooks/).
- **`policies-config.json`** — cho failproofai biết những chính sách nào để đánh giá và với tham số gì (chia sẻ trên tất cả các CLI agent)

Chuyển `--cli claude|codex|copilot|cursor|opencode|pi|gemini` để nhắm mục tiêu một agent cụ thể (cách nhau bằng dấu cách hoặc lặp lại cho bất kỳ tập hợp nào):

```bash
failproofai policies --install --cli codex --scope project
failproofai policies --install --cli copilot --scope project
failproofai policies --install --cli cursor --scope project
failproofai policies --install --cli opencode --scope project
failproofai policies --install --cli pi --scope project
failproofai policies --install --cli gemini --scope project
failproofai policies --install --cli claude codex copilot cursor opencode pi gemini
```

Khi `--cli` bị bỏ qua, `failproofai` phát hiện những CLI agent nào được cài đặt (`which claude` / `which codex` / `which copilot` / `which cursor-agent` / `which opencode` / `which pi` / `which gemini`):

- **Phát hiện một CLI** — tự động chọn CLI đó mà không cần nhắc.
- **Phát hiện nhiều CLI** trong một thiết bị đầu cuối tương tác — hiển thị dấu mũi tên đơn lựa chọn nhóm thành phần `Detected (N)` (với hàng tổng hợp `Install for all N detected` + mỗi CLI được phát hiện riêng lẻ) và phần `Not installed (M) · install hooks ahead of time` liệt kê mọi CLI hỗ trợ không được phát hiện như một tùy chọn cài đặt trước (↑↓ để di chuyển, Enter để chọn, ^C để thoát). Luồng gỡ cài đặt chỉ hiển thị phần Detected.
- **Phát hiện nhiều CLI** trong một lần chạy không tương tác (CI, không TTY) — cài đặt cho tất cả các CLI được phát hiện mà không cần nhắc.
- **Không phát hiện cái nào** — quay lại `claude`, với cảnh báo rằng không tìm thấy nhị phân agent nào trong PATH; lệnh hook vẫn được viết để nó kích hoạt ngay khi bạn cài đặt một cái.

Bạn có thể chỉnh sửa `policies-config.json` trực tiếp bất kỳ lúc nào; các thay đổi có hiệu lực ngay trên sự kiện hook tiếp theo mà không cần khởi động lại.

## Ví dụ: cấu hình cấp dự án với các giá trị mặc định của nhóm

Commit `.failproofai/policies-config.json` vào kho lưu trữ của bạn:

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "block-env-files"
  ],
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "release", "hotfix"]
    }
  }
}
```

Sau đó, mỗi nhà phát triển có thể tạo `.failproofai/policies-config.local.json` (được gitignore) cho các ghi đè cá nhân mà không ảnh hưởng đến các đồng nghiệp.
