## Configurazione di hook per Claude Code

Failproof AI si integra con Claude Code tramite il suo [sistema di hook](https://docs.anthropic.com/en/docs/claude-code/hooks). Quando esegui `failproofai policies --install`, registra comandi hook nel file `settings.json` di Claude Code che si attivano ad ogni chiamata di tool.

<Steps>
  <Step title="Installa failproofai">
    ```bash
    npm install -g failproofai
    ```
  </Step>
  <Step title="Abilita tutte le politiche integrate">
    ```bash
    failproofai policies --install
    ```
  </Step>
  <Step title="Verifica che gli hook siano registrati">
    ```bash
    cat ~/.claude/settings.json | grep failproofai
    ```

    Dovresti vedere le voci di hook per gli eventi `PreToolUse`, `PostToolUse`, `Notification` e `Stop`.
  </Step>
  <Step title="Esegui Claude Code">
    ```bash
    claude
    ```

    Le politiche ora vengono eseguite automaticamente ad ogni chiamata di tool. Prova a chiedere a Claude di eseguire `sudo rm -rf /` - verrà bloccato.
  </Step>
</Steps>

## Configurazione di hook per Agents SDK

Se stai costruendo con [Agents SDK](https://docs.anthropic.com/en/docs/agents-sdk), puoi utilizzare lo stesso sistema di hook a livello programmatico.

<Steps>
  <Step title="Installa failproofai nel tuo progetto">
    ```bash
    npm install failproofai
    ```
  </Step>
  <Step title="Configura gli hook nel tuo agente">
    Passa i comandi hook quando crei il tuo processo agente. Gli hook si attivano allo stesso modo di Claude Code - tramite JSON su stdin/stdout:

    ```bash
    failproofai --hook PreToolUse   # chiamato prima di ogni tool
    failproofai --hook PostToolUse  # chiamato dopo ogni tool
    ```
  </Step>
  <Step title="Scrivi una politica personalizzata per il tuo agente">
    ```javascript
    import { customPolicies, allow, deny } from "failproofai";

    customPolicies.add({
      name: "limit-to-project-dir",
      description: "Mantieni l'agente dentro la directory del progetto",
      match: { events: ["PreToolUse"] },
      fn: async (ctx) => {
        const path = String(ctx.toolInput?.file_path ?? "");
        if (path.startsWith("/") && !path.startsWith(ctx.session?.cwd ?? "")) {
          return deny("L'agente è limitato alla directory del progetto");
        }
        return allow();
      },
    });
    ```
  </Step>
  <Step title="Installa la politica personalizzata">
    ```bash
    failproofai policies --install --custom ./my-agent-policies.js
    ```
  </Step>
</Steps>

## Blocca comandi distruttivi

L'installazione più comune - previeni che gli agenti causino danni irreversibili.

```bash
failproofai policies --install block-sudo block-rm-rf block-force-push block-curl-pipe-sh
```

Cosa fa:
- `block-sudo` - blocca tutti i comandi `sudo`
- `block-rm-rf` - blocca l'eliminazione ricorsiva di file
- `block-force-push` - blocca `git push --force`
- `block-curl-pipe-sh` - blocca il piping di script remoti a shell

## Previeni la fuga di segreti

Impedisci agli agenti di vedere o disperdere credenziali nell'output dei tool.

```bash
failproofai policies --install sanitize-api-keys sanitize-jwt sanitize-connection-strings sanitize-bearer-tokens
```

Questi si attivano su `PostToolUse` - dopo l'esecuzione di un tool, puliscono l'output prima che l'agente lo veda.

## Ricevi avvisi Slack quando gli agenti hanno bisogno di attenzione

Utilizza l'hook di notifica per inoltrare gli avvisi di inattività a Slack.

```javascript
import { customPolicies, allow, instruct } from "failproofai";

customPolicies.add({
  name: "slack-on-idle",
  description: "Avvisa Slack quando l'agente è in attesa di input",
  match: { events: ["Notification"] },
  fn: async (ctx) => {
    const webhookUrl = process.env.SLACK_WEBHOOK_URL;
    if (!webhookUrl) return allow();

    const message = String(ctx.payload?.message ?? "L'agente è in attesa");
    const project = ctx.session?.cwd ?? "sconosciuto";

    try {
      await fetch(webhookUrl, {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({
          text: `*${message}*\nProgetto: \`${project}\``,
        }),
        signal: AbortSignal.timeout(5000),
      });
    } catch {
      // non bloccare mai l'agente se Slack non è raggiungibile
    }

    return allow();
  },
});
```

Installalo:

```bash
SLACK_WEBHOOK_URL=https://hooks.slack.com/... failproofai policies --install --custom ./slack-alerts.js
```

## Mantieni gli agenti su un branch

Previeni che gli agenti cambino branch o facciano push su branch protetti.

```javascript
import { customPolicies, allow, deny } from "failproofai";

customPolicies.add({
  name: "stay-on-branch",
  description: "Impedisci all'agente di fare checkout su altri branch",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const cmd = String(ctx.toolInput?.command ?? "");
    if (/git\s+checkout\s+(?!-b)/.test(cmd)) {
      return deny("Rimani sul branch corrente. Crea un nuovo branch con -b se necessario.");
    }
    return allow();
  },
});
```

## Richiedi test prima dei commit

Ricorda agli agenti di eseguire i test prima di fare commit.

```javascript
import { customPolicies, allow, instruct } from "failproofai";

customPolicies.add({
  name: "test-before-commit",
  description: "Ricorda all'agente di eseguire i test prima di fare commit",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const cmd = String(ctx.toolInput?.command ?? "");
    if (/git\s+commit/.test(cmd)) {
      return instruct("Esegui i test prima di fare commit. Usa `npm test` o `bun test` prima.");
    }
    return allow();
  },
});
```

## Blocca un repo di produzione

Effettua il commit di una configurazione a livello di progetto affinché ogni sviluppatore del tuo team ottenga le stesse politiche.

Crea `.failproofai/policies-config.json` nel tuo repo:

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-force-push",
    "block-push-master",
    "block-env-files",
    "sanitize-api-keys",
    "sanitize-jwt"
  ],
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "release", "production"]
    }
  }
}
```

Poi fai il commit:

```bash
git add .failproofai/policies-config.json
git commit -m "Aggiungi politiche failproofai del team"
```

Ogni membro del team che ha failproofai installato raccoglierà automaticamente queste regole.

## Costruisci uno standard di qualità a livello organizzativo con politiche di convenzione

L'installazione più efficace: effettua il commit di `.failproofai/policies/` nel tuo repo con politiche adattate al tuo progetto. Ogni membro del team le ottiene automaticamente — nessun comando di installazione, nessun cambio di configurazione.

<Steps>
  <Step title="Crea la directory e aggiungi le politiche">
    ```bash
    mkdir -p .failproofai/policies
    ```

    ```js
    // .failproofai/policies/team-policies.mjs
    import { customPolicies, allow, deny, instruct } from "failproofai";

    // Applica il package manager preferito del tuo team
    // (o abilita invece la politica integrata prefer-package-manager)
    customPolicies.add({
      name: "enforce-bun",
      match: { events: ["PreToolUse"] },
      fn: async (ctx) => {
        if (ctx.toolName !== "Bash") return allow();
        const cmd = String(ctx.toolInput?.command ?? "");
        if (/\bnpm\b/.test(cmd)) return deny("Usa bun invece di npm.");
        return allow();
      },
    });

    // Ricorda all'agente di eseguire i test prima di fare commit
    customPolicies.add({
      name: "test-before-commit",
      match: { events: ["PreToolUse"] },
      fn: async (ctx) => {
        if (ctx.toolName !== "Bash") return allow();
        if (/git\s+commit/.test(ctx.toolInput?.command ?? "")) {
          return instruct("Esegui i test prima di fare commit.");
        }
        return allow();
      },
    });
    ```
  </Step>
  <Step title="Effettua il commit su git">
    ```bash
    git add .failproofai/policies/
    git commit -m "Aggiungi politiche di qualità del team"
    ```
  </Step>
  <Step title="Continua a migliorare">
    Man mano che il tuo team incontra nuove modalità di guasto, aggiungi politiche e fai push. Tutti ricevono l'aggiornamento al prossimo `git pull`. Queste politiche diventano uno standard di qualità vivente che cresce con il tuo team.
  </Step>
</Steps>
