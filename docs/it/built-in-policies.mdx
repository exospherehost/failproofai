failproofai viene fornito con 39 politiche integrate che catturano le modalitĆ  di errore comuni degli agenti. Ogni politica si attiva su un tipo di evento hook specifico e sul nome dello strumento. Diciannove politiche accettano parametri che ti permettono di regolarne il comportamento senza scrivere codice. Cinque politiche di workflow applicano una pipeline commit ā†’ push ā†’ PR ā†’ CI prima che Claude si fermi.

## Panoramica

Le politiche sono raggruppate in categorie:

| Categoria | Politiche | Tipo di hook |

- **`block-`** ā€” impedisce all'agente di procedere.
- **`warn-`** ā€” fornisce all'agente contesto aggiuntivo in modo che possa autocorreggersi.
- **`sanitize-`** ā€” pulisce i dati sensibili dall'output dello strumento prima che l'agente li veda.

### Spazi dei nomi

Ogni politica risiede in uno slot `<namespace>/<name>`. Le politiche integrate appartengono allo spazio dei nomi
**`exospherehost/`** ā€” ad esempio, `exospherehost/sanitize-jwt`. Lo spazio dei nomi previene collisioni quando carichi anche politiche personalizzate o di terze parti
con nomi brevi simili.

Nella tua configurazione puoi fare riferimento a una politica integrata con il suo nome breve o il suo nome qualificato; entrambe le forme si risolvono nella stessa politica:

```json
{
  "enabledPolicies": [
    "sanitize-jwt",
    "exospherehost/block-rm-rf"
  ]
}
```

Se un nome non contiene `/`, failproofai lo tratta come appartenente allo spazio dei nomi predefinito `exospherehost`. I nomi che giĆ  contengono `/` (ad es. `myorg/foo`,
`custom/my-hook`) vengono mantenuti cosĆ¬ come sono.
- **`require-`** ā€” blocca l'evento Stop fino a quando le condizioni non vengono soddisfatte.

---

<Tip>
Ogni politica supporta un campo `hint` opzionale in `policyParams`. L'hint viene aggiunto al messaggio di deny o instruct che Claude vede, fornendo una guida pratica senza modificare il codice della politica. Funziona con le politiche integrate, personalizzate e di convenzione. Vedi [Configurazione ā†’ hint](/it/configuration#hint-cross-cutting) per i dettagli.
</Tip>

## Comandi pericolosi

Impedisci agli agenti di eseguire operazioni difficili da annullare o che potrebbero danneggiare il sistema host.

### `block-sudo`

**Evento:** PreToolUse (Bash)  
**Predefinito:** Nega qualsiasi comando `sudo`.

Blocca le invocazioni che includono la parola chiave `sudo`. L'abbinamento del pattern viene eseguito su token di comando analizzati, non sulla stringa grezza, per prevenire il bypass tramite iniezione di operatori shell.

**Parametri:**

| Parametro | Tipo | Predefinito | Descrizione |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | Prefissi di comando esatti consentiti. Ogni voce viene abbinata ai token argv analizzati. |

**Esempio:**

```json
{
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    }
  }
}
```

Con questa configurazione, `sudo systemctl status nginx` ĆØ consentito, ma `sudo rm /etc/hosts` ĆØ negato.

<Note>
I pattern vengono abbinati ai token analizzati, non alla stringa di comando grezza. Questo previene il bypass tramite operatori shell aggiunti (ad es. `sudo systemctl status x; rm -rf /` non corrisponde a `sudo systemctl status *`).
</Note>

**Evento:** PreToolUse (Bash)

|-------|------|---------|-------------|

**Esempio:**

## Comandi infra

Impedisci agli agenti di codifica di eseguire CLI di infrastruttura o di attivare pipeline CI/CD. Tutte le politiche in questa categoria sono **opt-in** (`defaultEnabled: false`) ā€” gli agenti che hanno legittimamente bisogno di chiamare `kubectl`, `terraform`, ecc. non verranno disturbati a meno che tu non abiliti la politica. Una volta abilitata, ogni invocazione della CLI corrispondente viene negata a meno che il comando non corrisponda a una voce in `allowPatterns`.

La grammatica dei pattern ĆØ la stessa di [`block-sudo`](#block-sudo): i token vengono abbinati ai argv analizzati, `*` ĆØ un wildcard per un token, e qualsiasi comando contenente un operatore shell standalone (`&&`, `||`, `|`, `;`) o un token con metacaratteri shell incorporati viene rifiutato prima dell'abbinamento della whitelist per prevenire bypass di iniezione.

**Evento:** PreToolUse (Bash)  
**Predefinito:** Nega qualsiasi invocazione `kubectl`.

**Parametri:**

| Parametro | Tipo | Predefinito | Descrizione |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | Prefissi di comando kubectl consentiti. |

**Esempio:**

```json
{
  "policyParams": {
    "block-kubectl": {
      "allowPatterns": ["kubectl get *", "kubectl describe *", "kubectl logs *"]

}
```

Con questa configurazione, `kubectl get pods` ĆØ consentito ma `kubectl apply -f deploy.yaml` ĆØ negato.

## Segreti (sanitizer)

Impedisci agli agenti di perdere credenziali nel loro contesto o output. Le politiche sanitizer si attivano su eventi **PostToolUse**. Quando Claude esegue un comando Bash, legge un file o chiama qualsiasi strumento, queste politiche controllano l'output prima che venga restituito a Claude. Se viene rilevato un pattern di segreto, la politica restituisce una decisione di negazione che impedisce all'output di essere passato indietro.

### `sanitize-jwt`

**Evento:** PostToolUse (tutti gli strumenti)  
**Predefinito:** Redige i token JWT (tre segmenti base64url separati da `.`).

Nessun parametro.

---

### `sanitize-api-keys`

**Evento:** PostToolUse (tutti gli strumenti)  
**Predefinito:** Redige i formati di chiavi API comuni: Anthropic (`sk-ant-`), OpenAI (`sk-`), GitHub PATs (`ghp_`), chiavi di accesso AWS (`AKIA`), chiavi Stripe (`sk_live_`, `sk_test_`) e chiavi API Google (`AIza`).

**Parametri:**

| Parametro | Tipo | Predefinito | Descrizione |
|-------|------|---------|-------------|
| `additionalPatterns` | `{ regex: string; label: string }[]` | `[]` | Pattern regex aggiuntivi da trattare come segreti. |

**Esempio:**

```json
{
  "policyParams": {
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo internal API key" },
        { "regex": "pat_[0-9a-f]{40}", "label": "Internal PAT" }
      ]
    }
  }
}
```

---

### `sanitize-connection-strings`

**Evento:** PostToolUse (tutti gli strumenti)  
**Predefinito:** Redige le stringhe di connessione del database che contengono credenziali incorporate (ad es. `postgresql://user:password@host/db`).

Nessun parametro.

---

### `sanitize-private-key-content`

**Evento:** PostToolUse (tutti gli strumenti)  
**Predefinito:** Redige i blocchi PEM (`-----BEGIN PRIVATE KEY-----`, `-----BEGIN RSA PRIVATE KEY-----`, ecc.).

Nessun parametro.

---

### `sanitize-bearer-tokens`

**Evento:** PostToolUse (tutti gli strumenti)  
**Predefinito:** Redige le intestazioni `Authorization: Bearer <token>` dove il token ĆØ 20 o piĆ¹ caratteri.

Nessun parametro.

## Accesso ai file

Mantieni gli agenti che lavorano all'interno dei confini del progetto e lontani dai file sensibili.

### `block-read-outside-cwd`

**Evento:** PreToolUse (Read, Bash)  
**Predefinito:** Nega la lettura dei file al di fuori della radice del progetto. Il confine ĆØ `CLAUDE_PROJECT_DIR` (impostato una volta per sessione da Claude Code), con un fallback alla directory di lavoro corrente della sessione quando la variabile non ĆØ impostata. L'uso della radice del progetto anzichĆ© del `cwd` live significa che il confine rimane stabile anche dopo che Claude si sposta in una sottodirectory.

**Parametri:**

| Parametro | Tipo | Predefinito | Descrizione |
|-------|------|---------|-------------|
| `allowPaths` | `string[]` | `[]` | Prefissi di percorso assoluto consentiti anche se al di fuori della radice del progetto. |

**Esempio:**

```json
{
  "policyParams": {
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company/config"]
    }
  }
}
```

---

### `block-secrets-write`

**Evento:** PreToolUse (Write, Edit)  
**Predefinito:** Nega le scritture nei file comunemente usati per chiavi private e certificati: `id_rsa`, `id_ed25519`, `*.key`, `*.pem`, `*.p12`, `*.pfx`.

**Parametri:**

| Parametro | Tipo | Predefinito | Descrizione |
|-------|------|---------|-------------|
| `additionalPatterns` | `string[]` | `[]` | Pattern di filename aggiuntivi (stile glob) da bloccare. |

**Esempio:**

```json
{

    "block-secrets-write": {
      "additionalPatterns": [".token", ".secret"]
    }
  }
}
```

## Git

Previeni i push accidentali, force-push e errori di branch difficili da annullare.

### `block-push-master`

**Evento:** PreToolUse (Bash)  
**Predefinito:** Nega `git push origin main` e `git push origin master`.

**Parametri:**

| Parametro | Tipo | Predefinito | Descrizione |
|-------|------|---------|-------------|
| `protectedBranches` | `string[]` | `["main", "master"]` | Nomi di branch che non possono essere pushati direttamente. |

**Esempio:**

```json
{
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "master", "release", "prod"]
    }
  }
}
```

<Tip>
Per consentire il push a tutti i branch (disabilitando effettivamente questa politica senza rimuoverla da `enabledPolicies`), imposta `protectedBranches: []`.
</Tip>

---

### `block-work-on-main`

**Evento:** PreToolUse (Bash)  
**Predefinito:** Nega `git commit`, `git merge`, `git rebase` e `git cherry-pick` mentre l'albero di lavoro ĆØ su `main` o `master`. La creazione e il passaggio di branch (`git checkout`, `git checkout -b`, `git switch`, `git switch -c`) non sono interessati.

**Parametri:**

| Parametro | Tipo | Predefinito | Descrizione |
|-------|------|---------|-------------|
| `protectedBranches` | `string[]` | `["main", "master"]` | Nomi di branch su cui commit/merge/rebase/cherry-pick viene negato. |

---

### `block-force-push`

**Evento:** PreToolUse (Bash)  
**Predefinito:** Nega `git push --force` e `git push -f`.

## Gestori di pacchetti

Applica quale gestore di pacchetti l'agente ĆØ autorizzato a utilizzare.

### `prefer-package-manager`

**Evento:** PreToolUse (Bash)  
**Predefinito:** Disabilitato. Quando abilitato, blocca qualsiasi comando del gestore di pacchetti non nella lista `allowed` e dice a Claude di riscrivere il comando usando un gestore consentito.

Rileva: pip, pip3, python -m pip, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo.

| Parametro | Tipo | Predefinito | Descrizione |
|-----------|------|---------|-------------|
| `allowed` | string[] | `[]` | Nomi dei gestori di pacchetti consentiti. Qualsiasi gestore rilevato non in questa lista viene bloccato. Quando vuota, la politica ĆØ un no-op. |
| `blocked` | string[] | `[]` | Nomi di gestori aggiuntivi da bloccare al di lĆ  della lista integrata (ad es. `['pdm', 'pipx']`). |

La lista di blocco integrata copre: pip, pip3, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo. Usa `blocked` per aggiungere gestori non in questa lista.

**Configurazione di esempio:**

```json
{
  "enabledPolicies": ["prefer-package-manager"],
  "policyParams": {
    "prefer-package-manager": {
      "allowed": ["uv", "bun"],
      "blocked": ["pdm", "pipx"]
    }
  }
}
```

Con questa configurazione, `pip install flask` e `pdm install flask` sono entrambi negati con un messaggio che dice a Claude di usare invece `uv` o `bun`. Comandi come `uv pip install flask` sono consentiti perchĆ© `uv` ĆØ nella whitelist e viene controllato per primo.

## Workflow

Applica un workflow end-of-session disciplinato. Queste politiche si attivano sull'evento **Stop** e negano l'agente dal fermarsi fino a quando ogni condizione non viene soddisfatta. Seguono una catena di dipendenza naturale: commit ā†’ push ā†’ PR ā†’ CI. Se una politica nega, le politiche successive nella catena vengono saltate (negare interrompe).

Tutte le politiche di workflow sono **fail-open**: se lo strumento richiesto non ĆØ disponibile (ad es. `gh` non installato, nessun git remote), la politica consente con un messaggio informativo che spiega perchĆ© il controllo ĆØ stato saltato.

### Semantica Stop per-CLI

L'applicazione Stop sembra leggermente diversa tra i sette CLI supportati perchĆ© ognuno espone un diverso contratto di hook "agente terminato". Il **risultato** ĆØ lo stesso ā€” l'agente non riesce a fermarsi mentre un gate del workflow sta fallendo ā€” ma la **meccanica** differisce. La tabella sottostante fornisce un riepilogo; solo Pi ha una stranezza visibile all'utente che vale la pena comprendere prima di abilitare una politica `require-*-before-stop`.

| CLI | Quando il gate si attiva | Quello che vedi |
|---|---|---|
| Claude Code | Stesso loop dell'agente, immediatamente | Claude continua a lavorare ā€” corregge il problema, quindi tenta nuovamente di terminare. Nessuna interruzione visibile per te. |
| Codex | Stesso loop dell'agente, immediatamente | Stesso di Claude. |
| GitHub Copilot CLI | Stesso loop dell'agente, immediatamente | Stesso di Claude (utilizza il canale di retry `{decision:"block", reason}` di Copilot ā€” verificato empiricamente contro Copilot CLI 1.0.41). |
| Cursor Agent | Stesso loop dell'agente, immediatamente | Stesso di Claude (utilizza il canale `{followup_message}` di Cursor ā€” limitato a `loop_limit`, predefinito 5 tentativi). |
| Gemini CLI | Stesso loop dell'agente, immediatamente | Stesso di Claude (utilizza il canale `{decision:"block", reason}` di Gemini su `AfterAgent`). |
| OpenCode | Stesso loop dell'agente, immediatamente | Stesso di Claude (utilizza la chiamata SDK `client.session.prompt(...)` di OpenCode instradato tramite `hookSpecificOutput.additionalContext`). |
| **Pi (pi-coding-agent)** | **Turno utente successivo** | **Pi si ferma visibilmente** quando il gate si attiva ā€” il suo loop dell'agente esce e sei restituito al prompt. Il gate quindi si attiva la prossima volta che invii un prompt: failproofai antepone una direttiva `MANDATORY ACTION REQUIRED` al prompt di sistema di quel turno, istruendo l'LLM di completare il passaggio del workflow (commit, push, ecc.) prima di fare tutto ciĆ² che hai chiesto. |

<Note>
**Limitazione Pi.** L'evento `AgentEndEvent` di Pi (l'equivalente upstream dell'hook `Stop` di Claude) non ha un tipo Result ā€” nel momento in cui si attiva, il loop dell'agente di Pi ĆØ giĆ  uscito. Pi non puĆ² essere forzato a riprovare lo stesso loop nel modo in cui Claude / Copilot / Cursor / Gemini / OpenCode puĆ². failproofai sposta il gate all'evento `before_agent_start` di Pi (che si attiva dopo il prompt utente successivo) in modo che il controllo del workflow sia comunque applicato, solo al turno successivo piuttosto che a quello corrente.

**Quello che questo significa in pratica:**

- Dopo che Pi si ferma, la ragione della negazione viene catturata in memoria con la chiave dell'id sessione Pi. Il prompt successivo che invii nella stessa sessione Pi lo drena: l'LLM vede la direttiva `MANDATORY ACTION REQUIRED` in cima al suo prompt di sistema, effettua il commit (o push / apri il PR / attendi CI) e solo allora continua con la tua richiesta. La ragione della negazione catturata ĆØ one-shot ā€” una volta drenata, il gate ĆØ chiaro.
- Il gate ĆØ limitato dal ciclo di vita del processo Pi. Se esegui `Ctrl+C` Pi o esci tra i turni, l'entry in memoria viene rimossa insieme al processo e il gate viene mancato. Claude, Copilot, Cursor, Gemini e OpenCode hanno lo stesso bound (uccidi l'agente e il gate viene mancato) ā€” Pi lo rende solo piĆ¹ visibile perchĆ© l'agente esce visibilmente prima che il gate si attivi.
- Un'entry negazione in sospeso viene anche azzerata su `session_shutdown` per qualsiasi motivo (`new` / `resume` / `fork` / `quit`), in modo che un gate stantio da una sessione precedente non possa trapelare in una sessione nuova iniziata nello stesso processo Pi.

Se hai bisogno di un retry nello stesso loop di Claude, esegui le tue politiche `Stop` su qualsiasi uno degli altri sei CLI supportati. Stiamo tracciando Pi upstream per un futuro tipo di Result su `AgentEndEvent` che ci permetterebbe di colmare questo gap.
</Note>

### `require-commit-before-stop`

**Evento:** Stop  
**Predefinito:** Nega il stop quando ci sono modifiche non committate (file modificati, in staging o non tracciati). Restituisce un messaggio informativo quando la directory di lavoro ĆØ pulita.

Nessun parametro.

### `require-push-before-stop`

**Evento:** Stop  
**Predefinito:** Nega il stop quando ci sono commit non pushati o quando il branch corrente non ha un branch di tracking remoto. Suggerisce `git push -u` per creare un branch di tracking se necessario. Fail open se nessun remote ĆØ configurato.

**Parametri:**

| Parametro | Tipo | Predefinito | Descrizione |
|-------|------|---------|-------------|
| `remote` | `string` | `"origin"` | Nome del remote verso cui pushare. |

**Esempio:**

```json
{
  "policyParams": {
    "require-push-before-stop": {
      "remote": "upstream"
    }
  }
}
```

### `require-pr-before-stop`

**Evento:** Stop  
**Predefinito:** Nega il stop quando nessuna pull request esiste per il branch corrente, o quando la PR esistente ĆØ chiusa senza essere mergiata. Istruisce Claude a creare un PR con `gh pr create`. Quando la PR ĆØ **mergiata**, la politica consente (il lavoro ĆØ stato spedito) e il messaggio suggerisce di passare al di fuori del branch (`git checkout main && git pull`).

Nessun parametro.

<Note>
Questa politica richiede [GitHub CLI](https://cli.github.com/) (`gh`) di essere installato e autenticato.
Esegui `gh auth login` con un token di accesso personale che ha lo scope `repo` per l'accesso in lettura alle
pull request. Se `gh` non ĆØ installato o non ĆØ autenticato, la politica fail open e segnala il motivo a Claude.
</Note>

### `require-no-conflicts-before-stop`

**Evento:** Stop  
**Predefinito:** Nega il stop quando il branch corrente non puĆ² essere mergiato in modo pulito nel branch di base. La politica prima conferma che ci sia una PR `OPEN` su GitHub per il branch ā€” senza una, non c'ĆØ nessun target di merge da applicare, quindi l'intera politica short-circuit per consentire. Una volta confermata una PR `OPEN`, due sonde indipendenti vengono eseguite:

1. **Locale** ā€” `git merge-tree --write-tree --name-only origin/<baseBranch> HEAD`. Su conflitto, il messaggio di negazione nomina i file in conflitto in modo che Claude sappia esattamente cosa risolvere.
2. **GitHub** ā€” riutilizza il risultato `gh pr view --json mergeable,state` giĆ  recuperato nel precontrollo. Cattura i conflitti che un `origin/<baseBranch>` locale stantio avrebbe mancato (ad es. qualcuno ha sbarcato una PR in conflitto su `main` dall'ultimo fetch). Un risultato `CONFLICTING` nega. Un risultato `UNKNOWN` nega anche e istruisce Claude ad aspettare ~10 secondi e ri-controllare prima di tentare di fermarsi di nuovo ā€” questo previene falsi negativi mentre GitHub ricalcola.

Salta interamente (consente) quando: `gh` non ĆØ installato, nessuna PR esiste per il branch, lo stato della PR non ĆØ `OPEN` (ad es. `MERGED`, `CLOSED`), o `gh pr view` restituisce output non parseable. Anche fail open quando `origin/<baseBranch>` ĆØ mancante localmente o quando nessun commit ĆØ davanti a base ā€” quei fallthrough Layer 1 consultano comunque la mergeability della PR in cache prima di consentire.

**Parametri:**

| Parametro | Tipo | Predefinito | Descrizione |
|-------|------|---------|-------------|
| `baseBranch` | `string` | `"main"` | Branch di base su cui controllare i conflitti. |

<Note>
GitHub CLI (`gh`) ĆØ richiesto per questa politica. La politica utilizza `gh pr view` per confermare
che esista una PR `OPEN` prima di eseguire qualsiasi sonda di conflitto ā€” senza `gh`, la politica
short-circuit per consentire. Esegui `gh auth login` con un token di accesso personale che ha
lo scope `repo` per l'accesso in lettura alle pull request.
</Note>

### `require-ci-green-before-stop`

**Evento:** Stop  
**Predefinito:** Nega il stop quando i controlli CI stanno fallendo o sono ancora in esecuzione sul branch corrente. Controlla sia i GitHub Actions workflow runs che i bot check di terze parti (ad es. CodeRabbit, SonarCloud, Codecov). Tratta le conclusioni `skipped` e `cancelled` come successo. Restituisce un messaggio informativo quando tutti i controlli passano.

Nessun parametro.

<Note>
Questa politica richiede [GitHub CLI](https://cli.github.com/) (`gh`) di essere installato e autenticato.
Esegui `gh auth login` con un token di accesso personale che ha lo scope `repo` per l'accesso in lettura ai
GitHub Actions workflow runs e all'API Checks. Se `gh` non ĆØ installato o non ĆØ autenticato, la politica fail open e segnala il motivo a Claude.
</Note>

## Disabilitare politiche individuali

Rimuovi una politica specifica da `enabledPolicies` nella tua configurazione, o disattivala nella scheda Policies del dashboard.

```json
{
  "enabledPolicies": [
    "block-rm-rf",
    "sanitize-api-keys"
  ]
}
```

Le politiche non elencate in `enabledPolicies` non vengono eseguite, anche se esistono entry `policyParams` per esse.
