failproofai использует JSON-файлы конфигурации для управления активными политиками, их поведением и местоположением пользовательских политик. Конфигурация разработана для удобного обмена в команде — добавьте её в репозиторий, и каждый разработчик получит одинаковую защиту агента.

---

## Уровни конфигурации

Существует три уровня конфигурации, оцениваемые по приоритету:

| Уровень | Путь файла | Назначение |
|---------|-----------|-----------|
| **project** | `.failproofai/policies-config.json` | Настройки для репозитория, добавляются в контроль версий |
| **local** | `.failproofai/policies-config.local.json` | Личные переопределения для репозитория, игнорируются .gitignore |
| **global** | `~/.failproofai/policies-config.json` | Пользовательские значения по умолчанию для всех проектов |

Когда failproofai получает событие хука, он загружает и объединяет все три файла, существующие в текущей рабочей директории.

### Правила слияния

**`enabledPolicies`** — объединение всех трёх уровней. Политика, активированная на любом уровне, становится активной.

```text
project:  ["block-sudo"]
local:    ["block-rm-rf"]
global:   ["block-sudo", "sanitize-api-keys"]

resolved: ["block-sudo", "block-rm-rf", "sanitize-api-keys"]  ← дедублицированное объединение
```

**`policyParams`** — первый уровень, определивший параметры для политики, полностью побеждает. Глубокого слияния значений в параметрах политики не происходит.

```text
project:  block-sudo → { allowPatterns: ["sudo apt-get update"] }
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo apt-get update"] }   ← побеждает project, global игнорируется
```

```text
project:  (нет записи block-sudo)
local:    (нет записи block-sudo)
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo systemctl status"] }  ← переходит на global
```

**`customPoliciesPath`** — первый уровень, определивший его, побеждает.

**`llm`** — первый уровень, определивший его, побеждает.

## Формат файла конфигурации

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "sanitize-jwt",
    "block-env-files",
    "block-read-outside-cwd"
  ],
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    },
    "block-push-master": {
      "protectedBranches": ["main", "release", "prod"]
    },
    "block-rm-rf": {
      "allowPaths": ["/tmp"]
    },
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company"]
    },
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo API key" }
      ]
    },
    "warn-large-file-write": {
      "thresholdKb": 512
    }
  },
  "customPoliciesPath": "/home/alice/myproject/my-policies.js"
}
```

## Справочник полей

### `enabledPolicies`

Тип: `string[]`

Список имён политик для активации. Имена должны точно совпадать с идентификаторами политик, показанными командой `failproofai policies`. Полный список см. в [Встроенные политики](/ru/built-in-policies).

Политики, не указанные в `enabledPolicies`, неактивны, даже если у них есть записи в `policyParams`.

### `policyParams`

Тип: `Record<string, Record<string, unknown>>`

Переопределения параметров для каждой политики. Внешний ключ — имя политики; внутренние ключи — специфичны для политики. Каждая политика документирует доступные параметры в [Встроенные политики](/ru/built-in-policies).

Если политика имеет параметры, но вы их не указали, используются встроенные значения по умолчанию политики. Пользователи, которые вообще не настраивают `policyParams`, получают идентичное поведение предыдущим версиям.

Неизвестные ключи в блоке параметров политики молча игнорируются во время запуска хука, но помечаются как предупреждения при выполнении `failproofai policies`.

#### `hint` (общий)

Тип: `string` (опционально)

Сообщение, добавляемое к причине, когда политика возвращает `deny` или `instruct`. Используйте его для предоставления Claude действенных рекомендаций без изменения самой политики.

Работает с любым типом политики — встроенной, пользовательской (`custom/`), соглашением проекта (`.failproofai-project/`), или соглашением пользователя (`.failproofai-user/`).

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "Попробуйте создать новую ветку."
    },
    "block-sudo": {
      "allowPatterns": ["sudo apt-get"],
      "hint": "Используйте apt-get напрямую без sudo."
    },
    "custom/my-policy": {
      "hint": "Сначала попросите одобрение у пользователя."
    }
  }
}
```

Когда `block-force-push` запрещает, Claude видит: *"Принудительная отправка заблокирована. Попробуйте создать новую ветку."*

Значения, не являющиеся строками, и пустые строки молча игнорируются. Если `hint` не установлен, поведение не меняется (обратная совместимость).

### `customPoliciesPath`

Тип: `string` (абсолютный путь)

Путь к JavaScript-файлу, содержащему пользовательские политики хуков. Устанавливается автоматически командой `failproofai policies --install --custom <path>` (путь разрешается в абсолютный перед сохранением).

Файл загружается заново при каждом событии хука — кэширования нет. Подробности см. в [Пользовательские политики](/ru/custom-policies).

### Политики на основе соглашений

В дополнение к явному `customPoliciesPath`, failproofai автоматически обнаруживает и загружает файлы политик из директорий `.failproofai/policies/`:

| Уровень | Директория | Область |
|---------|-----------|--------|
| Project | `.failproofai/policies/` | Совместно используется командой через контроль версий |
| User | `~/.failproofai/policies/` | Личное, применяется ко всем проектам |

**Соответствие файлов:** Загружаются только файлы, соответствующие `*policies.{js,mjs,ts}` (например `security-policies.mjs`, `workflow-policies.js`). Другие файлы в директории игнорируются.

**Конфигурация не требуется:** Политики на основе соглашений не нуждаются в записях в `policies-config.json`. Просто поместите файлы в директорию, и они будут обнаружены при следующем событии хука.

**Объединённая загрузка:** Сканируются обе директории соглашений — проекта и пользователя. Все соответствующие файлы с обоих уровней загружаются (в отличие от `customPoliciesPath`, который использует принцип первого уровня побеждает).

Подробности и примеры см. в [Пользовательские политики](/ru/custom-policies).

### `llm`

Тип: `object` (опционально)

Конфигурация LLM-клиента для политик, выполняющих вызовы AI. Не требуется для большинства настроек.

```json
{
  "llm": {
    "model": "claude-sonnet-4-6",
    "apiKey": "sk-ant-..."
  }
}
```

## Управление конфигурацией из CLI

Команды `policies --install` и `policies --uninstall` записывают в файл настроек хуков вашего CLI агента (точки входа хуков), в то время как `policies-config.json` — это файл, который вы управляете напрямую. Это два отдельных файла:

- **Настройки CLI агента** — указывает агенту вызывать `failproofai --hook <event>` при каждом использовании инструмента:
  - **Claude Code**: `~/.claude/settings.json` (пользователь), `<cwd>/.claude/settings.json` (проект), `<cwd>/.claude/settings.local.json` (локально)
  - **OpenAI Codex**: `~/.codex/hooks.json` (пользователь), `<cwd>/.codex/hooks.json` (проект) — Codex не имеет локального уровня
  - **GitHub Copilot CLI _(beta)_**: `~/.copilot/hooks/failproofai.json` (пользователь), `<cwd>/.github/hooks/failproofai.json` (проект) — Copilot не имеет локального уровня. Записи хуков используют поля команд Copilot с разделением по ОС `bash`/`powershell` и `timeoutSec`; файл содержит маркер верхнего уровня `version: 1`. Поддержка Copilot CLI находится в **бета** версии, так как мы проверяем схему записей `events.jsonl` (не указана в публичной документации) против большего количества реальных сессий.
  - **Cursor Agent _(beta)_**: `~/.cursor/hooks.json` (пользователь), `<cwd>/.cursor/hooks.json` (проект) — Cursor не имеет локального уровня. Записи хуков используют формат, подобный Claude, `{type, command, timeout}` (без разделения `bash`/`powershell`), но хранятся под ключами событий в camelCase (`preToolUse`, `beforeSubmitPrompt`, …) в плоском массиве согласно [схеме хуков](https://cursor.com/docs/hooks) Cursor; файл содержит маркер верхнего уровня `version: 1`. Обработчик канонизирует camelCase → PascalCase через `CURSOR_EVENT_MAP`, поэтому существующие встроенные политики работают без изменений. Поддержка Cursor Agent находится в **бета** версии, так как мы проверяем формат транскрипта Cursor на диске (не указан в публичной документации) против большего количества реальных установок.
  - **OpenCode _(beta)_**: `~/.config/opencode/opencode.json` + `~/.config/opencode/plugins/failproofai.mjs` (пользователь), `<cwd>/.opencode/opencode.json` + `<cwd>/.opencode/plugins/failproofai.mjs` (проект) — OpenCode не имеет локального уровня. В отличие от других шести CLI, OpenCode **не имеет внешней системы хуков команд**: он загружает встроенные JS/TS плагины, явно зарегистрированные через массив `plugin: []` в `opencode.json` (автооткрытие из `.opencode/plugins/` **не** работает в opencode v1.14.33). При установке создаётся небольшой сгенерированный шим плагина, который вызывает бинарный файл failproofai через подпроцесс и преобразует JSON-ответ бинарного файла (формата Claude) обратно в семантику плагина: `throw new Error()` для запрета события инструмента (отменяет вызов инструмента), `client.session.prompt(...)` для instruct и для запрета `Stop` / `SubagentStop` (отправляет причину запрета как следующее сообщение пользователя — единственный канал принудительного повтора, так как `session.idle` только уведомляет и выброс исключения из неё ни к чему не приводит), и отсутствие операции для разрешения. Шим канонизирует имена инструментов (lowercase → PascalCase через `OPENCODE_TOOL_MAP`) и ключи аргументов входного инструмента (camelCase → snake_case через `OPENCODE_TOOL_INPUT_MAP` для `Read` / `Write` / `Edit`, например `filePath` → `file_path`, `oldString` → `old_string`) перед отправкой в бинарный файл, поэтому встроенные политики проверки путей, такие как `block-read-outside-cwd`, `block-env-files` и `block-secrets-write`, работают без изменений на вызовах инструментов OpenCode. Сессии хранятся в БД SQLite OpenCode по адресу `~/.local/share/opencode/opencode.db`; средство просмотра сессий панели управления читает их через `opencode db --format json` и `opencode export <id>`. Поддержка OpenCode находится в **бета** версии, так как мы проверяем поведение в разных версиях и против большего количества реальных сессий. См. [документацию плагинов OpenCode](https://opencode.ai/docs/plugins/).
  - **Pi _(beta)_**: `~/.pi/agent/settings.json` (пользователь), `<cwd>/.pi/settings.json` (проект) — Pi не имеет локального уровня. Pi загружает пакеты расширений TypeScript при запуске; файл настроек — это плоский массив строк `{"packages": ["./relative/path", …]}`. failproofai записывает единую запись массива пакетов, указывающую на встроенную директорию `pi-extension/`. Расширение внутри подписывается на события Pi `tool_call` / `user_bash` / `input` / `session_start` и вызывает `failproofai --hook <Event> --cli pi`; обработчик канонизирует snake_case_нижний → PascalCase через `PI_EVENT_MAP`, поэтому существующие встроенные политики работают без изменений. Аргументы входного инструмента также канонизируются через `PI_TOOL_INPUT_MAP` (Pi's Read / Write / Edit передают `path` вместо `file_path`; сопоставление верхнего ключа позволяет работать `block-env-files` и `block-secrets-write` — `block-read-outside-cwd` уже имела резервный `path`). Поддержка Pi находится в **бета** версии, так как API расширений и макет журнала сессий Pi стабилизируются.
  - **Gemini CLI _(beta)_**: `~/.gemini/settings.json` (пользователь), `<cwd>/.gemini/settings.json` (проект) — Gemini не имеет локального уровня (документирует уровень `system` по адресу `/etc/gemini-cli/settings.json`, который failproofai не предоставляет). Записи хуков используют формат Claude `{type, command, timeout}`, обёрнутый в схему сопоставителя Gemini `{matcher, hooks: [...]}` с `matcher: "*"` по умолчанию. События имеют формат PascalCase (`SessionStart`, `BeforeAgent`, `AfterAgent`, `BeforeModel`, `AfterModel`, `BeforeToolSelection`, `BeforeTool`, `AfterTool`, `PreCompress`, `Notification`, `SessionEnd`); обработчик отображает их на канонические имена Claude через `GEMINI_EVENT_MAP`. Имена инструментов имеют формат snake_case (`run_shell_command`, `read_file`, `write_file`, `replace`, …) — обработчик канонизирует их через `GEMINI_TOOL_MAP`, поэтому существующие встроенные политики работают без изменений. Средство оценки политики выдаёт плоский формат Gemini `{decision: "deny", reason}` (предпочитаемый согласно Правилу Золота Gemini с контрактом exit-0), `{hookSpecificOutput: {hookEventName, additionalContext}}` для внедрения контекста в BeforeAgent / AfterTool / SessionStart, и `{decision: "block", reason}` в AfterAgent для семантики принудительного повтора. Поддержка Gemini CLI находится в **бета** версии, так как мы расширяем покрытие реальных случаев. См. [документацию хуков Gemini CLI](https://geminicli.com/docs/hooks/).
- **`policies-config.json`** — указывает failproofai, какие политики оценивать и с какими параметрами (общее для всех CLI агентов)

Передайте `--cli claude|codex|copilot|cursor|opencode|pi|gemini` для выбора конкретного агента (через пробел или повторно для любого подмножества):

```bash
failproofai policies --install --cli codex --scope project
failproofai policies --install --cli copilot --scope project
failproofai policies --install --cli cursor --scope project
failproofai policies --install --cli opencode --scope project
failproofai policies --install --cli pi --scope project
failproofai policies --install --cli gemini --scope project
failproofai policies --install --cli claude codex copilot cursor opencode pi gemini
```

Когда `--cli` опущен, `failproofai` определяет, какие CLI агентов установлены (`which claude` / `which codex` / `which copilot` / `which cursor-agent` / `which opencode` / `which pi` / `which gemini`):

- **Один CLI обнаружен** — автоматически выбирает этот CLI без запроса.
- **Несколько CLI обнаружено** в интерактивном терминале — показывает подсказку с единственным выбором с помощью клавиш стрелок, сгруппированную в раздел `Обнаружено (N)` (с строкой совокупного установки для всех N обнаруженных и каждым обнаруженным CLI отдельно) и раздел `Не установлено (M) · установить хуки заранее`, содержащий каждый необнаруженный поддерживаемый CLI как опцию предварительной установки (↑↓ для перемещения, Enter для выбора, ^C для выхода). Поток удаления показывает только раздел Обнаружено.
- **Несколько CLI обнаружено** в неинтерактивном запуске (CI, без TTY) — устанавливает для всех обнаруженных CLI без запроса.
- **Ничего не обнаружено** — возвращается к `claude` с предупреждением, что в PATH не найден бинарный файл агента; команда хука всё равно записывается, поэтому она активируется сразу же после установки.

Вы можете редактировать `policies-config.json` напрямую в любое время; изменения вступают в силу немедленно при следующем событии хука без необходимости перезагрузки.

## Пример: конфигурация уровня проекта с командными значениями по умолчанию

Добавьте `.failproofai/policies-config.json` в ваш репозиторий:

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "block-env-files"
  ],
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "release", "hotfix"]
    }
  }
}
```

После этого каждый разработчик может создать `.failproofai/policies-config.local.json` (игнорируется .gitignore) для личных переопределений без влияния на товарищей.
