failproofai использует JSON файлы конфигурации для управления тем, какие политики активны, как они работают и откуда загружаются пользовательские политики. Конфигурация разработана так, чтобы её было легко делиться с командой — добавьте её в репозиторий, и каждый разработчик получит одинаковую защиту агента.

---

## Области видимости конфигурации

Существует три уровня конфигурации, оцениваемые по приоритету:

| Уровень | Путь файла | Назначение |
|---------|-----------|-----------|
| **project** | `.failproofai/policies-config.json` | Настройки для репозитория, фиксируются в контроле версий |
| **local** | `.failproofai/policies-config.local.json` | Личные переопределения для репозитория, исключены из git |
| **global** | `~/.failproofai/policies-config.json` | Параметры по умолчанию на уровне пользователя для всех проектов |

Когда failproofai получает событие хука, он загружает и объединяет все три файла, которые существуют для текущей рабочей директории.

### Правила слияния

**`enabledPolicies`** — объединение всех трёх уровней. Политика, включённая на любом уровне, является активной.

```text
project:  ["block-sudo"]
local:    ["block-rm-rf"]
global:   ["block-sudo", "sanitize-api-keys"]

resolved: ["block-sudo", "block-rm-rf", "sanitize-api-keys"]  ← дедублировано
```

**`policyParams`** — первый уровень, который определяет параметры для конкретной политики, побеждает полностью. Глубокого слияния значений внутри параметров политики не происходит.

```text
project:  block-sudo → { allowPatterns: ["sudo apt-get update"] }
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo apt-get update"] }   ← project побеждает, global игнорируется
```

```text
project:  (нет записи block-sudo)
local:    (нет записи block-sudo)
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo systemctl status"] }  ← переходит на global
```

**`customPoliciesPath`** — первая область видимости, определяющая это, побеждает.

**`llm`** — первая область видимости, определяющая это, побеждает.

## Формат конфиг-файла

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "sanitize-jwt",
    "block-env-files",
    "block-read-outside-cwd"
  ],
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    },
    "block-push-master": {
      "protectedBranches": ["main", "release", "prod"]
    },
    "block-rm-rf": {
      "allowPaths": ["/tmp"]
    },
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company"]
    },
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo API key" }
      ]
    },
    "warn-large-file-write": {
      "thresholdKb": 512
    }
  },
  "customPoliciesPath": "/home/alice/myproject/my-policies.js"
}
```

## Справочник полей

### `enabledPolicies`

Тип: `string[]`

Список имён политик для включения. Имена должны точно соответствовать идентификаторам политик, показанным командой `failproofai policies`. Полный список см. в [Встроенные политики](/ru/built-in-policies).

Политики, не указанные в `enabledPolicies`, неактивны, даже если у них есть записи в `policyParams`.

### `policyParams`

Тип: `Record<string, Record<string, unknown>>`

Переопределения параметров для каждой политики. Внешний ключ — это имя политики; внутренние ключи — это специфичные для политики параметры. Каждая политика документирует свои доступные параметры в разделе [Встроенные политики](/ru/built-in-policies).

Если политика имеет параметры, но вы их не указываете, используются встроенные значения политики по умолчанию. Пользователи, которые вообще не настраивают `policyParams`, получают поведение, идентичное предыдущим версиям.

Неизвестные ключи внутри блока параметров политики молча игнорируются при срабатывании хука, но отмечаются как предупреждения при выполнении `failproofai policies`.

#### `hint` (кросс-функциональный)

Тип: `string` (опционально)

Сообщение, добавляемое к причине, когда политика возвращает `deny` или `instruct`. Используйте его для передачи Claude практического руководства без изменения самой политики.

Работает с любым типом политик — встроенными, пользовательскими (`custom/`), проектными соглашениями (`.failproofai-project/`) или пользовательскими соглашениями (`.failproofai-user/`).

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "Попробуйте создать новую ветку вместо этого."
    },
    "block-sudo": {
      "allowPatterns": ["sudo apt-get"],
      "hint": "Используйте apt-get непосредственно без sudo."
    },
    "custom/my-policy": {
      "hint": "Сначала попросите одобрение у пользователя."
    }
  }
}
```

Когда `block-force-push` отклоняет запрос, Claude видит: *«Force-pushing заблокирован. Попробуйте создать новую ветку вместо этого.»*

Не-строковые значения и пустые строки молча игнорируются. Если `hint` не установлен, поведение не изменяется (обратная совместимость).

### `customPoliciesPath`

Тип: `string` (абсолютный путь)

Путь к JavaScript файлу, содержащему пользовательские политики хуков. Устанавливается автоматически командой `failproofai policies --install --custom <path>` (путь преобразуется в абсолютный перед сохранением).

Файл загружается заново при каждом событии хука — кеширования нет. Подробнее см. в разделе [Пользовательские политики](/ru/custom-policies).

### Политики, основанные на конвенциях

Помимо явного `customPoliciesPath`, failproofai автоматически обнаруживает и загружает файлы политик из директорий `.failproofai/policies/`:

| Уровень | Директория | Уровень |
|---------|-----------|---------|
| Проект | `.failproofai/policies/` | Общедоступно для команды через контроль версий |
| Пользователь | `~/.failproofai/policies/` | Личное, применяется ко всем проектам |

**Сопоставление файлов:** Загружаются только файлы, соответствующие `*policies.{js,mjs,ts}` (например `security-policies.mjs`, `workflow-policies.js`). Другие файлы в директории игнорируются.

**Конфигурация не требуется:** Политики соглашений не требуют записей в `policies-config.json`. Просто поместите файлы в директорию, и они будут загружены при следующем событии хука.

**Объединённая загрузка:** Сканируются обе директории соглашений — проектная и пользовательская. Все соответствующие файлы из обоих уровней загружаются (в отличие от `customPoliciesPath`, который использует принцип первый-уровень-побеждает).

Подробнее см. в разделе [Пользовательские политики](/ru/custom-policies).

### `llm`

Тип: `object` (опционально)

Конфигурация клиента LLM для политик, которые делают вызовы AI. Не требуется для большинства конфигураций.

```json
{
  "llm": {
    "model": "claude-sonnet-4-6",
    "apiKey": "sk-ant-..."
  }
}
```

## Управление конфигурацией из CLI

Команды `policies --install` и `policies --uninstall` записывают в файл настроек хуков вашего CLI агента (точки входа хуков), а `policies-config.json` — это файл, который вы управляете напрямую. Это два отдельных файла:

- **Настройки CLI агента** — сообщает агенту вызывать `failproofai --hook <event>` при каждом использовании инструмента:
  - **Claude Code**: `~/.claude/settings.json` (пользователь), `<cwd>/.claude/settings.json` (проект), `<cwd>/.claude/settings.local.json` (локальный)
  - **OpenAI Codex**: `~/.codex/hooks.json` (пользователь), `<cwd>/.codex/hooks.json` (проект) — у Codex нет локального уровня
  - **GitHub Copilot CLI _(бета)_**: `~/.copilot/hooks/failproofai.json` (пользователь), `<cwd>/.github/hooks/failproofai.json` (проект) — у Copilot нет локального уровня. Записи хуков используют поля команд OS `bash`/`powershell` Copilot с `timeoutSec`; файл содержит маркер верхнего уровня `version: 1`. Поддержка Copilot CLI находится в **бета** версии, пока мы проверяем схему записи `events.jsonl` (которая не указана в открытой документации) на реальных сессиях.
  - **Cursor Agent _(бета)_**: `~/.cursor/hooks.json` (пользователь), `<cwd>/.cursor/hooks.json` (проект) — у Cursor нет локального уровня. Записи хуков используют форму `{type, command, timeout}`, подобную Claude (без разделения `bash`/`powershell`), но хранятся под ключами событий в camelCase (`preToolUse`, `beforeSubmitPrompt`, …) в плоском массиве согласно [схеме хуков](https://cursor.com/docs/hooks) Cursor; файл содержит маркер верхнего уровня `version: 1`. Обработчик канонизирует camelCase → PascalCase через `CURSOR_EVENT_MAP`, поэтому существующие встроенные политики срабатывают без изменений. Поддержка Cursor Agent находится в **бета** версии, пока мы проверяем формат транскрипта Cursor на диске (не указан в открытой документации) на реальных установках.
  - **OpenCode _(бета)_**: `~/.config/opencode/opencode.json` + `~/.config/opencode/plugins/failproofai.mjs` (пользователь), `<cwd>/.opencode/opencode.json` + `<cwd>/.opencode/plugins/failproofai.mjs` (проект) — у OpenCode нет локального уровня. В отличие от других четырёх CLI, OpenCode **не имеет системы внешних команд для хуков**: он загружает встроенные JS/TS плагины, явно зарегистрированные через массив `plugin: []` в `opencode.json` (автообнаружение из `.opencode/plugins/` **не** является способом загрузки плагинов в opencode v1.14.33). При установке создаётся небольшой генерируемый заглушка плагина, которая вызывает failproofai бинарный файл через subprocess и преобразует JSON ответ бинарного файла Claude-формы обратно в семантику плагина (`throw new Error()` для deny, `client.session.prompt(...)` для instruct, no-op для allow). Сессии хранятся в SQLite БД opencode в `~/.local/share/opencode/opencode.db`; просмотрщик сессий панели управления читает их через `opencode db --format json` и `opencode export <id>`. Поддержка OpenCode находится в **бета** версии, пока мы проверяем поведение в разных версиях на большем количестве реальных сессий. См. [документацию плагинов OpenCode](https://opencode.ai/docs/plugins/).
  - **Pi _(бета)_**: `~/.pi/agent/settings.json` (пользователь), `<cwd>/.pi/settings.json` (проект) — у Pi нет локального уровня. Pi загружает пакеты расширений TypeScript при запуске; файл настроек — это плоский массив строк `{"packages": ["./relative/path", …]}`. failproofai записывает одну запись массива packages, указывающую на свою директорию `pi-extension/`. Расширение внутренне подписывается на события Pi `tool_call` / `user_bash` / `input` / `session_start` и вызывает `failproofai --hook <Event> --cli pi`; обработчик канонизирует underscore_lower_snake_case → PascalCase через `PI_EVENT_MAP`, поэтому существующие встроенные политики срабатывают без изменений. Поддержка Pi находится в **бета** версии, пока API расширений Pi и макет лога сессии стабилизируются.
  - **Gemini CLI _(бета)_**: `~/.gemini/settings.json` (пользователь), `<cwd>/.gemini/settings.json` (проект) — у Gemini нет локального уровня (он документирует уровень `system` в `/etc/gemini-cli/settings.json`, который failproofai не предоставляет). Записи хуков используют форму `{type, command, timeout}` Claude обёрнутую в схему matcher Gemini `{matcher, hooks: [...]}` с `matcher: "*"` по умолчанию. События имеют формат PascalCase (`SessionStart`, `BeforeAgent`, `AfterAgent`, `BeforeModel`, `AfterModel`, `BeforeToolSelection`, `BeforeTool`, `AfterTool`, `PreCompress`, `Notification`, `SessionEnd`); обработчик отображает на канонические имена Claude через `GEMINI_EVENT_MAP`. Имена инструментов в формате snake_case (`run_shell_command`, `read_file`, `write_file`, `replace`, …) — обработчик канонизирует через `GEMINI_TOOL_MAP`, поэтому существующие встроенные политики срабатывают без изменений. Оценщик политик выдаёт форму Gemini `{decision: "deny", reason}` (предпочтительно согласно "Golden Rule" контракту exit-0 Gemini), `{hookSpecificOutput: {hookEventName, additionalContext}}` для внедрения контекста на BeforeAgent / AfterTool / SessionStart и `{decision: "block", reason}` на AfterAgent для семантики force-retry. Поддержка Gemini CLI находится в **бета** версии, пока мы расширяем охват реальных случаев. См. [документацию хуков Gemini CLI](https://geminicli.com/docs/hooks/).
- **`policies-config.json`** — сообщает failproofai, какие политики оценивать и с какими параметрами (общие для всех CLI агентов)

Передайте `--cli claude|codex|copilot|cursor|opencode|pi|gemini` для нацеливания на конкретного агента (разделённые пробелом или повторённые для любого подмножества):

```bash
failproofai policies --install --cli codex --scope project
failproofai policies --install --cli copilot --scope project
failproofai policies --install --cli cursor --scope project
failproofai policies --install --cli opencode --scope project
failproofai policies --install --cli pi --scope project
failproofai policies --install --cli gemini --scope project
failproofai policies --install --cli claude codex copilot cursor opencode pi gemini
```

Когда `--cli` опущен, `failproofai` определяет, какие CLI агентов установлены (`which claude` / `which codex` / `which copilot` / `which cursor-agent` / `which opencode` / `which pi` / `which gemini`):

- **Обнаружен один CLI** — автоматически выбирает этот CLI без подтверждения.
- **Несколько CLI обнаружено** в интерактивном терминале — показывает выбор с одной стрелкой, сгруппированный в раздел `Detected (N)` (с совокупной строкой `Install for all N detected` + каждый обнаруженный CLI отдельно) и раздел `Not installed (M) · install hooks ahead of time`, перечисляющий каждый необнаруженный поддерживаемый CLI как опцию предварительной установки (↑↓ для перемещения, Enter для выбора, ^C для выхода). Поток удаления показывает только раздел Detected.
- **Несколько CLI обнаружено** в неинтерактивном запуске (CI, no TTY) — устанавливает для всех обнаруженных CLI без подтверждения.
- **Ни один не обнаружен** — переходит к `claude`, с предупреждением о том, что двоичный файл агента не найден в PATH; команда хука всё ещё записывается, поэтому она активируется, как только вы установите его.

Вы можете редактировать `policies-config.json` напрямую в любое время; изменения вступают в силу немедленно при следующем событии хука без необходимости перезагрузки.

## Пример: конфигурация уровня проекта со стандартами для команды

Добавьте `.failproofai/policies-config.json` в ваш репозиторий:

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "block-env-files"
  ],
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "release", "hotfix"]
    }
  }
}
```

Каждый разработчик может затем создать `.failproofai/policies-config.local.json` (исключён из git) для личных переопределений без влияния на коллег.
