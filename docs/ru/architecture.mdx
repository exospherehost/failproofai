Этот документ объясняет, как failproofai работает изнутри: как система хуков перехватывает вызовы инструментов агента, как загружается и объединяется конфигурация, как оцениваются политики и как дашборд отслеживает активность агента.

---

## Обзор

failproofai состоит из двух независимых подсистем:

1. **Обработчик хуков** — быстрый CLI-подпроцесс, который Claude Code вызывает при каждом вызове инструмента агентом. Оценивает политики и возвращает решение.
2. **Монитор агентов (Дашборд)** — веб-приложение на Next.js для мониторинга сессий агентов и управления политиками.

Обе подсистемы используют общие файлы конфигурации в `~/.failproofai/` и директории `.failproofai/` проекта, однако запускаются как отдельные процессы и взаимодействуют только через файловую систему.

## Обработчик хуков

### Интеграция с Claude Code

При запуске `failproofai policies --install` в файл `~/.claude/settings.json` записываются следующие записи:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "failproofai --hook PreToolUse"
          }
        ]
      }
    ],
    "PostToolUse": [ ... ]
  }
}
```

Claude Code затем вызывает `failproofai --hook PreToolUse` как подпроцесс перед каждым вызовом инструмента, передавая JSON-payload через stdin.

### Формат payload

```json
{
  "session_id": "abc123",
  "transcript_path": "/home/user/.claude/projects/myproject/sessions/abc123.jsonl",
  "cwd": "/home/user/myproject",
  "permission_mode": "default",
  "hook_event_name": "PreToolUse",
  "tool_name": "Bash",
  "tool_input": { "command": "sudo apt install nodejs" }
}
```

Для событий `PostToolUse` payload также содержит `tool_result` с выводом инструмента.

Обработчик ограничивает размер stdin до 1 МБ. Payload, превышающий этот лимит, отбрасывается, и все политики неявно разрешают операцию.

### Формат ответа

**Запрет (PreToolUse):**
```json
{
  "hookSpecificOutput": {
    "permissionDecision": "deny",
    "permissionDecisionReason": "Blocked by failproofai: sudo command blocked"
  }
}
```

**Запрет (PostToolUse):**
```json
{
  "hookSpecificOutput": {
    "additionalContext": "Blocked by failproofai because: API key detected in output"
  }
}
```

**Инструкция (любое событие кроме Stop):**
```json
{
  "hookSpecificOutput": {
    "additionalContext": "Instruction from failproofai: Verify tests pass before committing."
  }
}
```

**Инструкция для события Stop:**
- Код выхода: `2`
- Причина записывается в stderr (не в stdout)

**Разрешение:**
- Код выхода: `0`
- Пустой stdout

**Разрешение с сообщением (бета):**

Начиная с v0.0.2-beta.3, `allow(message)` позволяет политике передавать информационный контекст обратно в Claude даже при разрешении операции. Обработчик хуков записывает следующий JSON в **stdout** (не в файл конфигурации — это ответ обработчика для Claude Code, аналогично ответам deny и instruct):

```json
// Written to stdout by the hook handler process
{
  "hookSpecificOutput": {
    "additionalContext": "All CI checks passed on branch 'feat/my-feature'."
  }
}
```
- Код выхода: `0` (операция разрешена)
- Если несколько политик возвращают `allow` с сообщением, их сообщения объединяются через символ новой строки в единую строку `additionalContext`
- Если ни одна политика не предоставляет сообщение, stdout остаётся пустым (как прежде)

### Конвейер обработки

`src/hooks/handler.ts` реализует полный конвейер:

```text
stdin JSON
  → разбор payload (макс. 1 МБ)
  → извлечение метаданных сессии (session_id, cwd, tool_name, tool_input и др.)
  → readMergedHooksConfig(cwd)    ← объединяет конфиг проекта + локальный + глобальный
  → регистрация включённых встроенных политик с разрешёнными параметрами
  → загрузка пользовательских политик из customPoliciesPath (если задан)
  → регистрация пользовательских политик в реестре политик
  → оценка всех политик (сначала встроенные, затем пользовательские)
      → первый deny прерывает цепочку
      → решения instruct накапливаются
      → сообщения allow накапливаются
  → запись JSON-решения в stdout
  → сохранение события в ~/.failproofai/hook-activity.jsonl
  → выход
```

Весь процесс выполняется менее чем за 100 мс для типичных payload без обращений к LLM.

## Загрузка конфигурации

`src/hooks/hooks-config.ts` реализует загрузку конфигурации с тремя уровнями приоритета.

```text
[1] {cwd}/.failproofai/policies-config.json        ← проект   (наивысший приоритет)
[2] {cwd}/.failproofai/policies-config.local.json  ← локальный
[3] ~/.failproofai/policies-config.json             ← глобальный (наименьший приоритет)
```

Логика объединения:
- `enabledPolicies` — дедуплицированное объединение из всех трёх файлов
- `policyParams` — ключ на уровне политики; побеждает первый файл, в котором он определён
- `customPoliciesPath` — побеждает первый файл, в котором он определён
- `llm` — побеждает первый файл, в котором он определён

Веб-дашборд использует `readHooksConfig()` (только глобальный уровень) для чтения и записи, поскольку не вызывается с рабочей директорией проекта.

## Оценка политик

`src/hooks/policy-evaluator.ts` выполняет политики по порядку.

Для каждой политики:

1. Поиск схемы `params` политики (если она есть).
2. Чтение `policyParams[policy.name]` из объединённой конфигурации.
3. Наложение пользовательских значений поверх значений по умолчанию из схемы для формирования `ctx.params`.
4. Вызов `policy.fn(ctx)` с разрешённым контекстом.
5. Если результат — `deny`, немедленно остановить и вернуть это решение.
6. Если результат — `instruct`, накопить сообщение и продолжить.
7. Если результат — `allow`, перейти к следующей политике.

После выполнения всех политик:
- Если был возвращён `deny`, сформировать ответ с запретом.
- Если были собраны ответы `instruct`, сформировать единый ответ instruct со всеми объединёнными сообщениями.
- В противном случае сформировать ответ с разрешением (пустой stdout, код выхода 0).

## Встроенные политики

`src/hooks/builtin-policies.ts` определяет все 26 встроенных политик в виде объектов `BuiltinPolicyDefinition`:

```typescript
interface BuiltinPolicyDefinition {
  name: string;
  description: string;
  fn: (ctx: PolicyContext) => PolicyResult;
  match: {
    events: HookEventType[];
    tools?: string[];
  };
  defaultEnabled: boolean;
  category: string;
  beta?: boolean;
  params?: PolicyParamsSchema;
}
```

Политики, принимающие `params`, объявляют `PolicyParamsSchema` с типами и значениями по умолчанию для каждого параметра. Оценщик политик внедряет разрешённые значения в `ctx.params` перед вызовом `fn`. Функции политик читают `ctx.params` без null-проверок, поскольку значения по умолчанию всегда применяются заранее.

Сопоставление шаблонов внутри политик использует разобранные токены команды (argv), а не сопоставление сырых строк. Это предотвращает обход через внедрение shell-операторов (например, шаблон для `sudo systemctl status *` нельзя обойти, добавив `; rm -rf /` к команде).

## Пользовательские политики

`src/hooks/custom-hooks-registry.ts` реализует реестр на основе `globalThis`:

```typescript
const REGISTRY_KEY = "__failproofai_custom_hooks__";

export const customPolicies = {
  add(hook: CustomHook): void { ... }
};

export function getCustomHooks(): CustomHook[] { ... }
export function clearCustomHooks(): void { ... }  // used in tests
```

`src/hooks/custom-hooks-loader.ts` загружает пользовательский файл политик:

1. Читает `customPoliciesPath` из конфигурации; пропускает, если отсутствует.
2. Преобразует в абсолютный путь; проверяет существование файла.
3. Переписывает все импорты `from "failproofai"` на реальный путь dist, чтобы `customPolicies` указывал на тот же реестр `globalThis`.
4. Рекурсивно переписывает транзитивные локальные импорты для обеспечения совместимости с ESM.
5. Записывает временные `.mjs`-файлы и загружает точку входа через `import()`.
6. Вызывает `getCustomHooks()` для получения зарегистрированных хуков.
7. Удаляет все временные файлы в блоке `finally`.

При любой ошибке (файл не найден, синтаксическая ошибка, ошибка импорта) ошибка записывается в `~/.failproofai/hook.log`, и загрузчик возвращает пустой массив. Встроенные политики не затрагиваются.

Пользовательские политики оцениваются после всех встроенных. Решение `deny` от пользовательской политики всё равно прерывает выполнение последующих пользовательских политик (однако к этому моменту все встроенные политики уже выполнены).

## Журналирование активности

После каждого события хука обработчик добавляет строку JSONL в `~/.failproofai/hook-activity.jsonl`:

```json
{
  "timestamp": "2026-04-06T12:34:56.789Z",
  "sessionId": "abc123",
  "eventType": "PreToolUse",
  "toolName": "Bash",
  "policyName": "block-sudo",
  "decision": "deny",
  "reason": "sudo command blocked by failproofai",
  "durationMs": 12
}
```

По одной строке на каждую политику, принявшую решение, отличное от allow. Решения allow не записываются (чтобы файл оставался компактным).

## Архитектура дашборда

Дашборд — это приложение на **Next.js 16**, использующее App Router с React Server Components и Server Actions.

```text
app/
  layout.tsx                  ← Корневой макет (тема, телеметрия, навигация)
  projects/page.tsx           ← Серверный компонент: список всех проектов Claude
  project/[name]/page.tsx     ← Серверный компонент: список сессий в проекте
  project/[name]/session/
    [sessionId]/page.tsx      ← Серверный компонент: просмотрщик сессии
  policies/page.tsx           ← Клиентский компонент: управление политиками + журнал активности
  actions/
    get-hooks-config.ts       ← Чтение конфигурации + списка политик
    update-hooks-config.ts    ← Включение/отключение политики
    update-policy-params.ts   ← Обновление параметров политики
    get-hook-activity.ts      ← Постраничный вывод/поиск по журналу активности
    install-hooks-web.ts      ← Установка/удаление хуков из браузера
  api/
    download/[project]/[session]/route.ts   ← Экспорт сессии в ZIP/JSONL
```

**Поток данных:**

- Компоненты страниц вызывают `lib/projects.ts` и `lib/log-entries.ts` для чтения данных проектов/сессий непосредственно из файловой системы (без API-слоя для чтения).
- Страница политик использует Server Actions для всех мутаций (переключение, обновление параметров, установка/удаление).
- Просмотрщик сессий разбирает формат JSONL-транскриптов Claude и отображает хронологию сообщений и вызовов инструментов.

**Ключевые архитектурные решения:**

- Без базы данных — всё персистентное состояние хранится в обычных файлах (`~/.failproofai/`, `~/.claude/projects/`).
- Server Actions для мутаций — REST API для операций CRUD не требуется.
- React Server Components для страниц чтения — более быстрая начальная загрузка, отсутствие клиентского бандла для получения данных.
- Клиентские компоненты только там, где требуется интерактивность (переключатели политик, поиск по активности, просмотр журнала).

## Структура файлов

```text
failproofai/
├── bin/
│   └── failproofai.mjs           # CLI-роутер (hook / dashboard / install и др.)
├── src/hooks/
│   ├── handler.ts                # Конвейер обработки событий хуков
│   ├── builtin-policies.ts       # 26 определений политик
│   ├── policy-evaluator.ts       # Движок выполнения политик
│   ├── policy-registry.ts        # Регистрация и поиск политик
│   ├── policy-types.ts           # TypeScript-интерфейсы
│   ├── hooks-config.ts           # Загрузка конфигурации с несколькими уровнями
│   ├── custom-hooks-registry.ts  # Реестр хуков на основе globalThis
│   ├── custom-hooks-loader.ts    # ESM-загрузчик для пользовательских JS-хуков
│   ├── manager.ts                # Операции установки / удаления / перечисления
│   ├── install-prompt.ts         # Интерактивный запрос выбора политик
│   ├── hook-logger.ts            # Запись журнала в hook.log
│   ├── hook-activity-store.ts    # Сохранение активности в hook-activity.jsonl
│   └── llm-client.ts             # LLM API-клиент (для политик на основе ИИ)
├── app/                          # Дашборд Next.js (страницы + server actions)
├── lib/                          # Общие утилиты
│   ├── projects.ts               # Перечисление проектов Claude из файловой системы
│   ├── log-entries.ts            # Разбор формата JSONL-транскриптов Claude
│   ├── paths.ts                  # Разрешение системных путей
│   └── ...
├── components/                   # Общие UI-компоненты React
├── contexts/                     # Провайдеры контекста React (тема, автообновление, телеметрия)
├── examples/                     # Примеры файлов пользовательских хуков
└── __tests__/                    # Модульные и E2E-тесты
```
