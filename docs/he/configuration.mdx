failproofai משתמש בקובצי הגדרות JSON כדי להשלוט אילו מדיניות פעילות, כיצד הן מתנהגות ומאיפה מדיניות מותאמת אישית נטענת. התצורה עוצבה להיות קלה לשיתוף עם הצוות שלך - התחייב אותה למאגר שלך וכל מפתח יקבל את אותה רשת בטיחות סוכן.

---

## היקפי תצורה

קיימים שלושה היקפי תצורה, המוערכים לפי סדר עדיפויות:

| היקף | נתיב קובץ | תכלית |
|-------|-----------|---------|
| **פרויקט** | `.failproofai/policies-config.json` | הגדרות לכל מאגר, מחויבות בשליטה בגרסה |
| **מקומי** | `.failproofai/policies-config.local.json` | עקיפות אישיות לכל מאגר, מתעלמות מ-git |
| **גלובלי** | `~/.failproofai/policies-config.json` | ברירות מחדל ברמת משתמש בכל פרויקטים |

כאשר failproofai מקבל אירוע hook, הוא טוען וממזג את כל שלושת הקובצים שקיימים בתיקייה עבודה הנוכחית.

### כללי מיזוג

**`enabledPolicies`** - התאחדות של כל שלושת ההיקפים. מדיניות שמופעלת בכל רמה פעילה.

```text
project:  ["block-sudo"]
local:    ["block-rm-rf"]
global:   ["block-sudo", "sanitize-api-keys"]

resolved: ["block-sudo", "block-rm-rf", "sanitize-api-keys"]  ← התאחדות ללא כפילויות
```

**`policyParams`** - ההיקף הראשון המגדיר פרמטרים למדיניות נתונה מנצח לחלוטין. אין מיזוג עמוק של ערכים בתוך פרמטרים של מדיניות.

```text
project:  block-sudo → { allowPatterns: ["sudo apt-get update"] }
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo apt-get update"] }   ← הפרויקט מנצח, הגלובלי מתעלם
```

```text
project:  (אין ערך block-sudo)
local:    (אין ערך block-sudo)
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo systemctl status"] }  ← נופל דרך לגלובלי
```

**`customPoliciesPath`** - ההיקף הראשון המגדיר אותו מנצח.

**`llm`** - ההיקף הראשון המגדיר אותו מנצח.

## תבנית קובץ הגדרות

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "sanitize-jwt",
    "block-env-files",
    "block-read-outside-cwd"
  ],
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    },
    "block-push-master": {
      "protectedBranches": ["main", "release", "prod"]
    },
    "block-rm-rf": {
      "allowPaths": ["/tmp"]
    },
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company"]
    },
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo API key" }
      ]
    },
    "warn-large-file-write": {
      "thresholdKb": 512
    }
  },
  "customPoliciesPath": "/home/alice/myproject/my-policies.js"
}
```

## הפניה שדות

### `enabledPolicies`

סוג: `string[]`

רשימת שמות מדיניות להפעלה. שמות חייבים להתאים בדיוק למזהי מדיניות המוצגים על ידי `failproofai policies`. ראה [מדיניות מובנות](/he/built-in-policies) לרשימה המלאה.

מדיניות שלא ב-`enabledPolicies` לא פעילה, גם אם יש להן ערכים ב-`policyParams`.

### `policyParams`

סוג: `Record<string, Record<string, unknown>>`

עקיפות פרמטרים לכל מדיניות. המפתח החיצוני הוא שם המדיניות; המפתחות הפנימיים הם ספציפיים למדיניות. כל מדיניות תעד את הפרמטרים הזמינים שלה ב[מדיניות מובנות](/he/built-in-policies).

אם למדיניות יש פרמטרים אך אתה לא מצביע עליהם, משתמשים בברירות המחדל המובנות של המדיניות. משתמשים שלא מגדירים `policyParams` כלל יקבלו התנהגות זהה לגרסאות קודמות.

מפתחות לא ידועים בתוך בלוק הפרמטרים של מדיניות מתעלמים בשקט בזמן הפעלת hook אך מסומנים כאזהרות כאשר אתה מריץ `failproofai policies`.

#### `hint` (חוצה)

סוג: `string` (אופציונלי)

הודעה המוסיפה לסיבה כאשר מדיניות מחזירה `deny` או `instruct`. השתמש בה כדי לתת Claude הוראות פעולה בלי לשנות את המדיניות עצמה.

עובד עם כל סוג מדיניות — מובנה, מותאם אישית (`custom/`), קונוונציה של פרויקט (`.failproofai-project/`), או קונוונציה של משתמש (`.failproofai-user/`).

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "נסה ליצור ענף טרי במקום זאת."
    },
    "block-sudo": {
      "allowPatterns": ["sudo apt-get"],
      "hint": "השתמש ב-apt-get ישירות ללא sudo."
    },
    "custom/my-policy": {
      "hint": "בקש אישור מהמשתמש תחילה."
    }
  }
}
```

כאשר `block-force-push` מורה, Claude רואה: *"Force-pushing חסום. נסה ליצור ענף טרי במקום זאת."*

ערכים שאינם מחרוזת ומחרוזות ריקות מתעלמים בשקט. אם `hint` לא מוגדר, התנהגות זהה (תאימות לאחור).

### `customPoliciesPath`

סוג: `string` (נתיב מוחלט)

נתיב לקובץ JavaScript המכיל מדיניות hook מותאמת אישית. זה מוגדר באופן אוטומטי על ידי `failproofai policies --install --custom <path>` (הנתיב מופתת למוחלט לפני שנשמר).

הקובץ נטען מחדש בכל אירוע hook - אין caching. ראה [מדיניות מותאמת אישית](/he/custom-policies) לפרטי תأليף.

### מדיניות מבוססות קונוונציה

בנוסף ל-`customPoliciesPath` המפורש, failproofai גוגל אוטומטית וטוען קובצי מדיניות מתיקיות `.failproofai/policies/`:

| רמה | תיקייה | היקף |
|-------|-----------|-------|
| פרויקט | `.failproofai/policies/` | משותף לצוות דרך שליטה בגרסה |
| משתמש | `~/.failproofai/policies/` | אישי, חל על כל פרויקטים |

**תאימת קובץ:** רק קובצים המתאימים ל-`*policies.{js,mjs,ts}` נטענים (למשל `security-policies.mjs`, `workflow-policies.js`). קובצים אחרים בתיקייה מתעלמים.

**אין צורך בהגדרות:** מדיניות קונוונציה אינה דורשת ערכים ב-`policies-config.json`. פשוט שם קובצים בתיקייה והם נבחרים בהאירוע hook הבא.

**טעינת התאחדות:** גם תיקיות קונוונציה של פרויקט וגם משתמש נסרקות. כל קובצים תאמים משתי הרמות נטענים (בשונה מ-`customPoliciesPath` אשר משתמש במנצח ראשון-היקף).

ראה [מדיניות מותאמת אישית](/he/custom-policies) לפרטים נוספים ודוגמאות.

### `llm`

סוג: `object` (אופציונלי)

הגדרות לקוח LLM למדיניות אשר מבצעות קריאות AI. לא נדרש ברוב ההגדרות.

```json
{
  "llm": {
    "model": "claude-sonnet-4-6",
    "apiKey": "sk-ant-..."
  }
}
```

## ניהול תצורה מ-CLI

הפקודות `policies --install` ו-`policies --uninstall` כותבות לקובץ הגדרות hook של ה-CLI של הסוכן (נקודות כניסה hook), בעוד `policies-config.json` הוא הקובץ שאתה מנהל ישירות. השניים נפרדים:

- **הגדרות סוכן CLI** — אומר לסוכן להתקשר ל-`failproofai --hook <event>` בכל שימוש בכלי:
  - **Claude Code**: `~/.claude/settings.json` (משתמש), `<cwd>/.claude/settings.json` (פרויקט), `<cwd>/.claude/settings.local.json` (מקומי)
  - **OpenAI Codex**: `~/.codex/hooks.json` (משתמש), `<cwd>/.codex/hooks.json` (פרויקט) — Codex אין לו היקף `local`
  - **GitHub Copilot CLI _(beta)_**: `~/.copilot/hooks/failproofai.json` (משתמש), `<cwd>/.github/hooks/failproofai.json` (פרויקט) — Copilot אין לו היקף `local`. ערכי Hook משתמשים בשדות הפקודה `bash`/`powershell` של Copilot עם `timeoutSec`; הקובץ נושא סימן `version: 1` ברמה העליונה. תמיכת CLI של Copilot היא **beta** בזמן שאנו מאמתים את סכמת רשומות `events.jsonl` (אשר הדוקס הציבורית לא מציינת) מול יותר הפעלות בעולם האמיתי.
  - **Cursor Agent _(beta)_**: `~/.cursor/hooks.json` (משתמש), `<cwd>/.cursor/hooks.json` (פרויקט) — Cursor אין לו היקף `local`. ערכי Hook משתמשים בצורה `{type, command, timeout}` בצורת Claude (אין חלוקת `bash`/`powershell`), אך מאוחסנים תחת מפתחות אירוע camelCase (`preToolUse`, `beforeSubmitPrompt`, …) במערך שטוח לפי [סכמת hooks של Cursor](https://cursor.com/docs/hooks); הקובץ נושא סימן `version: 1` ברמה העליונה. המטפל ממנה camelCase → PascalCase דרך `CURSOR_EVENT_MAP` כך שמדיניות builtin קיימות נשרפות ללא שינוי. תמיכת Cursor Agent היא **beta** בזמן שאנו מאמתים תמלול של Cursor על-דיסק (לא מוצא בדוקס הציבורית) מול יותר התקנות בעולם האמיתי.
  - **OpenCode _(beta)_**: `~/.config/opencode/opencode.json` + `~/.config/opencode/plugins/failproofai.mjs` (משתמש), `<cwd>/.opencode/opencode.json` + `<cwd>/.opencode/plugins/failproofai.mjs` (פרויקט) — OpenCode אין לו היקף `local`. בשונה משש CLI האחרים, OpenCode **אין לו מערכת hook פקודה חיצונית**: הוא טוען ישירות פלאגינים JS/TS רשומים באופן מפורש דרך מערך `plugin: []` ב-`opencode.json` (auto-discovery מ-`.opencode/plugins/` **אינו** הדרך שפלאגינים נטענים ב-opencode v1.14.33). Install שם shim plugin קטן שנוצר אשר מתקשר בתהליך משנה לבינארי failproofai ותורגם את התגובה JSON בצורת Claude של הבינארי חזרה לסמנטיקה של פלאגין: `throw new Error()` עבור tool-event deny (מבטל את קריאת הכלי), `client.session.prompt(...)` עבור instruct וגם `Stop` / `SubagentStop` deny (משדר את סיבת ה-deny כהודעת המשתמש הבאה — הערוץ החד-כיווני היחיד מאז `session.idle` הוא notification-only והשלכה ממנה היא no-op), וגם no-op עבור allow. ה-Shim ממנה גם שמות כלים (lowercase → PascalCase דרך `OPENCODE_TOOL_MAP`) וגם מפתחות arg input של כלים (camelCase → snake_case דרך `OPENCODE_TOOL_INPUT_MAP` ל-`Read` / `Write` / `Edit`, למשל `filePath` → `file_path`, `oldString` → `old_string`) לפני העברה לבינארי, כך שמדיניות בדיקת נתיבים מובנות כמו `block-read-outside-cwd`, `block-env-files`, וגם `block-secrets-write` נשרפות ללא שינוי בקריאות כלים של OpenCode. הפעלות גרות בבסיס נתונים SQLite של opencode ב-`~/.local/share/opencode/opencode.db`; מסך הבקרה של תצוגת הפעלה קורא אותן דרך `opencode db --format json` ו-`opencode export <id>`. תמיכת OpenCode היא **beta** בזמן שאנו מאמתים התנהגות בין גרסאות ומול יותר הפעלות בעולם האמיתי. ראה [דוקס פלאגינים של OpenCode](https://opencode.ai/docs/plugins/).
  - **Pi _(beta)_**: `~/.pi/agent/settings.json` (משתמש), `<cwd>/.pi/settings.json` (פרויקט) — Pi אין לו היקף `local`. Pi טוען חבילות הרחבה TypeScript בזמן יזימה; קובץ הגדרות הוא מערך מחרוזות שטוח `{"packages": ["./relative/path", …]}`. failproofai כותב ערך מערך חבילות יחיד המצביע על ספריית `pi-extension/` הכרוכה שלו. ההרחבה באופן פנימי מנויה לאירועי `tool_call` / `user_bash` / `input` / `session_start` של Pi וקולפת ל-`failproofai --hook <Event> --cli pi`; המטפל ממנה underscore_lower_snake_case → PascalCase דרך `PI_EVENT_MAP` כך שמדיניות builtin קיימות נשרפות ללא שינוי. ארגומנטים של input לכלים ממנים גם דרך `PI_TOOL_INPUT_MAP` (Read / Write / Edit של Pi משדרות `path` במקום `file_path`; מיפוי המפתח הרמה העליונה מאפשר `block-env-files` ו-`block-secrets-write` נשרפות — `block-read-outside-cwd` כבר היה fallback `path`). תמיכת Pi היא **beta** בזמן שה-API של הרחבה של Pi וסכמת session-log מייצבות.
  - **Gemini CLI _(beta)_**: `~/.gemini/settings.json` (משתמש), `<cwd>/.gemini/settings.json` (פרויקט) — Gemini אין לו היקף `local` (הוא תעד היקף `system` ב-`/etc/gemini-cli/settings.json` אשר failproofai לא חושף). ערכי Hook משתמשים בצורה `{type, command, timeout}` של Claude עטופה בסכמת matcher של Gemini `{matcher, hooks: [...]}` עם `matcher: "*"` כברירת מחדל. אירועים הם PascalCase (`SessionStart`, `BeforeAgent`, `AfterAgent`, `BeforeModel`, `AfterModel`, `BeforeToolSelection`, `BeforeTool`, `AfterTool`, `PreCompress`, `Notification`, `SessionEnd`); המטפל ממזג לשמות canonical של Claude דרך `GEMINI_EVENT_MAP`. שמות כלים הם snake_case (`run_shell_command`, `read_file`, `write_file`, `replace`, …) — המטפל ממנה דרך `GEMINI_TOOL_MAP` כך שמדיניות builtin קיימות נשרפות ללא שינוי. מעריך המדיניות משדר צורה שטוחה של Gemini `{decision: "deny", reason}` (מועדף לפי ה-Golden Rule של Gemini exit-0 חוזה), `{hookSpecificOutput: {hookEventName, additionalContext}}` עבור הזרקת הקשר ב-BeforeAgent / AfterTool / SessionStart, וגם `{decision: "block", reason}` על AfterAgent עבור סמנטיקה force-retry. תמיכת Gemini CLI היא **beta** בזמן שאנו מרחיבים כיסוי בעולם האמיתי. ראה [דוקס hooks של Gemini CLI](https://geminicli.com/docs/hooks/).
- **`policies-config.json`** — אומר ל-failproofai אילו מדיניות להעריך וכן מה פרמטרים (משותף בכל agent CLIs)

העבר `--cli claude|codex|copilot|cursor|opencode|pi|gemini` לממוצע סוכן ספציפי (רווח-מופרד או חזר לכל תת-קבוצה):

```bash
failproofai policies --install --cli codex --scope project
failproofai policies --install --cli copilot --scope project
failproofai policies --install --cli cursor --scope project
failproofai policies --install --cli opencode --scope project
failproofai policies --install --cli pi --scope project
failproofai policies --install --cli gemini --scope project
failproofai policies --install --cli claude codex copilot cursor opencode pi gemini
```

כאשר `--cli` מותער, `failproofai` מגלה אילו agent CLIs מותקנים (`which claude` / `which codex` / `which copilot` / `which cursor-agent` / `which opencode` / `which pi` / `which gemini`):

- **CLI יחיד מגולה** — בוחר באופן אוטומטי את ה-CLI הזה ללא הנחיה.
- **CLIs מרובים מגולים** בטרמינל אינטראקטיבי — מציג בחירה יחידה single-select prompt עם מקשי חץ מקובצת לסעיף `Detected (N)` (עם שורה מצטברת `Install for all N detected` + כל CLI מגולה בנפרד) וסעיף `Not installed (M) · install hooks ahead of time` המפרטת כל CLI לא מגולה כתמיכה כאפשרות forward-install (↑↓ להעברה, Enter לבחירה, ^C לחדל). זרימת ההסרה מציגה רק את סעיף Detected.
- **CLIs מרובים מגולים** בריצה לא אינטראקטיבית (CI, אין TTY) — מתקינות לכל CLIs מגולים ללא הנחיה.
- **אף אחד לא מגולה** — חוזר ל-`claude`, עם אזהרה שלא נמצא בינארי סוכן ב-PATH; פקודת hook עדיין כתובה כך היא מופעלת ברגע שאתה מתקין אחד.

אתה יכול לערוך `policies-config.json` ישירות בכל עת; שינויים נכנסים לתוקף מיד בהאירוע hook הבא ללא צורך בהפעלה מחדש.

## דוגמה: תצורת רמת-פרויקט עם ברירות מחדל של צוות

התחייב `.failproofai/policies-config.json` למאגר שלך:

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "block-env-files"
  ],
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "release", "hotfix"]
    }
  }
}
```

כל מפתח יכול אז ליצור `.failproofai/policies-config.local.json` (מתעלם מ-git) לעקיפות אישיות ללא השפעה על חברי צוות.
