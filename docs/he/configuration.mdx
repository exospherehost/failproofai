failproofai משתמש בקובצי תצורה JSON כדי לשלוט באילו מדיניויות פעילות, כיצד הן מתנהגות, ומאיפה מדיניויות מותאמות אישית נטענות. התצורה מעוצבת כדי להיות קלה לשיתוף עם הצוות שלך - בצור אותה למחסן הגרסאות שלך וכל מפתח יקבל את אותה רשת ביטחון של agents.

---

## היקפי תצורה

יש שלוש היקפי תצורה, המוערכים בסדר עדיפויות:

| היקף | נתיב קובץ | מטרה |
|-------|-----------|---------|
| **project** | `.failproofai/policies-config.json` | הגדרות לפי-מאגר, משוקללות לשליטת גרסאות |
| **local** | `.failproofai/policies-config.local.json` | עקיפות אישיות לפי-מאגר, מתוך gitignore |
| **global** | `~/.failproofai/policies-config.json` | ברירות מחדל ברמת משתמש בכל הפרויקטים |

כשsailproofai מקבל אירוע hook, היא טוענת ומזגת את כל שלוש הקבצים שקיימים לתיקיית העבודה הנוכחית.

### כללי מיזוג

**`enabledPolicies`** - האיחוד של כל שלושת ההיקפים. מדיניות שמופעלת בכל רמה פעילה.

```text
project:  ["block-sudo"]
local:    ["block-rm-rf"]
global:   ["block-sudo", "sanitize-api-keys"]

resolved: ["block-sudo", "block-rm-rf", "sanitize-api-keys"]  ← איחוד ללא כפילויות
```

**`policyParams`** - היקף ראשון המגדיר פרמטרים למדיניות מסוימת מנצח לחלוטין. אין מיזוג עמוק של ערכים בתוך פרמטרים של מדיניות.

```text
project:  block-sudo → { allowPatterns: ["sudo apt-get update"] }
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo apt-get update"] }   ← project מנצח, global מתעלם
```

```text
project:  (no block-sudo entry)
local:    (no block-sudo entry)
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo systemctl status"] }  ← נופל דרך ל-global
```

**`customPoliciesPath`** - היקף ראשון המגדיר אותו מנצח.

**`llm`** - היקף ראשון המגדיר אותו מנצח.

## פורמט קובץ תצורה

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "sanitize-jwt",
    "block-env-files",
    "block-read-outside-cwd"
  ],
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    },
    "block-push-master": {
      "protectedBranches": ["main", "release", "prod"]
    },
    "block-rm-rf": {
      "allowPaths": ["/tmp"]
    },
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company"]
    },
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo API key" }
      ]
    },
    "warn-large-file-write": {
      "thresholdKb": 512
    }
  },
  "customPoliciesPath": "/home/alice/myproject/my-policies.js"
}
```

## הפניה לשדות

### `enabledPolicies`

Type: `string[]`

רשימה של שמות מדיניויות להפעלה. שמות חייבים להתאים בדיוק למזהי המדיניויות שמוצגים על ידי `failproofai policies`. ראה [Built-in Policies](/he/built-in-policies) לקבלת הרשימה המלאה.

מדיניויות שאינן ב-`enabledPolicies` אינן פעילות, גם אם יש להן ערכים ב-`policyParams`.

### `policyParams`

Type: `Record<string, Record<string, unknown>>`

עקיפות פרמטרים לפי-מדיניות. המפתח החיצוני הוא שם המדיניות; המפתחות הפנימיים ספציפיים למדיניות. כל מדיניות מתעדת את הפרמטרים הזמינים שלה ב-[Built-in Policies](/he/built-in-policies).

אם למדיניות יש פרמטרים אך אתה לא מציין אותם, משתמשים בברירות המחדל המובנות של המדיניות. משתמשים שלא מתצרפים את `policyParams` בכלל מקבלים התנהגות זהה לגרסאות קודמות.

מפתחות לא ידועים בתוך בלוק הפרמטרים של מדיניות מתעלמים בשקט בזמן כיבוי-hook אך מסומנים כאזהרות כשאתה מפעיל `failproofai policies`.

#### `hint` (חוצה-קטעי)

Type: `string` (optional)

הודעה שצורפת לסיבה כשמדיניות מחזירה `deny` או `instruct`. השתמש בה כדי לתת ל-Claude הנחיות פעולה ללא שינוי המדיניות עצמה.

עובד עם כל סוג מדיניות — מובנה, מותאם אישית (`custom/`), כנסיה פרויקט (`.failproofai-project/`), או כנסיה משתמש (`.failproofai-user/`).

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "Try creating a fresh branch instead."
    },
    "block-sudo": {
      "allowPatterns": ["sudo apt-get"],
      "hint": "Use apt-get directly without sudo."
    },
    "custom/my-policy": {
      "hint": "Ask the user for approval first."
    }
  }
}
```

כשblock-force-push מוכח, Claude רואה: *"Force-pushing is blocked. Try creating a fresh branch instead."*

ערכים שאינם מחרוזת ומחרוזות ריקות מתעלמות בשקט. אם `hint` אינו מוגדר, ההתנהגות לא משתנה (תואמת לאחור).

### `customPoliciesPath`

Type: `string` (absolute path)

נתיב לקובץ JavaScript המכיל מדיניויות hook מותאמות אישית. זה נקבע באופן אוטומטי על ידי `failproofai policies --install --custom <path>` (הנתיב מומר לנתיב מוחלט לפני שאוחסן).

הקובץ נטען מחדש בכל אירוע hook - אין שמירת בזיכרון מ-cache. ראה [Custom Policies](/he/custom-policies) לפרטי כתיבה.

### מדיניויות מבוססות כנסיה

בנוסף ל-`customPoliciesPath` המפורש, failproofai גוזף ותוענת באופן אוטומטי קבצי מדיניות מספריות `.failproofai/policies/`:

| רמה | ספריה | היקף |
|-------|-----------|-------|
| Project | `.failproofai/policies/` | משותף עם צוות דרך שליטת גרסאות |
| User | `~/.failproofai/policies/` | אישי, חל על כל הפרויקטים |

**התאמה קובץ:** רק קבצים התואמים `*policies.{js,mjs,ts}` נטענים (לדוגמה `security-policies.mjs`, `workflow-policies.js`). קבצים אחרים בספריה מתעלמים.

**אין צורך בתצורה:** מדיניויות כנסיה אינן דורשות ערכים ב-`policies-config.json`. פשוט הכנס קבצים לספריה והם יבחרו בהאירוע hook הבא.

**טעינת אתחול:** שתי ספריות הכנסיה של project ו-user סרוקות. כל הקבצים התואמים משתי הרמות נטענים (בניגוד ל-`customPoliciesPath` שמשתמש בכללת-היקף-ראשון-מנצח).

ראה [Custom Policies](/he/custom-policies) לפרטים נוספים וקוד לדוגמה.

### `llm`

Type: `object` (optional)

תצורת LLM client עבור מדיניויות שמבצעות קריאות AI. לא נדרש לרוב ההגדרות.

```json
{
  "llm": {
    "model": "claude-sonnet-4-6",
    "apiKey": "sk-ant-..."
  }
}
```

## ניהול תצורה מה-CLI

הפקודות `policies --install` ו-`policies --uninstall` כותבות לקובץ הגדרות hook של CLI ה-agent שלך (נקודות כניסה hook), בעוד `policies-config.json` הוא הקובץ שאתה מנהל ישירות. השניים נפרדים:

- **הגדרות Agent CLI** — אומרת לה-agent להתקשר ל-`failproofai --hook <event>` בכל שימוש בכלי:
  - **Claude Code**: `~/.claude/settings.json` (user), `<cwd>/.claude/settings.json` (project), `<cwd>/.claude/settings.local.json` (local)
  - **OpenAI Codex**: `~/.codex/hooks.json` (user), `<cwd>/.codex/hooks.json` (project) — ל-Codex אין היקף `local`
- **`policies-config.json`** — אומרת ל-failproofai אילו מדיניויות להעריך ועם אילו פרמטרים (משותף בכל CLIs של agents)

עבור `--cli claude|codex` כדי למקד CLI ספציפי של agent (מופרד בחלל או חוזר שוב לשניהם):

```bash
failproofai policies --install --cli codex --scope project
failproofai policies --install --cli claude codex
```

כשמוטל `--cli` מעקה, `failproofai` מגלה אילו agent CLIs מותקנים (`which claude` / `which codex`):

- **CLI אחד מגולה** — בוחר אוטומטית את ה-CLI הזה ללא הודעה.
- **שני CLIs מגולים** בטרמינל אינטראקטיבי — מציג הנחייה בחירה יחידה עם חיצים: בחר `Both`, `Claude Code only`, או `OpenAI Codex only` (↑↓ להזיז, Enter לבחור, ^C לצאת).
- **שני CLIs מגולים** בהפעלה לא אינטראקטיבית (CI, אין TTY) — מתקין לשניהם ללא הודעה.
- **לא CLI מגולה** — חוזר ל-`claude`, עם אזהרה שלא נמצא בינארי של agent ב-PATH; הפקודה hook כתובה עדיין כדי שהיא תופעל ברגע שתתקין אחד.

אתה יכול לערוך `policies-config.json` ישירות בכל עת; שינויים נכנסים לתוקף בעדכון על האירוע hook הבא ללא צורך בהפעלה מחדש.

## דוגמה: תצורה ברמת פרויקט עם ברירות מחדל לצוות

בצור `.failproofai/policies-config.json` למאגר שלך:

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "block-env-files"
  ],
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "release", "hotfix"]
    }
  }
}
```

כל מפתח יכול אז ליצור `.failproofai/policies-config.local.json` (gitignored) לעקיפות אישיות ללא השפעה על חברים בצוות.
