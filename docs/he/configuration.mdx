failproofai משתמש בקובצי הגדרות JSON כדי לשלוט אילו מדיניויות פעילות, כיצד הן מתנהגות, ומהיכן מדיניויות מותאמות אישיות נטענות. ההגדרות מעוצבות כדי להיות קלות לשיתוף עם הצוות שלך - הפקד אותן למאגר שלך וכל מפתח יקבל את אותה רשת ביטחון לסוכנים.

---

## מתחמי הגדרות

יש שלוש מתחמי הגדרות, מוערכים לפי סדר עדיפויות:

| מתחם | נתיב קובץ | מטרה |
|-------|-----------|---------|
| **פרויקט** | `.failproofai/policies-config.json` | הגדרות לכל מאגר, מופקדות בבקרת גרסאות |
| **מקומי** | `.failproofai/policies-config.local.json` | עקיפות אישיות לכל מאגר, מתעלמות |
| **גלובלי** | `~/.failproofai/policies-config.json` | ברירות מחדל ברמת משתמש בכל הפרויקטים |

כאשר failproofai קולט אירוע קצה, הוא טוען ומיזוג את כל שלושת הקובצים שקיימים לספריית העבודה הנוכחית.

### כללי מיזוג

**`enabledPolicies`** - האיחוד של כל שלוש המתחמים. מדיניות המופעלת בכל רמה היא פעילה.

```text
project:  ["block-sudo"]
local:    ["block-rm-rf"]
global:   ["block-sudo", "sanitize-api-keys"]

resolved: ["block-sudo", "block-rm-rf", "sanitize-api-keys"]  ← איחוד מבטל כפילויות
```

**`policyParams`** - המתחם הראשון המגדיר פרמטרים למדיניות מסוימת מנצח כולו. אין מיזוג עמוק של ערכים בתוך הפרמטרים של מדיניות.

```text
project:  block-sudo → { allowPatterns: ["sudo apt-get update"] }
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo apt-get update"] }   ← הפרויקט מנצח, גלובלי מתעלם
```

```text
project:  (ללא ערך block-sudo)
local:    (ללא ערך block-sudo)
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo systemctl status"] }  ← נופל דרך לגלובלי
```

**`customPoliciesPath`** - המתחם הראשון המגדיר אותו מנצח.

**`llm`** - המתחם הראשון המגדיר אותו מנצח.

## פורמט קובץ הגדרות

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "sanitize-jwt",
    "block-env-files",
    "block-read-outside-cwd"
  ],
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    },
    "block-push-master": {
      "protectedBranches": ["main", "release", "prod"]
    },
    "block-rm-rf": {
      "allowPaths": ["/tmp"]
    },
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company"]
    },
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo API key" }
      ]
    },
    "warn-large-file-write": {
      "thresholdKb": 512
    }
  },
  "customPoliciesPath": "/home/alice/myproject/my-policies.js"
}
```

## ייחוס שדות

### `enabledPolicies`

סוג: `string[]`

רשימת שמות מדיניויות להפעלה. שמות חייבים להתאים בדיוק למזהי המדיניויות המוצגים על ידי `failproofai policies`. ראה [Built-in Policies](/he/built-in-policies) לרשימה המלאה.

מדיניויות שלא ב-`enabledPolicies` אינן פעילות, גם אם יש להן ערכים ב-`policyParams`.

### `policyParams`

סוג: `Record<string, Record<string, unknown>>`

עקיפות פרמטרים לכל מדיניות. המפתח החיצוני הוא שם המדיניות; המפתחות הפנימיים הם ספציפיים למדיניות. כל מדיניות מתעדת את הפרמטרים הזמינים שלה ב-[Built-in Policies](/he/built-in-policies).

אם למדיניות יש פרמטרים אך לא מציינת אותם, משתמשות ברירות המחדל המובנות של המדיניות. משתמשים שלא מגדירים `policyParams` כלל מקבלים התנהגות זהה לגרסאות קודמות.

מפתחות לא ידועים בתוך בלוק הפרמטרים של מדיניות מתעלמים שקט בזמן ירי קצה אך מסומנים כאזהרות כאשר אתה מריץ `failproofai policies`.

#### `hint` (חוצה-חיתוך)

סוג: `string` (אופציונלי)

הודעה שצוינה לסיבה כאשר מדיניות מחזירה `deny` או `instruct`. השתמש בה כדי לתת Claude הנחיות בר-ביצוע ללא שינוי המדיניות עצמה.

עובד עם כל סוג מדיניות - מובנה, מותאמת אישיות (`custom/`), כנס פרויקט (`.failproofai-project/`), או כנס משתמש (`.failproofai-user/`).

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "נסה ליצור ענף טרי במקום."
    },
    "block-sudo": {
      "allowPatterns": ["sudo apt-get"],
      "hint": "השתמש ב-apt-get ישירות ללא sudo."
    },
    "custom/my-policy": {
      "hint": "בקש אישור מהמשתמש קודם."
    }
  }
}
```

כאשר `block-force-push` דוחה, Claude רואה: *"כפיית דחיפה חסומה. נסה ליצור ענף טרי במקום."*

ערכים שאינם מחרוזת וחרוזיים ריקים מתעלמים שקט. אם `hint` לא מוגדר, ההתנהגות לא משתנה (תואם לאחור).

### `customPoliciesPath`

סוג: `string` (נתיב מוחלט)

נתיב לקובץ JavaScript המכיל מדיניויות קצה מותאמות אישיות. זה מוגדר באופן אוטומטי על ידי `failproofai policies --install --custom <path>` (הנתיב מחשב לערך מוחלט לפני אחסון).

הקובץ נטען טרי בכל אירוע קצה - אין קשירה לזכרון. ראה [Custom Policies](/he/custom-policies) לפרטי כתיבה.

### מדיניויות מבוססות על כנסים

בנוסף ל-`customPoliciesPath` המפורש, failproofai גילוי וטוען קובצי מדיניות באופן אוטומטי מספריות `.failproofai/policies/`:

| רמה | ספרייה | מתחם |
|-------|-----------|-------|
| פרויקט | `.failproofai/policies/` | משותף לצוות דרך בקרת גרסאות |
| משתמש | `~/.failproofai/policies/` | אישי, חל על כל הפרויקטים |

**התאמת קובץ:** רק קובצים התואמים `*policies.{js,mjs,ts}` נטענים (למשל `security-policies.mjs`, `workflow-policies.js`). קובצים אחרים בספרייה מתעלמים.

**ללא הגדרות נדרשות:** מדיניויות כנס אינן דורשות ערכים ב-`policies-config.json`. פשוט בידוד קובצים לספרייה והם נבחרים בחדש על האירוע הקצה הבא.

**טעינת איחוד:** שני מתחמי הכנס של הפרויקט והמשתמש סרוקים. כל קובצים התואמים משתי הרמות נטענים (בשונה מ-`customPoliciesPath` המשתמש בהנצחון-מתחם-ראשון).

ראה [Custom Policies](/he/custom-policies) לפרטים נוספים ודוגמאות.

### `llm`

סוג: `object` (אופציונלי)

הגדרות לקוח LLM למדיניויות המבצעות קריאות AI. לא נדרש לרוב ההתקנות.

```json
{
  "llm": {
    "model": "claude-sonnet-4-6",
    "apiKey": "sk-ant-..."
  }
}
```

## ניהול הגדרות מה-CLI

הפקודות `policies --install` ו-`policies --uninstall` כותבות לקובץ הגדרות הקצה של ה-CLI של הסוכן שלך (נקודות כניסה הקצה), בעוד `policies-config.json` הוא הקובץ שאתה מנהל ישירות. השניים נפרדים:

- **הגדרות CLI סוכן** — אומר לסוכן להתקשר ל-`failproofai --hook <event>` בכל שימוש בכלי:
  - **Claude Code**: `~/.claude/settings.json` (משתמש), `<cwd>/.claude/settings.json` (פרויקט), `<cwd>/.claude/settings.local.json` (מקומי)
  - **OpenAI Codex**: `~/.codex/hooks.json` (משתמש), `<cwd>/.codex/hooks.json` (פרויקט) — ל-Codex אין מתחם `local`
- **`policies-config.json`** — אומר ל-failproofai אילו מדיניויות להעריך וככל פרמטרים (משותף בכל ה-CLI של סוכנים)

עבור `--cli claude|codex` כדי לכוון ל-CLI סוכן ספציפי (מופרד או חוזר על שניהם):

```bash
failproofai policies --install --cli codex --scope project
failproofai policies --install --cli claude codex
```

כאשר `--cli` מושמט, `failproofai` מגלה אילו CLI סוכנים מותקנים (`which claude` / `which codex`):

- **CLI אחד מגולה** — בוחר אוטומטית את ה-CLI הזה ללא שאילתה.
- **שני CLI מגולים** בטרמינל אינטראקטיבי — מציג הנדסה של לחיצת מקש יחיד: התקן עבור [B]oth (ברירת מחדל), [C]laude רק, או co[D]ex רק.
- **שני CLI מגולים** בריצה לא-אינטראקטיבית (CI, אין TTY) — מתקנים לשניהם ללא שאילתה.
- **לא CLI מגולה** — חוזר ל-`claude`, עם אזהרה כי בינארי סוכן לא נמצא ב-PATH; פקודת הקצה כתובה עדיין כדי שהוא יופעל ברגע שתתקין אחד.

אתה יכול לערוך `policies-config.json` ישירות בכל זמן; שינויים נכנסים לתוקף מיד בחדש הקצה הבא ללא צורך בהפעלה מחדש.

## דוגמה: הגדרות ברמת פרויקט עם ברירות מחדל של צוות

הפקד `.failproofai/policies-config.json` למאגר שלך:

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "block-env-files"
  ],
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "release", "hotfix"]
    }
  }
}
```

כל מפתח יכול לאחר מכן ליצור `.failproofai/policies-config.local.json` (מתעלם) לעקיפות אישיות ללא השפעה על חברי צוות.
