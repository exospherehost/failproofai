failproofai משתמש בקובצי תצורת JSON כדי לשלוט אילו מדיניות פעילה, איך הם מתנהגים, והיכן מדיניות מותאמת אישית מוטענת מ. התצורה תוכננה כדי להיות קלה לשיתוף עם הצוות שלך - צור commit שלה למאגר שלך וכל מפתח מקבל את אותו רשת ביטחון של agents.

---

## היקפי תצורה

יש שלושה היקפי תצורה, המוערכים בסדר עדיפויות:

| היקף | נתיב הקובץ | מטרה |
|-------|-----------|---------|
| **project** | `.failproofai/policies-config.json` | הגדרות לכל repo, מקובצות בשליטת גרסה |
| **local** | `.failproofai/policies-config.local.json` | דריסות אישיות לכל repo, בignore של git |
| **global** | `~/.failproofai/policies-config.json` | ברירות מחדל ברמת משתמש בכל הפרויקטים |

כאשר failproofai מקבל אירוע hook, הוא טוען ומיזוג את שלושת הקובצים שקיימים בספריית העבודה הנוכחית.

### כללי מיזוג

**`enabledPolicies`** - האיחוד של כל שלושת ההיקפים. מדיניות המופעלת בכל רמה היא פעילה.

```text
project:  ["block-sudo"]
local:    ["block-rm-rf"]
global:   ["block-sudo", "sanitize-api-keys"]

resolved: ["block-sudo", "block-rm-rf", "sanitize-api-keys"]  ← איחוד ללא כפילויות
```

**`policyParams`** - ההיקף הראשון שמגדיר params עבור מדיניות נתונה מנצח לחלוטין. אין מיזוג עמוק של ערכים בתוך params של מדיניות.

```text
project:  block-sudo → { allowPatterns: ["sudo apt-get update"] }
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo apt-get update"] }   ← project מנצח, global מתעלם
```

```text
project:  (אין ערך block-sudo)
local:    (אין ערך block-sudo)
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo systemctl status"] }  ← נופל דרך ל-global
```

**`customPoliciesPath`** - ההיקף הראשון המגדיר אותו מנצח.

**`llm`** - ההיקף הראשון המגדיר אותו מנצח.

## פורמט קובץ הגדרה

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "sanitize-jwt",
    "block-env-files",
    "block-read-outside-cwd"
  ],
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    },
    "block-push-master": {
      "protectedBranches": ["main", "release", "prod"]
    },
    "block-rm-rf": {
      "allowPaths": ["/tmp"]
    },
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company"]
    },
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo API key" }
      ]
    },
    "warn-large-file-write": {
      "thresholdKb": 512
    }
  },
  "customPoliciesPath": "/home/alice/myproject/my-policies.js"
}
```

## הפניה שדות

### `enabledPolicies`

Type: `string[]`

רשימה של שמות מדיניות להפעלה. שמות חייבים להתאים בדיוק למזהי המדיניות המוצגים על ידי `failproofai policies`. ראה [מדיניות מובנות](/he/built-in-policies) לרשימה המלאה.

מדיניות שלא ב-`enabledPolicies` אינן פעילות, גם אם יש להן ערכים ב-`policyParams`.

### `policyParams`

Type: `Record<string, Record<string, unknown>>`

דריסות פרמטרים לכל מדיניות. המפתח החיצוני הוא שם המדיניות; המפתחות הפנימיים ספציפיים למדיניות. כל מדיניות מתעדת את הפרמטרים הזמינים שלה ב-[מדיניות מובנות](/he/built-in-policies).

אם למדיניות יש פרמטרים אך אתה לא מציין אותם, משתמשים בברירות המחדל המובנות של המדיניות. משתמשים שלא מגדירים `policyParams` בכלל מקבלים התנהגות זהה לגרסאות קודמות.

מפתחות לא ידועים בתוך בלוק params של מדיניות מתעלמים בשקט בזמן הדלקת hook אך מסומנים כאזהרות כשאתה מריץ `failproofai policies`.

#### `hint` (חוצה-חתך)

Type: `string` (optional)

הודעה שצורפת לסיבה כאשר מדיניות מחזירה `deny` או `instruct`. השתמש בה כדי לתת Claude הדרכה פעולה ללא שינוי המדיניות עצמה.

עובד עם כל סוג מדיניות — מובנה, מותאם אישית (`custom/`), קונוונציה של פרויקט (`.failproofai-project/`), או קונוונציה של משתמש (`.failproofai-user/`).

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "נסה ליצור ענף טרי במקום."
    },
    "block-sudo": {
      "allowPatterns": ["sudo apt-get"],
      "hint": "השתמש ב-apt-get ישירות ללא sudo."
    },
    "custom/my-policy": {
      "hint": "בקש מהמשתמש אישור ראשון."
    }
  }
}
```

כאשר `block-force-push` מסרב, Claude רואה: *"Force-pushing blocked. נסה ליצור ענף טרי במקום."*

ערכים שאינם מחרוזת ומחרוזות ריקות מתעלמים בשקט. אם `hint` לא מוגדר, ההתנהגות לא משתנה (תאום לאחור).

### `customPoliciesPath`

Type: `string` (absolute path)

נתיב לקובץ JavaScript המכיל מדיניות hook מותאמת אישית. זה מוגדר באופן אוטומטי על ידי `failproofai policies --install --custom <path>` (הנתיב מיוחל לערך מוחלט לפני שנשמר).

הקובץ טוען מחדש בכל אירוע hook — אין שום caching. ראה [מדיניות מותאמת אישית](/he/custom-policies) לפרטי כתיבה.

### מדיניות מבוססות קונוונציה

בנוסף ל-`customPoliciesPath` המפורש, failproofai גילוי וטוען באופן אוטומטי קובצי מדיניות מספריות `.failproofai/policies/`:

| רמה | ספריה | היקף |
|-------|-----------|-------|
| פרויקט | `.failproofai/policies/` | משותף עם צוות דרך שליטת גרסה |
| משתמש | `~/.failproofai/policies/` | אישי, חל על כל הפרויקטים |

**התאמת קובץ:** רק קובצים התואמים `*policies.{js,mjs,ts}` נטוענים (למשל `security-policies.mjs`, `workflow-policies.js`). קובצים אחרים בספריה מתעלמים.

**אין צורך בתצורה:** מדיניות קונוונציה לא דורשות ערכים ב-`policies-config.json`. פשוט זרוק קובצים לספריה והם נאספים בתוך אירוע ה-hook הבא.

**טעינה בחוד אחיד:** גם ספריות קונוונציה של פרויקט וגם משתמש נסרקות. כל קובצים תואמים משתי הרמות נטוענים (בניגוד ל-`customPoliciesPath` שמשתמש ב-first-scope-wins).

ראה [מדיניות מותאמת אישית](/he/custom-policies) לפרטים נוספים וקישוטים.

### `llm`

Type: `object` (optional)

תצורת קליינט LLM עבור מדיניות שמבצעות קריאות AI. לא נדרש ברוב ההתקנות.

```json
{
  "llm": {
    "model": "claude-sonnet-4-6",
    "apiKey": "sk-ant-..."
  }
}
```

## ניהול הגדרה מ-CLI

הפקודות `policies --install` ו-`policies --uninstall` כותבות לקובץ הגדרות ה-hook של ה-CLI שלך (נקודות ה-hook), בעוד `policies-config.json` הוא הקובץ שאתה מנהל ישירות. השניים נפרדים:

- **הגדרות ה-Agent CLI** — אומר ל-agent לקרוא ל-`failproofai --hook <event>` בכל שימוש בכלי:
  - **Claude Code**: `~/.claude/settings.json` (משתמש), `<cwd>/.claude/settings.json` (פרויקט), `<cwd>/.claude/settings.local.json` (local)
  - **OpenAI Codex**: `~/.codex/hooks.json` (משתמש), `<cwd>/.codex/hooks.json` (פרויקט) — Codex אין לו היקף `local`
  - **GitHub Copilot CLI _(beta)_**: `~/.copilot/hooks/failproofai.json` (משתמש), `<cwd>/.github/hooks/failproofai.json` (פרויקט) — Copilot אין לו היקף `local`. ערכי Hook משתמשים בשדות הפקודה `bash`/`powershell` המקושרים ל-OS של Copilot עם `timeoutSec`; הקובץ נושא סימן `version: 1` ברמה העליונה. תמיכת Copilot CLI היא **beta** בזמן שאנו מוודים את סכימת הרשומה `events.jsonl` (שהדוקס הציבור אינו מציינים) מול יותר מפגשים אמיתיים.
  - **Cursor Agent _(beta)_**: `~/.cursor/hooks.json` (משתמש), `<cwd>/.cursor/hooks.json` (פרויקט) — Cursor אין לו היקף `local`. ערכי Hook משתמשים בצורה `{type, command, timeout}` בצורת Claude (אין חלוקה `bash`/`powershell`), אך מאוחסנים תחת מפתחות אירוע camelCase (`preToolUse`, `beforeSubmitPrompt`, …) במערך שטוח לפי סכימת [hooks](https://cursor.com/docs/hooks) של Cursor; הקובץ נושא סימן `version: 1` ברמה העליונה. המטפל משחזר camelCase → PascalCase דרך `CURSOR_EVENT_MAP` כך שמדיניות מובנות קיימות מופעלות ללא שינוי. תמיכת Cursor Agent היא **beta** בזמן שאנו מוודים את תמלול על-דיסק של Cursor (לא מצוין בדוקס הציבור) מול יותר התקנות אמיתיות.
  - **OpenCode _(beta)_**: `~/.config/opencode/opencode.json` + `~/.config/opencode/plugins/failproofai.mjs` (משתמש), `<cwd>/.opencode/opencode.json` + `<cwd>/.opencode/plugins/failproofai.mjs` (פרויקט) — OpenCode אין לו היקף `local`. בניגוד לארבע ה-CLI האחרות, OpenCode אין **אין מערכת hook קריאה חיצונית**: הוא טוען בתהליך plug-ins JS/TS שנרשמו בהוקעה דרך מערך `plugin: []` ב-`opencode.json` (גילוי אוטומטי מ-`.opencode/plugins/` **לא** איך plug-ins טוענים ב-opencode v1.14.33). Install משליך אל קובץ shimroxy plugin קטן שתוך-process-קורא את failproofai binary ותרגום תגובת JSON בצורת Claude של ה-binary חזרה לסמנטיקה של plug-in (`throw new Error()` עבור deny, `client.session.prompt(...)` עבור instruct, no-op עבור allow). מפגשות חיות בסל של SQLite של opencode ב-`~/.local/share/opencode/opencode.db`; דוח לוח המחוונים קורא אותם דרך `opencode db --format json` ו-`opencode export <id>`. תמיכת OpenCode היא **beta** בזמן שאנו מוודים התנהגות על פני גרסאות ומול יותר מפגשים אמיתיים. ראה את [מסמכי plug-ins OpenCode](https://opencode.ai/docs/plugins/).
  - **Pi _(beta)_**: `~/.pi/agent/settings.json` (משתמש), `<cwd>/.pi/settings.json` (פרויקט) — Pi אין לו היקף `local`. Pi טוען חבילות הרחבה TypeScript בהתחלה; קובץ הגדרות הוא מערך מחרוזת שטוח `{"packages": ["./relative/path", …]}`. failproofai כותב כניסה מערך חבילות יחידה המצביעה על ספרייתה המבוקדת `pi-extension/`. ההרחבה פנימה מנויה ל-`tool_call` / `user_bash` / `input` / `session_start` אירועים של Pi ופחתות החוצה ל-`failproofai --hook <Event> --cli pi`; המטפל משחזר underscore_lower_snake_case → PascalCase דרך `PI_EVENT_MAP` כך שמדיניות מובנות קיימות מופעלות ללא שינוי. תמיכת Pi היא **beta** בזמן שה-API הרחבה של Pi ופריסת יומן מפגש מיצובות.
  - **Gemini CLI _(beta)_**: `~/.gemini/settings.json` (משתמש), `<cwd>/.gemini/settings.json` (פרויקט) — Gemini אין לו היקף `local` (הוא מתעד היקף `system` ב-`/etc/gemini-cli/settings.json` שfaiproofai לא חושף). ערכי Hook משתמשים בצורה `{type, command, timeout}` של Claude עטוף בסכימת matcher של Gemini `{matcher, hooks: [...]}` עם `matcher: "*"` כברירת מחדל. אירועים הם PascalCase (`SessionStart`, `BeforeAgent`, `AfterAgent`, `BeforeModel`, `AfterModel`, `BeforeToolSelection`, `BeforeTool`, `AfterTool`, `PreCompress`, `Notification`, `SessionEnd`); המטפל מתמפה לשמות קנוניים של Claude דרך `GEMINI_EVENT_MAP`. שמות הכלים הם snake_case (`run_shell_command`, `read_file`, `write_file`, `replace`, …) — המטפל משחזר דרך `GEMINI_TOOL_MAP` כך שמדיניות מובנות קיימות מופעלות ללא שינוי. מעריך המדיניות משדר צורת שטוח של Gemini `{decision: "deny", reason}` (מועדף לפי כלל הזהב של Gemini exit-0 contract), `{hookSpecificOutput: {hookEventName, additionalContext}}` להזרקת הקשר ב-BeforeAgent / AfterTool / SessionStart, ו-`{decision: "block", reason}` ב-AfterAgent לסמנטיקה force-retry. תמיכת Gemini CLI היא **beta** בזמן שאנו מרחיבים כיסוי אמיתי. ראה את [מסמכי hooks Gemini CLI](https://geminicli.com/docs/hooks/).
- **`policies-config.json`** — אומר ל-failproofai אילו מדיניות להעריך וכמה params (משותף על פני כל agent CLIs)

עבור `--cli claude|codex|copilot|cursor|opencode|pi|gemini` כדי להכוון ל-agent ספציפי (מופרד בחלל או חוזר עבור כל תת-קבוצה):

```bash
failproofai policies --install --cli codex --scope project
failproofai policies --install --cli copilot --scope project
failproofai policies --install --cli cursor --scope project
failproofai policies --install --cli opencode --scope project
failproofai policies --install --cli pi --scope project
failproofai policies --install --cli gemini --scope project
failproofai policies --install --cli claude codex copilot cursor opencode pi gemini
```

כשנחמד `--cli` מושמט, `failproofai` גוגל אילו agent CLIs מותקנות (`which claude` / `which codex` / `which copilot` / `which cursor-agent` / `which opencode` / `which pi` / `which gemini`):

- **CLi אחד גוגל** — בחירות אוטומטיות הcli זה ללא בקשה.
- **CLis מרובים גוגלו** בטרמינל יחיד — מראה בחירה יחידה עם חץ מקמפלט לפי `Detected (N)` (עם שורת `Install for all N detected` aggregated + כל CLI גוגל בנפרד) וחלק `Not installed (M) · install hooks ahead of time` רשימה כל CLi לא גוגל כאפציית forward-install (↑↓ לנוע, Enter בחר, ^C עזוב). זרימת ה-uninstall מראה רק החלק Detected.
- **CLis מרובים גוגלו** בריץ לא יחיד (CI, no TTY) — התקנות עבור כל CLis גוגל ללא בקשה.
- **אין גוגל** — נופל חזרה ל-`claude`, עם אזהרה שאין agent binary נמצא ב-PATH; הפקודה hook עדיין כתובה כך היא מופעלת ברגע שתתקין אחד.

אתה יכול לערוך `policies-config.json` ישירות בכל עת; שינויים תופסים בדיוק בתוך אירוע ה-hook הבא ללא צורך בהפעלה מחדש.

## דוגמה: תצורת רמת פרויקט עם ברירות מחדל צוות

Commit `.failproofai/policies-config.json` למאגר שלך:

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "block-env-files"
  ],
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "release", "hotfix"]
    }
  }
}
```

כל מפתח יכול אז ליצור `.failproofai/policies-config.local.json` (gitignored) לדריסות אישיות ללא השפעה על teammates.
