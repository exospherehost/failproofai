failproofai מגיע עם 39 מדיניות מובנות שתופסות מצבי כישלון נפוצים של סוכנים. כל מדיניות מופעלת על סוג אירוע hook ספציפי ושם כלי. תשע עשרה מדיניות מקבלות פרמטרים שמאפשרים לך לכוונן את ההתנהגות שלהן ללא כתיבת קוד. חמש מדיניות זרימת עבודה אוכפות קו-ביצוע של commit → push → PR → CI לפני שClaude עוצר.

## סקירה כללית

מדיניות מקובצות לקטגוריות:

| קטגוריה | מדיניות | סוג Hook |
|----------|----------|-----------|
| [פקודות מסוכנות](#dangerous-commands) | block-sudo, block-rm-rf, block-curl-pipe-sh, block-failproofai-commands | PreToolUse |
| [פקודות Infra](#infra-commands) | block-kubectl, block-terraform, block-aws-cli, block-gcloud, block-az-cli, block-helm, block-gh-pipeline | PreToolUse |
| [סודות (מטהרים)](#secrets-sanitizers) | sanitize-jwt, sanitize-api-keys, sanitize-connection-strings, sanitize-private-key-content, sanitize-bearer-tokens | PostToolUse |
| [סביבה](#environment) | block-env-files, protect-env-vars | PreToolUse |
| [גישת קבצים](#file-access) | block-read-outside-cwd, block-secrets-write | PreToolUse |
| [Git](#git) | block-push-master, block-work-on-main, block-force-push, warn-git-amend, warn-git-stash-drop, warn-all-files-staged | PreToolUse |
| [בסיס נתונים](#database) | warn-destructive-sql, warn-schema-alteration | PreToolUse |
| [אזהרות](#warnings) | warn-large-file-write, warn-package-publish, warn-background-process, warn-global-package-install | PreToolUse |
| [מנהלי חבילות](#package-managers) | prefer-package-manager | PreToolUse |
| [זרימת עבודה](#workflow) | require-commit-before-stop, require-push-before-stop, require-pr-before-stop, require-no-conflicts-before-stop, require-ci-green-before-stop | Stop |

- **`block-`** — עצור את הסוכן מהמשך.
- **`warn-`** — תן לסוכן הקשר נוסף כדי שיוכל לתקן את עצמו.
- **`sanitize-`** — טהר נתונים רגישים מפלט הכלי לפני שהסוכן רואה זאת.

### שמות קבוצה

כל מדיניות נמצאת בחריץ `<namespace>/<name>`. מדיניות מובנות שייכות
לשם קבוצה **`exospherehost/`** — לדוגמה, `exospherehost/sanitize-jwt`. שם הקבוצה
מונע התנגשויות כאשר אתה גם טוען מדיניות מותאמות או של צד שלישי
עם שמות קצרים דומים.

בתצורה שלך, אתה יכול להתייחס לבנייה על ידי שם קצר או שם
מתוקן; שני הצורות מתפקדות לאותה מדיניות:

```json
{
  "enabledPolicies": [
    "sanitize-jwt",
    "exospherehost/block-rm-rf"
  ]
}
```

אם לשם אין `/`, failproofai מתייחס אליו כשייך לשם הקבוצה הברירת המחדל
`exospherehost`. שמות שכבר מכילים `/` (לדוגמה `myorg/foo`,
`custom/my-hook`) נשמרים כפי שהם.
- **`require-`** — חסום את אירוע העצירה עד למילוי התנאים.

---

<Tip>
כל מדיניות תומכת בשדה `hint` אופציונלי ב-`policyParams`. ה-hint מצורף להודעת deny או instruct שClaude רואה, ונותן הנחיות פעולה ללא שינוי קוד מדיניות. עובד עם מדיניות מובנות, מותאמות ומדיניות של קונוונציה. ראה [Configuration → hint](/he/configuration#hint-cross-cutting) לפרטים.
</Tip>

## פקודות מסוכנות

מנע מסוכנים מהפעלת פעולות שקשה לבטל או שעלולות להזיק למערכת המארחת.

### `block-sudo`

**אירוע:** PreToolUse (Bash)  
**ברירת מחדל:** דוחה כל פקודת `sudo`.

חוסם הפעלות המכילות את מילת המפתח `sudo`. התאמת דפוסים נעשית על אסימוני פקודה מנותחים, לא על המחרוזת הגולמית, כדי למנוע עקיפה דרך הזרקת אופרטור shell.

**פרמטרים:**

| פרם | סוג | ברירת מחדל | תיאור |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | קידומות פקודה מדויקות המותרות. כל רשומה מובדקת מול אסימוני argv מנותחים. |

**דוגמה:**

```json
{
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    }
  }
}
```

עם תצורה זו, `sudo systemctl status nginx` מותר, אך `sudo rm /etc/hosts` דחוי.

<Note>
דפוסים מובדקים מול אסימונים מנותחים, לא מול מחרוזת הפקודה הגולמית. זה מונע עקיפה דרך אופרטורים shell מצורפים (לדוגמה `sudo systemctl status x; rm -rf /` לא תואם `sudo systemctl status *`).
</Note>

## פקודות Infra

עצור סוכני קידוד מהפעלת CLIs של תשתית או הפעלת צינורות CI/CD. כל מדיניות בקטגוריה זו היא **opt-in** (`defaultEnabled: false`) — סוכנים שבאמת צריכים לקרוא ל-`kubectl`, `terraform` וכו' לא יופרעו אלא אם אתה מפעיל את המדיניות. כאשר מופעל, כל הפעלה של ה-CLI המתואם מדוחה אלא אם הפקודה תואמת רשומה ב-`allowPatterns`.

דקדוק הדפוס זהה ל-[`block-sudo`](#block-sudo): אסימונים מובדקים מול argv מנותח, `*` הוא wildcard עבור אסימון אחד, וכל פקודה המכילה אופרטור shell עצמאי (`&&`, `||`, `|`, `;`) או אסימון עם תו shell מובנה דוחה לפני התאמה בהרשמת ההיתרים כדי למנוע עקיפות הזרקה.

### `block-kubectl`

**אירוע:** PreToolUse (Bash)  
**ברירת מחדל:** דוחה כל הפעלת `kubectl`.

**פרמטרים:**

| פרם | סוג | ברירת מחדל | תיאור |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | קידומות פקודת kubectl המותרות. |

**דוגמה:**

```json
{
  "policyParams": {
    "block-kubectl": {
      "allowPatterns": ["kubectl get *", "kubectl describe *", "kubectl logs *"]
    }
  }
}
```

עם תצורה זו, `kubectl get pods` מותר אך `kubectl apply -f deploy.yaml` דחוי.

## סודות (מטהרים)

עצור סוכנים מהדלפת חיזויים להקשר או פלט שלהם. מדיניות מטהר מופעלות על אירועי **PostToolUse**. כאשר Claude מריץ פקודת Bash, קורא קובץ או קורא לכל כלי, מדיניות אלה בוחנות את הפלט לפני שהוא מוחזר ל-Claude. אם דפוס סודי מזוהה, המדיניות מחזירה החלטת deny המונעת את הפלט מהעברה חזרה.

### `sanitize-jwt`

**אירוע:** PostToolUse (כל הכלים)  
**ברירת מחדל:** משחית את אסימוני JWT (שלוש קטעי base64url מופרדים על ידי `.`).

אין פרמטרים.

---

### `sanitize-api-keys`

**אירוע:** PostToolUse (כל הכלים)  
**ברירת מחדל:** משחית פורמטי מפתח API נפוצים: Anthropic (`sk-ant-`), OpenAI (`sk-`), GitHub PATs (`ghp_`), מפתחות גישה AWS (`AKIA`), מפתחות Stripe (`sk_live_`, `sk_test_`) ומפתחות Google API (`AIza`).

**פרמטרים:**

| פרם | סוג | ברירת מחדל | תיאור |
|-------|------|---------|-------------|
| `additionalPatterns` | `{ regex: string; label: string }[]` | `[]` | דפוסי regex נוספים להיחשב כסודות. |

**דוגמה:**

```json
{
  "policyParams": {
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo internal API key" },
        { "regex": "pat_[0-9a-f]{40}", "label": "Internal PAT" }
      ]
    }
  }
}
```

---

### `sanitize-connection-strings`

**אירוע:** PostToolUse (כל הכלים)  
**ברירת מחדל:** משחית מחרוזות חיבור למסד נתונים המכילות חיזויים מובנים (לדוגמה `postgresql://user:password@host/db`).

אין פרמטרים.

---

### `sanitize-private-key-content`

**אירוע:** PostToolUse (כל הכלים)  
**ברירת מחדל:** משחית בלוקי PEM (`-----BEGIN PRIVATE KEY-----`, `-----BEGIN RSA PRIVATE KEY-----` וכו').

אין פרמטרים.

---

### `sanitize-bearer-tokens`

**אירוע:** PostToolUse (כל הכלים)  
**ברירת מחדל:** משחית כותרות `Authorization: Bearer <token>` כאשר האסימון הוא 20 או יותר תווים.

אין פרמטרים.

## גישת קבצים

שמור על סוכנים הפועלים בתוך גבולות הפרויקט וחוק מקבצים רגישים.

### `block-read-outside-cwd`

**אירוע:** PreToolUse (Read, Bash)  
**ברירת מחדל:** דוחה קריאת קבצים מחוץ לשורש הפרויקט. הגבול הוא `CLAUDE_PROJECT_DIR` (מוגדר פעם ליד by Claude Code), עם fallback לספריית העבודה הנוכחית של ההפעלה כאשר משתנה זה לא מוגדר. שימוש בשורש הפרויקט ולא ב-`cwd` חי פירושו שהגבול נשאר יציב גם אחרי ש-Claude עובר `cd` לתת-ספריה.

**פרמטרים:**

| פרם | סוג | ברירת מחדל | תיאור |
|-------|------|---------|-------------|
| `allowPaths` | `string[]` | `[]` | קידומות נתיב מוחלט המותרות גם אם מחוץ לשורש הפרויקט. |

**דוגמה:**

```json
{
  "policyParams": {
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company/config"]
    }
  }
}
```

---

### `block-secrets-write`

**אירוע:** PreToolUse (Write, Edit)  
**ברירת מחדל:** דוחה כתיבות לקבצים המשמשים בדרך כלל למפתחות פרטיים ותעודות: `id_rsa`, `id_ed25519`, `*.key`, `*.pem`, `*.p12`, `*.pfx`.

**פרמטרים:**

| פרם | סוג | ברירת מחדל | תיאור |
|-------|------|---------|-------------|
| `additionalPatterns` | `string[]` | `[]` | דפוסי שם קובץ נוספים (בסגנון glob) לחסימה. |

**דוגמה:**

```json
{
  "policyParams": {
    "block-secrets-write": {
      "additionalPatterns": [".token", ".secret"]
    }
  }
}
```

## Git

מנע דחיפות אקראיות, דחיפות כפויות וטעויות ענף שקשה לבטל.

### `block-push-master`

**אירוע:** PreToolUse (Bash)  
**ברירת מחדל:** דוחה `git push origin main` ו-`git push origin master`.

**פרמטרים:**

| פרם | סוג | ברירת מחדל | תיאור |
|-------|------|---------|-------------|
| `protectedBranches` | `string[]` | `["main", "master"]` | שמות ענפים שלא ניתן לדחוף אליהם ישירות. |

**דוגמה:**

```json
{
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "master", "release", "prod"]
    }
  }
}
```

<Tip>
כדי להרשות דחיפה לכל הענפים (בעצם השבתת מדיניות זו ללא הסרתה מ-`enabledPolicies`), הגדר `protectedBranches: []`.
</Tip>

---

### `block-work-on-main`

**אירוע:** PreToolUse (Bash)  
**ברירת מחדל:** דוחה `git commit`, `git merge`, `git rebase` ו-`git cherry-pick` כאשר עץ העבודה נמצא ב-`main` או `master`. יצירה והחלפת ענף (`git checkout`, `git checkout -b`, `git switch`, `git switch -c`) אינן מושפעות.

**פרמטרים:**

| פרם | סוג | ברירת מחדל | תיאור |
|-------|------|---------|-------------|
| `protectedBranches` | `string[]` | `["main", "master"]` | שמות ענפים עליהם commit/merge/rebase/cherry-pick דחוי. |

---

### `block-force-push`

**אירוע:** PreToolUse (Bash)  
**ברירת מחדל:** דוחה `git push --force` ו-`git push -f`.

אין פרמטרים ספציפיים למדיניות. השתמש ב-[`hint`](/he/configuration#hint-cross-cutting) החוצה-חתך להצעת חלופות:

```json
{
  "policyParams": {
```

## מנהלי חבילות

אכוף אילו מנהלי חבילות הסוכן רשאי להשתמש.

### `prefer-package-manager`

**אירוע:** PreToolUse (Bash)  
**ברירת מחדל:** מושבת. כאשר מופעל, חוסם כל פקודת מנהל חבילות שלא ברשימה `allowed` ומורה ל-Claude לכתוב מחדש את הפקודה באמצעות מנהל מותר.

מזהה: pip, pip3, python -m pip, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo.

| פרמטר | סוג | ברירת מחדל | תיאור |
|-----------|------|---------|-------------|
| `allowed` | string[] | `[]` | שמות מנהל חבילות מותרים. כל מנהל מזוהה שלא ברשימה זו דחוי. כאשר ריק, המדיניות היא no-op. |
| `blocked` | string[] | `[]` | שמות מנהל נוספים לחסימה מעבר לרשימה המובנית (לדוגמה `['pdm', 'pipx']`). |

רשימת החסימה המובנית מכסה: pip, pip3, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo. השתמש ב-`blocked` להוספת מנהלים שאינם ברשימה זו.

**דוגמת תצורה:**

```json
{
  "enabledPolicies": ["prefer-package-manager"],
  "policyParams": {
    "prefer-package-manager": {
      "allowed": ["uv", "bun"],
      "blocked": ["pdm", "pipx"]
    }
  }
}
```

עם תצורה זו, `pip install flask` ו-`pdm install flask` שניהם דחויים עם הודעה המורה ל-Claude להשתמש ב-`uv` או `bun` במקום. פקודות כמו `uv pip install flask` מותרות כי `uv` ברשימת ההרשמה ונבדק ראשון.

## זרימת עבודה

אכוף זרימת עבודה משומרת בסוף ההפעלה. מדיניות אלה מופעלות על אירוע **Stop** ודחות את הסוכן מהעצירה עד שכל תנאי מתקיים. הם עוקבים אחר שרשרת תלות טבעית: commit → push → PR → CI. אם מדיניות דוחה, מדיניות מאוחרות בשרשרת דלנועות (deny short-circuits).

כל מדיניות זרימת עבודה היא **fail-open**: אם הכלי הנדרש לא זמין (לדוגמה `gh` לא מותקן, אין git remote), המדיניות מתירה עם הודעה אינפורמטיבית המסבירה מדוע בדיקה דלנועות.

### סמנטיקה Stop לכל CLI

אכיפת Stop נראית קצת אחרת על פני שבעת ה-CLIs הנתמכים כי כל אחד חושף חוזה hook אחר של סוף-הפעלה. ה-**תוצאה** זהה — הסוכן לא משתחרר מעצירה בזמן שער זרימת עבודה נכשל — אך ה-**מכניקה** שונה. הטבלה למטה מסכמת; רק Pi יש quirk ראוי למימון קודם שתאפשר מדיניות `require-*-before-stop`.

| CLI | כאשר השער מופעל | מה אתה רואה |
|---|---|---|
| Claude Code | אותה לולאת סוכן, מיד | Claude ממשיך לעבוד — מתקן את הבעיה, ואז מנסה להסתיים שוב. אין הפסקה נראית לך. |
| Codex | אותה לולאת סוכן, מיד | זהה ל-Claude. |
| GitHub Copilot CLI | אותה לולאת סוכן, מיד | זהה ל-Claude (משתמש בערוץ `{decision:"block", reason}` של Copilot — מאומת אמפירית כנגד Copilot CLI 1.0.41). |
| Cursor Agent | אותה לולאת סוכן, מיד | זהה ל-Claude (משתמש בערוץ `{followup_message}` של Cursor — מכופף ל-`loop_limit`, ברירת מחדל 5 ניסיונות). |
| Gemini CLI | אותה לולאת סוכן, מיד | זהה ל-Claude (משתמש בערוץ `{decision:"block", reason}` של Gemini ב-`AfterAgent`). |
| OpenCode | אותה לולאת סוכן, מיד | זהה ל-Claude (משתמש בקריאת `client.session.prompt(...)` SDK של OpenCode מנותב דרך `hookSpecificOutput.additionalContext`). |
| **Pi (pi-coding-agent)** | **תור משתמש הבא** | **Pi עוצר באופן נראות** כאשר השער מופעל — לולאת הסוכן שלה יוצאת ואתה חוזר להנחיה. השער אז מופעל בפעם הבאה שאתה שולח הנחיה: failproofai קדימה directive `MANDATORY ACTION REQUIRED` להנחיית המערכת של הפנייה זו, ומנחה את ה-LLM להשלים את שלב זרימת העבודה (commit, push וכו') לפני עשיית מה שביקשת. |

<Note>
**מגבלת Pi.** `AgentEndEvent` של Pi (המקבילה של Claude `Stop` hook) אין סוג Result — בתוך זמן שהוא מופעל, לולאת הסוכן של Pi כבר יצאה. Pi לא יכול להיכנס לניסיון שוב לאותה לולאה כדרך שClaude / Copilot / Cursor / Gemini / OpenCode יכולים. failproofai משנה את השער ל-`before_agent_start` אירוע של Pi (שמופעל אחרי ההנחיה הבאה של המשתמש) כדי שבדיקת זרימת העבודה עדיין אוכפת, רק בתור הבא ולא הנוכחי.

**מה זה אומר בפועל:**

- אחרי Pi עוצר, סיבת הדחייה נתפסת בזכרון מקודד לפי Pi session id. ההנחיה הבאה בדיוק שאתה משדר באותו תהליך Pi מניקה זאת: ה-LLM רואה את ה-directive `MANDATORY ACTION REQUIRED` בראש מערכת ההנחיה שלו, commits (או דוחף / פותח את ה-PR / חוכה ל-CI) ורק אז ממשיך עם בקשתך. סיבת הדחייה שנתפסה היא one-shot — ברגע שניקה, השער ברור.
- השער מתוחם על ידי אורך חיים של התהליך של Pi. אם `Ctrl+C` Pi או יוצא בין תורות, הרשומה בזכרון מושמטת יחד עם התהליך והשער מחומק. Claude, Copilot, Cursor, Gemini וOpenCode יש אותו כבול (הרוג את הסוכן והשער מחומק) — Pi רק עושה זאת יותר נראות כי הסוכן יוצא בצורה גלויה לפני שהשער מופעל.
- דחייה ממתנה גם היא מנוקה ב-`session_shutdown` לכל סיבה (`new` / `resume` / `fork` / `quit`), כדי שער stale משהפעלה קודמת לא יוכל להדליף לתוך הפעלה טריה שהופעלה באותו תהליך Pi.

אם אתה צריך Claude-style same-loop retry, הפעל את `Stop` מדיניות שלך תחת כל אחד משש ה-CLIs הנתמכות האחרות. אנחנו עוקבים אחרי Pi upstream לסוג Result עתידי ב-`AgentEndEvent` שיאפשר לנו לסגור את הפער זה.
</Note>

### `require-no-conflicts-before-stop`

**אירוע:** Stop  
**ברירת מחדל:** דוחה עצירה כאשר הענף הנוכחי לא יכול merge בצורה נקייה לענף הבסיס. המדיניות תחילה מאשרת שיש PR `OPEN` ב-GitHub לענף — ללא אחד, אין merge target לאכוף, כל המדיניות short-circuits להרשאה. ברגע ש-PR `OPEN` מאושרת, שני probes עצמאיים מריצים:

1. **Local** — `git merge-tree --write-tree --name-only origin/<baseBranch> HEAD`. בקונפליקט, הודעת הדחייה קוראה לקבצים בעימות כדי Claude יודע בדיוק מה לפתור.
2. **GitHub** — מחזור את `gh pr view --json mergeable,state` התוצאה כבר ממושך בהבדיקה המקדימה. תופס קונפליקטים שלוקאלי stale `origin/<baseBranch>` יחמיץ (לדוגמה מישהו נחת PR קונפליקט ב-`main` מאז הפנייה האחרונה). תוצאה `CONFLICTING` דוחה. תוצאה `UNKNOWN` גם דוחה ומורה ל-Claude להמתין ~10 שניות וחזור-בדיקה לפני ניסיון עצירה שוב — זה מונע false negatives בזמן GitHub recomputes.

דלנועות לחלוטין (מתירה) כאשר: `gh` לא מותקן, אין PR לענף, מצב ה-PR אינו `OPEN` (לדוגמה `MERGED`, `CLOSED`) או `gh pr view` מחזירה פלט unparseable. גם נכשל בפתיחות כאשר `origin/<baseBranch>` חסר לוקאלי או כאשר אין commits קדימה של בסיס — אלה Layer 1 fall-throughs עדיין יעצם את PR mergeability שמסומן לפני הרשאה.

**פרמטרים:**

| פרם | סוג | ברירת מחדל | תיאור |
|-------|------|---------|-------------|
| `baseBranch` | `string` | `"main"` | ענף בסיס לבדיקת קונפליקטים כנגד. |

<Note>
GitHub CLI (`gh`) נדרש למדיניות זו. המדיניות משתמשת ב-`gh pr view` לאשר
שקיים PR `OPEN` לפני הפעלת כל probe קונפליקט — ללא `gh`, המדיניות
short-circuits להרשאה. הרץ `gh auth login` עם personal access token שיש לו
`repo` scope לקריאה access לבקשות pull.
</Note>

## השבתת מדיניות בודדות

הסר מדיניות ספציפית מ-`enabledPolicies` בתצורה שלך או החליף אותה כבוי בטאב Policies של הלוח.

```json
{
  "enabledPolicies": [
    "block-rm-rf",
    "sanitize-api-keys"
  ]
}
```

מדיניות שלא מופיעות ב-`enabledPolicies` לא מריצות, גם אם רשומות `policyParams` קיימות עבורן.
