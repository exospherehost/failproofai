Bu belge failproofai'nin dahili olarak nasıl çalıştığını açıklar: hook sistemi agent araç çağrılarını nasıl kesintiye uğratır, konfigürasyon nasıl yüklenir ve birleştirilir, politikalar nasıl değerlendirilir ve dashboard agent aktivitesini nasıl izler.

---

## Genel Bakış

failproofai'nin iki bağımsız alt sistemi vardır:

1. **Hook handler** - Claude Code'un her agent araç çağrısında çalıştırdığı hızlı bir CLI alt işlemi. Politikaları değerlendirir ve bir karar döndürür.
2. **Agent Monitor (Dashboard)** - Agent oturumlarını izlemek ve politikaları yönetmek için bir Next.js web uygulaması.

Her iki alt sistem `~/.failproofai/` ve projenin `.failproofai/` dizinindeki konfigürasyon dosyalarını paylaşır, ancak ayrı işlemler olarak çalışır ve yalnızca dosya sistemi üzerinden iletişim kurar.

## Hook handler

### Claude Code ile entegrasyon

`failproofai policies --install` komutunu çalıştırdığınızda, `~/.claude/settings.json` içerisine şu şekilde girişler yazar:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "failproofai --hook PreToolUse"
          }
        ]
      }
    ],
    "PostToolUse": [ ... ]
  }
}
```

Claude Code daha sonra her araç çağrısından önce stdin üzerinde JSON yükü ileterek `failproofai --hook PreToolUse` komutunu bir alt işlem olarak çalıştırır.

### Yük formatı

```json
{
  "session_id": "abc123",
  "transcript_path": "/home/user/.claude/projects/myproject/sessions/abc123.jsonl",
  "cwd": "/home/user/myproject",
  "permission_mode": "default",
  "hook_event_name": "PreToolUse",
  "tool_name": "Bash",
  "tool_input": { "command": "sudo apt install nodejs" }
}
```

`PostToolUse` olayları için yük, araç çıktısını içeren `tool_result` de içerir.

Handler 1 MB stdin sınırını uygular. Bu sınırı aşan yükler atılır ve tüm politikalar örtülü olarak izin verir.

### Yanıt formatı

**Reddet (PreToolUse):**
```json
{
  "hookSpecificOutput": {
    "permissionDecision": "deny",
    "permissionDecisionReason": "Blocked by failproofai: sudo command blocked"
  }
}
```

**Reddet (PostToolUse):**
```json
{
  "hookSpecificOutput": {
    "additionalContext": "Blocked by failproofai because: API key detected in output"
  }
}
```

**Talimat ver (Stop dışındaki herhangi bir olay):**
```json
{
  "hookSpecificOutput": {
    "additionalContext": "Instruction from failproofai: Verify tests pass before committing."
  }
}
```

**Stop olayı talimatı:**
- Çıkış kodu: `2`
- Sebep stderr'e yazılmıştır (stdout değil)

**İzin ver:**
- Çıkış kodu: `0`
- Boş stdout

**Mesajlı izin ver:**

`allow(message)` bir politikanın operasyon izin verilse bile Claude'a bilgilendirici bağlam gönderebilmesini sağlar. Hook handler, hook'un Claude Code'a yanıtı (deny ve instruct yanıtları gibi) olan aşağıdaki JSON'u **stdout**'a yazar:

```json
// Hook handler işlemi tarafından stdout'a yazılır
{
  "hookSpecificOutput": {
    "additionalContext": "All CI checks passed on branch 'feat/my-feature'."
  }
}
```
- Çıkış kodu: `0` (operasyon izin verilir)
- Birden çok politika mesajlı `allow` döndürdüğünde, mesajları yeni satırlarla birleştirerek tek bir `additionalContext` dizesine katılır
- Hiçbir politika mesaj sağlamadığında, stdout boş olur (öncekiyle aynı)

### İşleme hattı

`src/hooks/handler.ts` tam işleme hattını uygular:

```text
stdin JSON
  → payload ayrıştırma (max 1 MB)
  → oturum meta verisi çıkarma (session_id, cwd, tool_name, tool_input, vb.)
  → readMergedHooksConfig(cwd)    ← proje + yerel + global config'i birleştirir
  → etkinleştirilen yerleşik politikaları çözümlenen parametrelerle kaydetme
  → customPoliciesPath'ten özel politikaları yükleme (ayarlanmışsa)
  → özel politikaları politika kaydına kaydetme
  → tüm politikaları değerlendirme (yerleşikler önce, ardından özel)
      → ilk deny kısa devre yapar
      → instruct kararları birikir
      → izin mesajları birikir
  → JSON kararı stdout'a yazma
  → olayı ~/.failproofai/hook-activity.jsonl'ye kalıcı kılma
  → çıkış
```

Tüm işlem, LLM çağrıları olmayan tipik yükler için 100ms altında çalışır.

## Yapılandırma Yükleme

`src/hooks/hooks-config.ts` üç kapsamlı config yükleme uygular.

```text
[1] {cwd}/.failproofai/policies-config.json        ← proje  (en yüksek öncelik)
[2] {cwd}/.failproofai/policies-config.local.json  ← yerel
[3] ~/.failproofai/policies-config.json             ← global   (en düşük öncelik)
```

Birleştirme mantığı:
- `enabledPolicies` - üç dosya arasında çoğaltılmış birleşim
- `policyParams` - politika başına anahtar, tanımlayan ilk dosya tamamen kazanır
- `customPoliciesPath` - tanımlayan ilk dosya kazanır
- `llm` - tanımlayan ilk dosya kazanır

Web dashboard, proje cwd'si ile çalıştırılmadığı için okuma ve yazma işlemi için `readHooksConfig()` (yalnızca global) kullanır.

## Politika Değerlendirmesi

`src/hooks/policy-evaluator.ts` politikaları sırayla çalıştırır.

Her politika için:

1. Politikanın `params` şemasını arayın (varsa).
2. Birleştirilmiş config'ten `policyParams[policy.name]` okuyun.
3. Şema varsayılanlarının üzerine kullanıcı tarafından sağlanan değerleri birleştirerek `ctx.params` üretin.
4. Çözümlenen bağlam ile `policy.fn(ctx)` öğesini çağırın.
5. Sonuç `deny` ise, hemen durdurun ve o kararı döndürün.
6. Sonuç `instruct` ise, mesajı biriktiğin ve devam edin.
7. Sonuç `allow` ise sonraki politikaya devam edin.

Tüm politikalar çalıştıktan sonra:
- Herhangi bir `deny` döndürüldüyse, deny yanıtını yayın.
- Herhangi bir `instruct` dönüşü toplanmışsa, tüm mesajları birleştirerek tek bir instruct yanıtı yayın.
- Aksi takdirde, izin yanıtı yayın (boş stdout, çıkış 0).

## Yerleşik Politikalar

`src/hooks/builtin-policies.ts` tüm 26 yerleşik politikayı `BuiltinPolicyDefinition` nesneleri olarak tanımlar:

```typescript
interface BuiltinPolicyDefinition {
  name: string;
  description: string;
  fn: (ctx: PolicyContext) => PolicyResult;
  match: {
    events: HookEventType[];
    tools?: string[];
  };
  defaultEnabled: boolean;
  category: string;
  beta?: boolean;
  params?: PolicyParamsSchema;
}
```

`params` kabul eden politikalar, her parametre için türler ve varsayılanlarla bir `PolicyParamsSchema` bildirir. Politika değerlendiricisi, `fn` öğesini çağırmadan önce çözümlenen değerleri `ctx.params`'e enjekte eder. Politika işlevleri, varsayılanlar her zaman ilk olarak uygulandığı için null-guard olmadan `ctx.params` öğesini okur.

Politikalar içindeki desen eşleştirmesi ham dize eşleştirmesi değil, ayrıştırılmış komut belirteçleri (argv) kullanır. Bu, shell operatörü enjeksiyonu yoluyla atlatılmayı engeller (örneğin, `sudo systemctl status *` için bir desen `; rm -rf /` komutunu ekleyerek atlatılamaz).

## Özel Politikalar

`src/hooks/custom-hooks-registry.ts` bir `globalThis`-destekli kayıt defteri uygular:

```typescript
const REGISTRY_KEY = "__failproofai_custom_hooks__";

export const customPolicies = {
  add(hook: CustomHook): void { ... }
};

export function getCustomHooks(): CustomHook[] { ... }
export function clearCustomHooks(): void { ... }  // used in tests
```

`src/hooks/custom-hooks-loader.ts` kullanıcının politika dosyasını yükler:

1. Config'ten `customPoliciesPath` öğesini okuyun; yoksa atlayın.
2. Mutlak yola çözün; dosya varolup olmadığını kontrol edin.
3. Tüm `from "failproofai"` içeri aktarımlarını `customPolicies` aynı `globalThis` kaydına çözümlenerek gerçek dist yoluna yeniden yazın.
4. ESM uyumluluğunu sağlamak için geçişli yerel içeri aktarımları yinelemeli olarak yeniden yazın.
5. Geçici `.mjs` dosyaları yazın ve giriş dosyasını `import()` yapın.
6. Kayıtlı hooks'u almak için `getCustomHooks()` öğesini çağırın.
7. `finally` bloğunda tüm geçici dosyaları temizleyin.

Herhangi bir hata (dosya bulunamadı, söz dizimi hatası, içeri aktarım hatası) durumunda, hata `~/.failproofai/hook.log` için kaydedilir ve loader boş bir dizi döndürür. Yerleşik politikalar etkilenmez.

Özel politikalar tüm yerleşik politikalardan sonra değerlendirilir. Özel politika `deny` hala daha fazla özel politikaları kısa devre yapar (ancak bu noktada tüm yerleşikler zaten çalıştırılmıştır).

## Aktivite günlüğü

Her hook olayından sonra, handler `~/.failproofai/hook-activity.jsonl` dosyasına bir JSONL satırı ekler:

```json
{
  "timestamp": "2026-04-06T12:34:56.789Z",
  "sessionId": "abc123",
  "eventType": "PreToolUse",
  "toolName": "Bash",
  "policyName": "block-sudo",
  "decision": "deny",
  "reason": "sudo command blocked by failproofai",
  "durationMs": 12
}
```

İzin vermeyen kararı alan her politika için bir satır. Allow kararları günlüğe kaydedilmez (dosyayı küçük tutmak için).

## Dashboard mimarisi

Dashboard, React Server Components ve Server Actions'ı kullanan App Router'ı kullanan bir **Next.js 16** uygulamasıdır.

```text
app/
  layout.tsx                  ← Kök layout (tema, telemetri, nav)
  projects/page.tsx           ← Sunucu bileşeni: tüm Claude projeleri listesi
  project/[name]/page.tsx     ← Sunucu bileşeni: projede oturumları listele
  project/[name]/session/
    [sessionId]/page.tsx      ← Sunucu bileşeni: oturum görüntüleyiciyi renderla
  policies/page.tsx           ← İstemci bileşeni: politika yönetimi + aktivite günlüğü
  actions/
    get-hooks-config.ts       ← Config + politika listesi oku
    update-hooks-config.ts    ← Politikayı aç/kapat
    update-policy-params.ts   ← Politika parametrelerini güncelle
    get-hook-activity.ts      ← Aktivite günlüğünü sayfalandır/ara
    install-hooks-web.ts      ← Tarayıcıdan hooks'u yükle/kaldır
  api/
    download/[project]/[session]/route.ts   ← CLI başına oturum dışa aktarımı (JSONL veya JSON)
```

**Veri akışı:**

- Sayfa bileşenleri, dosya sisteminden doğrudan proje/oturum verilerini okumak için `lib/projects.ts` ve `lib/log-entries.ts` öğesini çağırır (okuma için API katmanı yok).
- Policies sayfası tüm mutasyonlar için Server Actions'ı kullanır (aç/kapat, parametreler güncellemesi, yükle/kaldır).
- Oturum görüntüleyicisi, Claude'un JSONL transcript biçimini ayrıştırır ve mesajlar ve araç çağrılarının bir zaman çizelgesini renderlar.

**Temel tasarım kararları:**

- Veritabanı yok - tüm kalıcı durum düz dosyalarda (`~/.failproofai/`, `~/.claude/projects/`).
- Mutasyonlar için Server Actions - CRUD işlemleri için REST API gerekmez.
- Okuma sayfaları için React Server Components - daha hızlı ilk yük, veri getirme için istemci paketi yok.
- İstemci bileşenleri yalnızca etkileşim gerekli olduğunda (politika anahtarları, aktivite araması, günlük görüntüleyicisi).
