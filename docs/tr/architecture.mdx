Bu belge, failproofai'nin dahili olarak nasıl çalıştığını açıklar: hook sisteminin ajan araç çağrılarını nasıl yakaladığını, yapılandırmanın nasıl yüklenip birleştirildiğini, politikaların nasıl değerlendirildiğini ve dashboardın ajan etkinliğini nasıl izlediğini.

---

## Genel Bakış

failproofai iki bağımsız alt sistemden oluşur:

1. **Hook işleyici** - Claude Code'un her ajan araç çağrısında başlattığı hızlı bir CLI alt süreci. Politikaları değerlendirir ve bir karar döndürür.
2. **Ajan Monitörü (Dashboard)** - Ajan oturumlarını izlemek ve politikaları yönetmek için bir Next.js web uygulaması.

Her iki alt sistem de `~/.failproofai/` ve projenin `.failproofai/` dizinindeki yapılandırma dosyalarını paylaşır; ancak ayrı süreçler olarak çalışırlar ve yalnızca dosya sistemi aracılığıyla iletişim kurarlar.

## Hook işleyici

### Claude Code ile entegrasyon

`failproofai policies --install` komutunu çalıştırdığınızda, `~/.claude/settings.json` dosyasına şu şekilde girişler yazar:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "failproofai --hook PreToolUse"
          }
        ]
      }
    ],
    "PostToolUse": [ ... ]
  }
}
```

Claude Code ardından her araç çağrısından önce `failproofai --hook PreToolUse` komutunu bir alt süreç olarak başlatır ve stdin üzerinden JSON yükü iletir.

### Yük biçimi

```json
{
  "session_id": "abc123",
  "transcript_path": "/home/user/.claude/projects/myproject/sessions/abc123.jsonl",
  "cwd": "/home/user/myproject",
  "permission_mode": "default",
  "hook_event_name": "PreToolUse",
  "tool_name": "Bash",
  "tool_input": { "command": "sudo apt install nodejs" }
}
```

`PostToolUse` olayları için yük, ayrıca aracın çıktısını içeren `tool_result` alanını da barındırır.

İşleyici 1 MB stdin sınırı uygular. Bu sınırı aşan yükler atılır ve tüm politikalar örtük olarak izin verir.

### Yanıt biçimi

**Reddet (PreToolUse):**
```json
{
  "hookSpecificOutput": {
    "permissionDecision": "deny",
    "permissionDecisionReason": "Blocked by failproofai: sudo command blocked"
  }
}
```

**Reddet (PostToolUse):**
```json
{
  "hookSpecificOutput": {
    "additionalContext": "Blocked by failproofai because: API key detected in output"
  }
}
```

**Yönerge (Stop dışındaki tüm olaylar):**
```json
{
  "hookSpecificOutput": {
    "additionalContext": "Instruction from failproofai: Verify tests pass before committing."
  }
}
```

**Stop olayı yönergesi:**
- Çıkış kodu: `2`
- Neden stdout'a değil, stderr'e yazılır

**İzin ver:**
- Çıkış kodu: `0`
- Boş stdout

**Mesajlı izin ver (beta):**

v0.0.2-beta.3 sürümünden itibaren, `allow(message)` bir politikanın işlem izin verilse dahi Claude'a bilgilendirme bağlamı göndermesine olanak tanır. Hook işleyici aşağıdaki JSON'ı **stdout**'a yazar (bir yapılandırma dosyasına değil — bu, yukarıdaki reddet ve yönerge yanıtlarında olduğu gibi işleyicinin Claude Code'a verdiği yanıttır):

```json
// Hook işleyici süreci tarafından stdout'a yazılır
{
  "hookSpecificOutput": {
    "additionalContext": "All CI checks passed on branch 'feat/my-feature'."
  }
}
```
- Çıkış kodu: `0` (işleme izin verilir)
- Birden fazla politika mesajlı `allow` döndürdüğünde, mesajlar yeni satırlarla birleştirilerek tek bir `additionalContext` dizesi oluşturulur
- Hiçbir politika mesaj sağlamazsa stdout boş kalır (öncekiyle aynı)

### İşleme hattı

`src/hooks/handler.ts` tam hattı uygular:

```text
stdin JSON
  → yükü ayrıştır (maks. 1 MB)
  → oturum meta verilerini çıkar (session_id, cwd, tool_name, tool_input, vb.)
  → readMergedHooksConfig(cwd)    ← proje + local + global yapılandırmayı birleştirir
  → etkin yerleşik politikaları çözümlenmiş parametrelerle kaydet
  → customPoliciesPath'ten (ayarlanmışsa) özel politikaları yükle
  → özel politikaları politika kaydına ekle
  → tüm politikaları değerlendir (önce yerleşikler, sonra özel)
      → ilk deny kısa devre yapar
      → instruct kararları biriktirilir
      → allow mesajları biriktirilir
  → JSON kararını stdout'a yaz
  → olayı ~/.failproofai/hook-activity.jsonl dosyasına kaydet
  → çık
```

Tüm süreç, LLM çağrısı yapılmaksızın tipik yükler için 100ms'nin altında tamamlanır.

## Yapılandırma yükleme

`src/hooks/hooks-config.ts` üç kapsamlı yapılandırma yüklemesini uygular.

```text
[1] {cwd}/.failproofai/policies-config.json        ← proje  (en yüksek öncelik)
[2] {cwd}/.failproofai/policies-config.local.json  ← local
[3] ~/.failproofai/policies-config.json             ← global  (en düşük öncelik)
```

Birleştirme mantığı:
- `enabledPolicies` - üç dosyada yinelemesiz birleşim
- `policyParams` - politika başına anahtar; onu tanımlayan ilk dosya tamamen kazanır
- `customPoliciesPath` - onu tanımlayan ilk dosya kazanır
- `llm` - onu tanımlayan ilk dosya kazanır

Web dashboard, bir proje cwd'si ile çağrılmadığından okuma ve yazma için `readHooksConfig()` (yalnızca global) kullanır.

## Politika değerlendirmesi

`src/hooks/policy-evaluator.ts` politikaları sırayla çalıştırır.

Her politika için:

1. Politikanın `params` şemasına bakılır (varsa).
2. Birleştirilmiş yapılandırmadan `policyParams[policy.name]` okunur.
3. `ctx.params` oluşturmak için kullanıcı tarafından sağlanan değerler şema varsayılanlarıyla birleştirilir.
4. Çözümlenmiş bağlamla `policy.fn(ctx)` çağrılır.
5. Sonuç `deny` ise hemen durulur ve bu karar döndürülür.
6. Sonuç `instruct` ise mesaj biriktirilir ve devam edilir.
7. Sonuç `allow` ise bir sonraki politikaya geçilir.

Tüm politikalar çalıştıktan sonra:
- Herhangi bir `deny` döndürüldüyse, deny yanıtı iletilir.
- Herhangi bir `instruct` toplandıysa, tüm mesajlar birleştirilerek tek bir instruct yanıtı iletilir.
- Aksi takdirde, izin yanıtı iletilir (boş stdout, çıkış kodu 0).

## Yerleşik politikalar

`src/hooks/builtin-policies.ts`, `BuiltinPolicyDefinition` nesneleri olarak 26 yerleşik politikanın tamamını tanımlar:

```typescript
interface BuiltinPolicyDefinition {
  name: string;
  description: string;
  fn: (ctx: PolicyContext) => PolicyResult;
  match: {
    events: HookEventType[];
    tools?: string[];
  };
  defaultEnabled: boolean;
  category: string;
  beta?: boolean;
  params?: PolicyParamsSchema;
}
```

`params` kabul eden politikalar, her parametre için türler ve varsayılanlarla birlikte `PolicyParamsSchema` tanımlar. Politika değerlendiricisi, `fn` çağrılmadan önce çözümlenmiş değerleri `ctx.params` içine enjekte eder. Varsayılanlar her zaman önce uygulandığından, politika işlevleri `ctx.params` okurken null kontrolü yapmaz.

Politikalar içindeki desen eşleştirmesi, ham dize eşleşmesi yerine ayrıştırılmış komut belirteçleri (argv) kullanır. Bu, kabuk operatörü enjeksiyonu yoluyla atlatmayı engeller (örneğin, `sudo systemctl status *` için bir desen, komuta `; rm -rf /` eklenerek atlatılamaz).

## Özel politikalar

`src/hooks/custom-hooks-registry.ts`, `globalThis` destekli bir kayıt defteri uygular:

```typescript
const REGISTRY_KEY = "__failproofai_custom_hooks__";

export const customPolicies = {
  add(hook: CustomHook): void { ... }
};

export function getCustomHooks(): CustomHook[] { ... }
export function clearCustomHooks(): void { ... }  // testlerde kullanılır
```

`src/hooks/custom-hooks-loader.ts` kullanıcının politika dosyasını yükler:

1. Yapılandırmadan `customPoliciesPath` okunur; yoksa atlanır.
2. Mutlak yola çözümlenir; dosyanın var olup olmadığı kontrol edilir.
3. `customPolicies`'in aynı `globalThis` kaydına çözümlenmesi için tüm `from "failproofai"` içe aktarmaları gerçek dist yoluna yeniden yazılır.
4. ESM uyumluluğunu sağlamak için yerel geçişli içe aktarmalar yinelemeli olarak yeniden yazılır.
5. Geçici `.mjs` dosyaları yazılır ve giriş dosyası `import()` ile içe aktarılır.
6. Kayıtlı hook'ları almak için `getCustomHooks()` çağrılır.
7. `finally` bloğunda tüm geçici dosyalar temizlenir.

Herhangi bir hata durumunda (dosya bulunamadı, sözdizimi hatası, içe aktarma başarısızlığı), hata `~/.failproofai/hook.log` dosyasına kaydedilir ve yükleyici boş bir dizi döndürür. Yerleşik politikalar etkilenmez.

Özel politikalar, tüm yerleşik politikalardan sonra değerlendirilir. Özel bir politikanın `deny` kararı, sonraki özel politikaları kısa devre yaptırır (ancak tüm yerleşikler o noktada zaten çalışmış olur).

## Etkinlik günlüğü

Her hook olayının ardından, işleyici `~/.failproofai/hook-activity.jsonl` dosyasına bir JSONL satırı ekler:

```json
{
  "timestamp": "2026-04-06T12:34:56.789Z",
  "sessionId": "abc123",
  "eventType": "PreToolUse",
  "toolName": "Bash",
  "policyName": "block-sudo",
  "decision": "deny",
  "reason": "sudo command blocked by failproofai",
  "durationMs": 12
}
```

Allow dışı karar veren her politika için bir satır. Allow kararları günlüğe kaydedilmez (dosyayı küçük tutmak için).

## Dashboard mimarisi

Dashboard, React Server Components ve Server Actions kullanan App Router ile birlikte **Next.js 16** uygulamasıdır.

```text
app/
  layout.tsx                  ← Kök düzen (tema, telemetri, navigasyon)
  projects/page.tsx           ← Sunucu bileşeni: tüm Claude projelerini listeler
  project/[name]/page.tsx     ← Sunucu bileşeni: projedeki oturumları listeler
  project/[name]/session/
    [sessionId]/page.tsx      ← Sunucu bileşeni: oturum görüntüleyiciyi render eder
  policies/page.tsx           ← İstemci bileşeni: politika yönetimi + etkinlik günlüğü
  actions/
    get-hooks-config.ts       ← Yapılandırma + politika listesini oku
    update-hooks-config.ts    ← Politikayı aç/kapat
    update-policy-params.ts   ← Politika parametrelerini güncelle
    get-hook-activity.ts      ← Etkinlik günlüğünü sayfalandır/ara
    install-hooks-web.ts      ← Tarayıcıdan hook'ları yükle/kaldır
  api/
    download/[project]/[session]/route.ts   ← Oturumu ZIP/JSONL olarak dışa aktar
```

**Veri akışı:**

- Sayfa bileşenleri, proje/oturum verilerini doğrudan dosya sisteminden okumak için `lib/projects.ts` ve `lib/log-entries.ts` dosyalarını çağırır (okumalar için API katmanı yoktur).
- Politikalar sayfası, tüm değişiklikler için Server Actions kullanır (açma/kapama, parametre güncelleme, yükleme/kaldırma).
- Oturum görüntüleyici, Claude'un JSONL transkript biçimini ayrıştırır ve mesajlar ile araç çağrılarının zaman çizelgesini render eder.

**Temel tasarım kararları:**

- Veritabanı yok — tüm kalıcı durum düz dosyalarda tutulur (`~/.failproofai/`, `~/.claude/projects/`).
- Değişiklikler için Server Actions — CRUD işlemleri için REST API gerekmiyor.
- Okuma sayfaları için React Server Components — daha hızlı ilk yükleme, veri getirme için istemci paketi yok.
- İstemci bileşenleri yalnızca etkileşim gerektiğinde kullanılır (politika açma/kapama, etkinlik arama, günlük görüntüleyici).
