Özel politikalar, herhangi bir ajan davranışı için kural yazmanıza olanak tanır: proje kurallarını zorunlu kılın, sapmayı önleyin, yıkıcı işlemleri kapıya koyun, takılıp kalan ajanları tespit edin veya Slack, onay iş akışları ve daha fazlasıyla entegre edin. Yerleşik politikalarla aynı hook olay sistemini ve `allow`, `deny`, `instruct` kararlarını kullanırlar.

---

## Hızlı örnek

```js
// my-policies.js
import { customPolicies, allow, deny, instruct } from "failproofai";

customPolicies.add({
  name: "no-production-writes",
  description: "Block writes to paths containing 'production'",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Write" && ctx.toolName !== "Edit") return allow();
    const path = ctx.toolInput?.file_path ?? "";
    if (path.includes("production")) {
      return deny("Writes to production paths are blocked");
    }
    return allow();
  },
});
```

Yükleyin:

```bash
failproofai policies --install --custom ./my-policies.js
```

## Yükleme ve güncelleme

```bash
# Özel bir politika dosyasıyla yükleyin
failproofai policies --install --custom ./my-policies.js

# Politika dosyası yolunu değiştirin
failproofai policies --install --custom ./new-policies.js

# Özel politika yolunu yapılandırmadan kaldırın
failproofai policies --uninstall --custom
```

Çözümlenen mutlak yol, `policies-config.json` dosyasında `customPoliciesPath` olarak saklanır. Dosya her hook olayında yeniden yüklenir — olaylar arasında önbellekleme yapılmaz.

## API

### İçe aktarma

```js
import { customPolicies, allow, deny, instruct } from "failproofai";
```

### `customPolicies.add(hook)`

Bir politika kaydeder. Aynı dosyada birden fazla politika için bu işlevi istediğiniz kadar çağırabilirsiniz.

```ts
customPolicies.add({
  name: string;                         // zorunlu - benzersiz tanımlayıcı
  description?: string;                 // `failproofai policies` çıktısında gösterilir
  match?: { events?: HookEventType[] }; // olay türüne göre filtrele; tümünü eşleştirmek için atlayın
  fn: (ctx: PolicyContext) => PolicyResult | Promise<PolicyResult>;
});
```

### Karar yardımcıları

| Fonksiyon | Etkisi | Ne zaman kullanılır |
|----------|--------|----------|
| `allow()` | İşleme sessizce izin ver | Eylem güvenli, mesaja gerek yok |
| `deny(message)` | İşlemi engelle | Ajan bu eylemi gerçekleştirmemelidir |
| `instruct(message)` | Engellemeden bağlam ekle | Ajana yolda kalması için ek bağlam ver |

`deny(message)` - mesaj Claude'a `"Blocked by failproofai:"` öneki ile görünür. Tek bir `deny`, sonraki tüm değerlendirmeyi kısa devre yapar.

`instruct(message)` - mesaj, mevcut araç çağrısı için Claude'un bağlamına eklenir. İlk `instruct` kazanır — diğer politikalardan gelen sonraki `instruct` döndürmeleri yok sayılır.

### Bilgilendirici allow mesajları (beta)

<Note>
`allow(message)`, v0.0.2-beta.3 sürümünden itibaren kullanılabilen bir beta özelliğidir. API gelecekteki sürümlerde değişebilir. Önceki sürümler yalnızca argümansız `allow()` seçeneğini destekler.
</Note>

`allow(message)` işleme izin verir **ve** Claude'a bilgilendirici bir mesaj gönderir. Mesaj, hook işleyicisinin stdout yanıtında `additionalContext` olarak iletilir — `instruct` tarafından kullanılan mekanizmanın aynısı, ancak anlam olarak farklı: bu bir uyarı değil, durum güncellemesidir.

| Fonksiyon | Etkisi | Ne zaman kullanılır |
|----------|--------|----------|
| `allow(message)` | İzin ver ve Claude'a bağlam gönder | Bir kontrolün geçtiğini onaylayın veya neden atlandığını açıklayın |

Kullanım durumları:
- **Durum onayları:** `allow("All CI checks passed.")` — Claude'a her şeyin yolunda olduğunu bildirir
- **Başarısız-açık açıklamalar:** `allow("GitHub CLI not installed, skipping CI check.")` — Claude'a bir kontrolün neden atlandığını tam bağlamıyla bildirir
- **Birden fazla mesaj birikir:** birkaç politikanın her biri `allow(message)` döndürürse, tüm mesajlar yeni satırlarla birleştirilerek birlikte iletilir

```js
customPolicies.add({
  name: "confirm-branch-status",
  match: { events: ["Stop"] },
  fn: async (ctx) => {
    const cwd = ctx.session?.cwd;
    if (!cwd) return allow("No working directory, skipping branch check.");

    // ... dal durumunu kontrol et ...
    if (allPushed) {
      return allow("Branch is up to date with remote.");
    }
    return deny("Unpushed changes detected.");
  },
});
```

### `PolicyContext` alanları

| Alan | Tür | Açıklama |
|-------|------|-------------|
| `eventType` | `string` | `"PreToolUse"`, `"PostToolUse"`, `"Notification"`, `"Stop"` |
| `toolName` | `string \| undefined` | Çağrılan araç (örn. `"Bash"`, `"Write"`, `"Read"`) |
| `toolInput` | `Record<string, unknown> \| undefined` | Aracın giriş parametreleri |
| `payload` | `Record<string, unknown>` | Claude Code'dan gelen ham olay yükü |
| `session` | `SessionMetadata \| undefined` | Oturum bağlamı (aşağıya bakın) |

### `SessionMetadata` alanları

| Alan | Tür | Açıklama |
|-------|------|-------------|
| `sessionId` | `string` | Claude Code oturum tanımlayıcısı |
| `cwd` | `string` | Claude Code oturumunun çalışma dizini |
| `transcriptPath` | `string` | Oturumun JSONL transkript dosyasının yolu |

### Olay türleri

| Olay | Ne zaman tetiklenir | `toolInput` içeriği |
|-------|--------------|----------------------|
| `PreToolUse` | Claude bir araç çalıştırmadan önce | Aracın girişi (örn. Bash için `{ command: "..." }`) |
| `PostToolUse` | Bir araç tamamlandıktan sonra | Aracın girişi + `tool_result` (çıktı) |
| `Notification` | Claude bir bildirim gönderdiğinde | `{ message: "...", notification_type: "idle" \| "permission_prompt" \| ... }` - hook'lar her zaman `allow()` döndürmelidir, bildirimleri engelleyemezler |
| `Stop` | Claude oturumu sona erdiğinde | Boş |

## Değerlendirme sırası

Politikalar şu sırayla değerlendirilir:

1. Yerleşik politikalar (tanım sırasına göre)
2. Özel politikalar (`.add()` sırasına göre)

<Note>
İlk `deny`, sonraki tüm politikaları kısa devre yapar. İlk `instruct` kazanır — sonraki `instruct` döndürmeleri yok sayılır.
</Note>

## Geçişli içe aktarmalar

Özel politika dosyaları, göreli yollar kullanarak yerel modülleri içe aktarabilir:

```js
// my-policies.js
import { isBlockedPath } from "./utils.js";
import { checkApproval } from "./approval-client.js";

customPolicies.add({
  name: "approval-gate",
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const approved = await checkApproval(ctx.toolInput?.command, ctx.session?.sessionId);
    return approved ? allow() : deny("Approval required for this command");
  },
});
```

Giriş dosyasından erişilebilen tüm göreli içe aktarmalar çözümlenir. Bu, `from "failproofai"` içe aktarmalarını gerçek dist yoluna yeniden yazarak ve ESM uyumluluğunu sağlamak için geçici `.mjs` dosyaları oluşturarak hayata geçirilir.

## Olay türü filtreleme

Bir politikanın ne zaman tetikleneceğini sınırlamak için `match.events` kullanın:

```js
customPolicies.add({
  name: "require-summary-on-stop",
  match: { events: ["Stop"] },
  fn: async (ctx) => {
    // Yalnızca oturum sona erdiğinde tetiklenir
    // ctx.session.transcriptPath tam oturum günlüğünü içerir
    return allow();
  },
});
```

Her olay türünde tetiklemek için `match` seçeneğini tamamen atlayın.

## Hata yönetimi ve başarısızlık modları

Özel politikalar **fail-open** (başarısız-açık) çalışır: hatalar yerleşik politikaları hiçbir zaman engellemez veya hook işleyicisini çökertemez.

| Başarısızlık | Davranış |
|---------|----------|
| `customPoliciesPath` ayarlanmamış | Özel politika çalışmaz; yerleşikler normal şekilde devam eder |
| Dosya bulunamadı | `~/.failproofai/hook.log` dosyasına uyarı kaydedilir; yerleşikler devam eder |
| Sözdizimi/içe aktarma hatası | `~/.failproofai/hook.log` dosyasına hata kaydedilir; tüm özel politikalar atlanır |
| `fn` çalışma zamanında hata fırlatır | Hata kaydedilir; o hook `allow` olarak değerlendirilir; diğer hook'lar devam eder |
| `fn` 10 saniyeden uzun sürer | Zaman aşımı kaydedilir; `allow` olarak değerlendirilir |

<Tip>
Özel politika hatalarını ayıklamak için günlük dosyasını izleyin:

```bash
tail -f ~/.failproofai/hook.log
```
</Tip>

## Tam örnek: birden fazla politika

```js
// my-policies.js
import { customPolicies, allow, deny, instruct } from "failproofai";

// Prevent agent from writing to secrets/ directory
customPolicies.add({
  name: "block-secrets-dir",
  description: "Prevent agent from writing to secrets/ directory",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (!["Write", "Edit"].includes(ctx.toolName ?? "")) return allow();
    const path = ctx.toolInput?.file_path ?? "";
    if (path.includes("secrets/")) return deny("Writing to secrets/ is not permitted");
    return allow();
  },
});

// Keep the agent on track: verify tests before committing
customPolicies.add({
  name: "remind-test-before-commit",
  description: "Keep the agent on track: verify tests pass before committing",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const cmd = ctx.toolInput?.command ?? "";
    if (/git\s+commit/.test(cmd)) {
      return instruct("Verify all tests pass before committing. Run `bun test` if you haven't already.");
    }
    return allow();
  },
});

// Prevent unplanned dependency changes during freeze
customPolicies.add({
  name: "dependency-freeze",
  description: "Prevent unplanned dependency changes during freeze period",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const cmd = ctx.toolInput?.command ?? "";
    const isInstall = /^(npm install|yarn add|bun add|pnpm add)\s+\S/.test(cmd);
    if (isInstall && process.env.DEPENDENCY_FREEZE === "1") {
      return deny("Package installs are frozen. Unset DEPENDENCY_FREEZE to allow.");
    }
    return allow();
  },
});

export { customPolicies };
```
