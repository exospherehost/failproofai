failproofai, hangi politikaların etkin olduğunu, nasıl davrandığını ve özel politikaların nereden yükleneceğini kontrol etmek için JSON yapılandırma dosyalarını kullanır. Yapılandırma, takımınızla paylaşmak için tasarlanmıştır - depoyu sürüm kontrolüne kaydedin ve her geliştirici aynı ajan güvenlik ağını alır.

---

## Yapılandırma kapsamları

Üç yapılandırma kapsamı vardır ve öncelik sırasına göre değerlendirilir:

| Kapsam | Dosya yolu | Amaç |
|--------|-----------|------|
| **proje** | `.failproofai/policies-config.json` | Depo başına ayarlar, sürüm kontrolüne kaydedilir |
| **yerel** | `.failproofai/policies-config.local.json` | Kişisel depo başına geçersiz kılmalar, gitignored |
| **genel** | `~/.failproofai/policies-config.json` | Tüm projeler arasında kullanıcı düzeyinde varsayılanlar |

failproofai bir hook olayını aldığında, mevcut çalışma dizini için var olan üç dosyayı yükler ve birleştirir.

### Birleştirme kuralları

**`enabledPolicies`** - üç kapsamın tümünün birleşimi. Herhangi bir düzeyde etkinleştirilen bir politika etkindir.

```text
project:  ["block-sudo"]
local:    ["block-rm-rf"]
global:   ["block-sudo", "sanitize-api-keys"]

resolved: ["block-sudo", "block-rm-rf", "sanitize-api-keys"]  ← tekrarlanmayan birleşim
```

**`policyParams`** - belirli bir politika için parametreleri tanımlayan ilk kapsam tamamen kazanır. Bir politikanın parametreleri içinde derin birleştirme yoktur.

```text
project:  block-sudo → { allowPatterns: ["sudo apt-get update"] }
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo apt-get update"] }   ← proje kazanır, genel yok sayılır
```

```text
project:  (block-sudo girişi yok)
local:    (block-sudo girişi yok)
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo systemctl status"] }  ← genel ayara düşer
```

**`customPoliciesPath`** - bunu tanımlayan ilk kapsam kazanır.

**`llm`** - bunu tanımlayan ilk kapsam kazanır.

## Yapılandırma dosyası biçimi

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "sanitize-jwt",
    "block-env-files",
    "block-read-outside-cwd"
  ],
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    },
    "block-push-master": {
      "protectedBranches": ["main", "release", "prod"]
    },
    "block-rm-rf": {
      "allowPaths": ["/tmp"]
    },
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company"]
    },
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo API key" }
      ]
    },
    "warn-large-file-write": {
      "thresholdKb": 512
    }
  },
  "customPoliciesPath": "/home/alice/myproject/my-policies.js"
}
```

## Alan başvurusu

### `enabledPolicies`

Tür: `string[]`

Etkinleştirilecek politika adlarının listesi. Adlar, `failproofai policies` tarafından gösterilen politika tanımlayıcılarıyla tam olarak eşleşmelidir. Tam liste için [Yerleşik Politikalar](/tr/built-in-policies) bölümüne bakın.

`enabledPolicies` içinde olmayan politikalar, `policyParams` içinde girdileri olsa bile inaktiftir.

### `policyParams`

Tür: `Record<string, Record<string, unknown>>`

İlke başına parametre geçersiz kılmaları. Dış anahtar politika adıdır; iç anahtarlar politikaya özgüdür. Her politika, [Yerleşik Politikalar](/tr/built-in-policies) bölümünde kullanılabilir parametrelerini belgelendirir.

Bir politikanın parametreleri varsa ancak bunları belirtmezseniz, politikanın yerleşik varsayılanları kullanılır. `policyParams` yapılandırmayan kullanıcılar, önceki sürümlerle aynı davranışı alırlar.

Bir politikanın params bloğu içindeki bilinmeyen anahtarlar, hook tetiklendiğinde sessizce yok sayılır ancak `failproofai policies` çalıştırdığınızda uyarılar olarak işaretlenir.

#### `hint` (kesişmeyen)

Tür: `string` (isteğe bağlı)

Bir politika `deny` veya `instruct` döndürdüğünde nedene eklenen bir ileti. Politikanın kendisini değiştirmeden Claude'a eylem yapılabilir rehberlik vermek için kullanın.

Herhangi bir politika türüyle çalışır — yerleşik, özel (`custom/`), proje kuralı (`.failproofai-project/`) veya kullanıcı kuralı (`.failproofai-user/`).

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "Bunun yerine yeni bir dal oluşturmayı deneyin."
    },
    "block-sudo": {
      "allowPatterns": ["sudo apt-get"],
      "hint": "sudo olmadan doğrudan apt-get kullanın."
    },
    "custom/my-policy": {
      "hint": "Önce kullanıcıdan onay isteyiniz."
    }
  }
}
```

`block-force-push` reddettiğinde, Claude şunu görür: *"Force-pushing engellendi. Bunun yerine yeni bir dal oluşturmayı deneyin."*

String olmayan değerler ve boş diziler sessizce yok sayılır. `hint` ayarlanmamışsa, davranış değişmez (geriye dönük uyumlu).

### `customPoliciesPath`

Tür: `string` (mutlak yol)

Özel hook politikaları içeren bir JavaScript dosyasına giden yol. Bu, `failproofai policies --install --custom <path>` tarafından otomatik olarak ayarlanır (yol depolanmadan önce mutlak yola çözülür).

Dosya her hook olayında yeniden yüklenir - önbellek yoktur. Yazım ayrıntıları için [Özel Politikalar](/tr/custom-policies) bölümüne bakın.

### Kural tabanlı politikalar

Açık `customPoliciesPath`e ek olarak, failproofai `.failproofai/policies/` dizinlerinden politika dosyalarını otomatik olarak keşfeder ve yükler:

| Düzey | Dizin | Kapsam |
|-------|-------|--------|
| Proje | `.failproofai/policies/` | Takımla sürüm kontrolü aracılığıyla paylaşılır |
| Kullanıcı | `~/.failproofai/policies/` | Kişisel, tüm projeler için geçerli |

**Dosya eşleştirmesi:** Yalnızca `*policies.{js,mjs,ts}` ile eşleşen dosyalar yüklenir (ör. `security-policies.mjs`, `workflow-policies.js`). Dizindeki diğer dosyalar yok sayılır.

**Yapılandırma gerekmez:** Kural politikaları `policies-config.json` içinde girdi gerektirmez. Dosyaları dizine bırakın ve sonraki hook olayında seçilecektir.

**Birleşim yükleme:** Hem proje hem de kullanıcı kural dizinleri taranır. Her iki düzeyden de eşleşen tüm dosyalar yüklenir (`customPoliciesPath`'ten farklı olarak ilk kapsam kazanır).

Daha fazla ayrıntı ve örnekler için [Özel Politikalar](/tr/custom-policies) bölümüne bakın.

### `llm`

Tür: `object` (isteğe bağlı)

AI çağrıları yapan politikalar için LLM istemci yapılandırması. Çoğu kurulum için gerekli değildir.

```json
{
  "llm": {
    "model": "claude-sonnet-4-6",
    "apiKey": "sk-ant-..."
  }
}
```

## CLI'dan yapılandırmayı yönetme

`policies --install` ve `policies --uninstall` komutları, ajan CLI'ınızın hook ayarları dosyasına (hook giriş noktaları) yazarken, `policies-config.json` doğrudan yönettiğiniz dosyadır. İkisi ayrıdır:

- **Ajan CLI ayarları** — ajana her araç kullanımında `failproofai --hook <event>` çağırmasını söyler:
  - **Claude Code**: `~/.claude/settings.json` (kullanıcı), `<cwd>/.claude/settings.json` (proje), `<cwd>/.claude/settings.local.json` (yerel)
  - **OpenAI Codex**: `~/.codex/hooks.json` (kullanıcı), `<cwd>/.codex/hooks.json` (proje) — Codex'in yerel kapsamı yoktur
  - **GitHub Copilot CLI _(beta)_**: `~/.copilot/hooks/failproofai.json` (kullanıcı), `<cwd>/.github/hooks/failproofai.json` (proje) — Copilot'un yerel kapsamı yoktur. Hook girdileri, `timeoutSec` ile Copilot'un işletim sistemi anahtarlı `bash`/`powershell` komut alanlarını kullanır; dosya, üst düzey `version: 1` işaretçisi taşır. Copilot CLI desteği, `events.jsonl` kayıt şemasını (genel belgeler belirtmez) daha gerçek dünyada oturumlarla doğruladığımız sürece **beta** sürümündedir.
  - **Cursor Agent _(beta)_**: `~/.cursor/hooks.json` (kullanıcı), `<cwd>/.cursor/hooks.json` (proje) — Cursor'un yerel kapsamı yoktur. Hook girdileri Claude şeklindeki `{type, command, timeout}` formunu kullanır (`bash`/`powershell` bölümü yok), ancak Cursor'un [hooks şemasına](https://cursor.com/docs/hooks) göre düz bir dizi başına camelCase olay anahtarları (`preToolUse`, `beforeSubmitPrompt`, …) altında depolanır; dosya, üst düzey `version: 1` işaretçisi taşır. İşleyici, `CURSOR_EVENT_MAP` aracılığıyla camelCase → PascalCase'i kanonikleştirerek mevcut yerleşik politikaların değişmeden tetiklenmesini sağlar. Cursor Agent desteği, Cursor'un disk üzerinde transkript biçimini (genel belgeler belirtmez) daha gerçek dünyada yüklemelerle doğruladığımız sürece **beta** sürümündedir.
  - **OpenCode _(beta)_**: `~/.config/opencode/opencode.json` + `~/.config/opencode/plugins/failproofai.mjs` (kullanıcı), `<cwd>/.opencode/opencode.json` + `<cwd>/.opencode/plugins/failproofai.mjs` (proje) — OpenCode'un yerel kapsamı yoktur. Diğer dört CLI'den farklı olarak OpenCode **harici komut hook sistemi olmadığından**: `opencode.json` içindeki `plugin: []` dizisi aracılığıyla açıkça kaydedilen işlem içi JS/TS eklentilerini yükler (`.opencode/plugins/` otomatik keşfi, opencode v1.14.33'te eklentilerin nasıl yükleneceği **değildir**). Kurulum, ikili failproofai'yi çağıran ve ikili Claude şeklindeki JSON yanıtını eklenti semantiğine (deny için `throw new Error()`, instruct için `client.session.prompt(...)`, allow için no-op) çeviren küçük bir oluşturulan eklenti başlatma yı bırakır. Oturumlar `~/.local/share/opencode/opencode.db` konumundaki OpenCode'un SQLite DB'sinde yaşar; pano oturum görüntüleyicisi, `opencode db --format json` ve `opencode export <id>` aracılığıyla bunları okur. OpenCode desteği, davranışı sürümler arasında ve daha gerçek dünyada oturumlarla doğruladığımız sürece **beta** sürümündedir. [OpenCode eklentileri belgelerine](https://opencode.ai/docs/plugins/) bakın.
  - **Pi _(beta)_**: `~/.pi/agent/settings.json` (kullanıcı), `<cwd>/.pi/settings.json` (proje) — Pi'nin yerel kapsamı yoktur. Pi başlangıçta TypeScript uzantı paketlerini yükler; ayarlar dosyası, düz bir dize dizisi `{"packages": ["./relative/path", …]}` dir. failproofai, paketlenmiş `pi-extension/` dizinine işaret eden tek bir packages-array girdisi yazar. Uzantı dahili olarak Pi'nin `tool_call` / `user_bash` / `input` / `session_start` olaylarına abone olur ve `failproofai --hook <Event> --cli pi` komutunu çalıştırır; işleyici underscore_lower_snake_case → PascalCase'i `PI_EVENT_MAP` aracılığıyla kanonikleştirerek mevcut yerleşik politikaların değişmeden tetiklenmesini sağlar. Pi desteği, Pi'nin uzantı API'si ve oturum günlüğü düzeni stabilleşene kadar **beta** sürümündedir.
  - **Gemini CLI _(beta)_**: `~/.gemini/settings.json` (kullanıcı), `<cwd>/.gemini/settings.json` (proje) — Gemini'nin yerel kapsamı yoktur (varsayılan olarak `/etc/gemini-cli/settings.json` konumunda `system` kapsamını belgelendirir, failproofai bunu göstermez). Hook girdileri, Claude'un `{type, command, timeout}` formunu `{matcher, hooks: [...]}` matcher şemasında sarılı Gemini'nin içinde kullanır ve varsayılan olarak `matcher: "*"` ile. Olaylar PascalCase'tir (`SessionStart`, `BeforeAgent`, `AfterAgent`, `BeforeModel`, `AfterModel`, `BeforeToolSelection`, `BeforeTool`, `AfterTool`, `PreCompress`, `Notification`, `SessionEnd`); işleyici `GEMINI_EVENT_MAP` aracılığıyla Claude kanonik adlarıyla eşleştirir. Araç adları snake_case'tir (`run_shell_command`, `read_file`, `write_file`, `replace`, …) — işleyici `GEMINI_TOOL_MAP` aracılığıyla kanonikleştirerek mevcut yerleşik politikaların değişmeden tetiklenmesini sağlar. Politika değerlendiricisi Gemini'nin düz `{decision: "deny", reason}` şeklini yayınlar (Gemini'nin "Golden Rule" exit-0 sözleşmesine göre tercih edilir), BeforeAgent / AfterTool / SessionStart'ta bağlam enjeksiyonu için `{hookSpecificOutput: {hookEventName, additionalContext}}`, AfterAgent'te zorla yeniden deneme semantiği için `{decision: "block", reason}`. Gemini CLI desteği, daha gerçek dünyada kapsamı genişletene kadar **beta** sürümündedir. [Gemini CLI hooks belgelerine](https://geminicli.com/docs/hooks/) bakın.
- **`policies-config.json`** — failproofai'ye hangi politikaları değerlendireceğini ve ne parametrelerle değerlendireceğini söyler (tüm ajan CLI'ları arasında paylaşılan)

Belirli bir ajana hedef almak için `--cli claude|codex|copilot|cursor|opencode|pi|gemini` kullanın (boşluk ayrıldı veya herhangi bir alt küme için tekrarlanan):

```bash
failproofai policies --install --cli codex --scope project
failproofai policies --install --cli copilot --scope project
failproofai policies --install --cli cursor --scope project
failproofai policies --install --cli opencode --scope project
failproofai policies --install --cli pi --scope project
failproofai policies --install --cli gemini --scope project
failproofai policies --install --cli claude codex copilot cursor opencode pi gemini
```

`--cli` atlandığında, `failproofai` hangi ajan CLI'larının kurulu olduğunu algılar (`which claude` / `which codex` / `which copilot` / `which cursor-agent` / `which opencode` / `which pi` / `which gemini`):

- **Bir CLI algılandı** — sorulmadan bu CLI'yı otomatik olarak seçer.
- **Etkileşimli terminalde birden fazla CLI algılandı** — `Detected (N)` bölümüne gruplandırılmış ok tuşu tek seçim istemi gösterir (`Algılanan N'nin tümü için yükle` toplama satırı + algılanan her CLI bireysel olarak) ve her bir desteklenen CLI'yı önceden yükleme seçeneği olarak listelemek için `Yüklü değil (M) · hook'ları önceden yüklemeye hazır` bölümü (↑↓ hareket etmek, Enter seçmek, ^C çıkmak için). Kaldırma akışı yalnızca Algılanan bölümü gösterir.
- **Etkileşimli olmayan çalıştırmada birden fazla CLI algılandı** (CI, TTY yok) — sorulmadan algılanan tüm CLI'lar için yükler.
- **Hiçbiri algılanmadı** — `claude`'a geri döner ve PATH'de ajan ikili bulunamadığına dair bir uyarı verir; hook komutu hala yazıldığı için, bir tane kurduğunuz anda etkinleşir.

`policies-config.json` dosyasını istediğiniz zaman doğrudan düzenleyebilirsiniz; değişiklikler yeniden başlatma gerekmeksizin sonraki hook olayında hemen etkili olur.

## Örnek: takım varsayılanları ile proje düzeyinde yapılandırma

`.failproofai/policies-config.json` dosyasını deponuza kaydedin:

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "block-env-files"
  ],
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "release", "hotfix"]
    }
  }
}
```

Her geliştirici daha sonra, takım arkadaşlarını etkilemeden kişisel geçersiz kılmalar için `.failproofai/policies-config.local.json` (gitignored) oluşturabilir.
