Questo documento spiega come funziona internamente failproofai: come il sistema di hook intercetta le chiamate agli strumenti dell'agente, come viene caricata e unita la configurazione, come vengono valutate le policy e come la dashboard monitora l'attività dell'agente.

---

## Panoramica

failproofai dispone di due sottosistemi indipendenti:

1. **Gestore degli hook** - Un sottoprocesso CLI veloce che Claude Code invoca ad ogni chiamata agli strumenti dell'agente. Valuta le policy e restituisce una decisione.
2. **Monitor degli agenti (Dashboard)** - Un'applicazione web Next.js per monitorare le sessioni degli agenti e gestire le policy.

Entrambi i sottosistemi condividono i file di configurazione in `~/.failproofai/` e nella directory `.failproofai/` del progetto, ma vengono eseguiti come processi separati e comunicano solo attraverso il filesystem.

## Gestore degli hook

### Integrazione con Claude Code

Quando si esegue `failproofai policies --install`, vengono scritte voci come questa in `~/.claude/settings.json`:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "failproofai --hook PreToolUse"
          }
        ]
      }
    ],
    "PostToolUse": [ ... ]
  }
}
```

Claude Code invoca quindi `failproofai --hook PreToolUse` come sottoprocesso prima di ogni chiamata agli strumenti, passando un payload JSON su stdin.

### Formato del payload

```json
{
  "session_id": "abc123",
  "transcript_path": "/home/user/.claude/projects/myproject/sessions/abc123.jsonl",
  "cwd": "/home/user/myproject",
  "permission_mode": "default",
  "hook_event_name": "PreToolUse",
  "tool_name": "Bash",
  "tool_input": { "command": "sudo apt install nodejs" }
}
```

Per gli eventi `PostToolUse`, il payload contiene anche `tool_result` con l'output dello strumento.

Il gestore impone un limite di 1 MB su stdin. I payload che superano questo limite vengono scartati e tutte le policy consentono implicitamente.

### Formato della risposta

**Deny (PreToolUse):**
```json
{
  "hookSpecificOutput": {
    "permissionDecision": "deny",
    "permissionDecisionReason": "Blocked by failproofai: sudo command blocked"
  }
}
```

**Deny (PostToolUse):**
```json
{
  "hookSpecificOutput": {
    "additionalContext": "Blocked by failproofai because: API key detected in output"
  }
}
```

**Instruct (qualsiasi evento eccetto Stop):**
```json
{
  "hookSpecificOutput": {
    "additionalContext": "Instruction from failproofai: Verify tests pass before committing."
  }
}
```

**Instruct per evento Stop:**
- Codice di uscita: `2`
- Motivo scritto su stderr (non su stdout)

**Allow:**
- Codice di uscita: `0`
- stdout vuoto

**Allow con messaggio (beta):**

Dalla versione v0.0.2-beta.3, `allow(message)` consente a una policy di inviare contesto informativo a Claude anche quando l'operazione è permessa. Il gestore degli hook scrive il seguente JSON su **stdout** (non in un file di configurazione — questa è la risposta del gestore a Claude Code, esattamente come le risposte deny e instruct sopra):

```json
// Written to stdout by the hook handler process
{
  "hookSpecificOutput": {
    "additionalContext": "All CI checks passed on branch 'feat/my-feature'."
  }
}
```
- Codice di uscita: `0` (l'operazione è consentita)
- Quando più policy restituiscono `allow` con un messaggio, i messaggi vengono uniti con newline in un'unica stringa `additionalContext`
- Se nessuna policy fornisce un messaggio, stdout è vuoto (come prima)

### Pipeline di elaborazione

`src/hooks/handler.ts` implementa l'intera pipeline:

```text
stdin JSON
  → parse payload (max 1 MB)
  → estrazione metadati sessione (session_id, cwd, tool_name, tool_input, ecc.)
  → readMergedHooksConfig(cwd)    ← unisce configurazione progetto + locale + globale
  → registrazione policy builtin abilitate con parametri risolti
  → caricamento policy personalizzate da customPoliciesPath (se impostato)
  → registrazione policy personalizzate nel registro delle policy
  → valutazione di tutte le policy (prima le builtin, poi le personalizzate)
      → il primo deny interrompe il processo
      → le decisioni instruct si accumulano
      → i messaggi allow si accumulano
  → scrittura della decisione JSON su stdout
  → persistenza dell'evento in ~/.failproofai/hook-activity.jsonl
  → uscita
```

L'intero processo viene eseguito in meno di 100ms per payload tipici senza chiamate LLM.

## Caricamento della configurazione

`src/hooks/hooks-config.ts` implementa il caricamento della configurazione a tre livelli.

```text
[1] {cwd}/.failproofai/policies-config.json        ← progetto  (priorità più alta)
[2] {cwd}/.failproofai/policies-config.local.json  ← locale
[3] ~/.failproofai/policies-config.json             ← globale   (priorità più bassa)
```

Logica di unione:
- `enabledPolicies` - unione deduplicata tra tutti e tre i file
- `policyParams` - chiave per policy, vince interamente il primo file che la definisce
- `customPoliciesPath` - vince il primo file che la definisce
- `llm` - vince il primo file che la definisce

La dashboard web utilizza `readHooksConfig()` (solo globale) per lettura e scrittura, poiché non viene invocata con un cwd di progetto.

## Valutazione delle policy

`src/hooks/policy-evaluator.ts` esegue le policy in ordine.

Per ogni policy:

1. Si cerca lo schema `params` della policy (se presente).
2. Si legge `policyParams[policy.name]` dalla configurazione unita.
3. Si uniscono i valori forniti dall'utente sui valori predefiniti dello schema per produrre `ctx.params`.
4. Si chiama `policy.fn(ctx)` con il contesto risolto.
5. Se il risultato è `deny`, ci si ferma immediatamente e si restituisce quella decisione.
6. Se il risultato è `instruct`, si accumula il messaggio e si continua.
7. Se il risultato è `allow`, si passa alla policy successiva.

Dopo che tutte le policy sono state eseguite:
- Se è stato restituito un `deny`, viene emessa la risposta deny.
- Se sono stati raccolti ritorni `instruct`, viene emessa una singola risposta instruct con tutti i messaggi uniti.
- Altrimenti, viene emessa una risposta allow (stdout vuoto, uscita 0).

## Policy builtin

`src/hooks/builtin-policies.ts` definisce tutte le 26 policy integrate come oggetti `BuiltinPolicyDefinition`:

```typescript
interface BuiltinPolicyDefinition {
  name: string;
  description: string;
  fn: (ctx: PolicyContext) => PolicyResult;
  match: {
    events: HookEventType[];
    tools?: string[];
  };
  defaultEnabled: boolean;
  category: string;
  beta?: boolean;
  params?: PolicyParamsSchema;
}
```

Le policy che accettano `params` dichiarano un `PolicyParamsSchema` con tipi e valori predefiniti per ogni parametro. Il valutatore delle policy inietta i valori risolti in `ctx.params` prima di chiamare `fn`. Le funzioni delle policy leggono `ctx.params` senza null-guard perché i valori predefiniti vengono sempre applicati per primi.

La corrispondenza dei pattern all'interno delle policy utilizza token di comando analizzati (argv), non corrispondenza di stringhe grezze. Questo impedisce aggiramenti tramite iniezione di operatori shell (ad esempio, un pattern per `sudo systemctl status *` non può essere aggirato aggiungendo `; rm -rf /` al comando).

## Policy personalizzate

`src/hooks/custom-hooks-registry.ts` implementa un registro basato su `globalThis`:

```typescript
const REGISTRY_KEY = "__failproofai_custom_hooks__";

export const customPolicies = {
  add(hook: CustomHook): void { ... }
};

export function getCustomHooks(): CustomHook[] { ... }
export function clearCustomHooks(): void { ... }  // used in tests
```

`src/hooks/custom-hooks-loader.ts` carica il file delle policy dell'utente:

1. Si legge `customPoliciesPath` dalla configurazione; si salta se assente.
2. Si risolve il percorso assoluto; si verifica l'esistenza del file.
3. Si riscrivono tutti gli import `from "failproofai"` al percorso dist effettivo in modo che `customPolicies` si risolva nello stesso registro `globalThis`.
4. Si riscrivono ricorsivamente gli import locali transitivi per garantire la compatibilità ESM.
5. Si scrivono file `.mjs` temporanei e si esegue `import()` del file di ingresso.
6. Si chiama `getCustomHooks()` per recuperare gli hook registrati.
7. Si eliminano tutti i file temporanei in un blocco `finally`.

In caso di errore (file non trovato, errore di sintassi, errore di import), l'errore viene registrato in `~/.failproofai/hook.log` e il loader restituisce un array vuoto. Le policy integrate non vengono influenzate.

Le policy personalizzate vengono valutate dopo tutte le policy integrate. Un `deny` di una policy personalizzata interrompe comunque l'elaborazione delle policy personalizzate successive (ma tutte le policy integrate sono già state eseguite a quel punto).

## Logging dell'attività

Dopo ogni evento hook, il gestore aggiunge una riga JSONL a `~/.failproofai/hook-activity.jsonl`:

```json
{
  "timestamp": "2026-04-06T12:34:56.789Z",
  "sessionId": "abc123",
  "eventType": "PreToolUse",
  "toolName": "Bash",
  "policyName": "block-sudo",
  "decision": "deny",
  "reason": "sudo command blocked by failproofai",
  "durationMs": 12
}
```

Una riga per ogni policy che ha emesso una decisione non-allow. Le decisioni allow non vengono registrate (per mantenere il file di dimensioni ridotte).

## Architettura della dashboard

La dashboard è un'applicazione **Next.js 16** che utilizza l'App Router con React Server Components e Server Actions.

```text
app/
  layout.tsx                  ← Layout radice (tema, telemetria, nav)
  projects/page.tsx           ← Server component: elenco di tutti i progetti Claude
  project/[name]/page.tsx     ← Server component: elenco delle sessioni in un progetto
  project/[name]/session/
    [sessionId]/page.tsx      ← Server component: visualizzatore di sessione
  policies/page.tsx           ← Client component: gestione policy + log attività
  actions/
    get-hooks-config.ts       ← Lettura configurazione + elenco policy
    update-hooks-config.ts    ← Attivazione/disattivazione policy
    update-policy-params.ts   ← Aggiornamento parametri policy
    get-hook-activity.ts      ← Paginazione/ricerca nel log attività
    install-hooks-web.ts      ← Installazione/rimozione hook dal browser
  api/
    download/[project]/[session]/route.ts   ← Esportazione sessione come ZIP/JSONL
```

**Flusso dei dati:**

- I componenti di pagina chiamano `lib/projects.ts` e `lib/log-entries.ts` per leggere i dati di progetto/sessione direttamente dal filesystem (nessun livello API per le letture).
- La pagina Policies utilizza Server Actions per tutte le mutazioni (attivazione/disattivazione, aggiornamento parametri, installazione/rimozione).
- Il visualizzatore di sessione analizza il formato JSONL dei transcript di Claude e visualizza una timeline di messaggi e chiamate agli strumenti.

**Decisioni progettuali chiave:**

- Nessun database - tutto lo stato persistente è in file semplici (`~/.failproofai/`, `~/.claude/projects/`).
- Server Actions per le mutazioni - nessuna API REST necessaria per le operazioni CRUD.
- React Server Components per le pagine di lettura - caricamento iniziale più rapido, nessun bundle client per il recupero dati.
- Componenti client solo dove è necessaria l'interattività (toggle delle policy, ricerca nell'attività, visualizzatore log).

```text
failproofai/
├── bin/
│   └── failproofai.mjs           # Router CLI (hook / dashboard / install / ecc.)
├── src/hooks/
│   ├── handler.ts                # Pipeline degli eventi hook
│   ├── builtin-policies.ts       # 26 definizioni di policy
│   ├── policy-evaluator.ts       # Motore di esecuzione delle policy
│   ├── policy-registry.ts        # Registrazione e ricerca delle policy
│   ├── policy-types.ts           # Interfacce TypeScript
│   ├── hooks-config.ts           # Caricamento configurazione multi-scope
│   ├── custom-hooks-registry.ts  # Registro hook basato su globalThis
│   ├── custom-hooks-loader.ts    # Loader ESM per hook JS utente
│   ├── manager.ts                # Operazioni install / remove / list
│   ├── install-prompt.ts         # Prompt interattivo di selezione policy
│   ├── hook-logger.ts            # Logging su hook.log
│   ├── hook-activity-store.ts    # Persistenza attività su hook-activity.jsonl
│   └── llm-client.ts             # Client API LLM (per policy basate su AI)
├── app/                          # Dashboard Next.js (pagine + server actions)
├── lib/                          # Utility condivise
│   ├── projects.ts               # Enumerazione dei progetti Claude dal filesystem
│   ├── log-entries.ts            # Parsing del formato JSONL dei transcript Claude
│   ├── paths.ts                  # Risoluzione dei percorsi di sistema
│   └── ...
├── components/                   # Componenti UI React condivisi
├── contexts/                     # Provider di contesto React (tema, auto-refresh, telemetria)
├── examples/                     # File di esempio per hook personalizzati
└── __tests__/                    # Test unitari e E2E
```
