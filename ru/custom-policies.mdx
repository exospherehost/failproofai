Пользовательские политики позволяют создавать правила для любого поведения агента: применять соглашения проекта, предотвращать отклонения, блокировать деструктивные операции, обнаруживать зависших агентов или интегрироваться со Slack, рабочими процессами согласования и многим другим. Они используют ту же систему хук-событий и решения `allow`, `deny`, `instruct`, что и встроенные политики.

---

## Быстрый пример

```js
// my-policies.js
import { customPolicies, allow, deny, instruct } from "failproofai";

customPolicies.add({
  name: "no-production-writes",
  description: "Block writes to paths containing 'production'",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Write" && ctx.toolName !== "Edit") return allow();
    const path = ctx.toolInput?.file_path ?? "";
    if (path.includes("production")) {
      return deny("Writes to production paths are blocked");
    }
    return allow();
  },
});
```

Установка:

```bash
failproofai policies --install --custom ./my-policies.js
```

## Установка и обновление

```bash
# Установка с файлом пользовательских политик
failproofai policies --install --custom ./my-policies.js

# Замена пути к файлу политик
failproofai policies --install --custom ./new-policies.js

# Удаление пути к пользовательским политикам из конфигурации
failproofai policies --uninstall --custom
```

Разрешённый абсолютный путь сохраняется в `policies-config.json` как `customPoliciesPath`. Файл загружается заново при каждом хук-событии — кэширование между событиями не выполняется.

## API

### Импорт

```js
import { customPolicies, allow, deny, instruct } from "failproofai";
```

### `customPolicies.add(hook)`

Регистрирует политику. Вызывайте этот метод столько раз, сколько нужно для добавления нескольких политик в одном файле.

```ts
customPolicies.add({
  name: string;                         // обязательно — уникальный идентификатор
  description?: string;                 // отображается в выводе `failproofai policies`
  match?: { events?: HookEventType[] }; // фильтр по типу события; если не указан — совпадает со всеми
  fn: (ctx: PolicyContext) => PolicyResult | Promise<PolicyResult>;
});
```

### Вспомогательные функции для решений

| Функция | Эффект | Когда использовать |
|----------|--------|----------|
| `allow()` | Разрешить операцию без уведомлений | Действие безопасно, сообщение не требуется |
| `deny(message)` | Заблокировать операцию | Агент не должен выполнять это действие |
| `instruct(message)` | Добавить контекст без блокировки | Предоставить агенту дополнительный контекст для правильной работы |

`deny(message)` — сообщение отображается для Claude с префиксом `"Blocked by failproofai:"`. Единственный `deny` прерывает всю дальнейшую обработку.

`instruct(message)` — сообщение добавляется в контекст Claude для текущего вызова инструмента. Побеждает первый `instruct` — последующие возвраты `instruct` из других политик игнорируются.

### Информационные сообщения allow (бета)

<Note>
`allow(message)` — бета-функция, доступная начиная с v0.0.2-beta.3. API может измениться в будущих версиях. Более ранние версии поддерживают только `allow()` без аргументов.
</Note>

`allow(message)` разрешает операцию **и** отправляет информационное сообщение обратно Claude. Сообщение доставляется как `additionalContext` в ответе обработчика хука через stdout — тот же механизм, что использует `instruct`, но семантически отличный: это обновление статуса, а не предупреждение.

| Функция | Эффект | Когда использовать |
|----------|--------|----------|
| `allow(message)` | Разрешить и отправить контекст Claude | Подтвердить прохождение проверки или объяснить, почему проверка была пропущена |

Варианты использования:
- **Подтверждения статуса:** `allow("All CI checks passed.")` — сообщает Claude, что всё в порядке
- **Объяснения при пропуске проверок:** `allow("GitHub CLI not installed, skipping CI check.")` — сообщает Claude, почему проверка была пропущена, предоставляя полный контекст
- **Накопление нескольких сообщений:** если несколько политик возвращают `allow(message)`, все сообщения объединяются через символ новой строки и доставляются вместе

```js
customPolicies.add({
  name: "confirm-branch-status",
  match: { events: ["Stop"] },
  fn: async (ctx) => {
    const cwd = ctx.session?.cwd;
    if (!cwd) return allow("No working directory, skipping branch check.");

    // ... check branch status ...
    if (allPushed) {
      return allow("Branch is up to date with remote.");
    }
    return deny("Unpushed changes detected.");
  },
});
```

### Поля `PolicyContext`

| Поле | Тип | Описание |
|-------|------|-------------|
| `eventType` | `string` | `"PreToolUse"`, `"PostToolUse"`, `"Notification"`, `"Stop"` |
| `toolName` | `string \| undefined` | Вызываемый инструмент (например, `"Bash"`, `"Write"`, `"Read"`) |
| `toolInput` | `Record<string, unknown> \| undefined` | Входные параметры инструмента |
| `payload` | `Record<string, unknown>` | Полные необработанные данные события от Claude Code |
| `session` | `SessionMetadata \| undefined` | Контекст сессии (см. ниже) |

### Поля `SessionMetadata`

| Поле | Тип | Описание |
|-------|------|-------------|
| `sessionId` | `string` | Идентификатор сессии Claude Code |
| `cwd` | `string` | Рабочий каталог сессии Claude Code |
| `transcriptPath` | `string` | Путь к файлу JSONL-транскрипта сессии |

### Типы событий

| Событие | Когда срабатывает | Содержимое `toolInput` |
|-------|--------------|----------------------|
| `PreToolUse` | До запуска инструмента Claude | Входные данные инструмента (например, `{ command: "..." }` для Bash) |
| `PostToolUse` | После завершения работы инструмента | Входные данные инструмента + `tool_result` (вывод) |
| `Notification` | Когда Claude отправляет уведомление | `{ message: "...", notification_type: "idle" \| "permission_prompt" \| ... }` — хуки всегда должны возвращать `allow()`, они не могут блокировать уведомления |
| `Stop` | Когда сессия Claude завершается | Пусто |

## Порядок выполнения

Политики выполняются в следующем порядке:

1. Встроенные политики (в порядке определения)
2. Пользовательские политики (в порядке вызовов `.add()`)

<Note>
Первый `deny` прерывает все последующие политики. Первый `instruct` побеждает — последующие возвраты `instruct` игнорируются.
</Note>

## Транзитивные импорты

Файлы пользовательских политик могут импортировать локальные модули с использованием относительных путей:

```js
// my-policies.js
import { isBlockedPath } from "./utils.js";
import { checkApproval } from "./approval-client.js";

customPolicies.add({
  name: "approval-gate",
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const approved = await checkApproval(ctx.toolInput?.command, ctx.session?.sessionId);
    return approved ? allow() : deny("Approval required for this command");
  },
});
```

Разрешаются все относительные импорты, достижимые из входного файла. Это реализовано путём перезаписи импортов `from "failproofai"` на фактический путь к dist и создания временных файлов `.mjs` для обеспечения совместимости с ESM.

## Фильтрация по типу событий

Используйте `match.events`, чтобы ограничить срабатывание политики:

```js
customPolicies.add({
  name: "require-summary-on-stop",
  match: { events: ["Stop"] },
  fn: async (ctx) => {
    // Срабатывает только при завершении сессии
    // ctx.session.transcriptPath содержит полный журнал сессии
    return allow();
  },
});
```

Полностью опустите `match`, чтобы срабатывать на каждый тип события.

## Обработка ошибок и режимы отказа

Пользовательские политики работают в режиме **fail-open**: ошибки никогда не блокируют встроенные политики и не приводят к аварийному завершению обработчика хуков.

| Сбой | Поведение |
|---------|----------|
| `customPoliciesPath` не задан | Пользовательские политики не выполняются; встроенные продолжают работу в обычном режиме |
| Файл не найден | Предупреждение записывается в `~/.failproofai/hook.log`; встроенные продолжают работу |
| Ошибка синтаксиса/импорта | Ошибка записывается в `~/.failproofai/hook.log`; все пользовательские политики пропускаются |
| Исключение в `fn` во время выполнения | Ошибка записывается; этот хук обрабатывается как `allow`; остальные хуки продолжают работу |
| `fn` выполняется дольше 10 секунд | Таймаут записывается; обрабатывается как `allow` |

<Tip>
Для отладки ошибок пользовательских политик наблюдайте за файлом журнала:

```bash
tail -f ~/.failproofai/hook.log
```
</Tip>

## Полный пример: несколько политик

```js
// my-policies.js
import { customPolicies, allow, deny, instruct } from "failproofai";

// Prevent agent from writing to secrets/ directory
customPolicies.add({
  name: "block-secrets-dir",
  description: "Prevent agent from writing to secrets/ directory",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (!["Write", "Edit"].includes(ctx.toolName ?? "")) return allow();
    const path = ctx.toolInput?.file_path ?? "";
    if (path.includes("secrets/")) return deny("Writing to secrets/ is not permitted");
    return allow();
  },
});

// Keep the agent on track: verify tests before committing
customPolicies.add({
  name: "remind-test-before-commit",
  description: "Keep the agent on track: verify tests pass before committing",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const cmd = ctx.toolInput?.command ?? "";
    if (/git\s+commit/.test(cmd)) {
      return instruct("Verify all tests pass before committing. Run `bun test` if you haven't already.");
    }
    return allow();
  },
});

// Prevent unplanned dependency changes during freeze
customPolicies.add({
  name: "dependency-freeze",
  description: "Prevent unplanned dependency changes during freeze period",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const cmd = ctx.toolInput?.command ?? "";
    const isInstall = /^(npm install|yarn add|bun add|pnpm add)\s+\S/.test(cmd);
    if (isInstall && process.env.DEPENDENCY_FREEZE === "1") {
      return deny("Package installs are frozen. Unset DEPENDENCY_FREEZE to allow.");
    }
    return allow();
  },
});

export { customPolicies };
```
